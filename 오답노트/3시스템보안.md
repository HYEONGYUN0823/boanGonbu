## 1. 악성코드 & 관련 파일

### ❌ 오답노트: 트로이목마 틀린 설명
> **문제**: 트로이목마 설명으로 틀린 것은? (①백도어 종류 포함 ②유틸리티 위장 ③정보유출·자료파괴 ④자신을 복제한다)
> **정답**: ④

**왜?** 트로이목마는 게임·크랙·계산기 프로그램인 척 **위장**해서 사용자가 직접 설치하게 만드는 악성코드야. 그 안에 백도어가 들어있는 경우가 많고, 정보유출·자료파괴도 하지만, **스스로 자기 자신을 복제해서 퍼지지는 않아**. 자기복제는 웜의 특징이야.

> 💡 **한 방 구분**: 트로이목마 = 위장(설치당함), 웜 = 자기복제(스스로 퍼짐)

### ❌ 오답노트: 악성프로그램 설명 틀린 것 (백도어 vs 스파이웨어)
> **문제**: (①바이러스 ②웜 ③백도어 ④논리폭탄) 중 틀린 설명은?
> **정답**: ③ — 백도어를 "사용자 동의 없이 설치되어 개인정보를 수집한다"고 설명함

**왜?** 그 설명은 **스파이웨어(Spyware)**의 정의야. 백도어는 "몰래 들어오는 비밀 출입구" — 공격자가 원격접속·관리자 권한 획득을 위해 만들어두는 뒷문이 핵심이지, 개인정보 수집이 목적이 아니야.

> 💡 백도어 = 몰래 들어오는 문 / 스파이웨어 = 몰래 정보 수집

### USB 자동실행 파일
**정답**: `autorun.inf` — USB를 꽂자마자 지정된 프로그램이 자동 실행되도록 설정하는 파일. 악성코드(virus.exe 등)를 실행 대상으로 지정해두면 USB를 꽂는 순간 감염될 수 있어서 예전에 많이 악용됐어.

## 2. HTTP 요청구조 & 상태코드 / IE 보안영역

### ❌ 오답노트: HTTP 요청 메시지 순서
> **정답**: ① **Request Line → Header → Blank Line → Body**

> 💡 편지 비유: 봉투 겉면(요청줄) → 발신인 정보(헤더) → 빈 줄 → 편지 본문(바디)

### HTTP 상태코드 표

| 코드 | 의미 |
|---|---|
| 200 | OK (성공) |
| 301 | Moved Permanently |
| 302 | Found (임시 이동) |
| 304 | Not Modified (캐시 그대로 사용) |
| 400 | Bad Request |
| 401 | Unauthorized |
| 403 | Forbidden |
| 404 | Not Found |
| 408 | Request Timeout |
| 500 | Internal Server Error |
| 503 | Service Unavailable |

### ❌ 오답노트: HTTP 상태코드 틀린 것
> **문제**: (①200 ②304 ③400=Not Found ④408) 중 틀린 것은?
> **정답**: ③ — 400은 Not Found가 아니라 **Bad Request**

> 💡 400 = 요청 자체가 잘못됨(Bad Request) / 404 = 요청한 게 없음(Not Found)

### ❌ 오답노트: IE 보안영역
> **정답**: ③ — "보안등급 지정 가능한 사이트"는 존재하지 않는 개념

IE에서 설정 가능한 보안 영역은 딱 4개뿐이야: **인터넷 · 로컬 인트라넷 · 신뢰할 수 있는 사이트 · 제한된 사이트**.

> 💡 암기: **"인로신제"**

## 3. Windows 시스템 보안

### ❌ 오답노트: NTFS 파일시스템 옳지 않은 것
> **정답**: ④ — "NTFS 보안은 로컬에만 적용되고 네트워크 접근자에겐 적용 안 된다"는 설명

**왜?** 실제로는 **로컬 + 네트워크 접근 둘 다** NTFS 권한이 적용돼. 예를 들어 공유폴더에 "특정 사용자만 읽기 가능"으로 설정하면, 다른 컴퓨터에서 네트워크로 접속해도 그 권한 그대로 적용돼.

| | FAT32 | NTFS |
|---|---|---|
| 최대 파일 크기 | 약 4GB | 최대 16EB |
| 보안 기능 | 권한 설정 불가 | 파일 권한, 암호화(EFS), 감사(Audit) 가능 |

### ❌ 오답노트: SID에서 사용자 고유 ID
> **정답**: ④ (500)

SID 구조: `S - 버전 - 권한 - 컴퓨터(도메인)ID - RID`

맨 마지막 숫자가 **RID(Relative ID)**이고, 이게 사용자 고유번호야. 앞부분은 같은 컴퓨터라면 다 똑같고, 마지막 숫자만 사용자마다 달라.

> 💡 **500 = Administrator, 501 = Guest** — 시험에 자주 나옴

### SAM 파일 접근 가능 그룹
**정답**: ① — **Administrator, System**만 접근 가능

SAM(Security Account Manager)은 계정·패스워드(해시)를 저장하는 중요 파일이라 Power Users, Account Operators, Server Operators 같은 상위 권한 그룹도 접근할 수 없어.

### ❌ 오답노트: 레지스트리 설명 틀린 것
> **정답**: ② — "HKEY_CURRENT_USER는 전체 사용자들에 관한 정보"라는 설명

**왜?** HKCU(HKEY_CURRENT_USER)는 **현재 로그인한 사용자 한 명**의 설정만 저장해. 전체 사용자 정보는 **HKEY_USERS**에 있어.

| 레지스트리 | 의미 |
|---|---|
| **HKCU** | 현재 사용자(Current User) |
| **HKLM** | 컴퓨터 전체 설정(Local Machine) |
| **HKCR** | 파일 연결 정보(Classes Root, .txt→메모장 등) |
| **HKCC** | 현재 실행 중인 하드웨어 정보(Current Config) |

### 원격제어 도구와 거리가 먼 것
**정답**: ④ **BackOffice**

NetBus, pcAnywhere, VNC는 전부 원격제어 프로그램이지만, BackOffice는 마이크로소프트의 **서버 제품군(BackOffice Suite)**을 뜻해.

> 💡 참고로 이름이 비슷한 **'Back Orifice'**라는 실제 해킹 도구(백도어)가 따로 있어서 헷갈리기 쉬운데, 이 문제에서 묻는 'BackOffice'는 그것과 다른, 마이크로소프트의 정식 서버 제품을 가리켜.

### Windows 서버보안 강화 설정
**정답**: ② — Administrators 외 모든 계정에 "원격 시스템에서 강제 종료" 권한을 준 것

**왜?** 서버를 강제 종료할 수 있는 권한은 **Administrator 정도로만 제한**해야 해. 아무 계정이나 다 갖고 있으면 누구나 서버를 끌 수 있는 위험한 설정이 되는 거지.

## 4. 유닉스/리눅스 프로세스 관리 명령어

### 좀비 프로세스
**정답**: **`ps`로 STAT=Z 확인 후 `kill` 사용**

좀비 프로세스 = 실행은 끝났는데 부모 프로세스가 "얘 끝났네" 하고 정리(wait)를 안 해준 상태. 죽었는데 시체(프로세스 정보)만 남아있는 셈이야. `ps` 명령어로 STAT 값이 **Z**로 보이면 좀비.

### Device is busy — 사용 중인 프로세스 찾기
**정답**: **`lsof`** (List Open Files)

`umount` 하려는데 "Device is busy" 에러가 나면, 누군가 그 디스크를 아직 쓰고 있다는 뜻이야. `lsof /mnt` 처럼 실행하면 어떤 프로세스가 쓰고 있는지 보여줘.

### 소유권 없는 파일 찾기 명령어가 아닌 것
**정답**: `find / -ls 2>/dev/null` (①) — 이건 그냥 전체 파일 목록만 출력하는 거라, 소유자 없는 파일을 걸러내는 옵션이 없어.

> 💡 **`-nouser`** = 소유자 없는 파일 / **`-nogroup`** = 그룹 없는 파일

### CPU·메모리·부하율 모니터링
**정답**: **`top`** — CPU, Memory, Load Average, Process를 실시간으로 종합해서 보여줌 (df=디스크용량, iostat=디스크I/O, netstat=네트워크)

### 프로세스를 트리 형태로 표시
**정답**: **`pstree`** — 부모-자식 관계를 트리 구조로 보여줌 (ps=단순목록, netstat=네트워크, lsof=열린파일)

## 5. 유닉스/리눅스 파일 권한

### ❌ 오답노트: rwSr--r--에서 대문자 S
> **정답**: ① — SetUID를 **실행 권한이 없는** 파일에 설정할 경우 대문자 S로 표기

| 표기 | 의미 |
|---|---|
| **소문자 s** (`rws`) | SetUID/GID 설정 + 실행 권한(x)도 있음 |
| **대문자 S** (`rwS`) | SetUID/GID는 설정됐지만 실행 권한(x)은 없음 |

### ❌ 오답노트: 리눅스 특수권한 설명 틀린 것
> **문제**: Sticky Bit가 부여된 디렉터리는 실행 권한이 x에서 **s**로 변경된다? (①SetUID:x→s ②SetGID:x→s ③Sticky:x→s ④Sticky는 /tmp같은 공용디렉터리 문제 해결)
> **정답**: ③ — Sticky Bit는 x→s가 아니라 **x→t**로 바뀜

> 💡 **SetUID = s, SetGID = s, Sticky Bit = t** — Sticky Bit를 걸면 777 디렉터리(예: /tmp)에서도 **자기 파일만** 삭제할 수 있게 돼.

### umask 022일 때 생성 권한
**정답**: **디렉터리 755, 파일 644**

기본 생성 권한(디렉터리 777, 파일 666)에서 umask(022)를 빼면 돼: 777-022=755, 666-022=644.

## 6. 유닉스/리눅스 파일시스템 & 계정관리

### 패스워드를 암호화된 형태로 저장하는 파일
**정답**: **`/etc/shadow`**

예전엔 `/etc/passwd`에 저장했는데 누구나 읽을 수 있어서 위험했어. 그래서 패스워드(정확히는 해시값)만 `/etc/shadow`로 분리해서 관리해.

### shadow 필드 — 최대 사용기간 60일로 바꾸려면?
**정답**: **(ㄷ)** — shadow 필드 순서는 `마지막변경일(ㄱ) → 최소사용기간(ㄴ) → 최대사용기간(ㄷ) → 경고일(ㄹ)`이라서, "최대 사용기간"을 바꾸려면 (ㄷ) 자리의 숫자(99999)를 60으로 바꾸면 돼.

### ❌ 오답노트: 계정과 패스워드 설명 틀린 것
> **정답**: ④ — "`/etc/shadow`의 패스워드가 AES-256 알고리즘으로 **암호화**되어 저장된다"는 설명

**왜?** shadow에 저장되는 건 **암호화(Encryption)**가 아니라 **해시(Hash)**야 (SHA-512 등을 많이 씀). 암호화는 키로 복호화가 가능하지만, 해시는 원래 값으로 되돌릴 수 없다는 게 핵심 차이야.

### TCP Wrapper 설정 파일
**정답**: **`/etc/inetd.conf`** — 보호할 네트워크 서비스를 등록하는 설정 파일 (실제 허용/차단은 `/etc/hosts.allow`, `/etc/hosts.deny`도 함께 쓰임)

### ⚠️ 오답노트: EXT 파일시스템 (교재 vs 실제 구현 차이 주의!)
> **문제**: (ㄱ)파일시스템은 리눅스 초기 문제점을 해결하려고 고안됐다. (ㄴ)는 블록 크기·총 개수 등 기본 레이아웃 정보를 담는다. inode 테이블은 각 (ㄷ)에 1개씩만 존재한다.
> **정답**: ④ **(ㄱ)EXT (ㄴ)Super Block (ㄷ)파일시스템**

**⚠️ 주의할 점**: 실제 EXT2/3/4의 내부 구현에서는 inode 테이블이 **블록 그룹(Block Group)마다** 하나씩 존재해. 그런데 정보보안기사 교재에서는 "inode 테이블은 **파일시스템** 단위로 존재한다"는 좀 더 단순화된 설명을 기준으로 출제하기 때문에, 시험에서는 **교재 기준(파일시스템)**으로 외워야 해. 실무 지식과 살짝 달라서 헷갈리기 쉬운 문제야.

> 💡 **Super Block** = 파일시스템의 "주민등록증" — 블록 크기, 전체 블록 수, inode 개수 같은 핵심 정보를 담고 있어.

## 7. 로그 파일 분석 (Windows / Unix)

### ❌ 오답노트: Windows 이벤트 로그
> **문제**: (ㄱ)메모장으로 볼 수 있다 (ㄴ)확장자 *.evt/*.evtx (ㄷ)크기 변경 불가, 계속 저장 (ㄹ)보안로그=로그인정보 등. 옳은 것을 모두 고르면?
> **정답**: ④ **(ㄴ, ㄹ)**

- (ㄱ) ❌ 틀림 — evt/evtx는 바이너리 형식이라 메모장이 아니라 **이벤트 뷰어**로 봐야 해
- (ㄴ) ⭕ 맞음 — XP 이전은 `.evt`, Vista 이후는 `.evtx`
- (ㄷ) ❌ 틀림 — 로그 최대 크기는 **변경 가능**하고, 오래된 로그는 덮어쓰기·수동삭제도 가능해
- (ㄹ) ⭕ 맞음 — 보안 로그엔 로그인, 로그인 실패, 계정 생성, 파일 접근 등이 기록돼

### ⚠️ 오답노트: 유닉스 로그 파일 (utmp vs wtmp 주의!)
> **문제**: 유닉스 로그 분석 파일 내용으로 **옳지 않은 것**은? (①utmp: 성공한 로그인·로그아웃 정보 기록 ②btmp: 5번 이상 로그인 실패 시 기록 ③lastlog ④pacct)
> **정답**: ① — utmp에 대한 설명이 틀림

**⚠️ 왜 헷갈리기 쉬운가**: utmp는 **"현재" 로그인 중인 사용자** 정보만 담고 있어. 로그인·로그아웃의 **이력(History)**을 기록하는 건 **wtmp**의 역할이야. 그래서 ①번 설명은 utmp가 아니라 wtmp에 대한 설명이라 틀린 거지. (②번 "btmp: 5번 이상 실패 시 기록"은 실제로는 매 실패마다 기록하는 게 정확하지만, 정보보안기사 교재에서는 이 표현을 맞는 설명으로 취급하니 시험에서는 교재 기준으로 암기하면 돼.)

| 파일 | 내용 | 기억법 |
|---|---|---|
| **utmp** | 현재 로그인 중인 사용자 | Current |
| **wtmp** | 로그인·로그아웃 **이력** | History |
| **btmp** | 로그인 실패 기록 | Bad login |
| **lastlog** | 사용자별 마지막 로그인 정보 | Last |
| **pacct** | 실행한 명령어 기록 | Process accounting |

> 🔑 **`who`는 utmp를 읽고, `last`는 wtmp를 읽는다** — 이 둘을 헷갈리지 않는 게 핵심이야.

### last 명령어로 확인 가능한 로그파일
**정답**: **wtmp**

### syslog.conf — 커널 Emergency 로그만 저장
**정답**: **`kern.emerg /var/log/emerg.log`**

syslog 형식은 `facility.level`이야. `kern.emerg`는 "커널(kern)의 Emergency 등급만" 저장하고, `*.emerg`는 모든 facility의 Emergency를 저장한다는 차이가 있어.

## 8. 보안도구 & 방화벽

### 보안도구 구분 (취약점 점검 vs 침입탐지)
**정답**: **(ㄱ)취약점 점검도구**(SATAN, SAINT, COPS, Nessus, Nmap) / **(ㄴ)Snort**(대표적인 네트워크 침입탐지시스템, NIDS)

### 무결성 검사 도구가 아닌 것
**정답**: **`prstat`**

Tripwire, Fcheck, Samhain은 전부 파일 변조(무결성) 검사 도구인데, prstat은 Solaris에서 CPU·메모리·프로세스를 확인하는 명령어라 무결성 검사와는 무관해.

### 패킷 필터 접근통제 도구와 거리가 먼 것
**정답**: **`chmod`**

iptables, ipfwadm, ipchains는 전부 리눅스 방화벽(패킷 필터링) 계열이지만, chmod는 파일 권한을 바꾸는 명령어라 네트워크와는 관계없어.

## 9. 버퍼오버플로우 & 메모리 보호기법

### noexec_user_stack 설정 파일
**정답**: **`/etc/system`** (Solaris)

`set noexec_user_stack=1`을 설정하면 스택에 저장된 코드(셸코드 등)가 **실행되지 못하게** 막아서, 버퍼 오버플로우 공격이 스택에서 셸코드를 실행하는 걸 방지해.

### 버퍼오버플로우 보안 대책
**정답**: **경계 검사(Bounds Checking) 기능을 제공하는 컴파일러·링커 사용**

버퍼 크기를 넘는 데이터가 들어오면 컴파일러/링커 차원에서 미리 막아주는 방식이야.

### ASLR 설정값 확인 파일
**정답**: **`/proc/sys/kernel/randomize_va_space`**

### 스택·힙 셸코드 실행 방지 기법
**정답**: **DEP / NX bit**

| 기법 | 역할 |
|---|---|
| **ASLR** | 메모리 주소를 매번 랜덤하게 배치 (주소를 못 맞히게) |
| **DEP/NX bit** | 스택·힙 영역은 "실행 금지"로 표시 (셸코드를 넣어도 실행 자체가 안 됨) |
| **Stack Canary** | 리턴 주소 앞에 감시값을 둬서 덮어쓰기 여부를 탐지 |

## 10. 기타 공격기법 & 분석방법

### 포맷 스트링 취약점 — 직접적인 위험이 아닌 것
**정답**: **프로그램의 복제**

포맷 스트링(`printf(user)`처럼 잘못 작성 시)으로 가능한 건 메모리 읽기(`%x`), 메모리 덮어쓰기(`%n`), 프로그램 종료야. **"프로그램을 복제하는" 기능은 없어.**

### 포맷 스트링 점검 도구가 아닌 것(틀린 것)
**정답**: **`iptables`** (gdb, objdump, ltrace는 분석 도구, iptables는 방화벽)

### 자원 경쟁을 이용하는 공격
**정답**: **레이스 컨디션(Race Condition) 공격**

여러 프로세스가 동시에 같은 자원을 쓰려고 경쟁하는 상태를 이용하는 공격이야. 예를 들어 ATM 잔액을 동시에 두 번 출금 요청하면 둘 다 성공해버리는 취약점이 이런 원리야.

### 소스코드를 이해하고 분석하는 방법
**정답**: **화이트박스 분석**

| 분석 방법 | 소스코드 확인 여부 |
|---|---|
| 블랙박스 | ❌ 안 봄 (입력→출력만 관찰) |
| **화이트박스** | ⭕ 전부 봄 (버그·구현오류·취약점 찾기 좋음) |
| 그레이박스 | △ 일부만 봄 |

## 11. 최신 위협 & 클라우드 & IoT 보안

### IoT 기기 대상 DDoS 봇넷
**정답**: **미라이(Mirai)** — 2016년 발견, IP카메라·가정용 라우터 등 IoT 장비의 **기본 ID/PW(admin/admin 등)**를 이용해 감염시킨 후 DDoS에 악용

> 💡 참고로 Nimda는 이메일·웹서버를 감염시키는 오래된 웜, Stuxnet은 산업제어시스템 공격, SQL Slammer는 MS SQL 서버를 노린 웜이야.

### ❌ 오답노트: SECaaS 설명 적합하지 않은 것
> **정답**: ③ — "SECaaS는 넓은 의미의 **PaaS**로 볼 수 있다"는 설명

**왜?** SECaaS(Security as a Service)는 인증·안티바이러스·침입탐지 등을 클라우드로 빌려 쓰는 서비스인데, 이건 PaaS가 아니라 **SaaS 개념에 가까운** 보안 서비스야.

### 산업제어시스템 감시·제어
**정답**: **SCADA**

| | 역할 |
|---|---|
| **PLC** | 실제 기계를 직접 움직이는 제어장치 |
| **SCADA** | 여러 PLC를 모니터링·관리하는 상위 감시 시스템 |

### 내부정보 유출 방지 기술
**정답**: **DLP (Data Loss Prevention)** — USB·CD·블루투스 같은 매체·통신 제어를 통해 기밀문서 유출을 막는 솔루션 (DRM=문서 자체 암호화·권한관리, Firewall=네트워크 접근차단, IPS=공격차단과 구분)

### 원격 함수 호출 서비스
**정답**: **RPC (Remote Procedure Call)** — 다른 컴퓨터의 함수를 내 컴퓨터 함수처럼 실행하는 기술 (XML-RPC도 여기서 파생)

### ❌ 오답노트: 다크웹 설명 틀린 것
> **정답**: ② — "딥웹은 다크웹의 일부분이다"라는 설명

**왜?** 실제 관계는 반대야: **인터넷 ⊃ 딥웹 ⊃ 다크웹** — 다크웹이 딥웹의 일부분이야. (딥웹 = 검색엔진에 안 잡히는 모든 영역, 다크웹 = 그중에서도 Tor 같은 특수 브라우저로만 접근 가능한 영역)
