## 1. 왜 보안이 필요한가

전자상거래는 **서로 모르는 사람끼리** 인터넷에서 거래하는 구조야.  
오프라인과 달리 얼굴을 못 보기 때문에 다양한 위협이 생겨.

```
가짜 고객 → 주문 위조
가짜 상점 → 사기
도청자    → 카드번호 훔쳐보기
악의적 부인 → "나 결제한 적 없음" 주장
```

> 💡 **핵심:** 불특정 다수끼리 거래하므로 **비대칭키(공개키/개인키) 알고리즘**이 필수야.

## 2. 보안 공격 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **인증 공격** | 신분 위조 | 가짜 고객이 진짜인 척 로그인 |
| **송수신 부인 공격** | 거래 사실 부인 | "나 결제한 적 없어", "나 주문 못 받았어" |
| **기밀성 공격** | 내용 도청 | 카드번호, 주문 내용 훔쳐보기 |
| **무결성 공격** | 내용 변조 | 10,000원 → 100,000원으로 바꾸기 |

## 3. 보안 요구사항

### 3-1. 신분 확인 (인증)
- 고객이 진짜 그 고객인지
- 상점이 진짜 그 상점인지

### 3-2. 공증 / 부인방지 (Non-repudiation)
- 거래 사실을 나중에 **증명 가능**해야 함
- "내가 주문했다", "내가 결제받았다" → 증명 가능

### 3-3. 신뢰할 만한 제3자
- 전자상거래에서는 중간에서 믿을 수 있는 기관 필요
- **CA(인증기관)**, **은행**, **게이트웨이** 등이 그 역할
- SET에서는 **PKI(공개키 기반 구조)** 사용

## 4. 전자화폐

### 개요
- 가상공간에서 동전/지폐 역할을 하는 **디지털 데이터 기반의 화폐**
- 예: 문화상품권, 도서상품권, 전자캐시

### 생애 주기

```
[발행 단계]  → 은행/발행기관이 전자화폐 생성
      ↓
[지불 단계]  → 고객이 상점에 전자화폐 전달
      ↓
[결제 단계]  → 상점이 은행에서 정산
```

### 전자화폐 요구 조건 ⭐⭐

| 조건 | 설명 |
|------|------|
| **화폐가치** | 디지털 데이터 자체로 완벽한 화폐 가치 보유 |
| **익명성** ⭐⭐ | 구매 내역이 상점·은행이 짜고 봐도 노출되면 안 됨 (Privacy, Untraceability) |
| **오프라인성** | 매번 은행에 접속 없이 거래 가능 |
| **양도성** | 타인에게 양도 가능 |
| **분할 이용** | 큰 금액을 쪼개서 여러 번 사용 가능 (cf. n회 사용가능성: 정기권) |
| **익명성 취소** ⭐⭐ | 범죄·부정사용 시 **선택적으로** 익명성 해제 가능 |
| **이중사용 방지** | 같은 전자화폐를 복사해서 두 번 쓰는 것 → 은행이 검출해야 함 |

> 💡 **익명성 취소**는 익명성 조절 파라미터로 제공됨 — 평소엔 익명, 필요시만 해제

## 5. 전자지불 시스템

### 구성 주체
```
고객 (카드 소지자)
상점 (가맹점)
은행 (발행사 / 지불은행)
인증기관 (CA)
```

### 거래 흐름

```
[고객] ──── 지불 요청 ────→ [상점]
  ↑                            ↓
  │                       승인 요청
  │                            ↓
  └──── 승인 결과 ────── [은행/게이트웨이]
```

### 인출 / 지불 / 입금 흐름

```
인출: 고객이 은행에서 전자화폐로 전환 (현금 → 전자화폐)
지불: 고객이 상점에 전자화폐 결제
입금: 상점이 전자화폐를 은행에서 정산
```

### 지불 브로커 유무에 따른 분류

| 분류 | 설명 | 예시 |
|------|------|------|
| **Payment Broker 시스템** | 중간 브로커(신용카드사, 은행 등) 존재 | 신용카드, 계좌이체 |
| **전자화폐 시스템** | 브로커 없이 독립적 신용구조 | Mondex, E-cash |

### 전자지불 시스템 정보보호 요구사항

| 요구사항 | 설명 |
|----------|------|
| **위조불가능** | 가짜 전자화폐 생성 불가 |
| **부인방지** | 거래 사실 나중에 부인 불가 |
| **누명면제** | 구성요소들이 짜고 특정 구성요소에게 누명 씌우기 불가 |
| **무결성** | 결제 내용 변조 불가 |
| **프라이버시/익명성** | 누가 뭘 샀는지 노출 불가 |

## 6. SET (Secure Electronic Transaction)

### 개요
- **비자(VISA) + 마스터카드**가 공동 제정한 신용카드 전자결제 보안 표준
- **RSA 암호화** + **인증기술** 사용
- 핵심 기술: **전자봉투** + **이중서명**
- 현재까지 가장 안전한 전자결제 방안으로 평가됨

### SET의 목적

```
① 정보의 기밀성 확보
② 지불 정보의 무결성 확보
③ 상점 ↔ 고객의 상호 확인 (인증)
```

### SET 참여 주체 ⭐

| 주체 | 역할 |
|------|------|
| **카드 소지자** | 지불카드로 결제하는 고객 |
| **발행사** | 카드 발급 금융기관 |
| **가맹점** | 상품/서비스 제공 상점 |
| **지불은행** | 상점 쪽 정산 담당 금융기관 |
| **지불 게이트웨이** | 카드 결제를 기존 금융망으로 연결하는 통로 (장비) |
| **인증기관 (CA)** | SET 참여자에게 공개키 인증서 발급 |

### SET 구성 흐름

```
[카드 소지자]
      │
      │ 주문 + 암호화된 결제정보
      ↓
[가맹점] ──────────────→ [지불 게이트웨이] ──→ [지불은행]
      ↑                          ↑
      └──── CA 인증서 검증 ───────┘
                [CA]
```

> 💡 핵심: **상점은 주문정보만** 보고, **결제정보는 게이트웨이/은행만** 봄

### SET 장단점 ⭐⭐

**✅ 장점**
- 전자거래 사기 방지
- 기존 신용카드 인프라 그대로 활용
- SSL의 단점 보완 → SSL은 상점이 지불정보를 볼 수 있었음, SET는 이중서명으로 해결

**❌ 단점**
- 암호화 프로토콜이 너무 복잡함
- RSA 속도가 느려 전체 처리 속도 저하
- 카드 소지자에게 **전자지갑 소프트웨어** 필요
- 상점에도 별도 소프트웨어 필요
- 지불 게이트웨이에 별도 하드웨어/소프트웨어 필요

## 7. 이중서명 (매우 중요) ⭐⭐⭐

### 왜 필요한가?

```
고객의 딜레마:
  - 상점에게 → 주문 내용(O)은 보여줘야 함
  - 상점에게 → 카드번호(P)는 숨겨야 함
  - 은행에게  → 카드번호(P)는 보여줘야 함
  - 은행에게  → 주문 내용(O)은 안 보여줘도 됨
```

→ **이중서명**으로 해결: 두 정보를 **하나의 서명**으로 묶되, 각자 필요한 것만 확인

### 이중서명 구조

```
주문정보(O) ──→ Hash ──→ H(O) ──┐
                                  ├──→ [H(O) + H(P)] ──→ Hash ──→ 개인키로 서명
지불정보(P) ──→ Hash ──→ H(P) ──┘
```

> - `O`: Order Information (주문정보)  
> - `P`: Payment Information (지불정보)  
> - 두 해시값을 합쳐 다시 해시 → 카드 사용자의 **개인키로 최종 서명**

### 비유로 이해하기

```
봉투 A (주문 내용)   → 상점만 열어봄
봉투 B (결제 내용)   → 은행만 열어봄
두 봉투에 같은 도장  → 이중서명 (같은 거래임을 증명)
```

### 비밀키(세션키) 생성 과정

공개키 암호(RSA)는 느리기 때문에 실제 데이터는 **대칭키(비밀키)** 로 암호화해.

```
① 카드 사용자가 랜덤 비밀키 K 생성
② 지불정보(P)를 K로 암호화  →  {P}K
③ K 자체는 지불 게이트웨이의 공개키로 암호화  →  {K}PubGW
④ {P}K 와 {K}PubGW 를 상점으로 전송
```

> 💡 이게 바로 **전자봉투(Digital Envelope)** 구조야!

### 카드 사용자가 상점에 전송하는 데이터

```
┌─────────────────────────────────────────┐
│  ① 주문정보 O (평문)                    │
│  ② 암호화된 지불정보 {P}K              │
│  ③ 이중서명                             │
│  ④ 비밀키를 게이트웨이 공개키로 암호화 {K}PubGW │
│  ⑤ 카드 소지자 인증서                  │
└─────────────────────────────────────────┘
              ↓ 상점으로 전송
```

### 이중 해시값으로 구매정보 확인 (상점이 하는 일)

```
받은 주문정보 O
      ↓
    Hash
      ↓
   H(O) 계산
      ↓
이중서명 안의 H(O)와 비교 → 일치하면 주문 정상
```

### 지불정보 확인 (은행/게이트웨이가 하는 일)

```
{K}PubGW → 게이트웨이 개인키로 복호화 → K 획득
      ↓
{P}K → K로 복호화 → 지불정보 P 획득
      ↓
이중서명으로 O와 P가 같은 거래인지 확인
```

## 8. ebXML

### 개요
- 기업 간 전자상거래를 위한 **국제 표준 문서/교환 체계**
- 인터넷 표준 브라우저만으로 **장소 무관하게** 전자상거래 가능
- 저렴한 구현 비용, 개방된 네트워크 기반

### ebXML 구성 요소 ⭐

| 구성요소 | 설명 |
|----------|------|
| **비즈니스 프로세스** | 주문→승인→배송→결제 같은 거래 흐름 정의 |
| **핵심 컴포넌트** | 재사용 가능한 문서 부품 (주소, 상품명, 수량 등) |
| **등록저장소** | 표준 문서/컴포넌트 저장 공간 |
| **거래당사자** | 실제 거래에 참여하는 기업들 |
| **전송·교환·패키징** | 문서를 안전하고 규격에 맞게 전송하는 방식 |

### 사용 효과

**재활용성**  
→ 한 번 만든 컴포넌트를 반복 사용 → 중복 작업 제거, 비용 절감, 문서 표준화

**시나리오 수준 재활용**  
→ 문서뿐 아니라 **업무 흐름(시나리오) 자체**도 재사용 가능 (모델링 수준 아님)

## 9. WPKI (무선플랫폼 전자상거래 보안)

### 개요
- **무선망(모바일 환경)에서의 PKI**
- 스마트폰 등 모바일 기기에서 공개키 기반 인증을 구현한 구조

### WPKI 구성 요소 ⭐

| 구성요소 | 역할 |
|----------|------|
| **CA 서버 시스템** | 인증서 발급 및 관리 |
| **RA 서버 시스템** | 인증서 발급·관리 요청 중계 |
| **Client 시스템** | 사용자 단말기 (모바일 기기 등) |
| **Directory 서버 시스템** | CA가 발행한 인증서 정보 저장·관리 |

> 유선 PKI와 기본 구조 동일, 무선 환경에 맞게 최적화

## 10. RSA 복습

### 한 줄 정의
> **RSA** = 큰 수의 소인수분해가 어렵다는 수학적 성질을 이용한 **공개키 암호 방식**

### 핵심 아이디어
```
n = p × q  (큰 소수 두 개의 곱)
n을 만들기는 쉽지만, n만 보고 p, q를 찾기는 매우 어려움
→ 이 어려움이 RSA 안전성의 근거
```

### 암호화 / 복호화

```
암호화:  평문 → [상대방 공개키] → 암호문
복호화:  암호문 → [본인 개인키] → 평문
```

### 서명 / 검증

```
서명:  메시지 → [본인 개인키] → 서명값
검증:  서명값 → [발신자 공개키] → 정상 여부 확인
```

### 전자상거래에서 RSA 활용

| 활용 | 설명 |
|------|------|
| 인증서 | 공개키의 신뢰성 보증 |
| 전자서명 | 신원 확인 + 부인방지 |
| 세션키 전달 | RSA로 대칭키만 암호화하여 전달 |
| 신분 확인 | CA가 발급한 인증서 기반 |

> ⚠️ RSA는 **느리기 때문에** 본문 전체를 RSA로 암호화하지 않음  
> → **세션키(대칭키)만 RSA로** 암호화하고, 본문은 대칭키로 암호화 (→ 전자봉투)

## 📌 최종 요약 정리

```
전자상거래 보안 4대 목표
  ① 인증      → 누가 했는지 확인
  ② 무결성    → 내용이 안 바뀌었는지
  ③ 기밀성    → 남이 못 보게
  ④ 부인방지  → 나중에 부인 못 하게

전자화폐 핵심 조건
  - 익명성 (⭐ 상점+은행 짜도 추적 불가)
  - 익명성 취소 (범죄 시 선택적 해제)
```
