## 1. VPN 개요

### 등장 배경

| 방식 | 문제점 |
|------|--------|
| 전용선(사설망) | 비용이 너무 비쌈 |
| 공중망(인터넷) | 보안 없음 → 중요 데이터 전송 위험 |

> 💡 그래서 나온 게 **VPN**  
> → 싼 인터넷 + 전용선처럼 안전하게

### VPN 정의

> **공중 네트워크를 마치 전용회선처럼 사용하는 기술** 🔥

공중망을 경유하더라도 외부인으로부터 안전하게 보호하기 위해 다음 기능을 제공한다.

- 주소 및 라우터 체계의 비공개
- 데이터 암호화
- 사용자 인증
- 사용자 액세스 권한 제한

### 비유

```
일반 인터넷 = 엽서 (누구나 읽을 수 있음)
VPN       = 자물쇠 달린 금고에 넣어 보내는 것
```

## 2. VPN 구현 기술

### 핵심 4요소 🔥

| 구성 요소 | 역할 |
|-----------|------|
| 터널링 | 인터넷에 가상의 안전한 통로 만들기 |
| 캡슐화 | 데이터를 다른 패킷으로 감싸서 포장 |
| 암호화 | 내용을 숨김 (기밀성) |
| 인증 | 보낸 사람이 진짜인지 확인 |

### 터널링 & 캡슐화

- **캡슐화** : 데이터를 다른 패킷으로 감싸는 것 (포장)
- **터널링** : 캡슐화된 데이터를 공중망을 통해 안전하게 전달하는 것 (비밀 통로)
- 터널링되는 데이터를 **페이로드(Payload)** 라고 부름 → 터널링 중에는 내용 변경 없이 그냥 전송

```
[편지] → [봉투에 넣기(캡슐화)] → [비밀 통로로 이동(터널링)] → [도착 후 꺼내기(디캡슐화)]
```

### 암호화 방식

- 기밀성: **대칭키(DES, 3DES, AES, SEED, RC5 등)** 사용
- 키 교환: **공개키 방식(RSA, ElGamal 등)** 으로 대칭키를 안전하게 공유
- 메시지 인증: **MAC 또는 해시함수** 사용 → 출처 검증 + 변경 여부 확인
- 사용자 인증: 공유키, 공개키, 인증서, 전자서명 등 사용

> 💡 대칭키 = 공유키 (같은 말)  
> 💡 공유키 ≠ 공개키 (다른 개념!)

### VPN 기능 🔥

| 기능 | 설명 |
|------|------|
| 데이터 기밀성 | 암호화하여 제3자가 내용 파악 불가 |
| 데이터 무결성 | 전송 중 변경되지 않았음을 보장 (암호화 + 전자서명) |
| 데이터 근원 인증 | 수신 데이터가 해당 송신자로부터 온 것임을 확인 |
| 접근 통제 | 인증된 사용자에게만 접근 허용 |

## 3. VPN 분류

### 구성 형태에 따른 분류

#### Intranet VPN

> 본사 ↔ 지사 연결

```
[서울 본사] ←─── VPN ───→ [부산 지사]
    🏢                          🏢
    → 인터넷으로 연결하지만 같은 내부망처럼 사용
```

#### Extranet VPN

> 자사 + 고객 + 협력업체를 모두 VPN으로 연결

#### Remote Access VPN 🔥

> 재택근무자, 원격 접속자 → **보안이 가장 중요**

```
[집 노트북] ─── VPN ───→ [회사 서버]
   💻                        🏢
   → VPN 켜면 회사 내부망에 접속한 것처럼 동작
```

- 사용자가 언제 어디서 접속할지 모르기 때문에 철저한 인증이 필수

## 4. 터널링 프로토콜

### 계층별 분류 🔥

| 계층 | 프로토콜 | 구현 형태 | 캡슐화 대상 |
|------|----------|-----------|-------------|
| 2계층 | PPTP, L2TP, L2F | 클라이언트-서버 | IP, IPX, AppleTalk 등 다양 |
| 3계층 | IPSec, GRE | 호스트-호스트 (LAN-to-LAN) | IP만 |
| 2+3계층 | MPLS | 호스트-호스트 | IP |
| 세션 계층 | SOCKS v5 | - | - |

### 2계층 프로토콜 상세

#### PPTP (Microsoft)
- Microsoft가 개발
- IP, IPX, NetBEUI 페이로드를 암호화 후 IP 헤더로 캡슐화
- **단점**: 서버가 Windows NT 서버여야 함

#### L2F (Cisco)
- Cisco가 개발
- **NAS 개시형** → 사용자 별도 S/W 불필요
- 다중 접속 터널 지원

#### L2TP (PPTP + L2F 결합) 🔥
- Microsoft + Cisco 공동 지원
- 호환성 뛰어남
- 보안 강화를 위해 **IPSec과 결합**하여 사용

### Client 개시 vs NAS 개시 비교

| 구분 | Client 개시 | NAS 개시 |
|------|-------------|----------|
| 특징 | NAS가 인증/터널 형성에 관여 안 함 | NAS가 관여 |
| PC S/W | 필요 | 불필요 |
| 터널 | 단일 접속 | 다중 접속 |
| 프로토콜 | PPTP | L2F, L2TP |

### IPSec VPN vs SSL VPN 비교 🔥

| 구분 | IPSec VPN | SSL VPN |
|------|-----------|---------|
| 적용 계층 | L3 (네트워크) | L5 (세션~응용) |
| 소프트웨어 | 별도 설치 필요 | 웹 브라우저 자체 지원 |
| 적합한 환경 | Site-to-Site (LAN-to-LAN) | Site-to-Remote |
| 특징 | 강력한 보안 | 설치·관리 편리, 비용 절감 |

### SOCKS v5

- **세션 계층**의 프락시에서 동작
- IPSec보다 우수한 액세스 제어 + 인증 기능 제공
- v5부터 다양한 인증 방법 + UDP 프락시 지원

## 5. IPSec 개요

### 기본 개념

- **네트워크 계층(L3)** 보호를 위한 표준 프로토콜
- IPv4 + IPv6 모두 지원 (IPv6에서는 AH/ESP가 확장 헤더의 일부)
- **AH**와 **ESP**를 통해 인증, 무결성, 기밀성 제공

### IPSec 서비스

| 보안 서비스 | 설명 | 담당 |
|-------------|------|------|
| 접근 제어 | Bypass / Discard / Protect | SPD |
| 비연결 무결성 | MAC으로 각 패킷별 무결성 보장 | AH, ESP |
| 데이터 발신처 인증 | 올바른 출처에서 온 것임을 MAC으로 보장 | AH, ESP |
| 재전송 방지 | Sequence Number로 재전송 공격 차단 | AH, ESP |
| 기밀성(암호화) | 패킷 내용 암호화 | **ESP만** 🔥 |
| 트래픽 흐름 기밀성 | 터널 모드에서 원본 IP 헤더 암호화 | ESP/터널 모드 |

### IPSec 지원 알고리즘

| 서비스 | 알고리즘 |
|--------|----------|
| 암호화 | 56bit DES, 168bit 3-DES |
| 무결성 | HMAC-SHA-1, HMAC-MD5 |
| 키 협상 | IKE (Internet Key Exchange) |
| 키 분배 | Diffie-Hellman (DH 변형) |

## 6. AH 프로토콜

### AH = 인증 전용 (암호화 없음) 🔥

```
역할: 무결성 ✅ | 인증 ✅ | 암호화 ❌
비유: 👮 경비원 - "이거 변조 안 됐고, 보낸 사람 맞음" 도장 찍기
```

### AH 패킷 구조

```
┌─────────────┬────────────────┬──────────────────────────┐
│  IP Header  │   AH Header    │     원본 데이터           │
└─────────────┴────────────────┴──────────────────────────┘
```

### AH 헤더 필드

| 필드 | 설명 |
|------|------|
| Next Header | 다음 확장 헤더 유형 |
| Payload Len | AH 헤더 길이 |
| SPI (Security Parameter Index) | 보안 연계(SA) 식별자 (32bit) |
| Sequence Number | 순서 번호 → **재전송 공격 방지** 🔥 |
| Authentication Data (ICV) | 무결성 체크값, 32bit 배수 |

> 💡 **Mutable Field** : TTL, Header Checksum, NAT 환경의 Source IP 등  
> → 라우터 경유 시 값이 바뀔 수 있으므로 AH 계산 시 0으로 설정

## 7. ESP 프로토콜

### ESP = 암호화 + 인증 🔥

```
역할: 무결성 ✅ | 인증 ✅ | 암호화 ✅
비유: 📦🔐 금고 - 내용 숨기고 검증도 함
```

### ESP 처리 순서 (중요!)

```
1️⃣  데이터 뒤에 트레일러(Trailer) 추가
2️⃣  페이로드 데이터 + 트레일러 암호화 🔐
3️⃣  앞에 ESP 헤더 붙임
4️⃣  전체에 대해 인증값(ICV) 생성 후 붙임
```

### ESP 패킷 구조

```
┌───────────┬────────────┬──────────────────────┬──────────────────┐
│ IP Header │ ESP Header │  🔐 암호화된 데이터   │  인증값(ICV)     │
└───────────┴────────────┴──────────────────────┴──────────────────┘
```

### ESP 헤더 주요 필드

| 필드 | 설명 |
|------|------|
| SPI | 보안 연계(SA) 식별자 |
| Sequence Number | 재전송 공격 방지 |
| Payload Data | 암호화된 데이터 |
| Authentication Data (ICV) | MAC으로 생성한 무결성 체크값 |

### AH vs ESP 최종 비교 🔥

| 보안 서비스 | AH | ESP |
|-------------|:--:|:---:|
| 접근 제어 | ✅ | ✅ |
| 비연결 무결성 | ✅ | ✅ |
| 데이터 원본 인증 | ✅ | ✅ |
| 재전송 방지 | ✅ | ✅ |
| **기밀성 (암호화)** | ❌ | **✅** |
| 트래픽 흐름 기밀성 | ❌ | ✅ (터널 모드) |

> 🔥 **핵심 암기**  
> `AH = A → Authentication (인증/무결성만)`  
> `ESP = E → Encryption (암호화 포함)`

## 8. 전송 모드 vs 터널 모드

### 전송 모드 (Transport Mode)

> 호스트 ↔ 호스트 (내용만 보호)

```
┌─────────────┬──────────────────────────────┐
│  IP Header  │  🔐 보호된 데이터 (TCP/UDP)   │
└─────────────┴──────────────────────────────┘
  IP 헤더는 그대로 노출
```

- IP 헤더는 보호 안 됨 (주소 보임)
- 출발지/목적지가 최종 호스트

### 터널 모드 (Tunnel Mode)

> 보안 게이트웨이 간, 또는 LAN-to-LAN (전체 보호) 🔥

```
┌──────────────┬────────────────────────────────────────┐
│ New IP Header│  🔐 (원래 IP Header + 원본 데이터)      │
└──────────────┴────────────────────────────────────────┘
  새 IP 헤더는 보안 게이트웨이 주소
  원본 전체가 암호화됨
```

- 원본 IP 헤더까지 암호화 → 출발지/목적지 숨김
- **보안 통신의 한 쪽이 라우터(게이트웨이)이면 반드시 터널 모드 사용**

### 모드별 AH / ESP 적용 패킷 형태

#### AH 전송 모드
```
[IP 헤더] [AH] [TCP/데이터]
← 인증 범위(Immutable field 기준) →
```

#### AH 터널 모드
```
[New IP 헤더] [AH] [원래 IP 헤더] [TCP/데이터]
              ←────── 인증 범위 ──────────────→
```

#### ESP 전송 모드
```
[IP 헤더] [ESP 헤더] [🔐TCP/데이터 + 트레일러🔐] [ESP Auth]
                     ←── 암호화 ──→  ←── 인증 ───→
```

#### ESP 터널 모드
```
[New IP 헤더] [ESP 헤더] [🔐원래 IP 헤더 + TCP/데이터 + 트레일러🔐] [ESP Auth]
                         ←────────────── 암호화 ──────────────→  ←── 인증 ───→
```

## 9. 보안 연계 (SA)

### SA 개념

> **"이 통신에서는 어떤 방식으로 보안할지 미리 정해 둔 약속 (계약서)"**

SA에 포함되는 정보:
- 사용할 알고리즘 (AH / ESP)
- 사용할 키
- 전송 모드 / 터널 모드 여부
- Sequence Number

### SA의 핵심 특징 🔥

> **SA는 단방향이다**

- 양방향 통신이면 SA가 **2개** 필요 (A→B 용, B→A 용)
- 하나의 SA에는 AH 또는 ESP 중 **하나만** 포함

### SA Bundle

- AH + ESP 동시에 필요할 때 **SA 2개 이상을 묶어서 사용**
- 이를 **SA Bundle**이라 함

### SPD vs SAD

| 구분 | 역할 | 비유 |
|------|------|------|
| **SPD** (Security Policy DB) | 패킷을 Bypass / Discard / Protect 중 어떻게 처리할지 결정 | 문지기 규칙표 |
| **SAD** (Security Association DB) | 실제 SA 파라미터(키, 알고리즘 등) 저장 | 계약서 보관함 |

### 아웃바운드 패킷 처리 흐름

```
패킷 생성
   ↓
SPD 확인 (이 패킷 어떻게 처리?)
   ├─ Bypass  → 그냥 전송
   ├─ Discard → 버림
   └─ Protect → SAD에서 SA 정보 꺼냄
                   ↓
               AH / ESP 적용
                   ↓
               외부로 전송
```

### 인바운드 패킷 처리 흐름

```
외부에서 패킷 수신
   ↓
어떤 SA로 왔는지 확인 (SPI로 식별)
   ↓
AH / ESP 검증
 - 무결성 확인
 - 인증 확인
 - 재전송 공격 여부 확인 (Sequence Number)
   ↓
이상 없으면 복호화 / 복원
   ↓
상위 계층(TCP/UDP)으로 전달
```

## 10. IKE (인터넷 키 교환)

### IKE 개념

> **IPSec이 사용할 SA와 키를 협상하고 생성하는 프로토콜** 🔥

- IPSec과 독립적으로 구현
- ISAKMP : IKE가 SA를 만들 수 있도록 **틀(framework)을 제공**하는 규격

```
IKE = SA 만들기 + 키 교환 담당자
ISAKMP = 그 기반 프레임워크
IPSec = 실제 보안 처리 담당
```

### IKE 키 교환 방식

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **Preshared Key** | 양쪽 장비에 같은 키를 미리 입력 | 간단, 설정 쉬움 | 규모 커지면 확장성 약함 |
| **Public Key 방식** | 공개키/인증서/PKI 기반으로 교환 | 안전, 대규모 적합 | PKI 구축 필요, 복잡 |

### VPN 인증 방식 비교

| 방식 | 프로토콜 | 원격 접속 유형 | 특징 |
|------|----------|----------------|------|
| 피어 투 피어 | PKI | IPSec | 매우 뛰어난 보안, 구축 복잡 |

### VPN 핵심 한 줄
> **공중망을 마치 전용회선처럼 사용하는 기술**  
> = 터널링 + 캡슐화 + 암호화 + 인증

### AH / ESP 5초 암기
```
AH  = 인증/무결성만  (암호화 ❌)
ESP = 인증/무결성 + 암호화 ✅
기밀성(암호화) 는 ESP만 제공!
```

### 전송 모드 / 터널 모드
```
전송 모드 = 데이터(내용)만 보호   → PC↔PC
터널 모드 = 패킷 전체 보호        → GW↔GW (VPN 핵심)
```

### SA 3줄 정리
```
SA    = 보안 계약서 (단방향!)
SPD   = 패킷 처리 규칙표
SAD   = 계약서 보관함
```

### 터널링 프로토콜 계층
```
2계층  : PPTP, L2F, L2TP
3계층  : IPSec, GRE
세션층 : SOCKS v5
```
