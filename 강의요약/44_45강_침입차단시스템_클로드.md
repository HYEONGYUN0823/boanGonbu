# 🔥 침입차단시스템(Firewall) 정리

> 정보보안기사 대비 강의 노트  
> ⭐ = 시험 출제 빈도

---

## 1. 방화벽(Firewall) 개요

### 정의

- 공중 네트워크(외부)와 사설 네트워크(내부) 사이에 설치된 보안 장벽 (HW + SW 구성)
- 모든 트래픽이 반드시 **하나의 길목(Choke Point)** 을 통과하도록 만드는 것이 핵심
- 참조모니터(추상 개념)를 **네트워크 수준에서 구체화**한 것

> **비유** 🏢  
> 건물 정문 경비원. 누구냐 / 어디 가냐 / 들어와도 되냐 검사하는 것.

---

## 2. 접근제어 목록 (ACL)

| 종류 | 검사 기준 |
|---|---|
| Standard ACL | 출발지 IP 주소만 |
| Extended ACL | 출발지 IP + 목적지 IP + 포트번호 + 프로토콜 |

- 방화벽은 ACL(허용/차단 규칙표)을 보고 판단

---

## 3. 방화벽 제어 기능

| 기능 | 설명 |
|---|---|
| 서비스 제어 | 포트 번호 기준으로 허용 여부 결정 |
| 방향 제어 | 인바운드 / 아웃바운드 방향 제어 |
| 사용자 제어 | 특정 서비스에 접근 가능한 사용자 제어 |
| 행위 제어 | 서비스가 어떻게 사용되는지 제어 (예: FTP 명령어 제한) |

---

## 4. 방화벽 기능 ⭐

| 기능 | 설명 |
|---|---|
| 접근통제 | 정책에 따라 허용/차단 결정 |
| 사용자 인증 | 신분 증명 기능 |
| 로깅 & 감사추적 | 접근 기록 저장 (CCTV 역할) |
| 프라이버시 보호 | 이중 DNS(Zone Transfer) 정보 보호 |
| 데이터 암호화 | VPN 기능 활용 |

---

## 5. 방화벽 운영 정책 ⭐

| 정책 | 설명 | 특징 |
|---|---|---|
| **Permit All** | 기본 허용, 특정 트래픽만 차단 | 내부→외부 차단에 주로 사용 |
| **Deny All** | 기본 차단, 특정 트래픽만 허용 | 보안적으로 더 안전, 실무 권장 |

---

## 6. 방화벽 한계점 ⭐⭐

| 한계 | 이유 |
|---|---|
| 악성코드 탐지 어려움 | 내용까지 검사하면 오버헤드 발생, 속도 저하 |
| 내부 공격 방어 불가 | 경계 보안만 수행, 내부 공격자에게 정책 적용 불가 |
| 우회 통신 제어 불가 | 무선(LTE), 사설 선로는 방화벽 미통과 |
| 신종 공격 대응 불가 | 새로운 공격은 규칙이 없어 대응 어려움 |

---

## 7. UTM (Unified Threat Management) ⭐⭐⭐

- **All-In-One 통합 보안 솔루션**
- 방화벽 + IPS + VPN + Anti-Virus 등 하나의 장비에 탑재
- 발전 흐름: ESM → UTM → XTM

---

## 8. 방화벽 유형 분류

```
패킷 필터링    (L3, L4 동작)
스테이트풀     (L3, L4 + 상태추적)
프록시         (L7 동작, 응용계층)
```

---

## 9. 패킷 필터링 방화벽 ⭐⭐⭐

### 개념

- **스크리닝 라우터** 또는 **패킷 필터링 라우터** 라고도 불림
- 패킷을 **독립적으로** 검사 (이전 연결과의 연관성 고려 ❌)
- 패킷 헤더(3~4계층)만 검사

### 검사 항목

- 출발지 / 목적지 IP 주소
- 출발지 / 목적지 포트 번호
- 트래픽 방향 (인바운드 / 아웃바운드)
- 프로토콜 종류 (TCP, UDP, ICMP 등)
- 패킷 상태 (SYN=0, ACK=1)

> **ACK 포인트**  
> - 요청 패킷 → ACK = 0  
> - 응답 패킷 → ACK = 1

### 장단점 ⭐⭐⭐

| 장점 | 단점 |
|---|---|
| 구현 단순, 빠른 처리 | 연결 상태 추적 ❌ |
| 응용프로그램 독립적 | 헤더 외 내용 검사 ❌ |
| 높은 성능 | 사용자 인증 미지원 |
| - | Tiny Fragment 공격 취약 |
| - | 로깅/감사 기능 부족 |

---

## 10. NAT (Network Address Translation) ⭐⭐⭐

### 개념

- 내부(사설) IP 주소를 공인 IP 주소로 변환
- 공인 IP 1개를 여러 장비가 공유 가능
- L3으로 분류

> **비유** 🏢  
> 아파트 대표번호. 외부에선 대표번호 하나만 보이고 내부는 101호 / 102호로 구분.

### NAT 종류

| 종류 | 설명 | 비유 |
|---|---|---|
| **Static NAT** | 1:1 고정 변환 | 사장님 전용차 |
| **Dynamic NAT** | 공인 IP 풀에서 동적 할당 | 업무용 공용차 |
| **PAT** (Port Address Translation) | IP + 포트번호 조합으로 구분 | 공유기 방식 (가장 많이 사용) |

### PAT 예시

| 내부 IP | 변환 결과 |
|---|---|
| 192.168.0.2 | 1.1.1.1:3000 |
| 192.168.0.3 | 1.1.1.1:3001 |

## 11. Stateful Packet Inspection (SPI) ⭐⭐⭐

### 개념

- **이전 통신까지 기억하면서** 검사하는 방화벽
- TCP 연결 정보를 **상태 테이블**에 기록
- TCP 순서번호 추적 → **세션 하이재킹** 공격 방어 가능
- L3, L4에서 동작

### 패킷 필터링 vs SPI 비교

| 구분 | 패킷 필터링 | SPI |
|---|---|---|
| 검사 범위 | 현재 패킷만 | 과거 + 현재 |
| 연결 추적 | ❌ | ✅ |
| 보안 수준 | 낮음 | 높음 |

### 상태 테이블 예시

| 출발지 IP | 목적지 IP | 포트 | 상태 |
|---|---|---|---|
| 192.168.0.10 | 8.8.8.8 | 443 | ESTABLISHED |

- **ESTABLISHED** = 3-Way Handshake 완료, 정상 연결 상태

### 동작 원리

```
1. SYN 패킷 수신
   → 정책 허용 시 상태 테이블에 기록

2. 이후 패킷 수신
   → 상태 테이블 확인
      ├── 있음 → 통과
      └── 없음 → 접근제어 정책에 따라 결정
```

### 단점

- 첫 SYN 패킷이 공격에 노출되면 잘못된 상태 테이블 생성 가능

---

## 12. DPI (Deep Packet Inspection)

- 패킷 **내용(페이로드)까지** 검사
- 바이러스, 민감 데이터 탐지 가능
- 중요 파일에 높은 우선순위 부여

---

## 13. 프록시(Proxy) 방화벽 ⭐⭐⭐

### 개념

- **중간에서 대신 통신해주는** 방화벽
- 직접 연결 ❌ → 프록시를 통해서만 통신

```
클라이언트 → 프록시 → 인터넷
```

### 핵심 기능

| 기능 | 설명 |
|---|---|
| Relay | 데이터를 대신 전달 (중계자 역할) |
| IP 숨기기 | 내부 IP 외부에 노출 ❌ |
| 헤더 제거 | IP / ICMP / UDP 헤더 기반 공격 차단 |

### 장단점

| 장점 | 단점 |
|---|---|
| 보안 강함 | 트래픽 성능 저하 |
| 내부 시스템 보호 | 설정 복잡 (서비스별 별도 프록시 필요) |

---

## 14. 프록시 종류 비교 ⭐⭐⭐

| 구분 | 응용계층 프록시 (L7) | 회선계층 프록시 (L5) |
|---|---|---|
| 동작 계층 | L7 (Application) | L5 (Session) |
| 검사 범위 | 패킷 내용까지 | 연결만 관리 |
| 프록시 수 | 서비스마다 별도 필요 | 하나로 여러 서비스 처리 |
| 보안 | 높음 | 낮음 |
| 속도 | 느림 | 빠름 |
| 대표 예 | HTTP Proxy | SOCKS |

### SOCKS 프로토콜

- 회선 레벨 게이트웨이에서 사용하는 프록시 접속 프로토콜
- 터널링 기능 제공
- 인증 기반 접근제어 지원
- cf) 소켓(Socket) = IP + Port

## 15. Bastion Host ⭐⭐⭐

### 개념

- 외부에 노출된 **핵심 방어 서버**
- 방화벽 구성에서 가장 중요한 역할

> **비유** 🏢  
> 건물 로비 안내데스크. 외부 방문자는 여기까지만 접근 가능.

### 운영 원칙

- 불필요한 계정, 명령어, 파일 삭제 → **Lockdown 상태** 유지
- 모든 행위 로그 기록 → 감사 및 재발 방지

### 장단점

| 장점 | 단점 |
|---|---|
| 스크리닝 라우터보다 안전 | Bastion Host 손상 시 내부 보호 불가 |
| 로그 관리 용이 | - |

---

## 16. 방화벽 구축 형태 (구조 종류)

### ① 스크리닝 라우터 (Screening Router)

```
[외부] ─── [스크리닝 라우터] ─── [내부]
```

- 라우터에서 IP, 포트, SYN/ACK 검사
- **단점**: 인증 불가, 내부구조 노출, 로그 미지원, 방어 깊이 부족

---

### ② 듀얼-홈드 게이트웨이 (Dual-Homed Host) ⭐⭐⭐

```
[외부] ─── [Dual-Homed Host (랜선 2개)] ─── [내부]
```

- 인터페이스 2개 (외부용 1개 + 내부용 1개)
- **라우팅 기능 비활성화** → 직접 통신 불가, 반드시 거쳐야 함
- 정보지향적 공격 방어 가능

> **비유** 🚉  
> 환승역. 직통 ❌, 무조건 갈아타야 함.

---

### ③ 스크린드 호스트 게이트웨이 (Screened Host Gateway) ⭐⭐⭐

```
[외부] ─── [스크리닝 라우터] ─── [Bastion Host] ─── [내부]
```

- 스크리닝 라우터 + Bastion Host 결합
- Bastion Host: 인증 담당
- **장점**: 2단계 방어
- **단점**: 라우팅 테이블 변조 시 Bastion 우회, 내부 직접 침투 가능

---

### ④ 스크린드 서브넷 (Screened Subnet / DMZ) ⭐⭐⭐

```
[외부]
  │
[외부 방화벽]
  │
[DMZ: 웹서버 / 메일서버 / DNS]
  │
[내부 방화벽]
  │
[내부망: DB 등 중요 시스템]
```

- **DMZ(비무장지대)** = 중간 완충 구역
- DMZ에 위치: 웹서버, 메일서버, DNS
- 내부망에 위치: DB 등 핵심 시스템
- **허니팟(Honeypot)**: DMZ에 설치, 공격자 유인용 가짜 서버

**장단점**

| 장점 | 단점 |
|---|---|
| 가장 강력한 보안 | 구축 비용 높음 |
| 내부망 이중 보호 | 서비스 속도 저하 |

## 17. iptables ⭐⭐⭐

### 개념

- 리눅스 커널 내장 **netfilter** 관리 도구
- Rule 기반 패킷 필터링
- 상태 추적(Connection Tracking), NAT 기능 제공

### 주요 체인

| 체인 | 설명 |
|---|---|
| INPUT | 방화벽을 목적지로 하는 패킷 (들어오는 것) |
| OUTPUT | 방화벽을 출발지로 하는 패킷 (나가는 것) |
| FORWARD | 방화벽을 경유하는 패킷 |

### 주요 옵션

| 옵션 | 설명 |
|---|---|
| `-A` | 규칙 추가 |
| `-p` | 프로토콜 지정 |
| `-s` | 출발지 IP |
| `-d` | 목적지 IP |
| `--sport` | 출발지 포트 |
| `--dport` | 목적지 포트 |
| `-j` | 타깃 지정 (ACCEPT / DROP / REJECT) |

### DROP vs REJECT ⭐⭐⭐

| 타깃 | 동작 |
|---|---|
| **DROP** | 패킷 차단, 아무 응답 없음 (묵살) |
| **REJECT** | 패킷 차단 + ICMP 에러 응답 전송 |

### 포트 범위 표기법

| 표기 | 의미 |
|---|---|
| `:1023` | 1023 이하 모든 포트 |
| `1024:` | 1024 이상 모든 포트 |
| `1:1023` | 1 이상 1023 이하 |

### 사용 예시 ⭐⭐⭐

```bash
# 80번 포트 허용
iptables -A INPUT -p tcp --dport 80 -j ACCEPT

# 1023번 이하 포트 전부 차단
iptables -A INPUT -p tcp --dport :1023 -j DROP

# 특정 IP 차단
iptables -A INPUT -s 192.168.0.100 -j DROP
```

> 규칙은 **위에서부터 순서대로** 검사, 일치하면 해당 정책 실행.  
> 모든 규칙에 미해당 시 → **default 정책** 적용.
