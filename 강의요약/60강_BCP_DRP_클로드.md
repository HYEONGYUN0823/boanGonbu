## 1. BCP / DRP / BCM 개요

### 🔷 BCP (Business Continuity Plan) — 업무 연속성 계획

> **"IT만이 아니라, 회사 전체가 살아남기 위한 계획"**

- 인력, 설비, 자금, 업무, IT 등 **조직 전체 자원** 대상
- 재해 발생 시에도 **핵심 업무를 지속**하거나, **빠르게 복구**하는 것이 목표

### 🔷 DRP (Disaster Recovery Plan) — 재난 복구 계획

> **"재난이 생겼을 때, IT 시스템 중심으로 복구하는 계획"**

- 서버, 데이터, 네트워크, 애플리케이션 복구에 집중
- 대체 사이트에서 시스템 운영 재개가 핵심

### 🔷 BCM (Business Continuity Management) — 업무 연속성 관리

> **"BCP + DRP를 아우르는 전체 관리 체계"**

| 단계 | 내용 |
|------|------|
| 기획 단계 | 업무 연속성 관리 요구사항과 범위 설정 |
| 운영 단계 | 업무 연속성 계획 및 절차 개발·구현 |
| 검토 및 개선 단계 | 성과 모니터링·검토 후 경영진에게 보고 |

### 📊 세 개념의 포함 관계

```
BCM (전체 관리 체계)
 └─ BCP (회사 전체 생존 계획)
      ├─ 인력
      ├─ 설비
      ├─ 자금
      ├─ 업무
      └─ DRP (IT 복구 계획)
           ├─ 서버
           ├─ 데이터
           └─ 네트워크
```

## 2. BIA — 업무 영향 분석

> **BIA (Business Impact Analysis): "어떤 업무가 얼마나 중요한지 분석하는 단계"**  
> BCP의 **핵심 절차**이며, 복구 우선순위를 결정하는 과정

### BIA에서 하는 것

```
① 핵심 업무 프로세스 식별
② 재해 유형 식별 및 발생 가능성 평가
③ 업무 중단 지속 시간 평가
④ 업무별 손실 규모 평가
⑤ 복구 우선순위 및 대상 범위 설정
```

## 3. BCP 방법론 단계

### 📌 5단계 방법론 ⭐⭐⭐

| 순서 | 단계 |
|------|------|
| 1 | 프로젝트 범위 설정 및 계획 |
| 2 | 사업영향평가 (BIA) |
| 3 | **복구 전략 개발** — BIA 결과 기반으로 복구 자원·비용 산정 |
| 4 | **복구 계획 수립** — 명시적인 문서화 필수 |
| 5 | 프로젝트 수행 테스트 및 유지보수 |

### 📌 4단계 접근방법론 ⭐⭐⭐

| 순서 | 단계 |
|------|------|
| 1 | 프로젝트 범위 설정 및 기획 |
| 2 | 사업영향평가 |
| 3 | 사업연속성 계획 개발 |
| 4 | 계획 승인 및 실행 |

> 💡 **차이점**: 5단계는 복구 전략 개발과 복구 계획 수립을 분리한 더 세분화된 버전

## 4. 2차 사이트 종류

> ⭐⭐⭐⭐⭐ 시험 단골 — **비용, RTO, 방식** 세 가지 축으로 이해!

### 📊 한눈에 비교

```
구축 비용 / 구축 난이도
    높음 ↑
         │  Mirror Site  ← Active-Active (실시간 동시 운영)
         │  Hot Site     ← Active-Standby (실시간 대기)
         │  Warm Site    ← 부분 준비
    낮음 ↓  Cold Site    ← 장소만 확보

                  빠름 ──────────── 느림
                        복구 시간 (RTO)
```

---

### 🔴 미러 사이트 (Mirror Site) — Active-Active

| 항목 | 내용 |
|------|------|
| 방식 | 주 센터 = 재해복구센터, **둘 다 실시간 동시 운영** |
| RTO | **즉시** (거의 0) |
| 비용 | 가장 높음 |
| 적합 대상 | 웹 서비스 등 **데이터 업데이트 빈도가 낮은** 시스템 |
| 단점 | 데이터 업데이트가 많으면 과부하 |

```
주 센터  ════════ 실시간 동기화 ════════  미러 센터
 [Active]                                [Active]
     ↑                                       ↑
  서비스 중                              서비스 중
```

---

### 🟠 핫 사이트 (Hot Site) — Active-Standby

| 항목 | 내용 |
|------|------|
| 방식 | 재해복구센터를 **대기(Standby) 상태**로 유지, 실시간 복제 |
| RTO | **수 시간 이내** |
| 비용 | 높음 |
| 적합 대상 | DB, 애플리케이션 등 **데이터 갱신 빈도가 높은** 시스템 |
| 단점 | 미러 사이트와 비슷한 높은 비용, 보안 문제 |

```
주 센터  ──── 실시간 복제 ────▶  핫 사이트
 [Active]                        [Standby]
                                     │
                    재해 발생 시 ──▶ [Active 전환]
```

---

### 🟡 웜 사이트 (Warm Site)

| 항목 | 내용 |
|------|------|
| 방식 | **중요한 IT 자원만** 부분적으로 준비 |
| RTO | **수일 ~ 수주** |
| 비용 | 중간 (핫 사이트보다 저렴) |
| 단점 | 초기 복구 수준 부분적, 일부 데이터 손실 가능 |

```
주 센터                          웜 사이트
 [Active]     일부만 복제 ──▶   [핵심 자원만 대기]
                                  나머지는 재해 후 조달
```

---

### 🔵 콜드 사이트 (Cold Site)

| 항목 | 내용 |
|------|------|
| 방식 | **장소 또는 데이터만** 원격 보관, 자원은 재해 후 조달 |
| RTO | **수주 ~ 수개월** |
| 비용 | 가장 낮음 |
| 백업 주기 | 일 단위 또는 주 단위 백업 |
| 단점 | 복구 신뢰성 낮음, 테스트 어려움, 가장 느림 |

```
주 센터                          콜드 사이트
 [Active]  데이터만 백업 ──▶   [빈 공간 or 최소 설비]
                                  재해 시 장비 구매/조달
```

---

### 📋 사이트 비교 총정리 ⭐

| 유형 | 운영 방식 | RTO | 비용 | 특징 |
|------|-----------|-----|------|------|
| 미러 사이트 | Active-Active | 즉시 | ⭐⭐⭐⭐⭐ | 실시간 동시 서비스 |
| 핫 사이트 | Active-Standby | 수 시간 | ⭐⭐⭐⭐ | 실시간 복제, 대기 중 |
| 웜 사이트 | 부분 준비 | 수일~수주 | ⭐⭐⭐ | 핵심 자원만 준비 |
| 콜드 사이트 | 장소/데이터만 | 수주~수개월 | ⭐ | 재해 후 자원 조달 |

## 5. 복구 지표 — RTO / RPO / MTD / WRT

> ⭐⭐⭐⭐⭐ 가장 많이 나오는 개념 — **시간 축**으로 이해하자!

### 📌 용어 정리

| 용어 | 전체 이름 | 한마디 정의 |
|------|-----------|-------------|
| **RTO** | Recovery Time Objective | 시스템을 **얼마 안에** 복구해야 하는가 |
| **RPO** | Recovery Point Objective | 데이터를 **어느 시점까지** 잃어도 되는가 |
| **MTD** | Maximum Tolerable Downtime | 조직이 버틸 수 있는 **최대 중단 시간** |
| **WRT** | Work Recovery Time | 시스템 복구 후 **업무 정상화**까지 걸리는 시간 |

### 📊 시간 축 흐름

```
재해 발생
    │
    ▼
[서비스 중단] ◀──── RPO: 이 시점 이전 데이터까지 복구 목표
    │
    │◀──────────────── RTO: 이 시간 안에 시스템 복구 완료
    ▼
[시스템 복구 완료]
    │
    │◀──────────────── WRT: 데이터 검증 + 업무 정상화 소요 시간
    ▼
[정상 운영 재개]

│◀─────────────────────────────────────────────────────▶│
                   MTD = RTO + WRT
                 (최대 허용 중단 시간)
```

### 📌 관계식

```
MTD = RTO + WRT

RTO < MTD  (RTO가 반드시 MTD보다 작아야 함)
```

### 💡 직관적 비유

| 용어 | 비유 |
|------|------|
| RTO | 병원 응급실 — **몇 시간 안에** 처치해야 위험하지 않다 |
| RPO | 자동차 블랙박스 — **최소 몇 분치** 영상은 있어야 한다 |
| MTD | 환자의 골든타임 — **이 시간 넘으면 치명적** |
| WRT | 퇴원 후 재활 — 시스템은 살았는데, **정상화까지 걸리는 시간** |

## 6. 재난 복구 흐름

```
① System Unavailable      — 시스템 사용 불가 (재해 발생)
        │
② Begin System Recovery   — 복구 시작
        │
③ Complete System Recovery — 시스템 복구 완료
        │
④ Recover Lost Data       — 유실 데이터 복구
        │
⑤ Recover Work Backlog    — 밀린 업무 처리
        │
⑥ Test and Verify         — 데이터 / 시스템 검증
        │
⑦ Resume Normal Operation — 정상 운영 재개
```

## 7. 가용성 — MTTF / MTTR / MTBF

> SLA(서비스 수준 협약) 작성 시 기준이 되는 지표

### 📌 용어 정리

| 용어 | 전체 이름 | 의미 | 좋으려면 |
|------|-----------|------|----------|
| **MTTF** | Mean Time To Failure | 고장 나기까지 평균 시간 | 길수록 좋음 ↑ |
| **MTTR** | Mean Time To Repair | 고장 후 복구까지 평균 시간 | 짧을수록 좋음 ↓ |
| **MTBF** | Mean Time Between Failures | 고장과 고장 사이 평균 시간 | 길수록 좋음 ↑ |

### 📊 시간 축 흐름

```
[정상 운영]──────────────[고장]──[수리]──[정상 운영]──────────[고장]
│◀──────── MTTF ────────▶│     │◀MTTR▶│
│◀──────────────────── MTBF ──────────────────────────────────▶│
```

### 📌 가용성 공식

```
가용성 = MTTF / (MTTF + MTTR)
```

> 💡 **가용성 높이려면**: MTTF ↑ (잘 안 고장남) + MTTR ↓ (빨리 고침)

### 📌 SLA (Service Level Agreement)

> **이용자 ↔ 제공자** 사이의 서비스 수준에 관한 계약  
> 가동률, 장애 대응 시간, 백업 횟수 등을 명시

## 8. RAID 종류

> ⭐⭐ 디스크 여러 개를 묶어 **속도** 또는 **안정성**을 높이는 기술

### 핵심 키워드: **스트라이핑** vs **미러링** vs **패리티**

| 용어 | 의미 |
|------|------|
| 스트라이핑 | 데이터를 나눠서 여러 디스크에 분산 저장 (속도 ↑) |
| 미러링 | 같은 데이터를 복사해서 다른 디스크에 저장 (안전성 ↑) |
| 패리티 | 오류 검출/복구를 위한 데이터 (안전성 ↑) |

---

### 🔴 RAID 0 — Striping (스트라이핑)

```
데이터: [A][B][C][D]

디스크1: [A][C]
디스크2: [B][D]
```

| 장점 | 단점 |
|------|------|
| 빠름 (분산 저장) | 하나만 고장 나도 전체 데이터 손실 |

---

### 🟢 RAID 1 — Mirroring (미러링)

```
디스크1: [A][B][C][D]
디스크2: [A][B][C][D]  ← 완전 동일 복사본
```

| 장점 | 단점 |
|------|------|
| 안전 (하나 고장나도 OK) | 저장 공간 절반 낭비 |

---

### 🟡 RAID 2 — Hamming Code ECC

- 데이터를 모든 드라이브에 스트라이핑
- 오류 검출·수정을 위한 **해밍코드** 사용
- 현재는 거의 쓰이지 않음 (개념 이해 수준으로 OK)

---

### 🟠 RAID 3 — Parallel Transfer with Parity (바이트 단위)

```
디스크1: [A][D][G]
디스크2: [B][E][H]
디스크3: [C][F][I]
디스크4: [패리티]   ← 패리티 전담 디스크
```

- **바이트 단위** 스트라이핑
- 패리티를 별도 디스크에 집중 저장 → 패리티 디스크 병목 발생

---

### 🟤 RAID 4 — Block Level Parity (블록 단위)

- RAID 3과 동일한 구조, **블록 단위** 스트라이핑
- 패리티 디스크 병목 문제 동일하게 존재

---

### 🔵 RAID 5 — Independent Access with Parity ⭐⭐

```
디스크1: [데이터][데이터][패리티]
디스크2: [데이터][패리티][데이터]
디스크3: [패리티][데이터][데이터]
```

- **패리티를 모든 디스크에 분산** 저장 (RAID 3, 4의 병목 해소)
- 실무에서 가장 많이 쓰임

| 장점 | 단점 |
|------|------|
| 안정성 + 효율 모두 좋음 | 구현 복잡, 쓰기 성능 다소 손해 |

---

### 🟣 RAID 1+0 vs RAID 0+1

#### RAID 1+0 (Ten) — **미러링 먼저, 스트라이핑 나중**

```
[디스크1 ↔ 디스크2]   [디스크3 ↔ 디스크4]
    (미러링)               (미러링)
          └───── 스트라이핑 ─────┘
```

#### RAID 0+1 — **스트라이핑 먼저, 미러링 나중**

```
[디스크1 + 디스크2]  →  스트라이핑 세트
[디스크3 + 디스크4]  →  위의 미러링
```

> 💡 **RAID 1+0이 더 유리**: 디스크 하나 고장 시 복구 범위가 더 작음

| 유형 | 순서 | 안정성 |
|------|------|--------|
| RAID 1+0 | 미러링 → 스트라이핑 | 더 좋음 ✅ |
| RAID 0+1 | 스트라이핑 → 미러링 | 상대적으로 불리 |

---

### 📋 RAID 종류 요약 ⭐

| 레벨 | 방식 | 속도 | 안전성 | 특징 |
|------|------|------|--------|------|
| RAID 0 | 스트라이핑 | ⭐⭐⭐⭐⭐ | ❌ | 빠르지만 위험 |
| RAID 1 | 미러링 | ⭐⭐ | ⭐⭐⭐⭐⭐ | 안전하지만 공간 손해 |
| RAID 3 | 바이트 스트라이핑 + 패리티 디스크 | ⭐⭐⭐ | ⭐⭐⭐ | 패리티 디스크 병목 |
| RAID 5 | 블록 스트라이핑 + 분산 패리티 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 균형형, 실무 표준 |
| RAID 1+0 | 미러 후 스트라이핑 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 속도 + 안전 둘 다 |

## 9. 데이터 백업 유형

| 유형 | 백업 대상 | 백업 시간 | 복구 시간 | 아카이브 비트 |
|------|-----------|-----------|-----------|----------------|
| **완전 백업** | 전체 데이터 | 느림 🐢 | 빠름 🚀 | 해제 |
| **차등 백업** | 마지막 완전 백업 이후 변경분 | 보통 | 보통 | 유지 |
| **증분 백업** | 마지막 백업 이후 변경분 | 빠름 🚀 | 느림 🐢 | 해제 |

### 📊 시각화

```
[월] 완전 백업  ← 전체 백업

[화] 차등 백업  ← 월 이후 변경분 전체
[수] 차등 백업  ← 월 이후 변경분 전체 (점점 커짐)

[화] 증분 백업  ← 월 이후 변경분만
[수] 증분 백업  ← 화 이후 변경분만 (가장 작음)
```

> 💡 **복구 시**: 증분은 완전백업 + 모든 증분백업 순서대로 적용해야 해서 느림

## 10. BCP/DRP 테스트 종류

> ⭐⭐ 실제 재난을 가정해 대응 능력을 검증하는 훈련

### 📋 테스트 비교

| 테스트 종류 | 1차 사이트 | 2차 사이트 | 특징 |
|-------------|-----------|-----------|------|
| **체크리스트 / 구조적 워크스루** | ✅ | ✅ | 문서만 검토, 실제 시스템 미가동 |
| **시뮬레이션** | ✅ | ❌ | 재난 상황 가정해 1차 사이트에서 훈련 |
| **병행 테스트** | ✅ | ✅ | 두 사이트 동시 운영하며 비교 |
| **완전 중단 테스트** | ❌ | ✅ | 1차 중단 후 2차만 운영, 가장 현실적 |

### 🔺 주의사항

```
✔ 반드시 사전에 공지
✔ 핵심 인력 포함 전 직원 참여
✔ 업무 폭주 시간대는 피할 것
```

## 11. 핵심 요약 암기표

### 🗂️ BCP / DRP / BCM

```
BCP = 회사 전체 생존 계획 (IT 포함)
DRP = IT 복구 계획
BCM = BCP + DRP를 관리하는 전체 체계
BIA = 핵심 업무 식별 + 복구 우선순위 결정
```

### 🗂️ 2차 사이트

```
미러  : Active-Active  / RTO 즉시     / 비용 최고 / 실시간 동시 운영
핫    : Active-Standby / RTO 수 시간  / 비용 높음 / 실시간 대기
웜    : 부분 준비      / RTO 수일~수주 / 비용 중간 / 핵심만 대기
콜드  : 장소만         / RTO 수주~수개월 / 비용 최저 / 재해 후 조달
```

### 🗂️ 복구 지표

```
RTO  = 시스템 복구 목표 시간
RPO  = 허용 데이터 손실 시점
WRT  = 업무 정상화 소요 시간
MTD  = 최대 허용 중단 시간 = RTO + WRT
```

### 🗂️ 가용성

```
MTTF = 고장까지 걸리는 시간 (길수록 좋음)
MTTR = 수리에 걸리는 시간  (짧을수록 좋음)
MTBF = 고장 간격 평균 시간 (길수록 좋음)
가용성 = MTTF / (MTTF + MTTR)
```

### 🗂️ RAID

```
RAID 0  = 빠름, 위험 (스트라이핑)
RAID 1  = 안전, 공간 낭비 (미러링)
RAID 5  = 균형형 (분산 패리티) ← 실무 표준
RAID 1+0 = 미러링 후 스트라이핑 ← 0+1보다 유리
```

### 🗂️ 백업

```
완전 백업  : 백업 느림 / 복구 빠름
차등 백업  : 중간 / 중간
증분 백업  : 백업 빠름 / 복구 느림
```
