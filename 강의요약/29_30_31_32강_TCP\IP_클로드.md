## 1. 계층별 역할 요약

| 계층 | 통신 단위 | 핵심 역할 |
|------|-----------|-----------|
| 물리 계층 | 비트 | 실제 전기신호 전송 (물리적) |
| 데이터링크 계층 | 프레임 | 노드-대-노드 전달 |
| 네트워크 계층 | 패킷(데이터그램) | 목적지까지 경로 탐색 (라우팅) |
| 전송 계층 | 세그먼트(TCP) / 데이터그램(UDP) | 종단-대-종단 신뢰성 |
| 응용 계층 | 메시지 | 사용자 서비스 (HTTP, FTP 등) |

> 응용·전송·네트워크·데이터링크 = **논리적** 통신  
> 물리 계층만 = **물리적** 통신

- 데이터와 신호는 **아날로그 또는 디지털** 형태 가능

## 3. 데이터링크 계층

### 핵심 개념

- **노드(Node)**: 네트워크에 연결된 모든 장치 (PC, 공유기, 서버 등)
- **통신 방식**: 노드-대-노드 (한 칸씩 이동)
- **링크**: 노드와 노드 사이의 구간

### 제공 서비스

| 서비스 | 설명 |
|--------|------|
| 프레임 짜기 | 데이터를 프레임 단위로 포장 |
| 흐름 제어 | 송신 속도 조절 |
| 오류 제어 | 전기신호 오류 탐지·복구 |
| 혼잡 제어 | ❌ 데이터링크 계층 담당 아님 → 네트워크/전송 계층 |

> 전기신호는 오류에 취약 → 프레임도 오류에 취약

## 4. 네트워크 계층

### 라우팅 vs 포워딩

| 구분 | 설명 | 비유 |
|------|------|------|
| 라우팅 | 전체 경로를 계산하고 지도를 만드는 것 | 내비게이션 경로 설정 |
| 포워딩 | 패킷이 도착했을 때 라우터가 어디로 보낼지 결정 | 내비 보고 핸들 꺾기 |

- 포워딩 테이블 = 라우팅 테이블 (같은 말)
- 라우터는 **3계층(네트워크 계층) 장비**

## 5. IP 주소 체계

### 클래스 기반 주소

| 클래스 | 첫 비트 | 호스트 수 | 범위 |
|--------|---------|-----------|------|
| A | 0 | 약 1,677만 개 (2²⁴-2) | 0.x.x.x ~ 127.x.x.x |
| B | 10 | 약 6만 5천 개 | 128.x.x.x ~ 191.x.x.x |
| C | 110 | 254개 | 192.x.x.x ~ 223.x.x.x |

**문제점**: 주소가 낭비되거나 부족 → **주소 고갈**

### 서브네팅 / 슈퍼네팅

- **서브네팅**: 큰 네트워크를 필요에 따라 작게 쪼개는 것
  - 단점: 기반 배분 어려움
- **슈퍼네팅**: 여러 네트워크를 하나로 묶는 것
  - 단점: 라우팅 복잡해짐
- 둘 다 근본적인 주소 고갈 해결 못 함

### CIDR (클래스리스)

- 클래스 권한 제거 → 필요한 만큼만 할당
- **슬래시 표기법**: `192.168.1.0/24`
- 인터넷 라우팅 테이블 비대화 방지
- 여러 addressing domain으로 나눠 라우팅 정보량 감소

| 지원 여부 | 프로토콜 |
|-----------|----------|
| CIDR 지원 | BGP4, OSPF |
| CIDR 미지원 | RIPv1, IGRP |

### VLSM

- 서브네팅을 여러 번 반복하여 크기가 다른 서브네트워크 계층으로 구분하는 기법

### 특수 주소 (시험 중요)

| 종류 | 주소 | 설명 |
|------|------|------|
| 루프백 | 127.0.0.0/8 | 자기 자신에게, 외부로 안 나감. `ping 127.0.0.1`로 TCP/IP 활성화 확인 |
| 제한된 브로드캐스트 | 255.255.255.255 | 같은 네트워크 전체 전송. 라우터 밖으로 못 나감 |
| 멀티캐스트 | 224.0.0.0 ~ 239.255.255.255 | 특정 그룹에게만 전송 |

### 사설 주소 (Private IP)

| 클래스 | 주소 블록 | 범위 |
|--------|-----------|------|
| A | 10.0.0.0/8 | 10.x.x.x 전체 |
| B | 172.16.0.0/12 | 172.16.x.x ~ 172.31.x.x |
| C | 192.168.0.0/16 | 192.168.x.x 전체 |

> 인터넷 공인 IP 아님 — 내부망 전용

## 6. IPv4 데이터그램

### 기본 특성

- **비신뢰적**: 도착 보장 안 함 (신뢰성은 TCP가 담당)
- **비연결형**: 미리 연결 없이 그냥 전송
- **최선형(Best Effort)**: 최선을 다하지만 보장은 없음
- 구조: **헤더 + 페이로드(데이터)**

### 헤더 필드 정리

| 필드 | 크기 | 설명 |
|------|------|------|
| VER | 4비트 | IP 버전 (= 4) |
| HLEN | 4비트 | 헤더 길이 |
| 서비스 유형 | 8비트 | 우선순위 (급배송 vs 일반) |
| 전체 길이 | 16비트 | 헤더 + 데이터 합산 크기 |
| 식별자 | 16비트 | 단편화된 조각들의 공통 번호 |
| 플래그 | 3비트 | 조각 더 있음 / 마지막 조각 표시 |
| 단편화 오프셋 | 13비트 | 이 조각이 원래 데이터에서 몇 번째 위치인지 |
| TTL | 8비트 | 라우터 통과마다 1 감소, 0이 되면 폐기 |
| 프로토콜 | 8비트 | 상위 계층 프로토콜 식별 (다중화/역다중화) |
| 헤더 체크섬 | 16비트 | 헤더 오류 검출 (오류 시 수정 없이 폐기) |
| 출발지 IP | 32비트 | 보내는 주소 |
| 목적지 IP | 32비트 | 받는 주소 |
| 옵션 | 가변 | 선택적 필드 |

### TTL

- 목적: 패킷이 네트워크에서 **무한 루프** 도는 것 방지
- 라우터 지날 때마다 1씩 감소
- 0이 되면 즉시 폐기
- 이름은 "Time"이지만 실제로는 **라우터 홉 횟수** 카운트

### 프로토콜 필드와 다중화

- 프로토콜 필드 = 상위 계층 프로토콜 구분 태그
- **다중화**: 여러 프로토콜 데이터를 하나의 IP로 함께 전송
- **역다중화**: 도착 후 프로토콜 필드 보고 해당 프로토콜로 분배

| 번호 | 프로토콜 |
|------|----------|
| 1 | ICMP |
| 6 | TCP |
| 17 | UDP |

### 헤더 체크섬

- 헤더 **오류 검출**만 함 (오류 수정 ❌)
- 오류 발생 시 → **그냥 버림**
- 라우터 통과마다 TTL 변경 → 체크섬 재계산 필요
- 데이터(페이로드)는 검사 안 함 → 상위 계층(TCP 등)이 담당

### 단편화 (Fragmentation)

- 데이터가 MTU(최대 전송 단위)보다 크면 쪼개서 전송
- 식별자: 같은 원본에서 나온 조각들은 같은 번호
- 플래그: 뒤에 조각 더 있는지 표시
- 오프셋: 원본에서 이 조각의 위치

## 7. ARP / RARP / GARP

### ARP (Address Resolution Protocol)

- **역할**: IP 주소 → MAC 주소 변환
- 실제 전송에는 IP 주소뿐 아니라 **물리 주소(MAC)** 필요

#### 동작 흐름

```
1. ARP 요청 (브로드캐스트)
   → "이 IP 주소를 가진 장치 누구야?"

2. ARP 응답 (유니캐스트)
   → "나야! 내 MAC 주소는 이거야"
```

- **ARP Cache**: 찾은 MAC 주소를 저장해두는 공간 (매번 물어보는 비효율 방지)

#### TCP/IP에서 ARP 위치

- IP(네트워크 계층)와 MAC(데이터링크 계층) **사이의 다리 역할**

### RARP

- ARP 반대 방향: **MAC 주소 → IP 주소** 변환

### GARP (Gratuitous ARP)

- Sender IP = Target IP인 ARP 요청 (자기 자신에게)
- 브로드캐스트로 네트워크에 **자신의 존재를 알림**

#### 목적

1. IP 충돌 감지
2. 다른 장비의 ARP Cache 갱신

#### 취약점 ⚠️

- 인증 없이 ARP Cache 갱신 → **ARP 스푸핑 공격** 가능
- 공격자가 가짜 MAC 주소로 응답 → 데이터가 공격자에게 전달

## 8. ICMP

### 개요

- **역할**: 네트워크 오류 보고 및 상태 진단
- 오류 **수정은 안 함** — 보고만 하고, 수정은 상위 계층에 맡김
- 계층 위치: **네트워크 계층** (IP와 같은 레벨)

### ICMP 메시지 구조

- **유형(Type)**: 메시지 대분류
- **코드(Code)**: 세부 원인 (소분류)

### 오류 보고 메시지

| 메시지 | 설명 |
|--------|------|
| 목적지 도달 불가 (Destination Unreachable) | 최종 목적지에 도달 못 함 |
| 근원지 억제 (Source Quench) | 데이터그램 폐기 알림 (IPv6에서 삭제) |
| 재지정 (Redirect) | 더 적절한 라우터로 변경 요청. **ICMP Redirect 공격** 가능 |
| 시간 경과 (Time Exceeded) | TTL=0으로 폐기됨 알림. Code 0: 폐기, Code 1: 재조합 타임아웃 |

### ICMP 메시지가 생성되지 않는 경우

- 브로드캐스트 주소를 가진 데이터그램
- 멀티캐스트 주소를 가진 데이터그램
- 첫 번째 단편이 아닌 단편 데이터그램
- 특수 주소 (127.0.0.0, 0.0.0.0 등)

### 질의 메시지

| 타입 | 메시지 | 설명 |
|------|--------|------|
| 8 | Echo Request | 연결 확인 요청 (ping) |
| 0 | Echo Reply | Echo Request에 대한 응답 |

### ping & traceroute

- **ping**: ICMP Echo Request/Reply로 호스트 생존 확인
- **traceroute**: TTL을 1부터 1씩 늘려가며 각 라우터의 위치 확인

## 9. IPv6

### 등장 배경

- IPv4 주소 고갈 (32비트 → 약 43억 개)
- 보안 문제, 라우팅 효율성, QoS 부족

### IPv4 vs IPv6 비교 (매우 중요)

| 구분 | IPv4 | IPv6 |
|------|------|------|
| 주소 길이 | 32비트 | 128비트 |
| 표시 방법 | 8비트 × 4부분, 10진수 | 16비트 × 8부분, 16진수 |
| 주소 개수 | 약 43억 개 | 약 3.4 × 10³⁸ 개 |
| 주소 할당 | 클래스 단위 비순차 할당 | 네트워크 규모에 따라 순차 할당 |
| 브로드캐스트 | 있음 | 없음 (멀티캐스트로 대체) |
| 헤더 크기 | 가변 (20~60바이트) | 고정 (40바이트) |
| QoS | 미흡 | 제공 |
| 보안 | IPSec 별도 설치 | IPSec 자체 지원 |
| NAT | 필요 | 불필요 |

### IPv6 주요 특징

#### 1. 확장된 주소 공간
- 2¹²⁸개 주소 → NAT 불필요
- 유니캐스트, 애니캐스트, 멀티캐스트 주소 형태

#### 2. 새로운 헤더 포맷
- 헤더 **고정 길이** (40바이트) → 라우터 처리 빠름
- **체크섬 삭제** (데이터링크 계층에서 이미 수행하므로 중복 제거)
- 단편화 → 라우팅 부담 줄임
- 옵션 → **확장 헤더**로 분리

#### 3. 보안 기능 내장
- IPv4: IPSec 프로토콜 별도 설치 필요
- IPv6: 프로토콜 내에 보안 기능 탑재
  - 인증(AH), 기밀성(ESP) 확장 헤더로 제공

#### 4. 향상된 서비스
- **플로우 레이블(Flow Label)** 필드로 QoS 지원

### IPv6 주소 종류

| 종류 | 설명 |
|------|------|
| 유니캐스트 | 1:1 전송 |
| 멀티캐스트 | 그룹 전송. 브로드캐스트를 멀티캐스트의 특수한 경우로 처리 |
| 애니캐스트 | 그룹 중 **가장 가까운 1개**에게만 전송. 송신 주소로 사용 불가 |

### IPv4 → IPv6 전환 방법

| 방법 | 설명 | 특징 |
|------|------|------|
| 이중 스택 (Dual Stack) | 한 장비가 IPv4 + IPv6 동시 지원 | 가장 추천, 장기적 방법 |
| 터널링 (Tunneling) | IPv6 패킷을 IPv4로 포장해서 통과 | IPv4 구간 우회 |
| 헤더 변환 (Translation) | IPv4 ↔ IPv6 헤더 완전 변환 | 대부분 IPv6 전환 후 일부 IPv4만 남았을 때 |

### IPv4가 아직 많이 쓰이는 이유

1. 기존 인프라 (서버, 라우터 등) 전부 IPv4 기반
2. IPv4 ↔ IPv6 직접 통신 불가 → 변환 장비 필요
3. **NAT** 덕분에 주소 부족 문제 임시 해결 중
4. 전환 비용 (장비 교체, 시스템 수정) 큼
5. 일반 사용자 입장에서 체감 차이 없음

## 10. 전송 계층 — UDP vs TCP

### UDP (User Datagram Protocol)

- **비연결형**, 비신뢰적
- 헤더: **8바이트 고정** (4개 필드)
- 확인 응답 없음 → 빠름

#### UDP 헤더 필드

| 필드 | 크기 | 설명 |
|------|------|------|
| 출발지 포트 | 16비트 | 보내는 포트 |
| 목적지 포트 | 16비트 | 받는 포트 |
| 전체 길이 | 16비트 | 헤더 + 데이터 크기 |
| 체크섬 | 16비트 | 오류 확인 |

#### UDP 사용 예시

- 스트리밍, 온라인 게임, DNS, RIP (30초마다 업데이트, 한두 번 못 받아도 무방)

---

### TCP (Transmission Control Protocol)

- **연결형**, 신뢰적, 순서 보장
- 헤더: **20바이트 이상** (필드 많음)
- 스트림 전송 → 세그먼트 단위로 쪼개서 전송

#### TCP 헤더 필드

| 필드 | 크기 | 설명 |
|------|------|------|
| 출발지 포트 | 16비트 | 보내는 포트 |
| 목적지 포트 | 16비트 | 받는 포트 |
| 순서 번호 | 32비트 | 세그먼트 순서 추적 |
| 확인응답 번호 (ACK) | 32비트 | 다음에 받아야 할 바이트 번호 |
| HLEN | 4비트 | 헤더 길이 |
| 예약 | 6비트 | 사용 안 함 |
| 제어 플래그 | 6비트 | SYN / ACK / FIN / RST / PSH / URG |
| 윈도우 크기 | 16비트 | 한 번에 받을 수 있는 데이터 크기 (흐름 제어) |
| 체크섬 | 16비트 | 오류 확인 |
| 긴급 포인터 | 16비트 | URG 플래그와 함께 사용 |

#### 제어 플래그 6가지

| 플래그 | 의미 |
|--------|------|
| SYN | 연결 요청 "나랑 대화할래?" |
| ACK | 확인 응답 "잘 받았어" |
| FIN | 정상 종료 "나 다 말했어, 끊자" |
| RST | 강제 리셋 "뭔 소리야, 끊어!" |
| PSH | 즉시 처리 요청 "이거 급해, 빨리!" |
| URG | 긴급 데이터 "이거 진짜 중요해!" |

#### 3-Way Handshake (연결 수립)

```
클라이언트                        서버
    |  ── SYN ──────────────→  |   "나 연결할게"
    |  ←──── SYN + ACK ──────  |   "응, 나도 연결할게"
    |  ── ACK ──────────────→  |   "확인!"
    |  (이후 데이터 전송 시작)   |
```

#### 4-Way Handshake (연결 종료)

```
클라이언트                        서버
    |  ── FIN ──────────────→  |   "나 다 보냈어"
    |  ←──── ACK ────────────  |   "알겠어"
    |  ←──── FIN ────────────  |   "나도 다 보냈어"
    |  ── ACK ──────────────→  |   "확인, 종료!"
```

#### 절반 폐쇄 (Half-Close)

- 한쪽은 전송 끝냈는데, 반대쪽은 아직 보낼 게 남은 상태
- 한 방향만 닫고, 나머지 방향은 열어두는 것

### UDP vs TCP 비교

| 구분 | TCP | UDP |
|------|-----|-----|
| 연결 | 연결 먼저 (3-way) | 그냥 전송 |
| 신뢰성 | 보장 | 없음 |
| 순서 | 번호 매겨서 보장 | 미보장 |
| 속도 | 느림 | 빠름 |
| 헤더 크기 | 20바이트 이상 | 8바이트 고정 |
| 사용 예 | HTTP, FTP, 이메일 | 스트리밍, 게임, DNS |

## 핵심 암기 포인트 요약

- **CIDR 지원**: BGP4, OSPF / **미지원**: RIPv1, IGRP
- **루프백**: 127.0.0.0/8 — 자기 자신, 외부로 안 나감
- **사설 IP**: 10.x.x.x / 172.16~31.x.x / 192.168.x.x
- **TTL**: 라우터 지날 때마다 1 감소, 0이면 폐기
- **ARP**: 브로드캐스트 요청 → 유니캐스트 응답
- **GARP 취약점**: 인증 없이 ARP Cache 갱신 → ARP 스푸핑
- **ICMP**: 오류 보고만, 수정 안 함
- **UDP 헤더**: 8바이트 고정, 4개 필드
- **TCP 헤더**: 20바이트 이상, 플래그 6개 (SYN/ACK/FIN/RST/PSH/URG)
- **IPv6 헤더**: 고정 40바이트, 체크섬 삭제
