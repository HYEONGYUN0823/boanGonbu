## 1. 보안제품 평가 기준 개요

### 1-1. 흐름 정리 ⭐

나라마다 기준이 달라서 불편 → 국제표준 만들자!

```
미국  →  TCSEC (오렌지북)
유럽  →  ITSEC
캐나다 → CTCPEC
         ↓
   국제표준 통합
         ↓
   CC (Common Criteria)
   ISO/IEC 15408
```

> **한 줄 요약:** TCSEC → ITSEC → CC(현재 국제표준)

---

### 1-2. 3종 비교 ⭐⭐

| 구분 | TCSEC | ITSEC | CC |
|------|-------|-------|----|
| 별명 | 오렌지북 | - | - |
| 제정 | 미국 | 유럽 4개국 | ISO/IEC (국제) |
| 표준 번호 | - | - | **ISO/IEC 15408** |
| 적용 범위 | 미국 내 | 유럽 | 세계 공통 |
| 기능성/보증성 | 묶어서 평가 | **분리** 평가 | 분리 평가 |
| CIA 중 중점 | 기밀성만 | 기밀+무결+가용 | 기밀+무결+가용 |
| 네트워크 | ❌ (OS만) | ✅ | ✅ |

## 2. TCSEC (오렌지북)

### 2-1. 개요

- 미국이 만든 **최초의 보안 평가 기준**
- TCB, 참조모니터, 보안커널 개념 도입
- Secure OS 평가에 집중

### 2-2. 등급 체계 ⭐⭐⭐

> 암기법: **M D C L S S V**

| 등급 | 이름 | 핵심 내용 | 예시 |
|------|------|-----------|------|
| D | Minimal | 보안 설정 없음 | MS-DOS, Windows 3.1 |
| C1 | Discretionary | 로그인 존재, 파일 권한 설정 | - |
| C2 | Controlled | 계정별 로그인, 감사 로그, 접근 거부 가능 | Windows NT 4.0, Unix |
| B1 | Labeled | 데이터에 보안 레벨 설정 가능 (대외비, 기밀 등) | - |
| B2 | Structured | 정형화된 보안 정책 존재 | - |
| B3 | Security | 불필요한 부분 제거, 모듈별 테스트 가능 | - |
| A1 | Verified | 이론상 완벽한 보안 (최고 등급) | - |

```
보안 강도
 A1  ███████  (최고)
 B3  ██████
 B2  █████
 B1  ████
 C2  ███
 C1  ██
 D   █         (최저)
```

> 정부 요구 수준: 1992년까지 C2 이상 → 2003년 B3 이상으로 강화

---

### 2-3. TCSEC 문제점 ⭐

| 문제점 | 설명 |
|--------|------|
| OS 중심 | 네트워크, DB 등 미고려 |
| 기밀성만 | 무결성·가용성 미고려 |
| 군사용 중심 | 민간 상업 환경에 적용 어려움 |
| 독립형 시스템 | 네트워크 환경 미반영 |

---

### 2-4. 레드북 (TNI)

- 오렌지북의 단점 보완: **네트워크 환경** 평가 기준
- Red Book = 책 표지 색 때문에 붙은 별명

```
오렌지북  →  OS 1대
레드북    →  네트워크 환경
```

## 3. ITSEC

### 3-1. 개요

- **독일, 프랑스, 네덜란드, 영국** 4개국이 만든 유럽 보안 기준

### 3-2. TCSEC vs ITSEC ⭐⭐

| 항목 | TCSEC | ITSEC |
|------|-------|-------|
| 기능성/보증성 | 묶어서 평가 | **분리** 평가 |
| CIA | 기밀성만 | **기밀+무결+가용** |
| 네트워크 | ❌ 독립형(stand-alone) | ✅ 네트워크 환경 |
| 유연성 | 낮음 | 더 높음 |

---

## 4. CC (Common Criteria) / ISO IEC 15408

### 4-1. 개요 ⭐⭐⭐

- 정식 명칭: **ISO/IEC 15408**
- TCSEC + ITSEC + CTCPEC를 통합한 국제 표준
- 1996년 Ver 1.0 → 1998년 Ver 2.0 발간

---

### 4-2. 핵심 구성 요소 ⭐⭐⭐

```
고객  →  PP 작성  →  업체  →  ST 작성  →  제품 개발  →  평가  →  EAL 부여
(요구사항)              (제품 설계서)         (TOE)
```

| 구성요소 | 영어 | 역할 |
|----------|------|------|
| **PP** | Protection Profile | 요구사항 모음 (What I Want?) |
| **ST** | Security Target | 제품 설계서, 벤더가 작성 (What I Have?) |
| **TOE** | Target Of Evaluation | 실제 평가받는 제품 |

---

### 4-3. PP vs ST 비교 ⭐⭐⭐

| 항목 | PP (보호 프로파일) | ST (보안 목표 명세서) |
|------|------------------|----------------------|
| 작성자 | 고객/사용자 측 | **벤더(개발사)** |
| 구현 독립성 | 독립적 | 종속적 |
| 적용 대상 | 제품군 (방화벽 전체) | 특정 제품 (A사 방화벽) |
| 표현 방법 | **What I Want?** | **What I Have?** |
| 오퍼레이션 | 미완료 가능 | 모두 완료 필수 |

```
PP  →  ST의 템플릿 역할
PP  →  ST를 수용 (O)
ST  →  PP를 수용 (X)
```

> PP 종류: 방화벽, IDS, VPN 등 약 10여 개

---

### 4-4. EAL 등급 ⭐⭐

> CC의 보증 수준 (7단계)

| 등급 | 이름 | 첫글자 |
|------|------|--------|
| EAL1 | 기능적 시험 | **F**unctionally |
| EAL2 | 구조적 시험 | **S**tructurally |
| EAL3 | 방법론적 시험 및 검사 | **M**ethodically |
| EAL4 | 방법론적 설계, 시험 및 검토 | **M**ethodically |
| EAL5 | 준정형적 설계 및 시험 | **S**emiformally |
| EAL6 | 준정형적 검증된 설계 및 시험 | **S**emiformally |
| EAL7 | 정형적 검증된 설계 및 시험 | **F**ormally |

```
EAL7  ███████  (최고, 가장 엄격)
EAL6  ██████
EAL5  █████
EAL4  ████
EAL3  ███
EAL2  ██
EAL1  █          (최저)
```

---

### 4-5. ISO/IEC 15408 문서 구성 ⭐

| 파트 | 내용 |
|------|------|
| Part 1 | 소개 및 일반 평가 모델 |
| Part 2 | **보안 기능** 요구사항 (클래스 > 패밀리 > 구성요소) |
| Part 3 | **보안 보증** 요구사항 (EAL 단계별) |

- **Part 2 보안 기능 예시:** 식별·인증, 암호 지원, 보안 감사, 접근 통제, 사용자 정보보호
- **Part 3 보증 요구사항 예시:** 개발, 시험, 형상관리, 취약점 평가, 배포 및 운영

---

### 4-6. 각 인증 등급 비교 (전부 7단계)

| 기준 | 등급 범위 |
|------|-----------|
| TCSEC | D → C1 → C2 → B1 → B2 → B3 → A1 |
| ITSEC | E0 → E1 → ... → E6 |
| CC | EAL1 → EAL2 → ... → EAL7 |

## 5. CCRA (국제상호인증협정)

- CC 기반 국제 상호 인정 협정
- 국가 간 교역 장벽을 낮춰 수출 용이하게 함
- **인증서 발행국(CAP)** vs **수용국(CCP)**
- 우리나라: **2006년 CAP 자격(발행국)으로 가입**
- 국내 평가기관: **한국인터넷진흥원(KISA)**
- 인증서 발행: **국가정보원**

## 6. ISMS-P 개요

### 6-1. ISMS-P가 뭔데? ⭐⭐⭐

```
회사:  "우리 보안 열심히 하고 있습니다!"
  ↓
국가:  "진짜?"
  ↓
     점검
  ↓
     합격
  ↓
ISMS-P 인증!
```

- 정보보호 및 개인정보보호를 위한 조치가 인증기준에 적합함을 증명하는 제도
- 근거: 정보통신망법 제47조 + 개인정보보호법 제32조의2
- 2018년 11월 7일 시행

---

### 6-2. ISMS vs ISMS-P 차이 ⭐⭐⭐

| 구분 | ISMS | ISMS-P |
|------|------|--------|
| 포함 | 정보보호만 | 정보보호 + 개인정보보호 |
| 인증 항목 | **80개** (1+2) | **101개** (1+2+3) |

```
① 관리체계 수립 및 운영  →   4개 분야, 16개 항목
② 보호대책 요구사항      →  12개 분야, 64개 항목
③ 개인정보 처리단계      →   5개 분야, 21개 항목

ISMS   = ① + ②      = 16 + 64 = 80개
ISMS-P = ① + ② + ③  = 16 + 64 + 21 = 101개
```

## 7. 관리체계 수립 및 운영

> PDCA 사이클: Plan → Do → Check → Act

```
관리체계 기반 마련 (6개)
        ↓
    위험관리 (4개)
        ↓
  관리체계 운영 (3개)
        ↓
관리체계 점검 및 개선 (3개)
```

---

### 7-1. 관리체계 기반 마련

> 핵심: 책임자 뽑고, 조직 만들고, 정책 세우고, 예산 줘라

| 항목 | 내용 |
|------|------|
| 경영진 참여 | 의사결정에 적극 참여, 보고·검토·승인 절차 수립 |
| 최고책임자 지정 | CISO(정보보호), CPO(개인정보) 임원급으로 지정 |
| 조직 구성 | 전문성 있는 실무조직 구성 |
| 범위 설정 | 관리체계 적용 범위 문서화 |
| 정책 수립 | 경영진 의지, 역할·책임, 활동 근거 포함 |
| 자원 할당 | 전문 인력 확보, 예산 할당 |

> **CISO** = 정보보호 최고책임자 (망법 근거)  
> **CPO** = 개인정보보호 책임자 (개인정보보호법 근거)

---

### 7-2. 위험관리

#### 흐름
```
정보자산 식별
     ↓
현황 및 흐름 분석
     ↓
  위험 평가 (연 1회 이상)
     ↓
보호대책 선정
```

#### 위험 처리 전략 (AAMT) ⭐⭐

| 전략 | 영어 | 예시 |
|------|------|------|
| **회피** | Avoid | 위험한 사업 아예 안 함 |
| **수용** | Accept | 그냥 감수 |
| **감소** | Mitigation | 방화벽 설치 |
| **전가** | Transfer | 보험 가입 |

---

### 7-3. 관리체계 운영

| 항목 | 내용 |
|------|------|
| 보호대책 구현 | 계획한 보호대책 실제 구현 |
| 보호대책 공유 | 관련 내용 공유·교육 |
| 운영현황 관리 | 지속적 운영 상태 관리 |

---

### 7-4. 관리체계 점검 및 개선

- 법적 요구사항 준수 검토 (컴플라이언스)
- 내부 감사·점검 수행
- 개선 조치 이행

## 8. 보호대책 요구사항

> 12개 분야, 64개 인증기준  
> 실제 보안 조치를 수행하는 영역

### 8-1. 12개 분야 암기

```
정책 → 조직 → 자산 → 인적보안 → 외부자 → 물리
인증 → 접근 → 암호 → 개발 → 운영 → 사고대응
```

---

### 8-2. 인적 보안

#### 직무분리 ⭐

- 개발자 ≠ 승인자 ≠ 배포자 (한 사람이 다 하면 위험)
- 직무분리 어려울 때 보완대책: **상호검토**, **책임추적성 확보**

#### 보안 서약

| 시점 | 내용 |
|------|------|
| 입사 시 | 정보보호 서약서 |
| 퇴직 시 | **비밀유지 서약서** (NDA 징구) |
| 외주 직원 | 보안 서약서 |

> **NDA 징구** = 비밀유지계약서(Non-Disclosure Agreement)를 **받는** 것

---

### 8-3. 외부자 보안 ⭐

- 외부자 현황 파악 및 위험 평가
- 계약서에 보안 요구사항 명시
- 계약 만료 시 **NDA 징구**

---

### 8-4. 물리 보안

#### 보호구역 3단계 ⭐⭐

```
통제구역  >  제한구역  >  접견구역
  (강)                      (약)
```

| 구역 | 장소 | 특징 |
|------|------|------|
| 통제구역 | 전산실, 서버실, 통신실 | 최소 인원 + 추가 절차 필요 |
| 제한구역 | 사무실 | 출입증 필요 |
| 접견구역 | 로비, 회의실 | 일반 출입 가능 |

#### 보호 설비

| 설비 | 용도 |
|------|------|
| UPS (무정전 전원장치) | 정전 시 임시 전원 공급 |
| 비상발전기 | 장시간 정전 대비 |
| 디가우저 | 자기장으로 데이터 완전 삭제 |

---

### 8-5. 인증 및 권한관리

```
누구냐? (인증) → 들어와도 되냐? (접근통제) → 뭘 할 수 있냐? (권한)
```

- **최소 권한의 원칙**: 업무 목적에 따른 최소한의 권한만 부여
- 접근권한 부여·변경·삭제 이력 관리 및 주기적 검토

---

### 8-6. 접근통제 7가지 ⭐⭐

> 암기: **네·정·응·데·무·원·인**

| 순서 | 접근통제 영역 |
|------|-------------|
| 네 | 네트워크 접근 (DMZ, 서버팜, DB존 분리) |
| 정 | 정보시스템 접근 |
| 응 | 응용프로그램 접근 |
| 데 | 데이터베이스 접근 |
| 무 | 무선 네트워크 접근 |
| 원 | 원격접근 통제 |
| 인 | 인터넷 접속 통제 |

---

### 8-7. DMZ ⭐⭐

> Demilitarized Zone (비무장지대)

```
인터넷
   ↓
[방화벽]
   ↓
  DMZ     ←  웹서버, 공개 서버
   ↓
[방화벽]
   ↓
내부망    ←  DB, 내부 시스템
```

- 공개서버는 **반드시 DMZ**에 배치
- 내부망과 분리된 영역

---

### 8-8. 암호키 관리

- 생성 → 이용 → 보관 → 배포 → **파기** 전 단계 관리
- 복구 가능하도록 **별도 서버에 분리 보관**
- 암호키 접근권한 최소화

---

### 8-9. 로그 관리 ⭐⭐

#### 접속기록 필수 항목 (계·접·접·수)

| 항목 | 예시 |
|------|------|
| **계**정 및 식별자 | admin |
| **접**속일시 | 2026-05-29 12:00 |
| **접**속지 정보 | 192.168.0.1 |
| **수**행업무 | 회원정보 조회 |

#### 시간 동기화

- **NTP** (Network Time Protocol) 사용
- 모든 시스템 시각 통일 → 신뢰성 있는 로그 분석 가능

### 8-10. 정보자산 완전 삭제 방법 ⭐⭐⭐

> 복원 불가능한 3가지 방법

| 방법 | 설명 |
|------|------|
| 물리 파괴 | 소각, 파쇄 |
| **디가우저** | 강한 자기장으로 데이터 삭제 |
| 덮어쓰기 | 00000, 11111 반복 덮어쓰기 (Low Level Format) |

### 8-11. BCP / DRP ⭐⭐

| 구분 | BCP | DRP |
|------|-----|-----|
| 전체 명칭 | Business Continuity Plan | Disaster Recovery Plan |
| 번역 | 업무 연속성 계획 | 재해 복구 계획 |
| 범위 | **회사 전체** (인력, 건물, 자금, IT) | **IT 시스템** 복구 |
| 목적 | 재난 상황에서도 서비스 지속 | 망가진 시스템 복구 |

```
BCP
 ├ 인력
 ├ 건물
 ├ 자금
 └ IT (DRP)
```

#### RTO / RPO / MTO ⭐⭐

| 용어 | 영어 | 의미 |
|------|------|------|
| **RTO** | Recovery Time Objective | **복구목표시간** (얼마나 빨리 복구?) |
| **RPO** | Recovery Point Objective | **복구목표시점** (얼마나 데이터 손실 허용?) |
| **MTO** | Maximum Tolerable Outage | 최대 허용 중단 시간 |

```
MTO = RTO + WRT
WRT = Work Recovery Time (업무 재개 시간)
```

## 10. 기타 인증 및 보안 활동

### 10-1. GDPR ⭐⭐

- **EU 개인정보보호법** (2018년 시행)
- EU 역내 개인정보의 자유로운 이동 보장

#### 핵심 내용

| 항목 | 내용 |
|------|------|
| 적용 범위 | EU 거주자 개인정보 처리 시 적용 (EU 사업장 없어도 적용) |
| 아동 동의 | **만 16세 미만** 시 부모 동의 필요 (최저 13세까지 낮출 수 있음) |
| **DPO** | Data Protection Officer, 개인정보보호 책임자 (우리나라 CPO와 유사) |
| one-stop-shop | 주 사업장 국가의 감독기구 1곳만 대응하면 됨 |

#### 제재 규정 ⭐⭐

| 위반 수준 | 과징금 |
|-----------|--------|
| 일반 위반 | 전년도 전세계 매출액 **2%** 또는 **1,000만 유로** 중 큰 금액 |
| 심각한 위반 | 전년도 전세계 매출액 **4%** 또는 **2,000만 유로** 중 큰 금액 |

#### 정보주체의 권리 ⭐⭐

| 권리 | 설명 |
|------|------|
| 정보를 제공받을 권리 | 처리 내용 고지 받을 권리 |
| 접근권 | 본인 정보 확인 요구 |
| 정정권 | 부정확한 정보 수정 요구 |
| **삭제권 (잊힐 권리)** | 본인 정보 삭제 요구 ← **시험 핵심** |
| 개인정보 이동권 | 다른 컨트롤러에게 정보 전송 요구 |
| 자동화 의사결정 거부 | 프로파일링 기반 결정 거부 권리 |

### 10-2. OECD 정보보호 9원칙 ⭐

> 암기: **인책대윤민 / 위보보재**

| 분류 | 원칙 |
|------|------|
| 정책 수준 | **인**식, **책**임, **대**응, **윤**리, **민**주주의 |
| 운영 수준 | **위**험평가, **보**안설계 및 이행, **보**안관리, **재**평가 |

### 10-3. 사이버 위기 경보 ⭐⭐⭐

> 암기: **초·파·노·주·빨**

```
정상(초록) → 관심(파랑) → 주의(노랑) → 경계(주황) → 심각(빨강)
```

| 단계 | 색상 | 주요 상황 |
|------|------|-----------|
| 정상 | 🟢 초록 | 평상시 |
| 관심 | 🔵 파랑 | 웜/바이러스 피해 가능성, 해외 사이버공격 국내 유입 우려 |
| 주의 | 🟡 노랑 | 일부 네트워크 장애, 일부 기관 침해사고 발생 |
| **경계** | 🟠 주황 | 복수 ISP망 장애·마비, 국지적 침해사고, 대규모 피해 가능성 ← **자주 출제** |
| 심각 | 🔴 빨강 | 국가적 네트워크 사용 불가, 국가적 공동 대처 필요 |
