## 1. 데이터베이스 기본 개념

### ✅ 데이터베이스(DB)란?

> 여러 사람이 함께 사용하는 **정리된 데이터 창고**

| 정의 | 설명 | 예시 |
|------|------|------|
| **통합된 데이터** | 흩어진 데이터를 한 곳에 모음 | 학생정보 + 성적정보 통합 |
| **저장된 데이터** | 컴퓨터 저장장치(HDD/SSD)에 보관 | 전원 꺼져도 유지됨 |
| **운영 데이터** | 실제 업무에 사용되는 데이터 | 은행 계좌, 쇼핑몰 주문 |
| **공용 데이터** | 여러 사용자가 동시에 사용 | 교수·학생·행정실 모두 사용 |

---

### ✅ DBMS (Database Management System)

DB를 **관리해주는 프로그램** (Oracle, MySQL, MSSQL 등)

```
[사용자]  →  "3학년 학생 보여줘"
   ↓
[DBMS]   →  질의 처리 및 결과 반환
   ↓
[DB]     →  실제 데이터 저장소
```

| 구성요소 | 역할 |
|---------|------|
| DB | 데이터 저장 |
| DBMS | 관리 소프트웨어 |
| 사용자 | 데이터 요청/사용 |

> **OS vs DBMS 접근제어 비교**  
> - OS: 파일 단위 읽기/쓰기만 제어 가능 (특정 레코드·필드 제한 불가)  
> - DBMS: 특정 행(row), 열(column), 필드 단위까지 **세밀하게** 접근 제어 가능

## 2. 키(Key)의 유형 ⭐⭐⭐ 매우중요

> 데이터를 **구별하기 위한 값**

### 📌 키 유형 한눈에 비교

| 키 이름 | 유일성 | 최소성 | 설명 |
|--------|:------:|:------:|------|
| **슈퍼키** (Super Key) | ✅ | ❌ | 유일성만 만족 (불필요한 속성 포함 가능) |
| **후보키** (Candidate Key) | ✅ | ✅ | 유일성 + 최소성 모두 만족 |
| **기본키** (Primary Key) | ✅ | ✅ | 후보키 중 **대표로 선택된 키** |
| **대체키** (Alternate Key) | ✅ | ✅ | 후보키 중 **기본키가 되지 못한 나머지** |
| **외래키** (Foreign Key) | - | - | 다른 테이블의 **기본키를 참조**하는 키 |

### 📌 예시로 이해하기

```
학생 테이블
┌────────┬──────────┬─────────────┐
│ 학번   │  이름    │  주민번호   │
├────────┼──────────┼─────────────┤
│  1001  │  김철수  │  990101-1   │
│  1002  │  이영희  │  030101-2   │
└────────┴──────────┴─────────────┘

- 슈퍼키  : {학번}, {주민번호}, {학번+이름}, {학번+주민번호} ...
- 후보키  : {학번}, {주민번호}  ← 유일하면서 최소한의 컬럼
- 기본키  : {학번}              ← 대표로 선택
- 대체키  : {주민번호}          ← 후보였지만 탈락
```

```
수강 테이블
┌──────────┬──────────┐
│ 수강번호 │  학번(FK) │
├──────────┼──────────┤
│   A1     │   1001   │  ← 학생 테이블의 기본키 참조
└──────────┴──────────┘

- 외래키: 수강테이블의 [학번] → 학생테이블의 [학번(PK)] 참조
```

## 3. 무결성의 종류

### 📌 참조 무결성

> 외래키는 **NULL이거나**, 참조하는 테이블의 **기본키 값과 같아야 함**

```
✅ 정상: 수강테이블 학번 = 1001  →  학생테이블에 1001 존재
❌ 오류: 수강테이블 학번 = 9999  →  학생테이블에 9999 없음
```

| 무결성 종류 | 설명 |
|------------|------|
| **참조 무결성** | 외래키는 NULL이거나 참조 PK 값이어야 함 |
| **운영적 무결성** | 트랜잭션 병행 처리 중 논리적 일관성 보장 |
| **의미적 무결성** | 데이터 허용값 통제 (예: 대학교 5학년은 없음) |

## 4. 트랜잭션(Transaction) ⭐⭐⭐ 매우중요

### ✅ 트랜잭션이란?

> **논리적인 작업의 한 단위** - 전부 성공하거나 전부 실패해야 함

```
[계좌이체 예시]
A계좌 -5,000원   +   B계좌 +5,000원
→ 둘 다 성공해야 정상 / 하나라도 실패하면 전부 취소
```

### 📌 ACID 성질 (시험 핵심 암기)

| 문자 | 이름 | 설명 | 예시 |
|------|------|------|------|
| **A** | 원자성 (Atomicity) | 전부 성공 or 전부 실패 | 출금만 되고 입금 안되면 전부 취소 |
| **C** | 일관성 (Consistency) | DB 규칙 항상 유지 | 잔액이 음수가 되면 안됨 |
| **I** | 격리성 (Isolation) | 동시 작업 간 간섭 없음 | 두 사람이 동시 송금해도 꼬이면 안됨 |
| **D** | 영속성 (Durability) | 완료된 작업은 영구 저장 | 전원 꺼져도 결과 유지 |

## 5. 데이터베이스 보안 개요

> 핵심: **해커가 DB를 탈취해도 암호화되어 있으면 볼 수 없다**

### 보안 요구사항

| 요구사항 | 설명 |
|---------|------|
| 사용자 인증 | 허가된 사용자만 접근 |
| 접근통제 | 권한 범위 내에서만 데이터 접근 |
| 무결성 보호 | 인가되지 않은 변경으로부터 보호 |
| 감사 | 접근 및 변경 이력 기록 |
| 비밀 데이터 보호 | 민감 정보 암호화 관리 |

## 6. 보안 위협 ⭐⭐⭐ 매우중요

### 📌 집성 (Aggregation)

> 개별적으로는 민감하지 않은 정보들을 **조합해서 비밀 정보를 생성**하는 공격

```
지사A 매출: 1억  (공개 정보)
지사B 매출: 2억  (공개 정보)
지사C 매출: 3억  (공개 정보)
        ↓ 합산
회사 총매출: 6억  ← 대외비였던 정보!
```

- 대응책: 아직 완전한 해결책 없음 (출제 포인트!)

---

### 📌 추론 (Inference)

> **통계 데이터나 간접 정보**를 통해 비밀 정보를 **추측**하는 방법

```
예시 1)
"전과 여부 있는 학생 수: 1명"
+ "우리 반 학생: 김철수만 있음"
→ 김철수가 전과 있다는 걸 추론 가능

예시 2)
"유권자 수: 1,000명"
"여당 지지자: 650명"
→ 야당 지지율 35% 추론 가능
```

#### 추론 대응책

| 방법 | 설명 |
|------|------|
| Polyinstantiation | 낮은 권한 사용자에게 가짜 정보 제공 |
| Partition | 데이터를 분리하여 집합적 추론 방지 |
| Cell suppression | 민감 셀의 값을 아예 숨김 |
| Noise | 통계에 의도적 오차 삽입 |
| Perturbation (섭동) | 데이터를 약간 변조하여 정확한 추론 방지 |

---

### 📌 다중사례화 (Polyinstantiation)

> **동일한 객체에 대해 두 가지 버전**을 만들어 낮은 권한 사용자에게 **가짜 정보(Cover Story)** 를 제공

```
관리자가 보는 정보:  "미사일 위치 = 부산항"
일반 사용자가 보는 정보: "미사일 위치 = 정보 없음"
```

- 추론 및 집성 공격 방어에 활용

## 7. 데이터베이스 보안 통제 ⭐⭐⭐ 매우중요

> 3가지 핵심 방어 수단: **흐름제어, 추론제어, 접근제어**

### 📌 흐름제어 (Flow Control)

> 높은 보안 수준의 정보가 **낮은 보안 수준 객체로 흘러가는 것**을 차단

```
[1급 비밀 데이터]
        ↓ ❌ 차단
[일반 사용자 영역]
```

---

### 📌 추론제어 (Inference Control)

> 통계나 간접 정보로부터 **비밀 정보가 추론되는 것**을 방지

#### 추론 채널 종류

| 채널 | 설명 |
|------|------|
| 간접 접근 | 직접 접근 없이 간접적으로 정보 획득 |
| 상관 데이터 | 의미적으로 연관된 데이터를 통해 추론 |
| 미싱 데이터 | 데이터가 없다는 사실 자체로 비밀 데이터 존재 파악 |

#### 일반적인 추론 방지 방법
- 사용자가 보유한 데이터를 고려하여 질의 허용 여부 결정
- 부정확하거나 일관성 없는 질의 결과 제공 (의도적 오차)

---

### 📌 접근제어 (Access Control)

> 정책에 따라 **인가된 사용자만** 시스템 객체에 접근 허용

```
사용자      권한
────────────────
학생     →  조회(SELECT)만 가능
교수     →  조회 + 수정(UPDATE) 가능
관리자   →  전체 권한
```

## 8. DBMS 접근통제

### 📌 데이터베이스 언어

| 분류 | 명령어 | 역할 |
|------|--------|------|
| **DML** (Data Manipulation Language) | SELECT, INSERT, UPDATE, DELETE | 데이터 조작 |
| **DDL** (Data Definition Language) | CREATE, ALTER, DROP | 구조(테이블) 정의 |
| **DCL** (Data Control Language) | GRANT, REVOKE, DENY | 권한 제어 |

---

### 📌 SQL 기반 접근통제 - GRANT / REVOKE ⭐

```sql
-- 권한 부여 (GRANT)
GRANT SELECT ON student TO kim;
-- → kim에게 student 테이블 조회 권한 부여

-- 권한 회수 (REVOKE)
REVOKE SELECT ON student FROM kim;
-- → kim의 조회 권한 회수
```

#### WITH GRANT OPTION

> 권한을 받은 사용자가 **다른 사용자에게도 권한을 부여**할 수 있음

```
DBA
 ↓ GRANT (WITH GRANT OPTION)
철수
 ↓ GRANT
영희
```

#### 계단식 권한 회수 (CASCADE)

```
DBA → 철수 → 영희

DBA가 철수 권한 회수
        ↓
영희 권한도 자동으로 함께 회수됨
```

---

### 📌 뷰(View) 기반 접근통제 ⭐

> **하나 이상의 테이블로부터 유도된 가상 테이블** (실행 시간에만 구체화)

```
원본 테이블
┌────────┬──────────┬─────────────┐
│  학번  │  이름    │  주민번호   │
├────────┼──────────┼─────────────┤
│  1001  │  김철수  │  990101-1   │
└────────┴──────────┴─────────────┘

뷰 생성 (이름 컬럼만 공개)
┌──────────┐
│   이름   │
├──────────┤
│  김철수  │
└──────────┘
→ 주민번호는 일반 사용자에게 숨겨짐
```

| 특징 | 설명 |
|------|------|
| 논리적 독립성 | 기본 테이블 구조 변경 없이 다른 형태로 제공 |
| 보안성 | 민감 컬럼/행 숨김 처리 가능 |
| 가상 테이블 | 실제 저장되는 건 아님 (물리적 독립성 X) |

---

### 📌 DBMS 보안 점검 사항

| 구분 | 점검 항목 |
|------|---------|
| 기본 점검 | 디폴트 계정 패스워드 변경 (설치 시 기본 계정 해킹 악용 주의) |
| 기본 점검 | DBA 권한 제한 (일반 계정에 DBA 권한 부여 금지) |
| 추가 점검 | 사용하지 않는 계정 삭제 |
| 추가 점검 | 패스워드 일방향 암호화 (해시함수) 여부 확인 |

---

### 📌 다중 수준 보안 (MLS)

> 분류 세분화 정도(Granularity)에 따라 접근 제어 복잡도 증가

```
세분화 수준 (아래로 갈수록 세밀함)
━━━━━━━━━━━━━━━━━━
전체 데이터베이스
개별 테이블 (관계)
개별 열 (속성/컬럼)
개별 행 (튜플/레코드)
개별 요소 (값)
━━━━━━━━━━━━━━━━━━
```

## 9. DB 암호화 방법

> 크게 **컬럼 암호화 방식**과 **블록 암호화 방식**으로 나뉨

### 📌 방식 비교 한눈에 보기

| 방식 | 암호화 위치 | 특징 |
|------|------------|------|
| **Plug-In** | DB 서버 내부 | 앱 수정 거의 불필요, 구현 용이 |
| **API** | 애플리케이션 서버 | 통신 구간도 암호화, 차세대 시스템 적합 |
| **Hybrid** | APP + DB 서버 둘 다 | Plug-In 배치 성능 저하 보완 |
| **TDE** | DBMS 커널 내장 | 앱·스키마 수정 불필요, DBMS 최적화 |
| **파일 암호화** | OS 레벨 | 비정형 데이터(이미지, 영상, 로그) 암호화 |

---

### 📌 Plug-In 방식

```
[애플리케이션]
      ↓
[DB 서버]
      ↓ (Plug-In 호출)
[암호화 모듈]
      ↓
[DB 저장소]
```

- 암/복호화 모듈이 **DB 서버**에 설치
- DBMS가 플러그인 형태로 모듈 호출
- **앱 수정 거의 불필요** → 구현 용이

---

### 📌 API 방식 ⭐⭐⭐ 매우중요

```
[애플리케이션]
      ↓ (API 라이브러리로 암호화)
[암호화된 데이터]
      ↓ (암호화 상태로 전송)
[DB 서버 저장]
```

- 암/복호화 모듈이 **애플리케이션 서버**에 API 라이브러리로 설치
- **통신 구간까지 암호화 상태 유지** → 더 안전
- 차세대 시스템 개발에 적합

---

### 📌 Hybrid 방식

```
[애플리케이션]  ← API 암호화 모듈
      ↓
[DB 서버]       ← Plug-In 암호화 모듈
```

- Plug-In 방식의 **배치(batch) 업무 성능 저하**를 API 방식으로 보완
- APP + DB 서버 양쪽에 모두 암호화 모듈 설치

---

### 📌 TDE 방식 (Transparent Data Encryption) ⭐⭐⭐ 매우중요

```
[사용자/애플리케이션]
          ↓
[DBMS 커널 (TDE 내장)]
          ↓ 자동 암/복호화 처리
[디스크 저장 - 암호화 상태]
```

- DBMS에 **내장된 암호화 기능** 사용
- **Transparent(투명)**: 앱이나 사용자는 암호화 여부를 의식하지 않음
- 기존 **앱 수정 불필요**, DB **스키마 변경 불필요**
- DBMS 엔진에 최적화된 성능 제공

---

### 📌 파일 암호화 방식 (운영체제 암호화)

```
[OS 레벨]
    ↓ 파일 자체를 암호화
[DB 파일 / 로그 파일 / 이미지 / 영상]
```

- OS 상에서 **파일 단위** 암호화
- **비정형 데이터** 암호화에 유용 (로그, 이미지, 음성, 영상)

## 10. 인증 방식

### 📌 SQL Server 인증 방식 비교 ⭐

| 방식 | 설명 | 보안성 |
|------|------|:------:|
| **Windows 인증** | 윈도우 로그인 계정으로 SQL Server 자동 접속 (SID 사용) | ✅ 권장 |
| **SQL Server 인증** | DB 전용 ID/PW 입력 | ⚠️ 계정 노출 위험 |
| **혼합 인증** | Windows 인증 + SQL Server 인증 혼용 | - |

```
Windows 인증 흐름
Windows 로그인 성공 → SQL Server 별도 로그인 없이 자동 접속

SQL Server 인증 흐름
별도 ID: sa / PW: ****  → 계정 노출 시 보안 취약점
```

---

### 📌 고정 데이터베이스 역할 (Role)

| 역할 | 권한 |
|------|------|
| `db_owner` | 모든 권한 (소유자) |
| `db_datareader` | 읽기(SELECT)만 가능 |
| `db_datawriter` | 추가(INSERT), 수정(UPDATE), 삭제(DELETE) 가능 |

## 📝 시험 핵심 요약

### 키 유형

| 키 | 핵심 키워드 |
|----|------------|
| 후보키 | **유일성 + 최소성** |
| 슈퍼키 | 유일성만 (최소성 없음) |
| 기본키 | 후보키 중 **대표로 선택** |
| 대체키 | 기본키 탈락 후보키 |
| 외래키 | 다른 테이블 PK 참조 |

### ACID

| 기호 | 이름 | 핵심 |
|------|------|------|
| A | 원자성 | 전부 성공 or 전부 실패 |
| C | 일관성 | DB 규칙 항상 유지 |
| I | 격리성 | 동시 작업 간 간섭 없음 |
| D | 영속성 | 완료 작업 영구 저장 |

### 보안 위협

| 공격 | 핵심 |
|------|------|
| 집성 | 사소한 정보 조합 → 민감 정보 생성 |
| 추론 | 통계/간접 정보 → 비밀 정보 추측 |
| 다중사례화 | 낮은 권한자에게 가짜 정보 제공 (방어) |

### DB 암호화

| 방식 | 암호화 위치 | 특징 |
|------|------------|------|
| Plug-In | DB 서버 | 앱 수정 불필요 |
| API | APP 서버 | 통신 구간도 암호화 |
| Hybrid | APP + DB | Plug-In 성능 보완 |
| TDE | DBMS 커널 | Transparent, 앱·스키마 수정 불필요 |
| 파일 암호화 | OS 레벨 | 비정형 데이터 |
