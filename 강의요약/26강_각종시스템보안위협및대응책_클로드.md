## 1. 메모리 구조

프로그램이 실행되면 메모리는 4개 영역으로 나뉜다.

```
높은 주소 (0xFFFF...)
┌─────────────────────┐
│      Stack 영역      │  ← 위에서 아래로 성장 ↓
│  (지역변수, 반환주소) │
├─────────────────────┤
│         ↕           │  (Stack과 Heap이 서로 마주보며 성장)
├─────────────────────┤
│      Heap 영역       │  ← 아래에서 위로 성장 ↑
│    (동적 할당 공간)   │
├─────────────────────┤
│      Data 영역       │
│  (전역변수, 정적변수) │
├─────────────────────┤
│      Text 영역       │
│  (실행코드, 상수)     │  ← 읽기 전용
└─────────────────────┘
낮은 주소 (0x0000...)
```

| 영역 | 저장 내용 | 특징 |
|------|-----------|------|
| Text | 실행 코드, 상수 | 읽기 전용, 수정 불가 |
| Data | 전역변수, 정적변수 | 프로그램 시작~종료까지 유지 |
| Heap | 동적 할당 메모리 | malloc/free로 직접 관리 |
| Stack | 지역변수, 반환주소, FP | 함수 호출마다 프레임 생성 |

### 정적 vs 동적

- **정적**: 코드 작성 시 크기가 고정 → `char buf[10]`
- **동적**: 실행 중에 필요한 만큼 요청 → `malloc(n)`

## 2. 버퍼 오버플로우

### 스택 프레임 구조

함수가 실행될 때 스택에 아래와 같이 쌓인다.

```
높은 주소
┌─────────────────────┐
│   반환주소 (RET)     │  ← 함수 끝나면 여기로 점프
├─────────────────────┤
│   프레임포인터 (FP)  │  ← 이전 함수 위치 기억
├─────────────────────┤
│   지역변수 (buf)     │  ← 낮은 주소에 위치
└─────────────────────┘
낮은 주소
```

- **buf**: 데이터를 담는 그릇 (예: `char buf[10]`)
- **FP**: 이전 함수로 돌아갈 기준점 (책갈피)
- **RET**: 함수 종료 후 점프할 주소 (복귀 티켓)

### 반환주소 원리

CPU는 함수를 호출하기 전에 "돌아올 주소"를 스택에 저장한다.

```
1. main()이 func() 호출
2. CPU가 "func 끝나면 main의 줄 4로 돌아와" 라고 스택에 저장
3. func() 실행
4. func() 끝 → 스택에서 반환주소 꺼냄 → main 줄 4로 점프
```

### 공격 원리

`buf`가 10칸인데 30칸을 넣으면 넘친 데이터가 위쪽을 덮어씀.

```
정상:  [buf 10칸] [FP] [RET → main]
공격:  [AAAAAAAAAA넘침💥] [FP 덮임] [RET → 셸코드 주소]
```

### 공격 절차

1. **1단계**: 셸코드를 buf에 저장
2. **2단계**: 오버플로우로 반환주소를 buf 주소로 덮어씀
3. **3단계**: 함수 종료 시 buf로 점프 → 셸코드 실행 → root 권한 획득

### 취약한 함수 vs 안전한 함수

```c
// 취약 - 길이 체크 없음
strcpy(buf, argv[1]);
gets(buf);
scanf("%s", buf);

// 안전 - 길이 한도 지정
strncpy(buf, argv[1], sizeof(buf));
fgets(buf, sizeof(buf), stdin);
scanf("%9s", buf);
```

### SetUID와의 관계

- **SetUID**: 누가 실행하든 파일 소유자(root) 권한으로 실행되는 설정
- SetUID가 설정된 프로그램을 공격하면 → 셸코드가 **root 권한**으로 실행됨

### 셸코드란?

`/bin/sh`(쉘)을 실행하는 기계어 코드.  
쉘을 띄우면 시스템에서 명령어를 마음대로 실행할 수 있어서 해커의 최종 목표가 된다.

## 3. 힙 오버플로우

힙도 버퍼 오버플로우와 개념은 동일하다.  
할당받은 크기보다 더 많이 쓰면 옆 공간을 침범한다.

```
[데이터A][다음주소→] → [데이터B][다음주소→] → [데이터C]
```

힙에는 연결리스트 형태로 데이터가 저장되는데, 데이터A가 오버플로우 나면 옆의 포인터(다음주소)를 덮어써서 공격에 활용할 수 있다.

| 구분 | 스택 오버플로우 | 힙 오버플로우 |
|------|----------------|--------------|
| 대상 | 반환주소 덮어씀 | 연결리스트 포인터 덮어씀 |
| 성장방향 | 높은→낮은 주소 | 낮은→높은 주소 |

## 4. 시스템 자원 고갈 공격 (DoS)

시스템이 가진 자원을 모두 소모시켜 정상 서비스를 불가능하게 만드는 공격.

### 4-1. 디스크 고갈 공격

```c
#include <unistd.h>
#include <sys/file.h>
void main() {
    int fd;
    char buf[1000];
    fd = creat("/root/tempfile", 0777);
    while(1) write(fd, buf, sizeof(buf));  // 1000바이트씩 무한 저장
}
```

- **원리**: 파일에 1000바이트씩 계속 써서 디스크를 꽉 채움
- **결과**: 디스크 가득 참 → 시스템 파일 저장 불가
- **대책**: `quota` 설정 (사용자별 디스크 사용량 한도 설정)

### 4-2. 메모리 고갈 공격

```c
#include <stdio.h>
void main() {
    char *m;
    while(1)
        m = malloc(1000);  // free() 없이 계속 할당만
}
```

- **원리**: `malloc()`으로 메모리를 계속 요청하되 `free()`로 반납하지 않음
- **결과**: RAM 전체 소모 → 시스템 마비 (터미널 창 하나 여는 데 몇 분)
- **핵심**: `malloc` = 메모리 요청, `free` = 반납. 반납 없이 계속 요청하면 고갈

### 4-3. 프로세스 고갈 공격 (포크 폭탄)

```c
#include <unistd.h>
void main() {
    while(1)
        fork();  // 프로세스를 무한 복제
}
```

- **fork()**: 현재 프로세스를 복사해 자식 프로세스를 생성하는 함수
- **원리**: 1 → 2 → 4 → 8 → 16... 기하급수적으로 늘어남
- **결과**: 가용 프로세스 소진 → OS 재설치 필요할 수도 있음

### 4-4. 프로세스 강제 종료 공격

```bash
#!/bin/sh
sync
kill -15 1  # PID 1 이상 모든 프로세스에 종료 신호
```

- **조건**: root 권한 필요
- **결과**: 실행 중인 모든 프로세스 강제 종료

## 5. 백도어와 탐지

### 백도어란?

해커가 시스템에 몰래 설치하는 **비밀 원격 접속 통로**.  
정상 프로세스 이름을 사칭해 탐지를 피한다.

### 주요 사칭 대상 프로세스

| 프로세스 | 정상 역할 |
|----------|-----------|
| `csrss.exe` | 윈도우 콘솔 관리 |
| `explorer.exe` | 파일탐색기, 바탕화면 |
| `lsass.exe` | 로그인 인증 |
| `svchost.exe` | DLL 기반 서비스 실행 (여러 개 존재해 특히 사칭 多) |
| `winlogon.exe` | 로그인 화면 |
| `services.exe` | 서비스 목록 관리 |

### 가짜 프로세스 탐지법

경로가 다르면 가짜다.

```
정상: C:\Windows\System32\svchost.exe
가짜: C:\Temp\svchost.exe  ← 악성
```

### H-IDS (호스트 기반 침입탐지시스템)

백도어 탐지를 위한 "어깨 형님" 역할. 시스템 옆에서 항상 감시한다.

| 방법 | 내용 |
|------|------|
| 열린 포트 확인 | 몰래 열린 포트 감시 |
| 무결성 검사 | 시스템 파일 변조 여부 확인 |
| 로그 분석 | 수상한 접속 기록 탐지 |

## 6. 리버스 엔지니어링과 난독화

### 리버스 엔지니어링

완성된 프로그램을 분해해서 내부 구조/취약점을 파악하는 기술.

```
완성된 프로그램 → 분석 → 취약점 발견 → 공격 코드 제작
```

> 비유: 경쟁사 라면을 먹어보고 레시피를 역으로 알아내는 것

### 난독화 (Obfuscation)

리버스 엔지니어링을 어렵게 만들기 위해 **코드를 읽기 어렵게** 만드는 기술.

```c
// 원래 코드
if (password == "1234")

// 난독화 후
if (x9a2 == "\x31\x32\x33\x34")
```

| 종류 | 설명 |
|------|------|
| 소스코드 난독화 | 개발자가 작성한 코드를 꼬아놓음 |
| 바이너리 난독화 | 컴파일된 실행파일을 꼬아놓음 |

> 비유: 레시피를 훔쳐봐도 해석 못하게 암호로 적어두는 것

## 7. 루트킷 (Rootkit)

### 개념

시스템에 설치되어 **존재를 숨기면서** 공격자가 언제든 **관리자 권한으로 접근**할 수 있도록 비밀통로를 유지하는 프로그램 집합.

> 비유: 건물 경비원을 부하로 교체하고 마스터키를 복사해 언제든 드나드는 것

### 특징

- 탐지가 매우 어려움 (자기 존재를 숨김)
- `ls`, `ps`, `netstat` 등 **정상 명령어를 가짜로 교체**
- "수상한 파일 있어?" → 가짜 명령어가 "없어요~" 라고 거짓 응답

### 루트킷 탐지가 어려운 이유

정상 프로그램 자체를 변조된 버전으로 교체해버리기 때문에, 시스템 내부 명령어로는 탐지 불가. 외부 신뢰할 수 있는 도구(부팅 CD 등)로 검사해야 한다.

## 8. 논리폭탄 (Logic Bomb)

### 개념

특정 조건(트리거)이 만족되면 악성 행동(페이로드)을 실행하는 코드.

| 구성요소 | 설명 | 비유 |
|----------|------|------|
| **트리거** | 폭탄이 터지는 조건 | 타이머의 뇌관 |
| **페이로드** | 실제 악성 행동 | 폭탄 자체 |

### 예시

```
트리거:  "매년 3월 1일 자정이 되면"
페이로드: "모든 파일을 삭제해"
```

> 비유: 타이머 폭탄. 평소엔 조용히 있다가 특정 조건이 되는 순간 터진다.

### 트리거 종류

- 특정 날짜/시간
- 특정 파일 실행
- 특정 사용자 로그인
- 특정 파일 삭제 등

## 핵심 요약

| 공격 기법 | 핵심 |
|-----------|------|
| 스택 버퍼 오버플로우 | buf 넘쳐서 반환주소 변조 → 셸코드 실행 |
| 힙 오버플로우 | 동적 할당 공간 넘쳐서 포인터 변조 |
| 디스크 고갈 | 무한 쓰기로 디스크 꽉 채움 |
| 메모리 고갈 | malloc만 하고 free 안 함 |
| 프로세스 고갈 | fork() 무한 반복 (포크 폭탄) |
| 백도어 | 정상 프로세스 사칭, H-IDS로 탐지 |
| 리버스 엔지니어링 | 프로그램 분해 → 취약점 파악, 난독화로 대응 |
| 루트킷 | 시스템 명령어 변조, 존재 숨김 |
| 논리폭탄 | 트리거 조건 충족 시 페이로드 실행 |
