## 1. 메일 서버 구조

### 구성 요소

| 구성요소 | 비유 | 역할 |
|----------|------|------|
| **MUA** | 이메일 앱 (아웃룩, 썬더버드) | 메일 작성 / 읽기 |
| **MTA** | 우체국 | 메일 전달 (릴레이) |
| **MDA** | 우편함 관리자 | 메일함에 저장 |
| **MRA** | 우편 배달원 | 사용자에게 메일 꺼내줌 |

### 메일 전송 흐름

```
[보내는 사람]
     │
    MUA (아웃룩 등)
     │ SMTP
     ▼
    MTA (내 메일 서버)  ← DNS MX 조회로 목적지 찾음
     │ SMTP
     ▼
    MTA (상대 메일 서버)
     │ MDA
     ▼
    메일함 (Mail Box)
     │ POP3 / IMAP
     ▼
    MRA
     │
    MUA (상대방 앱)
```

> **DNS MX 레코드**: `user@gmail.com` 으로 메일 보낼 때, 서버가 DNS에 물어봄  
> → `"gmail.com 메일 어디로 보내?"` → `"gmail-smtp.google.com 으로 보내!"`

## 2. SMTP

### 개요

- **Simple Mail Transfer Protocol**
- MTA 클라이언트 ↔ 서버 간 메일 전송 프로토콜
- **두 곳**에서 사용됨
  1. 사용자 → 메일 서버 (MUA → MTA)
  2. 메일 서버 → 메일 서버 (MTA → MTA)

### 특징

- 기본적으로 **암호화 없음** (평문 전송 → 엽서처럼 누구나 볼 수 있음)
- DNS MX 레코드로 수신 서버 탐색
- **ESMTP**로 확장 가능

### SMTP → ESMTP 진화

```
SMTP  →(자동변환)→  ESMTP  →  PEM / PGP / S/MIME
(암호화 없음)       (인증, 기밀성 추가)
```

**ESMTP 추가 기능**
- 배달 확인 (Delivery Status Notification)
- 메시지 크기 제한
- 전송 중단 시 재전송
- TLS 암호화 지원

### 주요 명령어

| 명령어 | 의미 |
|--------|------|
| `HELO` | 세션 초기화 (기존 SMTP) |
| `EHLO` | 세션 초기화 (ESMTP, 확장 기능 지원 선언) |
| `MAIL FROM` | 보내는 사람 주소 |
| `RCPT TO` | 받는 사람 주소 |
| `VRFY` | 편지함 사용 가능 여부 확인 |
| `NOOP` | 통신 상태 확인 (아무 역할 없음) |
| `DATA` | 메일 내용 전송 |
| `QUIT` | 세션 종료 |

> 응답은 **3자리 숫자 코드** (HTTP 응답 코드와 유사한 구조)

## 3. POP3 vs IMAP

### POP3

- 서버에서 메일을 **다운로드 후 가져감** (우체통에서 편지 꺼내 집에 가져가는 느낌)
- 서버에 메일이 남지 않을 수도 있음
- 여러 기기 동기화 불편

| 항목 | 값 |
|------|----|
| 포트 | **110** |
| 암호화 버전 (POP3S) | **995** (over SSL/TLS) |

### IMAP4

- 메일을 **서버에 둔 채 동기화** (여러 기기가 같은 보관함 공유)
- 폴더 관리, 부분 다운로드, 헤더 검색 가능

| 항목 | 값 |
|------|----|
| 포트 | **143** |
| 암호화 버전 (IMAPS) | **993** (over SSL/TLS) |
| 참고 | IMAP3 = 220번 포트 |

### 비교 그림

```
POP3                        IMAP
 메일 서버                   메일 서버
    │                       ├─ PC
    ▼                       ├─ 스마트폰
   PC로 다운로드              └─ 태블릿
 (서버에서 없어질 수도)         전부 동기화
```

## 4. PGP

### 개요

- **Phil Zimmermann**이 개발한 이메일 / 파일 암호화 시스템
- 다양한 플랫폼에서 활용 가능한 소프트웨어

### PGP가 제공하는 5가지 서비스 ⭐

| 서비스 | 알고리즘 | 설명 |
|--------|----------|------|
| **기밀성** | CAST-128, IDEA, 3DES + RSA | 세션키로 메시지 암호화, 수신자 공개키로 세션키 암호화 |
| **인증 (전자서명)** | RSA/DSS + SHA-1/MD5 | 해시(다이제스트) 생성 후 송신자 개인키로 암호화 → 부가형 전자서명 |
| **압축** | ZIP | 서명 후, 암호화 전에 압축 |
| **전자메일 호환성** | Radix-64 (Base64) | 바이너리 데이터를 ASCII 문자로 변환, CRC 오류 검출 |
| **단편화** | - | Radix-64 변환 이후 큰 메일을 쪼갬 (S/MIME은 미제공) |

> **전자서명 효과**: 무결성 + 인증 + 송신부인방지

### PGP 동작 흐름 (암호화 + 서명)

```
[송신자]
메시지 작성
   │ 해시 (SHA-1 등)
   ▼
메시지 다이제스트
   │ 송신자 개인키(PR_a)로 RSA/DSS 암호화
   ▼
전자서명 생성
   │
메시지 + 서명 → ZIP 압축
   │
세션키(K_s) 생성 (일회용 대칭키)
   │ K_s로 대칭키 암호화 (CAST-128/IDEA/3DES)
   ▼
암호화된 메시지
   │
K_s를 수신자 공개키(PU_b)로 RSA 암호화
   │
Radix-64 변환
   ▼
[암호화된 세션키] + [암호화된 메시지] 전송
```

```
[수신자]
수신자 개인키(PR_b)로 세션키 복호화
   │
세션키로 메시지 복호화
   │
ZIP 압축 해제
   │
송신자 공개키(PU_a)로 전자서명 복호화 → 다이제스트 A
   │
메시지 다시 해시 → 다이제스트 B
   │
A == B? → 검증 완료
```

> **왜 하이브리드?** 공개키 암호는 느림 → 실제 내용은 빠른 대칭키로, 대칭키만 공개키로 보호

### PGP 키링 (Key Ring)

```
내 컴퓨터
├─ 개인 키링 (내 것)
│   ├─ 내 공개키
│   └─ 내 개인키
│
└─ 공개 키링 (남들 것)
    ├─ 철수 공개키
    ├─ 영희 공개키
    └─ 민수 공개키
```

### PGP 인증서 & Web of Trust

- **CA 불필요** → **Web of Trust (신뢰망)** 사용
- 링에 속한 누구든 인증서에 서명 가능
- 계층 구조 없음 (트리 없음)

```
[PKI 방식]          [PGP 방식]
     CA              철수 ↔ 영희 ↔ 민수
   / | \               ↘        ↗
사용자들                  지훈
중앙 집중                분산형
```

| 구분 | PGP | PKI (X.509) |
|------|-----|-------------|
| 중앙기관 | 없음 | 있음 (CA) |
| 신뢰 방식 | Web of Trust | CA 기반 |
| 인증서 서명 | 사용자끼리 | CA |
| 구조 | **분산형** | **중앙집중형** |

### 부가형 전자서명이란?

원본 메시지에 전자서명을 **붙여서** 함께 전송하는 방식

```
전송 데이터 = [원본 메시지] || [전자서명]
```

수신자 검증:
1. 메시지 다시 해시 → 값 A
2. 전자서명을 송신자 공개키로 복호화 → 값 B
3. A == B 이면 → 위변조 없음 + 진짜 송신자 확인

## 5. MIME & S/MIME

### MIME 등장 배경

- 기존 SMTP는 **ASCII 텍스트만** 전송 가능
- 사진, 동영상, 한글, PDF 등 전송 불가
- MIME: 텍스트 아닌 데이터를 **ASCII 형태로 변환해서 전송**

### MIME 변환 흐름

```
[cat.jpg 원본]
   │ MIME (Base64) 변환
   ▼
"QWxhZGRpbg==..."  (ASCII 문자열)
   │ SMTP 전송
   ▼
인터넷
   │ MIME 복원
   ▼
[cat.jpg 복원]
```

### MIME 헤더

| 헤더 | 의미 |
|------|------|
| `MIME-Version` | MIME 버전 |
| `Content-Type` | 데이터 종류 |
| `Content-Transfer-Encoding` | 변환 방식 (Base64 등) |
| `Content-ID` | 데이터 식별자 |
| `Content-Description` | 설명 |

**Content-Type 주요 유형 (7가지)**

`Text` / `Image` / `Audio` / `Video` / `Application` / `Multipart` / `Message`

- **Multipart**: 여러 종류 데이터 혼합 (본문 + 사진 첨부 등)
- **Message**: 메시지 안에 다른 메시지 포함

### Base64 (Radix-64)

- 바이너리 데이터를 ASCII 문자로 변환하는 **형식 변환** 도구
- ⚠️ **암호화가 아님** → 누구나 복원 가능, 패스워드 보호 불가

```
바이너리: 101010101110
   │ Base64 변환
   ▼
"QWxhZGRpbg=="
```

### PEM / PGP / S/MIME 비교

| 기술 | 개발자 | 특징 |
|------|--------|------|
| **PEM** | IETF | 중앙집중형 키 인증, 이론 중심, 복잡해서 잘 안씀 |
| **PGP** | Phil Zimmermann | 분산형 (Web of Trust), 실사용 많음 |
| **S/MIME** | RSA Data Security | MIME 기반, X.509 인증서, 기업 메일에 많이 쓰임 |

### S/MIME

- **MIME + 암호화 + 전자서명** = S/MIME
- PEM의 복잡성, PGP의 보안성 한계를 보완
- **X.509 인증서** 사용 → CA 필요 (중앙형)

**제공하는 보안 서비스** ⭐

| 서비스 | 내용 |
|--------|------|
| 기밀성 | 메시지 내용 암호화 |
| 무결성 | 변조 여부 확인 |
| 사용자 인증 | 전자서명으로 신원 확인 |
| 송신부인방지 | 발뺌 불가 |

**전자서명**: DSS (서명 전용) 사용  
**세션키 암호화**: Diffie-Hellman 사용

### S/MIME 동작 흐름

```
메일 작성
   │ 선택: 전자서명 / 암호화 / 둘 다
   ▼
S/MIME 처리
   │ Base64 변환
   ▼
SMTP 전송  ← 중간에서 봐도 이미 암호화됨
   ▼
수신자: 복호화 → 서명 검증
```

### S/MIME 콘텐츠 유형

#### 1. 동봉된 데이터 (기밀성)

```
메시지
   │ 세션키(K_s)로 대칭키 암호화
   ▼
암호문
   │ K_s를 수신자 공개키로 암호화
   ▼
[암호화된 메시지] + [암호화된 K_s] 전송
```

> **RecipientInfo 블록**: 수신자 공개키 인증서 식별자 + 알고리즘 식별자 + 암호화된 세션키

#### 2. 서명된 데이터 (무결성 + 인증)

```
메시지
   │ 해시 (SHA-256 등)
   ▼
다이제스트
   │ 송신자 개인키로 RSA 암호화
   ▼
전자서명
   │ Base64 부호화
   ▼
[메시지 + 전자서명] 전송
```

> S/MIME 기능 있는 수신자만 서명 검증 가능

#### 3. 명문-서명 데이터

| 항목 | 처리 |
|------|------|
| 메시지 내용 | 평문 (Base64 없음) |
| 전자서명 | Base64 변환 O |

→ S/MIME 없는 수신자도 **내용은 읽을 수 있음**, 서명 검증은 불가

#### 4. 서명 + 암호화 (가장 안전)

```
메시지 → 전자서명 → 암호화
```

### S/MIME 인증서 클래스 (VeriSign)

| 클래스 | 인증 방법 |
|--------|----------|
| 클래스 1 | 이메일 주소로 본인 확인 |
| 클래스 2 | 제3자 데이터베이스 확인 |
| 클래스 3 | 대면 인증 + 신분증 확인 (가장 엄격) |

## 6. 스팸 메일 대응 기술

### SPF (Sender Policy Framework)

**메일 서버 등록제** — 발신 서버 IP를 DNS에 미리 등록해두고 일치 여부 확인

```
[메일 수신]
   │
수신 서버가 DNS 조회
"이 IP 진짜 네이버 메일 서버 맞아?"
   │
맞으면 통과 / 아니면 차단
```

### DKIM (DomainKeys Identified Mail)

**도메인 키 인증 메일** — 메일 헤더 + 바디에 전자서명 삽입, 위변조 여부 검증

```
[발신 서버]
메일 내용 + 헤더
   │ 전자서명 생성
   ▼
헤더에 서명 첨부
   │ SMTP 전송
   ▼
[수신 서버]
DNS에서 공개키 조회 → 서명 검증
```

> ⚠️ DKIM은 MUA가 아닌 **서버 레벨**에 구현 → 사용자에게 투명  
> S/MIME은 메시지 내용만 서명, DKIM은 **헤더 + 바디** 모두 서명

### SPF vs DKIM

| 기술 | 확인 대상 |
|------|----------|
| SPF | 보낸 서버 IP |
| DKIM | 메일 내용 위변조 |

### DMARC

**SPF + DKIM을 통합 관리**하는 이메일 인증 정책

```
메일 수신
   ├─ SPF 검사
   └─ DKIM 검사
        │
   정책 판단 (통과 / 차단)
        │
   DMARC 보고서 → 발신 측에 전달 (모니터링 가능)
```

### 스팸 필터 솔루션

메일 서버 **앞단**에서 프록시 메일 서버로 동작

**필터링 종류**

| 필터 | 내용 |
|------|------|
| 메일 헤더 필터링 | 헤더 정보 분석 |
| 제목 필터링 | 제목 키워드 차단 |
| 본문 필터링 | 특정 단어/패턴 탐지 (ex. Bayesian filtering) |
| 첨부파일 필터링 | `.exe`, `.com`, `.dll`, `.bat` 등 실행파일 차단 |
| 송신자 필터링 | IP/이메일 주소 블랙리스트 관리 |

## 7. Sendmail

Unix 환경의 **MTA 소프트웨어**

### 주요 파일 / 디렉터리 ⭐

| 경로 | 역할 |
|------|------|
| `/var/spool/mqueue` | sendmail 큐 디렉터리 (수신 메일 임시 저장) |
| `/etc/mail/access` | 스팸메일 및 릴레이 차단 설정 파일 |
| `/etc/mail/sendmail.cf` | sendmail 전체 설정 파일 |

> `access` 파일은 텍스트 파일이며, 실제 적용을 위해 `makemap` 명령으로 `access.db` 파일 생성 필요

### `/etc/mail/access` 제어 옵션

| 옵션 | 의미 |
|------|------|
| `RELAY` | 지정 대상의 메일 릴레이 허용 |
| `REJECT` | 지정 대상의 메일 모두 거부 |
| `DISCARD` | 지정 대상의 메일 조용히 폐기 |
| `OK` | 조건 없이 모두 허용 |
| `501 "메시지"` | 지정 주소 메일 거부 + 거부 메시지 회신 |

**예시**

```
sulinux.net              RELAY
linux.co.kr              RELAY
211.223.191.123          REJECT
do@hyung.co.kr           501 You send email to Spammer. Delete Email Address in your list
```

> sendmail 기본 설정: 로컬 호스트 외 릴레이는 허용하지 않음 → 필요한 도메인/IP는 RELAY 명시 필요

## 8. 핵심 비교표 정리

### 이메일 프로토콜

| 프로토콜 | 역할 | 포트 |
|----------|------|------|
| SMTP | 메일 전송 | 25 (SMTPS: 465) |
| POP3 | 메일 다운로드 | 110 (POP3S: 995) |
| IMAP4 | 메일 동기화 | 143 (IMAPS: 993) |

### 이메일 보안

| 기술 | 신뢰 방식 | 암호화 | 특징 |
|------|----------|--------|------|
| PGP | Web of Trust (분산형) | 하이브리드 | ZIP 압축, Base64, 단편화 제공 |
| S/MIME | CA / X.509 (중앙형) | 하이브리드 | 기업 표준, 단편화 미제공 |
| PEM | CA (중앙형) | - | 이론 중심, 잘 안씀 |

### 스팸 대응

| 기술 | 확인 대상 | 방법 |
|------|----------|------|
| SPF | 발신 서버 IP | DNS에 허용 IP 등록 |
| DKIM | 메일 위변조 여부 | 헤더+바디 전자서명 |
| DMARC | SPF + DKIM 통합 | 정책 관리 + 보고서 |

### 전자서명 vs 암호화 핵심 차이

| 구분 | 암호화 (기밀성) | 전자서명 (인증) |
|------|----------------|----------------|
| 목적 | 내용 숨기기 | 신원 확인 |
| 사용하는 키 | 수신자 공개키 | 송신자 개인키 |
| 복호화 키 | 수신자 개인키 | 송신자 공개키 |

---

> 💡 **전체 흐름 한줄 정리**  
> `SMTP`(메일 전송) → `MIME`(첨부파일 가능) → `S/MIME`(보안 추가) → `SPF/DKIM/DMARC`(위조메일 방지)
