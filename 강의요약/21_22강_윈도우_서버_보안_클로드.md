## 1. 윈도우 링 구조 & 이중 연산 모드

### 링 구조란?

권한 수준을 나눈 보안 구조. 번호가 낮을수록 권한이 강하다.

```
Ring 3  응용 프로그램      ← 사용자 모드
─────────────────────────── (경계선)
Ring 2  각종 관리자        ← 커널 모드
Ring 1  마이크로 커널      ← 커널 모드
Ring 0  HAL               ← 커널 모드
        하드웨어
```

| 링 | 이름 | 역할 | 모드 |
|---|---|---|---|
| Ring 3 | 응용 프로그램 | 브라우저, 게임 등 일반 앱 | 사용자 모드 |
| Ring 2 | 각종 관리자 | I/O·메모리·프로세스·보안 관리자 | 커널 모드 |
| Ring 1 | 마이크로 커널 | 스케줄링, 인터럽트 처리, 동기화 | 커널 모드 |
| Ring 0 | HAL | 하드웨어 ↔ 소프트웨어 번역기 | 커널 모드 |

---

### HAL (Hardware Abstraction Layer)

> 💡 **비유**: 한국 콘센트 표준(220V)처럼, 어떤 제조사 하드웨어든 HAL 표준만 맞추면 윈도우에서 동작한다.

- 하드웨어와 소프트웨어 사이의 **번역기** 역할
- 윈도우가 제시한 **표준 인터페이스**
- 덕분에 앱 개발자는 하드웨어 종류를 몰라도 됨 → **추상화**

---

### 이중 연산 모드 (Dual Mode Operation)

| 모드 | 범위 | 특징 |
|---|---|---|
| **사용자 모드** | Ring 3 | 하드웨어 직접 접근 불가, 시스템 콜로 커널에 요청 |
| **커널 모드** | Ring 0~2 | 하드웨어 직접 제어 가능, 오류 시 블루스크린(BSOD) |

> 💡 **왜 나누냐?** 앱이 하드웨어를 직접 건드리는 걸 막아 OS가 통제권을 유지하기 위해

#### 오류 발생 시 차이점

- **사용자 모드 오류** → 해당 앱만 종료 (다른 앱은 정상)
- **커널 모드 오류** → 메모리 덤프 후 시스템 전체 재시작 (블루스크린)

> 💡 **메모리 덤프** = 오류 직전 메모리 상태를 파일로 저장해두는 것 (원인 분석용)

## 2. 파일 시스템

### FAT16 / FAT32 / NTFS 비교

| | FAT16 | FAT32 | NTFS |
|---|---|---|---|
| 테이블 크기 | 16비트 | 32비트 | - |
| 최대 파티션 | **2GB** | 대용량 OK | 대용량 OK |
| 호환성 | 낮음 | **높음** (윈도우·맥·리눅스) | 윈도우 위주 |
| 접근제어 | ❌ | ❌ | ✅ |
| 암호화 | ❌ | ❌ | ✅ |
| 자동 압축 | ❌ | ❌ | ✅ |
| 결함 관리 | ❌ | ❌ | ✅ |
| 보안성 | 낮음 | **낮음** | **높음** |

#### FAT 장단점

- **장점**: 호환성 우수, 저용량 볼륨 최적화, 구조 단순
- **단점**: 보안 취약 (접근제어·암호화 불가)

#### NTFS 장단점

- **장점**: 대용량 지원, 강력한 보안, 자동 압축, 결함 관리
- **단점**: Windows NT 계열 외 호환 불가, 저용량에서 FAT보다 느림, 구조 복잡

#### NTFS 암호화 특징

- 폴더 암호화 시 → 폴더 내 **모든 파일 자동 암호화**
- 암호화된 파일을 복사/이동해도 → **암호화 속성 유지**

#### FAT → NTFS 변환

```bash
CONVERT C: /FS:NTFS
```

> ⚠️ 단방향만 가능 (FAT → NTFS O, NTFS → FAT X)

---

### MBR / VBR / MFT

#### MBR (Master Boot Record)

> 💡 **비유**: 건물 입구 안내판 ("1층 소설, 2층 과학...")

- 물리적 디스크의 **첫 번째 섹터**에 위치
- 크기: **512 byte**
- 파티션 정보 + 부트 코드 저장
- 부팅 시 제일 먼저 읽혀서 어느 파티션으로 부팅할지 결정

#### VBR (Volume Boot Record)

> 💡 **비유**: 각 층 엘리베이터 앞 안내판

- **파티션마다 하나씩** 존재
- 해당 파티션의 OS를 실제로 기동시키는 역할
- **마지막 섹터에 백업본** 보유 → 손상 시 복구 가능

#### MFT (Master File Table)

> 💡 **비유**: 도서관 도서 목록 DB ("홍길동전 → 3층 52번 책장")

- **NTFS 전용**, 가장 중요한 영역
- 볼륨의 모든 파일·디렉터리 정보 저장
- 저장 정보: 파일 이름, 위치, 크기, 생성·수정 날짜

#### 디스크 파괴형 악성코드

| 공격 대상 | 피해 | 복구 방법 |
|---|---|---|
| **MBR 파괴** | 부팅 자체 불가 | 부트 코드 복사 + 파티션 테이블 재작성 |
| **VBR 파괴** | OS 기동 불가 | 백업본으로 덮어쓰기 |

## 3. 부팅 순서 (XP / 2000 / 2003)

```
1단계  POST 실행
2단계  기본 부팅 설정 로드
3단계  MBR 로드
4단계  NTLDR 실행
5단계  NTDETECT.COM 실행
6단계  NTOSKRNL.EXE 실행
```

| 단계 | 이름 | 역할 | 비유 |
|---|---|---|---|
| 1 | **POST** | 하드웨어 자가진단 (CPU·RAM·키보드 등) | 출근 전 몸 상태 체크 |
| 2 | **기본 설정 로드** | BIOS가 부팅 장치 설정 읽어옴 | 출근 준비 |
| 3 | **MBR 로드** | 어느 파티션으로 부팅할지 결정 | 건물 입구 안내판 확인 |
| 4 | **NTLDR** | boot.ini 읽어 부팅 옵션 표시 | 사무실 열고 할 일 목록 확인 |
| 5 | **NTDETECT.COM** | 하드웨어 검사 및 목록 작성 | 사무실 장비 점검 |
| 6 | **NTOSKRNL.EXE** | 커널 실행 + HAL.DLL 로드 | 실제 업무 시작 |

> 💡 **NTLDR** = NT Loader, 부팅 관리자 역할  
> 💡 **NTOSKRNL** = NT OS Kernel, 윈도우 핵심 커널

## 4. 계정 & 권한 체계

### OS별 최고 관리자 계정

| OS | 계정 이름 | 번호 |
|---|---|---|
| Unix/Linux | **root** | 0번 |
| Windows | **Administrator** | 500번 |
| Netware | **Supervisor** | - |

### 윈도우 기본 계정 3종류

| 계정 | 비유 | 특징 |
|---|---|---|
| **Administrator** | 건물 주인 | 모든 권한 |
| **System** | 건물 자체 관리 시스템 | OS 내부에서 사용, Administrator보다 높음 |
| **Guest** | 방문객 임시 출입증 | 제한된 권한, 기본 비활성화 |

---

### SID (Security Identifier)

윈도우가 계정을 구분하는 **고유 번호** (동명이인 구분용)

| SID | 의미 |
|---|---|
| S-1-0-0 | SID 모를 때 사용 |
| S-1-1-0 | Everyone (모든 사용자) |
| S-1-5-domain-**500** | Administrator |
| S-1-5-domain-**501** | Guest |
| S-1-5-domain-**1000~** | 일반 사용자 |

> 💡 500~999번은 윈도우 시스템 예약 번호 → 일반 사용자는 1000번부터 부여

---

### 윈도우 기본 그룹

| 그룹 | 역할 |
|---|---|
| **Administrators** | 모든 권한 (대표이사) |
| **Account Operators** | 계정 생성·삭제·관리 (인사팀) |
| **Backup Operators** | 백업·복구 목적 파일 접근 |
| **Guests** | 설정 변경 불가, 도메인 권한 제한 |
| **Print Operators** | 프린터 관리 |
| **Power Users** | 일반 유저보다 권한 조금 더 있음 |
| **Replicator** | 파일 복제·동기화 전용 |
| **Server Operators** | 서버 관리 (재시작, 공유 설정 등) |
| **Users** ⭐ | 일반 사용자 그룹 |

#### Users 그룹 상세

- ✅ 할 수 있는 것: 일반 작업, 개인 환경 설정
- ❌ 못 하는 것: 시스템 서비스 시작/종료, 디렉터리 공유 설정, 시스템 전체 설정 변경

> 💡 **시스템 서비스** = 방화벽, 원격 데스크톱, 윈도우 업데이트 같은 백그라운드 프로그램들

---

### Unix/Linux UID 체계

| 이름 | 의미 |
|---|---|
| **UID** | 사용자 식별 번호 (root=0) |
| **RUID** | 실제 로그인한 사람 (절대 안 바뀜) |
| **EUID** | 현재 실제로 적용되는 권한 (SetUID로 잠깐 바뀔 수 있음) |

> 💡 **예시**: 일반 사용자가 `passwd` 명령어 실행 시  
> RUID = 일반 사용자, EUID = root (비밀번호 파일 수정 위해 잠시 권한 상승)

---

### 프로세스 권한 탈취

- **방법**: Administrator/System 권한으로 실행 중인 프로세스에 **악성 코드 삽입**
- **결과**: 그 프로세스가 내 코드를 실행 → Administrator 권한으로 명령 실행됨
- **위험**: 방화벽 끄기, 시스템 파일 수정, 계정 생성 등 가능

## 5. LSA / SAM / SRM

### 역할 요약

| | 비유 | 역할 |
|---|---|---|
| **SAM** | 직원 명부 DB | 계정·비밀번호(해시) 저장 |
| **LSA** | 보안 담당자 | 로그인 확인 총괄 + 감사 메시지 **저장** |
| **SRM** | 출입문 경비원 | SID 부여 + 접근 허용/차단 + 감사 메시지 **생성** |

---

### 전체 인증 흐름

```
1. 사용자 아이디/비밀번호 입력
        ↓
2. LSA → SAM에 "이 계정 맞아?" 확인 요청
        ↓
3. SAM → 일치 확인 후 SRM에 알림
        ↓
4. SRM → 사용자에게 SID 부여
        ↓
5. 파일/폴더 접근 시도
        ↓
6. SRM → SID 기반으로 접근 허용/차단
        + 감사 메시지 생성
        ↓
7. LSA → 감사 메시지 저장
```

---

### SAM 파일 접근 통제

> SAM 파일 = 모든 계정 정보가 담긴 핵심 파일 → 해커의 주요 공격 대상

```
Administrators  ✅ 접근 가능
System          ✅ 접근 가능
일반 Users      ❌ 접근 불가
Guests          ❌ 접근 불가
```

> ⚠️ SAM 파일 탈취 시 비밀번호 해시값 획득 → 크래킹 공격으로 비밀번호 노출 위험

## 6. 네트워크 드라이브 & 파일·폴더 권한

### 네트워크 드라이브

> 💡 **비유**: 다른 컴퓨터(서버)의 폴더를 내 D드라이브처럼 쓰는 것

```bash
net use Z: \\192.168.0.1\공유폴더$
```

명령프롬프트에서 위 명령어 입력 후 계정·비밀번호 입력하면 Z드라이브로 연결됨.

---

### 파일·폴더 권한 6가지

| 권한 | 폴더 | 파일 | 할 수 있는 것 |
|---|:---:|:---:|---|
| **모든 권한** | ✅ | ✅ | 읽기·쓰기·수정·삭제·권한설정까지 전부 |
| **수정** | ✅ | ✅ | 읽기·쓰기·수정·삭제 (권한설정만 못함) |
| **읽기 및 실행** | ✅ | ✅ | 읽고 실행만 가능, 수정·삭제 불가 |
| **폴더 내용 보기** | ✅ | ❌ | 폴더 안 목록만 볼 수 있음 |
| **읽기** | ✅ | ✅ | 내용 읽기만 가능 |
| **쓰기** | ✅ | ✅ | 새 파일·폴더 생성 가능, 권한 확인 가능 |

> 💡 **폴더 내용 보기**가 파일엔 없는 이유 → 파일 안에 또 다른 파일이 없으니까!

권한 세기 순서:
```
모든 권한 > 수정 > 읽기 및 실행 > 폴더 내용 보기 > 읽기 > 쓰기
```

#### 헷갈리는 것 정리

- **쓰기 vs 수정**: 쓰기는 새로 만들기만 / 수정은 만들기 + 기존 파일 수정·삭제까지
- **읽기 vs 읽기 및 실행**: 읽기는 내용 보기만 / 읽기 및 실행은 프로그램 실행까지

---

### NTFS 접근 권한 3가지 규칙

#### 규칙 1 — 권한은 누적된다

> 💡 출입증이 여러 개면 권한도 합쳐짐

```
Users 그룹      → 읽기 권한
Developers 그룹 → 수정 권한
두 그룹 모두 속하면 → 읽기 + 수정 둘 다 가능
```

#### 규칙 2 — 파일 권한 > 디렉터리 권한

> 💡 건물 출입 금지여도 특정 방은 들어갈 수 있음

```
C:\비밀폴더\         → Users 그룹 접근 거부
C:\비밀폴더\공개.txt → Users 그룹 읽기 허용
→ 폴더엔 못 들어가지만 공개.txt는 읽을 수 있음
```

#### 규칙 3 — 거부가 허용보다 우선

> 💡 블랙리스트에 올라있으면 VIP여도 입장 금지

```
Users 그룹   → 읽기 허용
Blocked 그룹 → 읽기 거부
두 그룹 모두 속하면 → 읽기 불가! 거부가 이김
```

| 규칙 | 핵심 |
|---|---|
| **규칙 1** | 여러 권한 → 다 합쳐짐 |
| **규칙 2** | 파일 권한이 폴더 권한보다 강함 |
| **규칙 3** | 거부가 허용보다 무조건 우선 |

## 7. 공유 폴더 관리

### $ 기호 — 공유 폴더 숨기기

```
공유 이름: L    → 네트워크 목록에 보임 👀
공유 이름: L$   → 네트워크 목록에서 안 보임 🙈
```

> 단, 주소 직접 입력하면 접근 가능: `\\192.168.0.1\L$`

숨긴 공유 포함 전체 목록 확인:
```bash
net share
```

### 윈도우 기본 숨김 공유

| 공유 이름 | 의미 | 위험성 |
|---|---|---|
| **C$, D$** | 드라이브 전체 열쇠 | 드라이브 통째로 접근 가능 |
| **ADMIN$** | 윈도우 시스템 폴더 열쇠 | 핵심 시스템 파일 접근 가능 |
| **IPC$** | 컴퓨터끼리 대화하는 통로 | 널 세션 공격에 악용 가능 |

> ⚠️ C$, D$, ADMIN$ → **비활성화 권장**
> ⚠️ 단순 중지 후 재시작하면 OS가 자동으로 다시 만들어버림 → **레지스트리에서 자동 생성 막아야** 완전 비활성화

공유 제거 명령어:
```bash
net share 공유이름 /delete
```

---

### IPC$ & 널 세션 (Null Session)

**널 세션** = 아이디·비밀번호 없이 IPC$에 연결하는 것

```bash
net use \\192.168.0.1\IPC$ "" /user:""
```

널 세션으로 뽑아낼 수 있는 정보:
- 컴퓨터 이름
- 사용자 계정 목록 ← 해커가 엄청 좋아함
- 그룹 목록
- 공유 폴더 목록
- 패스워드 정책

> ⚠️ **IPC$는 제거 불가** → 제거 시 네트워크 서비스 일부 작동 안 함

차단 방법 — 레지스트리 수정:
```
RestrictAnonymous = 1  →  익명 사용자 정보 열람 제한
RestrictAnonymous = 2  →  널 세션 자체를 완전 차단
```

### EFS / BitLocker 암호화

| | **EFS** | **BitLocker** |
|---|---|---|
| 암호화 범위 | 파일·폴더 단위 | 드라이브 전체 |
| 도난 시 | 취약할 수 있음 | 강력하게 보호 |
| 사용 목적 | 특정 파일 보호 | 기기 분실·도난 대비 |

> 💡 **같이 쓰면 더 강력** — BitLocker로 드라이브 전체 잠그고, EFS로 중요 파일 추가 잠금!

## 8. 레지스트리

> 💡 **비유**: 윈도우의 모든 설정 정보가 담긴 창고

편집기: `regedit` / `regedt32` (XP 이후 사실상 동일)

> ⚠️ 잘못 건드리면 윈도우가 안 켜질 수 있으므로 **함부로 수정 금지!**

### 5대 루트 키

```
HKCR  → 파일 확장자 연결 정보 (.hwp → 한글로 열기)
HKCU  → 현재 로그인한 사용자 설정 (바탕화면, 즐겨찾기 등)
HKLM  → 컴퓨터 전체 설정 (가장 중요!)
 ├─ HARDWARE  → CPU, RAM 등 하드웨어 정보
 ├─ SAM       → 계정·비밀번호 정보 (민감!)
 ├─ SECURITY  → 보안 정책 설정 (민감!)
 ├─ SOFTWARE  → 설치된 프로그램 정보
 └─ SYSTEM    → 부팅·드라이버·서비스 설정
HKU   → 이 컴퓨터 모든 계정의 설정 전부
HKCC  → 현재 사용 중인 하드웨어 구성 정보
```

> ⚠️ SAM, SECURITY는 **SYSTEM 계정만** 접근 가능할 정도로 민감한 정보!

## 9. 윈도우 보안 설정

### 계정 잠금 정책

> 💡 **비유**: ATM 비밀번호 3회 오류 → 카드 잠김

Brute Force(자동 비밀번호 대입 공격)를 막기 위한 설정

| 설정 | 의미 |
|---|---|
| **잠금 임계값** | 몇 번 틀리면 잠글지 (예: 5회) |
| **잠금 기간 = 30분** | 30분 후 자동 잠금 해제 |
| **잠금 기간 = 0** | 관리자가 직접 풀어줄 때까지 잠금 유지 (더 안전) |

---

### 패스워드 복잡도 설정

```
❌ 취약한 비밀번호: 1234, abcd, 이름+생년월일
✅ 복잡도 만족: 영문 대소문자 + 숫자 + 특수문자 조합 (예: Abc@1234!)
```

복잡도 설정을 **"사용"** 으로 켜두면 조건 불만족 시 비밀번호 설정 자체가 안 됨.

---

### 공유 권한 설정

```
기본 설정 (위험!)
Everyone → 접근 가능 ❌

올바른 설정
Everyone  → 제거
필요한 계정만 → 접근 허용 ✅
```

> 안 쓰는 서비스 = 열려있는 창문 → 공격 경로가 됨
