## 1. ping

> **⭐ 매우 중요**

### 1-1. 개요

`ping`은 원격 호스트가 정상적으로 동작하는지 확인하는 **네트워크 진단 도구**다.

`ping`이 정상 응답을 반환하면 **L1(물리) ~ L3(네트워크) 계층까지 이상 없음**을 의미한다.

`ping`으로 할 수 있는 것들:
- 대상 호스트의 **생존 여부** 확인
- 네트워크 **속도(RTT)** 측정
- 도메인 이름으로 **IP 주소** 조회

### 1-2. 동작 원리

`ping`은 **ICMP(Internet Control Message Protocol)** 를 이용한다.

```
로컬호스트  ──── ICMP Echo Request (Type 8) ────▶  원격 호스트
로컬호스트  ◀─── ICMP Echo Reply   (Type 0) ────   원격 호스트
```

- **Echo Request (Type 8)**: "야, 살아있냐?"
- **Echo Reply (Type 0)**: "응, 살아있어"

### 1-3. ping 명령 실행 결과 분석

```
ping 192.168.0.7

Reply from 192.168.0.7: bytes=32 time=27ms TTL=128
Reply from 192.168.0.7: bytes=32 time=3ms  TTL=128
Reply from 192.168.0.7: bytes=32 time=99ms TTL=128

Packets: Sent = 4, Received = 4, Lost = 0 (0% loss)
Minimum = 3ms, Maximum = 99ms, Average = 27ms
```

| 항목 | 설명 |
|---|---|
| `bytes` | ICMP 패킷 크기 |
| `time` | Echo Request 전송 ~ Echo Reply 수신까지의 **왕복 시간(RTT)** |
| `TTL` | 패킷이 라우터를 지날 때마다 1씩 감소 |
| `Lost` | 손실된 패킷 수 및 손실률 |
| `Minimum / Maximum / Average` | 왕복 시간 통계 |

### 1-4. TTL (Time To Live)

TTL은 패킷이 네트워크를 **무한 순환하지 않도록** 막는 값이다.  
라우터 1개를 통과할 때마다 **1씩 감소**하고, 0이 되면 패킷을 폐기한다.

#### OS별 TTL 초기값

| OS | TTL 초기값 |
|---|---|
| Windows | **128** |
| Unix / Linux | **255** |

#### OS Fingerprinting

TTL 잔여값을 보고 OS를 추측할 수 있다.

- TTL ≈ 100대 → **Windows** 계열
- TTL ≈ 200대 → **Unix/Linux** 계열

> 예: TTL=122라면 128에서 6개의 라우터를 거쳤고, Windows 계열일 가능성이 높다.

### 1-5. ping 관련 보안 설정

서버에 따라 **방화벽에서 ping 응답을 차단**하도록 설정하는 경우가 있다.  
이는 해킹이나 서버 부하를 줄이기 위한 목적이다.

### 1-6. 스위핑 (Sweeping)

여러 IP에 자동으로 ping을 날려서 **살아있는 호스트를 탐색**하는 행위.

- 브로드캐스트 ping 또는 자동화 도구를 이용
- 대표 도구: **`fping`**

## 2. traceroute / tracert

> **⭐ 매우 중요**

### 2-1. 개요

`traceroute`는 출발지에서 목적지까지의 **경로(라우터)를 단계별로 추적**하는 도구다.

중간 경로 어디서 느려지는지, 어디서 패킷이 막히는지 파악할 수 있어 **네트워크 Troubleshooting**에 매우 유용하다.

### 2-2. 동작 원리

TTL을 **1씩 증가**시키며 패킷을 반복 전송한다.  
각 TTL 단계에서 라우터가 ICMP 응답을 돌려보내기 때문에, 경로를 하나씩 알 수 있다.

#### 단계별 흐름 예시

아래 구간을 추적한다고 가정한다.

```
내 PC → 라우터1 → 라우터2 → 라우터3 → 목적지 서버
```

```
TTL=1 전송 → 라우터1 도착 → TTL=0 → 라우터1이 ICMP Time Exceeded 응답
              결과: "첫 번째 경로 = 라우터1"

TTL=2 전송 → 라우터1(TTL=1) → 라우터2(TTL=0) → 라우터2가 ICMP Time Exceeded 응답
              결과: "두 번째 경로 = 라우터2"

TTL=3 전송 → 라우터1 → 라우터2 → 라우터3(TTL=0) → 라우터3이 ICMP Time Exceeded 응답
              결과: "세 번째 경로 = 라우터3"

TTL=4 전송 → 라우터1 → 라우터2 → 라우터3 → 목적지 도착!
              목적지가 ICMP Port Unreachable 응답
              결과: "목적지 도달 확인"
```

#### 목적지 도달 판단 방법

목적지 서버에는 **UDP 포트 33434** 로 패킷을 전송한다.  
이 포트는 **일부러 없는 포트**로, 목적지 서버가 아래와 같이 응답한다.

```
"그 포트 없는데?" → ICMP Destination Unreachable (Port Unreachable)
```

이 응답을 받으면 **목적지에 도달했다**고 판단한다.

> 우연한 도달을 방지하기 위해 각 TTL마다 **UDP 패킷을 정확히 3번** 전송한다.

### 2-3. ICMP 메시지 정리

| 상황 | ICMP 메시지 |
|---|---|
| 중간 라우터에서 TTL=0 | **Time Exceeded** |
| 목적지 서버 도달 (없는 포트) | **Port Unreachable** |

### 2-4. OS별 차이

| 구분 | 명령어 | 사용 프로토콜 |
|---|---|---|
| Linux / Unix | `traceroute` | **UDP 포트 33434** |
| Windows | `tracert` | **ICMP Echo Request / Reply** |

### 2-5. 실행 결과 분석

```
traceroute to 203.0.113.1

1  10.0.0.1    1ms   2ms   1ms
2  20.0.0.1    5ms   4ms   6ms
3  30.0.0.1   10ms   9ms  11ms
4  203.0.113.1 15ms  14ms  16ms
```

| 필드 | 설명 |
|---|---|
| 첫 번째 숫자 | TTL 값 (= 몇 번째 경로인지) |
| IP 주소 | 해당 구간의 라우터 IP |
| ms 3개 | 같은 구간을 3번 측정한 왕복 시간 |

> 기본 TTL: **30**, 패킷 크기: **40 Bytes**  
> 5초 이내에 응답 없으면 해당 왕복 시간은 `*`로 표시된다.

### 2-6. `* * *` 표시 의미

```
3   * * *
```

`*` 가 표시되는 경우:

1. **방화벽(ACL)** 이 traceroute의 UDP 패킷을 차단
2. 실제 해당 구간에 **장애 발생**

### 2-7. Troubleshooting 활용

- 지정한 주소가 실제 존재하는지 확인
- 응답이 느린 구간이 어딘지 파악
- 패킷이 적절한 경로로 라우팅되는지 확인
- `*` 표시 구간 식별

## 3. TCP 3-way Handshake

> **⭐ 약간 중요**

### 3-1. 개요

TCP는 데이터를 주고받기 전에 **연결을 먼저 확립**한다.  
이 과정이 3번의 메시지 교환으로 이루어지기 때문에 **3-way Handshake**라고 한다.

### 3-2. 연결 흐름

```
클라이언트 (Active Open)              서버 (Passive Open)
        |                                    |
        |──── SYN (Seq=x) ───────────────────▶|   "나 x부터 시작할게, 연결하자"
        |                                    |
        |◀─── SYN + ACK (Seq=y, Ack=x+1) ───|   "알겠어, 다음엔 x+1 줘. 나는 y부터 시작"
        |                                    |
        |──── ACK (Seq=x+1, Ack=y+1) ───────▶|   "확인. 다음엔 y+1 줘"
        |                                    |
                    ESTABLISHED ✅
```

### 3-3. Seq / Ack 번호 의미

**ACK 번호 = "다음에 받을 데이터 번호"** 즉, 받은 번호 + 1이다.

| 메시지 | Seq | Ack | 의미 |
|---|---|---|---|
| SYN | x | - | "나는 x부터 시작" |
| SYN+ACK | y | x+1 | "x 받았어, 다음은 x+1 줘" |
| ACK | x+1 | y+1 | "y 받았어, 다음은 y+1 줘" |

> **SYN도 번호를 1개 소비**한다.  
> 따라서 데이터가 없어도 항상 +1 처리된다.

#### 쉬운 비유

```
클라: "1번 상자 보냄"
서버: "1번 받았어, 다음은 2번 줘" → Ack = 2 (= 1+1)
```

### 3-4. 상태 변화

| 상태 | 대상 | 의미 |
|---|---|---|
| `SYN-SENT` | 클라이언트 | SYN 전송 후 응답 대기 |
| `SYN-RECEIVED` | 서버 | SYN 받고 ACK 대기 중 |
| `ESTABLISHED` | 양쪽 | 연결 완료 |

> **SYN Flood 공격**은 `SYN-RECEIVED` 상태를 대량으로 만들어  
> 서버의 연결 대기 큐(Backlog Queue)를 고갈시키는 공격이다.

## 4. TCP 4-way Handshake

> **⭐ 약간 중요**

### 4-1. 개요

TCP 연결을 **종료**할 때 거치는 4단계 과정이다.

3-way와 달리 **양측이 독립적으로 종료**를 선언하기 때문에 4번의 교환이 필요하다.

### 4-2. 종료 흐름

```
클라이언트                             서버
        |                                    |
        |──── FIN (Seq=x) ───────────────────▶|   "나 다 보냈어, 끊을게"
        |                                    |
        |◀─── ACK (Ack=x+1) ─────────────────|   "알겠어"     → 서버: CLOSE-WAIT
        |                                    |
        |◀─── FIN (Seq=y) ───────────────────|   "나도 끊을게"
        |                                    |
        |──── ACK (Ack=y+1) ─────────────────▶|   "확인"       → 클라: TIME-WAIT
        |                                    |
                      CLOSED ✅
```

### 4-3. 왜 3-way가 아니라 4-way인가?

| 구분 | 이유 |
|---|---|
| **연결(3-way)** | 서버가 SYN+ACK를 **동시에** 보낼 수 있음 |
| **종료(4-way)** | 서버가 ACK 보낸 후 **남은 데이터 처리**가 필요할 수 있어 FIN을 따로 보냄 |

### 4-4. Seq / Ack 번호

FIN도 SYN과 마찬가지로 **번호를 1개 소비**한다.

| 메시지 | Seq | Ack | 의미 |
|---|---|---|---|
| FIN | x | - | "나 끊을게" |
| ACK | - | x+1 | "FIN 받았어" |
| FIN | y | - | "나도 끊을게" |
| ACK | - | y+1 | "FIN 받았어, 확인" |

### 4-5. 상태 변화

| 상태 | 대상 | 의미 |
|---|---|---|
| `CLOSE-WAIT` | 서버 | 클라 FIN 받고 자신의 데이터 전송 마무리 중 |
| `FIN-WAIT` | 클라이언트 | FIN 전송 후 상대 FIN 대기 |
| `TIME-WAIT` | 클라이언트 | 마지막 ACK 전송 후 **2MSL** 동안 대기 |
| `LAST-ACK` | 서버 | 자신의 FIN에 대한 ACK 대기 |
| `CLOSED` | 양쪽 | 완전히 종료 |

#### TIME-WAIT란?

마지막 ACK를 보낸 후 **바로 종료하지 않고 일정 시간 대기**하는 상태다.

이유: 마지막 ACK가 유실됐을 경우 서버가 FIN을 재전송할 수 있는데,  
클라이언트가 이미 종료된 상태라면 재전송된 FIN을 처리할 수 없기 때문이다.

> 대기 시간: **2MSL (Maximum Segment Lifetime × 2)**  
> MSL = 패킷이 네트워크에서 살아있을 수 있는 최대 시간

## 5. netstat

> **⭐ 매우 중요**

### 5-1. 개요

`netstat`은 시스템의 **네트워크 연결 상태, 라우팅 테이블, 소켓 정보**를 확인하는 진단 도구다.

### 5-2. 보안 활용

- 현재 열려 있는 **포트 목록** 확인
- **외부에서 접속한 호스트** 파악
- 비정상적인 연결 감지 (악성코드, 백도어 탐지)

### 5-3. 주요 옵션

| 옵션 | 설명 |
|---|---|
| `-i [interval]` | 네트워크 인터페이스 정보 출력 |
| `-s` | 프로토콜별 통계 (IP, TCP, UDP, ICMP) |
| `-a` | 모든 소켓 정보 출력 (LISTEN 포함) |
| `-n` | 주소를 숫자(IP)로 표시 |
| `-r` | 라우팅 테이블 출력 |

> `-s` 옵션은 **3~4계층 프로토콜(IP, TCP, UDP, ICMP)** 통계만 출력한다.  
> MAC 주소(2계층)는 포함되지 않는다.

### 5-4. 결과 필드 설명

| 필드 | 설명 |
|---|---|
| `Proto` | 사용 프로토콜 (tcp, udp 등) |
| `Recv-Q` | 수신 대기 중인 데이터 크기 |
| `Send-Q` | 전송 대기 중인 데이터 크기 |
| `Local Address` | 로컬 IP:포트 |
| `Foreign Address` | 연결된 원격 IP:포트 |
| `State` | 소켓 상태 |

### 5-5. State 값 전체 정리

| State | 의미 |
|---|---|
| `LISTEN` | 서버가 연결 요청 대기 중 |
| `SYN-SENT` | 클라이언트가 SYN 전송, 응답 대기 |
| `SYN-RECV` | 서버가 SYN 받고 ACK 대기 (SYN Flood 관련) |
| `ESTABLISHED` | 연결 완료, 데이터 전송 중 |
| `CLOSE-WAIT` | 상대 FIN 받고 자신의 종료 준비 중 |
| `FIN-WAIT-1` | FIN 전송 후 ACK 대기 |
| `FIN-WAIT-2` | ACK 받고 상대 FIN 대기 |
| `TIME-WAIT` | 마지막 ACK 전송 후 2MSL 대기 |
| `LAST-ACK` | 자신의 FIN에 대한 마지막 ACK 대기 |
| `CLOSED` | 완전히 종료된 상태 |

## 6. tcpdump

> **⭐ 약간 중요**

### 6-1. 개요

`tcpdump`는 네트워크 인터페이스를 통과하는 **패킷을 캡처하고 출력**하는 도구다.

스니핑(Sniffing) 도구의 일종으로, 자신의 컴퓨터로 들어오는 **모든 패킷 내용을 도청**할 수 있다.

| 운영체제 | 도구 |
|---|---|
| Unix / Linux | `tcpdump` |
| Windows | `WinDump` |

### 6-2. 활용 목적

- **공격 유형 분석**: 어떤 패킷이 들어왔는지 확인
- **공격자 추적**: 비정상적인 트래픽 발원지 파악
- **네트워크 트러블슈팅**: 패킷 손실, 지연 구간 분석

### 6-3. 보안 주의

`ifconfig` 결과에서 **`PROMISC`** 플래그가 보인다면,  
해당 인터페이스가 **무차별 모드(Promiscuous Mode)** 로 동작 중이라는 의미다.  
→ 스니핑 공격 가능성을 의심해야 한다.

## 7. ifconfig / ipconfig

### 7-1. ifconfig (Linux / Unix)

네트워크 인터페이스의 설정 정보 조회 및 변경 도구.  
인터페이스 활성화 / 비활성화도 가능하다.

#### 실행 결과 예시

```
eth0: flags=4163<UP,BROADCAST,RUNNING,PROMISC,MULTICAST>  mtu 1500
      inet 192.168.0.10  netmask 255.255.255.0  broadcast 192.168.0.255
      Metric:1
```

| 항목 | 설명 |
|---|---|
| `UP` | 인터페이스 활성화 상태 |
| `PROMISC` | 무차별 모드 (스니핑 주의) |
| `MTU: 1500` | 최대 전송 단위 1500 Bytes (이더넷 기본값) |
| `Metric:1` | 라우팅 메트릭 값 |

### 7-2. ipconfig (Windows)

Windows에서는 대부분 설정을 **제어판**에서 수행하며,  
`ipconfig`는 주로 **조회 및 DHCP 관련 작업**에 사용한다.

#### 주요 옵션

| 옵션 | 설명 |
|---|---|
| `/all` | 인터페이스 상세 정보 출력 (MAC 주소 포함) |
| `/release [어댑터]` | DHCP IP 주소 반납 |
| `/renew [어댑터]` | DHCP IP 주소 갱신 |
| `/displaydns` | DNS 캐시 내용 출력 |
| `/flushdns` | DNS 캐시 삭제 |
| `/registerdns` | DNS 강제 재등록 |

## 8. route

### 8-1. 개요

라우팅 테이블에 경로를 **추가하거나 삭제**하는 명령어.

| 방식 | 설명 |
|---|---|
| **정적 라우팅** | 관리자가 `route` 명령으로 수동 설정 |
| **동적 라우팅** | RIP, OSPF, BGP 등 라우팅 프로토콜이 자동 관리 |

### 8-2. 주요 명령

```bash
route add [목적지] [게이트웨이]   # 경로 추가
route del [목적지]                # 경로 삭제
```

## 9. 시험 핵심 요약

### 도구별 핵심 정리

| 도구 | 프로토콜 | 핵심 키워드 |
|---|---|---|
| `ping` | ICMP (Type 8 / 0) | 생존 확인, TTL, OS Fingerprinting |
| `traceroute` (Linux) | UDP 33434 + ICMP | TTL 1씩 증가, Time Exceeded / Port Unreachable |
| `tracert` (Windows) | ICMP | 동일 원리, ICMP 사용 |
| `netstat` | - | 연결 상태, 열린 포트, State 값 |
| `tcpdump` | - | 패킷 캡처, 스니핑, PROMISC |
| `ifconfig` | - | 인터페이스 설정, PROMISC 확인 |
| `ipconfig` | - | DHCP 갱신/반납, DNS 캐시 |
| `route` | - | 라우팅 테이블 추가/삭제 |

### 자주 틀리는 포인트 ❌ → ✅

| 틀리기 쉬운 내용 | 올바른 내용 |
|---|---|
| traceroute = TCP | traceroute = **UDP 33434** (Linux) |
| ACK = 받은 번호 | ACK = **다음에 받을 번호** (받은 번호 + 1) |
| FIN은 번호 소비 안 함 | FIN도 **번호 1개 소비** → 항상 +1 |
| TIME-WAIT = 서버 | TIME-WAIT = **클라이언트** |
| CLOSE-WAIT = 클라이언트 | CLOSE-WAIT = **서버** |
| 4-way도 SYN+ACK 합쳐서 | 4-way는 ACK, FIN **따로** 전송 |
| `*` = 무조건 장애 | `*` = 방화벽 차단 **또는** 장애 |

### 핵심 흐름 빠른 암기

```
3-way: SYN → SYN+ACK → ACK
4-way: FIN → ACK → FIN → ACK
```

```
ping       → ICMP Type 8 (Request) / Type 0 (Reply)
traceroute → 중간: Time Exceeded / 도착: Port Unreachable
```

```
TTL: Windows=128, Unix=255 (라우터마다 -1)
```
