## 1. 거버넌스 구조 — 큰 그림

거버넌스는 3단계 포함 관계로 이루어진다.

```
기업 거버넌스 (회사 전체의 방향/정책)
    └── IT 거버넌스 (IT를 통해 가치 창출 + 위험 관리)
            └── 정보보호 거버넌스 (보안 위험을 적절한 수준으로 관리)
```

| 단계 | 한 줄 설명 |
|------|-----------|
| 기업 거버넌스 | 회사 전체의 방향, 정책, 목표를 설정하는 것 |
| IT 거버넌스 | IT와 IT 프로세스의 위험·수익 균형을 맞추어 기업 목적을 달성하는 체계 |
| 정보보호 거버넌스 | 이사회·경영진이 보안 위험이 적절히 관리되고 있음을 감독하는 메커니즘 |

> 💡 **비유** : 회사를 큰 배라고 하면 —  
> 기업 거버넌스 = 배가 어디로 갈지 결정  
> IT 거버넌스 = 항해 장비를 어떻게 쓸지 결정  
> 정보보호 거버넌스 = 해적·사고·침몰 위험을 막는 방법 결정

## 2. 정보보호 거버넌스

### 2-1. 등장 배경

이사회와 경영진이 **"회사의 위험이 적절하게 관리되고 있는가?"** 를 직접 감독할 수 있어야 한다는 필요성에서 등장.  
즉, 보안은 기술자만의 문제가 아니라 **경영 문제**가 된 것이다.

---

### 2-2. 목표 — ABC ⭐⭐⭐

| 영문 | 한글 | 의미 |
|------|------|------|
| **A**ccountability | 책임성 | 보안 사고 시 누가 책임지는가? 승인자·관리자·결재자를 명확히 함 |
| **B**usiness Alignment | 비즈니스 연계성 | 보안이 회사 목표 달성에 기여하는가? 너무 불편해도, 너무 느슨해도 안 됨 |
| **C**ompliance | 준거성 | 법·규제·내부 정책을 제대로 준수하는가? |

---

### 2-3. 이사회 vs 경영진 역할 ⭐⭐

| 구분 | 역할 | 특징 |
|------|------|------|
| **이사회** | 보안 정책의 **방향** 설정 | 추상적, 큰 틀 ("무엇을 할지") |
| **경영진** | 보안 정책의 **개발·구현** | 구체적, 실행 ("어떻게 할지") |

> ⚠️ 최고경영진이 수정에 동의하지 않으면 보안 정책은 수행할 수 없다.

---

### 2-4. 정보보호 거버넌스 구현 요건 5가지 ⭐⭐⭐

암기 키워드 : **전 · 위 · 자 · 성 · 가**

| 요건 | 설명 |
|------|------|
| **전**략적 연계 | 비즈니스 목표 ↔ IT 목표 ↔ 보안 전략이 서로 연계되어야 함 |
| **위**험 관리 | 조직에 맞는 위험 관리 체계 수립, 수용 가능한 수준으로 지속 관리 |
| **자**원 관리 | 중요 정보 자산·인프라를 포함한 전사적 정보보안 아키텍처 확보 |
| **성**과 관리 | 모니터링·보고·평가를 통한 성과 측정 체계 운영 |
| **가**치 전달 | 구성원에게 보안의 중요성과 가치를 교육 (사장 설득 + 직원 교육) |

## 4. ISO/IEC 27000 시리즈

### 전체 구조 한눈에 보기

```
ISO/IEC 27000 시리즈
├── 27000  개관 및 용어
├── 27001  요구사항 (ISMS 인증 기준) ← 핵심
├── 27002  보안 통제 실무 가이드 (구 ISO 17799)
├── 27003  구현 가이드
├── 27004  측정 및 보고
├── 27005  위험 관리 프로세스 지침
├── 27014  정보보호 거버넌스
├── 27017  클라우드 보안
└── 27018  클라우드 개인식별정보(PII) 보호
```

### 번호별 상세 ⭐⭐⭐

| 번호 | 주제 | 핵심 키워드 |
|------|------|------------|
| **27000** | 개관 및 용어 | 정의, 개념 |
| **27001** | 요구사항 | ISMS 인증, 필수 준수 |
| **27002** | 실무 가이드 | Best Practice, 구 ISO 17799 |
| **27003** | 구현 방법 | 구축 절차 상세 |
| **27004** | 측정 | 통제 효율성 수치화·보고 |
| **27005** | 위험 관리 | ISO 13335-3/4 기반 |
| **27014** | 거버넌스 | 정보보호 거버넌스 |
| **27017** | 클라우드 보안 | 클라우드 서비스 |
| **27018** | 클라우드 PII | 클라우드 개인정보 |

## 5. ISO 27001 — ISMS 핵심 표준

### 5-1. 개요

- 영국 BSI의 **BS 7799** 를 기반으로 제정된 국제 표준
- **ISMS(정보보호관리체계)** 를 체계적으로 운영하고 있음을 증명하는 인증 기준
- "보안을 잘 하고 있는 것 같다" → **"체계적으로 관리하고 있다는 것을 증명"**

---

### 5-2. IT 보안관리 프로세스 — PDCA ⭐⭐⭐

ISO 27001의 핵심은 **반복(PDCA 사이클)** 이다.  
구체적 적용은 **ISO 27005(위험 관리)** 에 명시되어 있다.

```
  ┌─────────────────────────────────┐
  │                                 │
  ▼                                 │
Plan (계획) ──► Do (실행) ──► Check (점검) ──► Act (개선)
  수립           이행          모니터링·유지      개선·유지
```

| 단계 | 내용 |
|------|------|
| **Plan** | 보안 목표·정책·절차 수립 |
| **Do** | 계획 이행 |
| **Check** | 모니터링, 측정, 유지보수 |
| **Act** | 결과 분석 후 개선 |

---

### 5-3. ISO 27001:2013 구조

**7개 관리 과정** + **PDCA 연결**

| 관리 과정 | PDCA |
|----------|------|
| 조직의 상황 | **Plan** |
| 리더십 | **Plan** |
| 기획 | **Plan** |
| 지원 | **Do** |
| 운영 | **Do** |
| 성과평가 | **Check** |
| 개선 | **Act** |

- 통제 요구사항 : **14개 영역, 114개 통제 항목** ⭐

---

### 5-4. ISO 27001:2022 — 변경 사항 ⭐⭐

2013 버전의 14개 영역을 **4개 주제**로 재분류

| 분류 | 설명 |
|------|------|
| **조직적 (Organizational)** | 정책, 역할, 책임 등 조직 관련 통제 |
| **사람 (People)** | 인적 자원 관련 통제 |
| **물리적 (Physical)** | 물리적 환경 및 설비 보안 통제 |
| **기술적 (Technological)** | 기술적 보안 통제 |

## 6. 보안 문서 계층 구조

### 6-1. 5단계 계층 ⭐⭐⭐

```
정책 (Policy)          ← 가장 넓고 추상적
    └── 표준 (Standard)        ← 의무 규칙
            └── 기준선 (Baseline)   ← 최소 보호 수준
                    └── 지침 (Guideline)  ← 권장 방법 (강제성 없음)
                            └── 절차 (Procedure)  ← 단계별 실행 방법
```

---

### 6-2. 각 계층 상세 비교

| 계층 | 강제성 | 대상 | 특징 | 예시 |
|------|--------|------|------|------|
| **정책** | 높음 | 전사 | 큰 원칙만, 세부 없음, 경영진 승인 필요 | "허가된 사용자만 접근 가능하다" |
| **표준** | 높음 | 전사 | 정책 지원, 의무 규칙 | "비밀번호 최소 12자 이상" |
| **기준선** | 중간 | 시스템별 | 최소 보호 수준 정의, 비교 기준점 | OS 설치 후 보안 옵션 체크리스트 |
| **지침** | 낮음 | 실무자 | 권장사항, 유연 적용 가능 | KISA 보안 가이드라인 |
| **절차** | 높음 | 실무자 | 단계별 작업, 가장 구체적 | 계정 생성 → 승인 → 발급 → 로그 기록 |

> 💡 **표준 vs 기준선 차이**  
> - 표준 : "OS 설치 시 강력한 보안 옵션을 사용해야 한다" (규칙)  
> - 기준선 : "설치 후 해당 옵션이 실제로 적용됐는지 검사하는 체크리스트" (기준점)

> 💡 **지침 vs 절차 차이**  
> - 지침 : "이렇게 하면 좋다" → 강제성 없음  
> - 절차 : "이 순서대로 해라" → 구체적 단계 제시

---

### 6-3. 정보보호 정책 요구사항

정책이 효력을 발휘하려면 아래 3가지가 필요하다.

1. **경영진에 의한 승인** — 최고경영진이 동의하지 않으면 정책은 수행 불가
2. **포괄적·일반적·개괄적으로 기술** — 세부 사항은 하위 문서(표준·절차)에서 다룸
3. **문서화 + 직원 교육** — 교육까지 완료해야 법적 보호를 받을 수 있음

## 7. IT 보안 계획 vs 보안 정책

### 비교 ⭐⭐

| 항목 | 보안 정책 | IT 보안 계획 |
|------|----------|------------|
| **성격** | 방향·원칙 | 실행 설계도 |
| **내용** | "무엇을 지켜야 하는가" | "어떻게, 누가, 언제 실행할 것인가" |
| **특징** | 넓고 추상적 | 구체적, 자원·책임·일정 포함 |
| **기간** | 장기 | 1년 단위, 주기적 검토 |

### IT 보안 계획 포함 내용

- 식별된 위험 목록
- 권고 통제 사항 및 우선순위
- 선택된 통제와 필요한 자원
- 책임 담당자
- 이행 시작일 / 종료일
- 유지보수 요구사항

> 📌 최소 **1년에 한 번** 검토, 인수합병·아웃소싱 등 변화 발생 시 추가 검토

## 8. 보안 조직 및 역할

### 8-1. 구성원 역할 ⭐⭐⭐

| 역할 | 책임 |
|------|------|
| **최고경영자** | 보안 전체 총괄책임 |
| **정보보호 관리자** | 정책·표준·대책·절차의 설계·구현·관리·조사 |
| **데이터 관리자** | 데이터의 정확성과 무결성 유지 |
| **프로세스 관리자** | 업무 절차가 보안 정책에 맞게 운영되도록 보증 |
| **정보시스템 감사자** | 독립적 입장에서 보안 활동 검증 및 보증 |

> 💡 **정보보안 전문가의 핵심 역할 = 촉진자(Facilitator)**  
> 혼자 다 하는 게 아니라, 보안이 잘 돌아가도록 조율하고 연결하는 역할

> ⚠️ 중요 : 애플리케이션 구축 **후** 보안 기능 추가 시 보통 **10배 이상의 비용** 발생  
> → 처음부터 보안을 설계에 포함해야 한다 (Security by Design)

---

### 8-2. 전사적 보안감독 위원회

- **구성** : 중간관리자급
- **역할** : 보안 활동을 조직 전체에 지속적으로 인식시키는 역할
- **다루는 주제** : 비전 선언문, 미션 선언문, 보안 이슈 조율
- **회의 주기** : 조직 문화에 따라 다름

---

### 8-3. 경영진 설득 방법 3가지 ⭐

| 방법 | 풀네임 | 설명 |
|------|--------|------|
| **FUD** | Fear, Uncertainty, Doubt | 위험을 직접 보여줌 — "이거 안 하면 큰일 납니다" |
| **Due Care** | 법적 주의 의무 | 법적 책임 강조 — "이건 회사의 당연한 의무입니다" |
| **Team Up** | 팀 협력 설득 | 실무자 + 법무팀이 함께 경영진을 설득 |

## 9. 인적 자원 보안

### 9-1. 고용 전 → 재직 중 → 퇴직 흐름

```
[고용 전]            [재직 중]              [퇴직]
배경조사             보안 교육               장비 반납
레퍼런스 체크  ──►  직무순환/분리      ──►  계정 삭제
NDA 서명             강제휴가                접근권한 회수
                     지속적 인식 훈련
```

---

### 9-2. 고용 전 보안

| 항목 | 내용 |
|------|------|
| **배경 검사** | 이력서 허위 기재 방지, 신원 확인 |
| **레퍼런스 체크** | 경력·능력·팀워크·리더십 검증 |
| **고용협정(NDA)** | 기밀 유지 조항, 영업비밀·지적재산권 보호 포함 |

> NDA는 퇴직 시 검토 서류, Pen-Test 수행 시에도 활용됨

---

### 9-3. 재직 중 통제 ⭐⭐⭐

| 통제 | 목적 | 적용 예 | 통제 유형 |
|------|------|---------|----------|
| **직무순환** | 부정·실수 발견, 공모 위험 감소 | 금융기관에서 보편적 적용 | 예방 + 탐지 |
| **직무분리** | 한 사람에게 모든 권한 집중 방지 | 결제 신청자 ≠ 결제 승인자 | 예방적·관리적 통제 |
| **강제휴가** | 부정 행위 탐지 | 직원 부재 중 이상 작업 드러냄 | 탐지 통제 |

> ⚠️ 소규모 조직에서 직무순환·분리가 어려운 경우  
> → **부서장 감독 강화 + 제3자 감사추적 검토**로 보완

## 10. 핵심 요약 & 시험 포인트

### ✅ 거버넌스

- 기업 ⊃ IT ⊃ 정보보호 거버넌스 (포함 관계)
- 정보보호 거버넌스 목표 = **ABC** (Accountability / Business Alignment / Compliance)
- 구현 요건 5개 = **전·위·자·성·가**

### ✅ 표준

- COBIT = IT 거버넌스 프레임워크 (ISACA/ITGI 개발)
- ISO 27001 = ISMS 인증 핵심
- ISO 27002 = 보안통제 실무 가이드 (구 ISO 17799)
- ISO 27005 = 위험 관리

### ✅ ISO 27001

- 2013 : 7개 관리 과정 / 14개 영역 / 114개 통제 항목
- 2022 : 조직적·사람·물리적·기술적 4개로 재분류
- PDCA 사이클이 핵심

### ✅ 보안 문서 계층

```
정책 → 표준 → 기준선 → 지침 → 절차
(추상적)                        (구체적)
```

### ✅ 인적 보안

- 해고 시 : **통보 전 접근권한 먼저 삭제** ← 시험 단골
- 직무분리 : 예방적 통제
- 강제휴가 : 탐지 통제
- NDA = 고용 전·재직 중·퇴직 시 모두 활용

### ✅ 경영진 설득

- FUD → Due Care → Team Up
