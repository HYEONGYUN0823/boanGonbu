# 🚪 접근통제 (Access Control)

---

## 📌 먼저 읽기! 핵심 개념 한눈에

> 💡 **비유 : 회사 건물 출입 시스템**
> 1. 사원증 태그 → "나 홍길동이야!" (식별)
> 2. 비밀번호 입력 → "진짜 홍길동 맞아!" (인증)
> 3. 출입 허용/거부 → "3층까지만 들어갈 수 있어!" (인가)

---

## 1. 접근통제 기본 개념 ⭐⭐⭐

접근통제는 두 가지를 모두 막아야 해!

```
❌ 비인가된 사용자가 정보자원에 접근하는 것
❌ 인가된 사용자가 비인가된 방식으로 접근하는 것
❌ 인가된 사용자가 실수/의도적으로 잘못 접근해서 자산을 훼손하는 것
```

### 주요 용어

| 용어 | 역할 | 설명 |
|---|---|---|
| **주체 (Subject)** | 능동적 (행위자) | 접근을 **요청**하는 개체 (사람, 프로세스 등) |
| **객체 (Object)** | 수동적 (제공자) | 접근의 **대상**이 되는 개체 (파일, DB, 서버 등) |
| **접근 (Access)** | - | 주체와 객체 사이의 정보 흐름 (읽기/쓰기/삭제/수정) |

---

## 2. 접근통제 절차 3단계 ⭐⭐⭐⭐⭐

> 💡 **비유 : 햄버거 가게 알바 출근**
> 1. 이름표 착용 → "나 김철수야!" (식별 - 공개)
> 2. 지문 찍기 → "진짜 김철수 맞아!" (인증 - 사설정보)
> 3. "주방만 들어갈 수 있어" → (인가 - 권한 부여)

```
[식별] ──→ [인증] ──→ [인가]
  ID       패스워드     ACL
 (공개)   (사설정보)  (권한목록)
```

### 단계별 상세

| 단계 | 목적 | 접근 매체 | 핵심 |
|---|---|---|---|
| **식별 (Identification)** | 신원 표명 | 사용자명, 계정번호, 메모리카드 | 책임추적성(Log) 분석에 중요! |
| **인증 (Authentication)** | 신원 검증 | 패스워드, PIN, 토큰, 스마트카드, 생체인증 | 진짜 본인인지 증명(verify) |
| **인가 (Authorization)** | 권한 부여 | 접근제어목록(ACL), 보안 등급 | 인증된 주체에게 업무 수행 권리 부여 |

> 💡 **알 필요성 (Need-to-Know)**
> 인가 단계에서 적용되는 원칙
> "이 정보가 이 사람에게 필요한가?"를 따지는 것
> = 최소 권한 원칙과 같은 의미

---

## 3. 책임추적성 (Accountability)

```
식별자(ID) = 각 개인의 신원을 나타냄
           → 사용자 행위를 로그(Log)로 추적 가능

⚠️ 책임추적이 가장 어려운 경우
   → 하나의 계정을 여러 명이 공유할 때!
   (누가 했는지 특정 불가능)
```

---

## 4. 접근통제 요구사항 ⭐⭐⭐

### 1) 입력의 신뢰성 (Input Reliability)

```
접근제어 시스템은 입력되는 사용자 정보를 신뢰할 수 있어야 함
→ 접근제어 앞단(전단)에 충분한 사용자 인증 기능이 있어야 함!

순서 : [인증] → [접근제어]
인증이 먼저 되어야 접근제어가 의미 있음
```

### 2) 최소 권한 부여 (Least Privilege) ⭐⭐⭐

> 💡 **비유 : 햄버거 가게 알바생**
> 알바생은 주방 들어가고 조리하는 권한만 필요해
> 회계장부 열람, 재고 시스템 수정 권한은 필요 없잖아!
> → 필요한 것만 딱 주는 게 최소 권한!

```
허가받은 업무를 수행하기 위한 최소한의 권한만 부여
→ 권한 남용으로 인한 피해 최소화

= Need-to-Know (알 필요성)
↔ 최대권한 (Maximum Privilege Policy) 과 반대 개념
```

### 3) 직무분리 (Separation of Duty) ⭐⭐⭐

> 💡 **비유 : 햄버거 가게 운영**
> 주문 받는 사람 따로 / 돈 받는 사람 따로 / 조리하는 사람 따로
> 한 사람이 다 하면 몰래 돈 빼돌리기 쉬움!
> → 역할을 나누면 부정행위 어려워짐

```
업무의 발생부터 완료까지 한 사람이 처리할 수 없게 하는 보안 정책

직무분리 예시
  보안 ↔ 감사
  개발 ↔ 운영
  암호키 관리 ↔ 암호키 변경

⚠️ 직무분리의 원칙을 깨는 것 = 공모 (두 명 이상이 짜고 치는 것)
✅ 해결 방법 = 직무순환 (주기적으로 담당 업무를 바꿈)
```

---

## 📌 최종 핵심 암기 요약

```
✅ 접근통제 3단계
   식별(ID) → 인증(패스워드/생체) → 인가(ACL)
   식별 = 공개, 인증 = 사설정보

✅ 책임추적성
   ID = 로그 분석의 핵심
   계정 공유 → 책임추적 불가!

✅ 접근통제 요구사항 3가지
   입력의 신뢰성 : 인증이 먼저, 그 다음 접근제어
   최소 권한     : 필요한 권한만 딱! = Need-to-Know
   직무분리      : 한 사람이 전체 업무 처리 불가
                   공모로 깰 수 있음 → 직무순환으로 해결

✅ 최소 권한 vs 최대 권한
   최소 권한 = Need-to-Know ← 우리가 지향해야 할 것
   최대 권한 = 반대 개념 (위험!)
```
