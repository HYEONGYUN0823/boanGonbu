## 1. 네트워크 공격 분류

### 수동적 공격 vs 능동적 공격

| 구분 | 수동적 공격 | 능동적 공격 |
|------|------------|------------|
| 특징 | 몰래 보기만 함 | 변조, 위조, 실행 |
| 예시 | 스니핑, 도청 | 세션 하이재킹, DoS |
| 탐지 | 어려움 | 쉬움 |
| 대응 | 암호화 | 무결성 검증 + 암호화 |

> 💡 **비유:** 수동적 = "몰래 엿듣는 도청범" / 능동적 = "통신에 끼어들어 장난치는 놈"

## 2. DoS / DDoS 공격

### 개념

- **DoS (Denial of Service):** 단일 컴퓨터로 서버에 과도한 부하를 주어 서비스를 마비시키는 공격
- **DDoS (Distributed DoS):** 다수의 좀비 PC(봇넷)를 이용해 분산 공격
- **DRDoS (Distributed Reflection DoS):** 정상 서버를 반사(증폭) 서버로 악용하는 공격

### DDoS 구성요소

```
공격자(Bot Master)
    ↓ 명령
C&C 서버 (Command & Control)
    ↓ 지시
좀비 PC들 (Agent / Bot)
    ↓ 공격 트래픽
피해 서버 🎯
```

### 봇넷 (Botnet)

- **Bot:** 보안 결함을 이용해 원격에서 시스템을 제어하는 프로그램
- **Botnet:** 봇에 감염된 다수의 좀비 PC/디바이스로 구성된 네트워크
- 분류: 중앙집중형(IRC, HTTP 봇넷) / 분산형(P2P 봇넷)

### DNS 싱크홀 (DNS Sinkhole) — 대응책

- 감염된 PC가 C&C 서버로 접속하려 할 때, **싱크홀 서버**로 유도하여 차단
- 악성 트래픽을 미리 파놓은 "함정 서버"로 보내는 방식

### 봇넷 보안장비 우회 기법

| 기법 | 설명 |
|------|------|
| **Fast Flux** | C&C 서버의 IP를 빠르게 변경하여 탐지 회피 |
| **DGA** | 알고리즘으로 무작위 도메인을 생성하여 접속 |
| **Domain Shadowing** | 합법적인 도메인의 서브도메인을 악용 |

## 3. 주요 DoS 공격 기법

### 3-1. TCP SYN Flooding ⭐⭐⭐

#### 3-Way Handshake 정상 흐름

```
클라이언트                서버
   SYN      ──────────>
             <──────────  SYN+ACK
   ACK      ──────────>
                         연결 완료 ✅
```

#### SYN Flooding 공격 흐름

```
공격자(가짜 IP)           서버
   SYN      ──────────>
             <──────────  SYN+ACK
   (ACK 없음 ❌)          Half-Open 상태로 대기...
   SYN      ──────────>
   SYN      ──────────>  연결 테이블 꽉 참 💥
   SYN      ──────────>  정상 사용자 접속 불가
```

#### Half-Open 상태란?
- 한쪽(서버)은 연결 대기 중인데, 다른 쪽(클라이언트)은 연결하지 않은 상태
- 서버는 Backlog Queue에 대기 항목을 쌓아두다가 결국 고갈됨

#### 방어 대책

| 방법 | 설명 |
|------|------|
| **SYN Cookie** ⭐ | SYN 수신 시 메모리 저장 대신, 암호화된 시퀀스 번호를 SYN+ACK에 담아 전송. ACK가 올 때만 연결 생성 |
| **임계치 설정** | 동일 IP의 SYN 요청 횟수 제한 (iptables 활용) |
| **IP 필터링** | 위조된 출발지 IP 패킷 차단 |
| **연결 테이블 확장** | 임시방편. 근본 해결책 아님 |
| **First SYN Drop** | 첫 SYN은 버리고 재요청 패킷 여부 확인 |

```bash
# Linux SYN Cookie 활성화
sysctl -w net.ipv4.tcp_syncookies=1

# iptables 동일 IP 5개 초과 SYN 차단
iptables -A INPUT -p TCP --dport 80 --syn -m connlimit --connlimit-above 5 -j DROP
```

### 3-2. SMURF Attack ⭐⭐

#### 공격 구조

```
공격자
  └─ 출발지 IP = 피해자 IP로 위조
  └─ 브로드캐스트 주소로 ICMP Echo Request 전송

증폭 네트워크 내 모든 PC
  └─ 피해자 IP로 ICMP Echo Reply 응답 폭탄 💥
```

> 💡 **비유:** "단체 채팅방에 피해자 번호로 '다들 나한테 연락해!' 를 보내는 것"

- **Fraggle Attack:** SMURF와 동일하나 ICMP 대신 **UDP** 사용

#### 방어 대책

```bash
# 라우터에서 Directed Broadcast 차단
(config-if)# no ip directed-broadcast
```
- 호스트가 브로드캐스트 ICMP 패킷에 응답하지 않도록 설정

---

### 3-3. LAND Attack ⭐⭐

- **출발지 IP = 목적지 IP** 로 설정하여 전송 (자기 자신에게 패킷 보내기)
- 시스템 혼란 → 다운

#### 방어
- 출발지 IP와 목적지 IP가 동일한 패킷 차단

---

### 3-4. Ping of Death

- ICMP 패킷을 정상 크기(65,535 bytes)보다 **크게** 만들어 전송
- 패킷은 네트워크를 거치며 분할(Fragment)되고, 수신 측에서 조립 시 오버플로우 발생
- **방어:** 분할된 ICMP 패킷을 공격으로 의심하여 탐지

---

### 3-5. Teardrop Attack ⭐⭐⭐

#### Fragment / Offset 개념

```
원본 데이터를 쪼개서 전송
[0~100] [100~200] [200~300]
  ↓ 각 조각에 Offset(위치 정보)을 붙임
[0번] [100번] [200번]
수신 측에서 Offset을 보고 재조립 ✅
```

#### 공격 방법

```
정상:
  조각1: [0~100]
  조각2: [100~200]  ← 딱 이어짐 ✅

공격 (Teardrop):
  조각1: [0~100]
  조각2: [50~150]   ← 겹침 💥 → 재조립 오류 → 시스템 다운
```

> 💡 **비유:** "퍼즐 조각이 겹쳐서 끼워 맞출 수 없는 상태"
> 결과: **Blue Screen of Death (BSOD)**

#### 관련 변종 공격

| 공격명 | 특징 |
|--------|------|
| **Bonk** | 모든 패킷의 순서번호를 1번으로 조작 |
| **Boink** | 비정상적인 시퀀스 번호 전송 |
| **Nestea / Newtear** | Teardrop 유사 |
| **Jolt** | Ping of Death 유사 (Windows 95/NT 대상) |

---

### 3-6. IP Fragmentation 우회 공격 (DoS 아님!)

| 공격 | 설명 |
|------|------|
| **Tiny Fragment** | IP 헤더보다 작은 조각을 만들어 방화벽이 분석 못하게 함 → 우회 침입 |
| **Fragment Overlap** | Offset을 조작해 재조합 후 방화벽이 막은 포트에 접근 가능하게 만듦 |

> ⚠️ 둘 다 **DoS 공격이 아닌 방화벽 우회 공격**

### 3-7. DRDoS ⭐⭐

```
공격자
  └─ 출발지 IP = 피해자 IP로 위조
  └─ 반사 서버(DNS, NTP, SNMP 등)로 요청 전송

반사 서버
  └─ 피해자 IP로 대용량 응답 전송 💥
```

#### UDP 기반 DRDoS 종류

| 종류 | 반사 서버 | 특징 |
|------|----------|------|
| **DNS 증폭** | DNS 서버 | ANY Type 질의로 대용량 응답 유도 |
| **NTP 증폭** | NTP 서버 | `monlist` 명령 악용 |
| **SNMP 증폭** | SNMP 서버 | `GetBulkRequest` 악용 |
| **SSDP 증폭** | IoT 기기 | UPnP 프로토콜 악용 |
| **Memcached 증폭** | Memcached 서버 | 캐시 데이터 대량 요청 |
| **CLDAP 증폭** | LDAP 서버 | UDP LDAP 악용 |

### 3-8. 웹/DB 부하 공격 (Slow 공격 3형제) ⭐⭐

| 공격 | 방법 | 핵심 |
|------|------|------|
| **Slowloris** (Slow HTTP Header DoS) | HTTP 헤더를 조금씩 전송, 마지막 개행문자 안보냄 | 서버가 헤더 안끝났다 판단 → 연결 유지 |
| **RUDY** (Slow HTTP POST DoS) | Content-Length를 크게 설정 후 데이터를 아주 천천히 전송 | 서버가 POST 완료 기다리며 연결 유지 |
| **Slow Read** | TCP Window Size를 낮게 설정 | 서버가 천천히 전송하느라 연결 유지 |

> 💡 **공통 핵심:** "연결을 오래 붙잡아서 서버 자원(Connection 수) 고갈"

---

### 3-9. GET Flooding 관련

| 공격 | 설명 |
|------|------|
| **GET Flooding** | 세션 맺은 후 HTTP GET 요청 폭탄 |
| **CC Attack** (Cache-Control) | Cache-Control 헤더 조작으로 캐시 무력화 → 웹서버 직접 처리 강제 |
| **동적 HTTP Request Flooding** | 요청 페이지를 계속 바꿔가며 차단 우회 |
| **HulkDos** | URL을 지속 변경하며 GET Flooding |
| **HashDos** | 조작된 파라미터로 해시 충돌 유발 → CPU 소진 |

## 4. DDoS 공격 유형 분류

### 3가지 유형 핵심 정리 ⭐⭐⭐

| 유형 | 특징 지표 | 대표 공격 | 피해 |
|------|----------|----------|------|
| **대역폭 공격** | bps 높음 | UDP/ICMP Flooding | 회선 전체 마비 |
| **자원 소진 공격** | pps 높음 | SYN Flooding | 서버 CPU/메모리 고갈 |
| **웹/DB 부하 공격** | Connection 높음 | GET Flooding | 웹/DB 서버 과부하 |

> 💡 **bps** = bit per second (데이터 양) / **pps** = packet per second (패킷 개수)

### 전통적 DDoS 도구 발전 순서

```
Trinoo (UDP flood)
  ↓
TFN (UDP + SYN + ICMP + SMURF)
  ↓
Stacheldraht (TFN + 암호화)
  ↓
TFN2K (포트 임의 사용 + 암호화)
```

## 5. 네트워크 스캐닝

### 정보 수집 3단계

```
풋프린팅 → 스캐닝 → 목록화(Enumeration)
(주변조사)  (직접두드리기)  (공격 목록 정리)
```

### 포트 스캔 종류 ⭐⭐⭐

| 스캔 종류 | 방법 | 로그 남김 | 특징 |
|----------|------|----------|------|
| **TCP Connect (Full Open)** | 3-way handshake 완성 | ✅ 남김 | 정확하지만 탐지됨 |
| **TCP SYN (Half Open)** | SYN → RST로 끊음 | 거의 안남김 | 빠르고 조용 |
| **UDP 스캔** | UDP 전송 후 ICMP Unreachable 확인 | - | 신뢰도 낮음 |
| **FIN 스캔** | FIN 패킷 전송 | ❌ 안남김 | 닫힌 포트 → RST |
| **NULL 스캔** | 플래그 모두 OFF | ❌ 안남김 | 닫힌 포트 → RST |
| **XMAS 스캔** | 여러 플래그 ON | ❌ 안남김 | 닫힌 포트 → RST |
| **ACK 스캔** | ACK 전송 | - | 방화벽 정책 파악용 |
| **Decoy 스캔** | 가짜 IP 여러 개 섞기 | - | 공격자 추적 어렵게 |
| **FTP Bounce 스캔** | FTP 서버를 중간 경유지로 사용 | - | 방화벽 우회 |

> 💡 **FIN/NULL/XMAS 공통점:** 열린 포트는 무응답, 닫힌 포트는 RST 응답

### TCP 플래그 (UAPRSF)

| 플래그 | 이름 | 의미 |
|--------|------|------|
| **U** | URG | 긴급 데이터 |
| **A** | ACK | 수신 확인 |
| **P** | PSH | 즉시 전달 |
| **R** | RST | 강제 연결 종료 |
| **S** | SYN | 연결 시작 요청 |
| **F** | FIN | 연결 정상 종료 |

### Nmap 주요 옵션

```bash
nmap -sS 192.168.0.1       # TCP SYN 스캔
nmap -sT 192.168.0.1       # TCP Connect 스캔
nmap -sU 192.168.0.1       # UDP 스캔
nmap -O  192.168.0.1       # OS 탐지
nmap -p 80,443 192.168.0.1 # 특정 포트 스캔
nmap -v  192.168.0.1       # 상세 출력
nmap -T0 ~ -T5             # 스캔 속도 조절 (T0=느림, T5=빠름)
```

### OS Fingerprinting

- **목적:** 대상 OS 종류/버전 파악 → 맞춤 취약점 공격
- **방법:**
  - TCP 응답 패턴 분석
  - **배너 그래빙 (Banner Grabbing):** 서비스가 응답할 때 보내는 자기소개 문구 확인
    ```
    telnet [대상IP] [포트번호]
    ```

## 6. 스니핑 (Sniffing)

### 개념

- 네트워크 통신 패킷을 **몰래 엿듣는** 공격
- **소극적(수동적) 공격** — 탐지 어려움
- 도구: Wireshark, Dsniff 등

### Promiscuous Mode (무차별 모드)

- 일반 모드: 자기한테 온 패킷만 수신
- **Promiscuous 모드:** 자기 것이 아닌 패킷도 전부 수신

> 💡 **비유:** 일반 = "내 편지만 받음" / Promiscuous = "우편함 편지 다 확인"

### 환경별 스니핑

| 환경 | 특징 | 공격 방법 |
|------|------|----------|
| **허브** | 모든 포트로 패킷 퍼뜨림 | Promiscuous 모드만으로 가능 |
| **스위치** | 지정 포트로만 전송 | MAC Flooding으로 허브처럼 동작하게 유도 |

### Switch Jamming (MAC Flooding) ⭐

- 스위치의 **MAC Address Table**을 가짜 MAC으로 가득 채움
- 테이블 오버플로우 → 스위치가 **허브처럼 모든 포트로 전송**
- 스위치는 **Fail Open** 정책 (고장 시 → 전체 허용)

| | Fail Open | Fail Close |
|--|-----------|------------|
| 우선 | 가용성 | 보안성 |
| 예 | 비상구, 네트워크 스위치 | 금고, 중요 보안 시스템 |

### 스니핑 탐지 방법

| 방법 | 원리 |
|------|------|
| **ping 이용** | 존재하지 않는 MAC으로 ping → 응답 있으면 Promiscuous 상태 의심 |
| **ARP 이용** | ping과 유사한 원리로 이상 반응 탐지 |
| **DNS 방법** | Inverse DNS Lookup으로 이상 호스트 탐지 |
| **Decoy 방법** | 가짜 계정/패스워드를 뿌려서 사용 시도 탐지 |
| **ARP watch** | IP-MAC 매핑 감시, 변경 시 관리자 알림 |

### 스니핑 방어 대책

- **능동적 대응:** 탐지 (위의 탐지 방법 활용)
- **수동적 대응(근본 대책):** 암호화

| 서비스 | 암호화 프로토콜 |
|--------|--------------|
| 웹 | SSL/TLS |
| 이메일 | PGP, PEM, S/MIME |
| 원격 접속 | SSH |
| 네트워크 전체 | VPN (L2TP, IPSec 등) |

## 7. 스푸핑 (Spoofing)

### 개념

- 자신의 정체를 **다른 주소/신분으로 속이는** 공격
- **중간자 공격 (MITM)** 또는 **DoS** 공격에 활용

### ARP (Address Resolution Protocol) 기본

```
PC1: "192.168.0.10의 MAC 주소가 뭐야?" (ARP Request - 브로드캐스트)
PC2: "나야! MAC은 AA:BB:CC:DD:EE:FF" (ARP Reply - 유니캐스트)
PC1: ARP Cache에 저장
     192.168.0.10 → AA:BB:CC:DD:EE:FF
```

> 💡 **IP = 집 주소 / MAC = 초인종 이름표 / ARP = "이 집 이름표가 뭐야?" 라고 묻는 것**

### ARP 스푸핑 ⭐⭐⭐

- 가짜 **ARP Reply**를 지속적으로 전송 → 피해자의 ARP Cache 오염
- **ARP Cache Poisoning**이라고도 함

```
원래:
피해자 ──────────────→ 서버

ARP 스푸핑 후:
피해자 ──→ 공격자 ──→ 서버
            (중간에서 패킷 도청 가능)
```

**방어:**
```bash
# 정적 ARP 설정 (재부팅 시 초기화됨)
arp -s [IP주소] [MAC주소]
```

### ARP Redirect

- 공격자가 **라우터인 척** MAC을 위조하여 ARP Reply를 브로드캐스트
- 피해자의 트래픽이 공격자를 거쳐 라우터로 전달됨

### ICMP Redirect

- 라우터가 호스트에게 **더 나은 경로**를 알려주는 기능 (3계층)
- **공격:** 공격자가 "내가 더 좋은 라우터야"라고 속여 경로 변조

| | ARP Redirect | ICMP Redirect |
|--|-------------|---------------|
| 계층 | 2계층 | 3계층 |
| 변조 대상 | ARP Cache Table | 라우팅 테이블 |
| 속이는 것 | MAC 주소 | 경로(라우터) |

### IP 스푸핑 ⭐⭐

- 신뢰 관계(Trust)가 있는 시스템의 **IP를 위장**하여 서버에 접근
- 1995년 **케빈 미트닉**이 최초 시도

```
공격 절차:
1. 신뢰 관계 확인 (서버 A는 서버 B를 신뢰)
2. 서버 B에게 SYN Flooding → 응답 불가 상태로 만듦
3. 공격자가 서버 B의 IP로 위장하여 서버 A에 접속
```

**신뢰 관계 설정 파일:**
- `/etc/hosts.equiv` — 시스템 전체 영향
- `$HOME/.rhosts` — 사용자별 설정

**방어:**
- 외부에서 내부 IP를 출발지로 하는 패킷 차단
- 트러스트 관계 사용 금지
- IPSec (VPN) 사용 ← 근본적 대책

### DNS 스푸핑

- 실제 DNS 서버보다 **빠르게 가짜 DNS 응답**을 전송
- DNS는 UDP(53번 포트) → 먼저 도착한 응답을 신뢰 → 시간차 공격

```
피해자 → "aaa.com IP 알려줘" (DNS Query)
공격자 → "192.168.0.100이야!" (가짜 DNS Response, 더 빨리 도착)
피해자 → 가짜 사이트로 접속 💥
```

**방어:** `/etc/hosts` 파일에 중요 사이트 IP를 직접 등록

### 스푸핑 유형 정리

| 유형 | 계층 | 설명 |
|------|------|------|
| **IP 스푸핑** | 3계층 | 출발지 IP 위조 |
| **ARP 스푸핑** | 2계층 | MAC 주소 위조 |
| **DNS 스푸핑** | 7계층 | 잘못된 IP로 유도 |
| **웹 스푸핑** | 7계층 | 웹 페이지 내용 변조 |
| **이메일 스푸핑** | 7계층 | 발신자 위장 |

## 8. 세션 하이재킹

### 개념 ⭐⭐

- 인증이 완료된 **세션을 가로채서** 별도 인증 없이 통신을 계속하는 공격
- OTP, Challenge/Response 같은 인증도 무력화 가능한 위험한 공격

> 💡 **비유:** "이미 문 열어둔 집에 몰래 들어가 주인인 척"

### TCP 연결 하이재킹

- 스니핑과 달리 **명령 실행까지 가능** → 적극적(능동적) 공격
- 도구: HUNT, Juggernaut

#### 순서번호 추론 기반 공격

- TCP 시퀀스 번호를 예측하여 가짜 패킷을 끼워 넣는 방식
- **방어:** 시작 순서번호를 **랜덤(Random)** 으로 생성

#### 스니핑 기반 TCP 하이재킹

```
1. 공격자가 통신 스니핑 → 정확한 시퀀스 번호 획득
2. RST 패킷으로 서버 쪽 연결만 끊음
3. 새로운 시퀀스 번호로 서버에 재연결
4. 공격자 ↔ 서버 간 세션 성립 (클라이언트는 모름)
```

**ACK Storm:** 비동기 상태에서 클라이언트와 서버가 서로 ACK를 계속 주고받는 현상

### 세션 하이재킹 방어 ⭐

| 방법 | 설명 |
|------|------|
| **암호화** ← 최우선 | 데이터 암호화로 내용 보호 |
| **지속적 인증** | 일정 간격으로 재인증 |
| **비동기화 상태 탐지** | 이상 세션 감지 |
| **ACK Storm 탐지** | 비정상적인 ACK 폭증 감지 |
| **패치** | TCP/IP 취약점 수정 |

### HTTP 세션 하이재킹 (Session ID 하이재킹)

- 웹 서버는 **Session ID** 로 로그인 상태를 구분
- 이 Session ID를 훔치면 → 인증 없이 해당 사용자 권한으로 접근 가능
- **방어:** 추측하기 어렵고 긴 세션 식별자 사용

## 9. Remote Attack

### 분류

| 구분 | 설명 |
|------|------|
| **Local Attack** | 공격 대상 시스템에서 직접 실행 |
| **Remote Attack** | 네트워크를 통해 다른 컴퓨터를 공격 |

### Trojan vs Exploit ⭐⭐

| | Trojan | Exploit |
|--|--------|---------|
| 정의 | 정상처럼 보이는 악성 프로그램 | 취약점을 실제로 공격하는 코드/프로그램 |
| 비유 | "겉은 선물, 속은 폭탄" | "자물쇠 따는 도구" |
| 특징 | 원격 조종, 정보 탈취 | 로컬/원격 취약점 공격 |

## 📝 시험 직전 초압축 정리

### 공격 유형 한눈에

```
수동적 공격 = 도청(스니핑)     탐지 어려움
능동적 공격 = 변조/실행         탐지 쉬움

DoS       = 한 대로 공격
DDoS      = 좀비 여러 대로 공격
DRDoS     = 정상 서버를 반사체로 이용

SYN Flood  = Half-Open 대량 생성 → 연결 테이블 고갈
SMURF      = 브로드캐스트 + 피해자 IP 위조 → ICMP 폭탄
LAND       = 출발지IP = 목적지IP
Teardrop   = Offset 겹치게 → 재조립 오류
Tiny Frag  = 방화벽 우회 (DoS 아님!)
```

### 스캔 종류 핵심

```
Sweep        = 살아 있는 호스트 확인
Full Open    = 끝까지 연결 → 로그 남음
Half Open    = SYN만 → 로그 거의 없음
FIN/NULL/XMAS = 플래그 조작 → 스텔스
ACK 스캔     = 방화벽 정책 파악
```

### 스푸핑 vs 하이재킹

```
ARP 스푸핑   = ARP Cache 오염 → MITM
ICMP Redirect = 라우팅 테이블 변조 → MITM
IP 스푸핑    = 신뢰 관계 악용
DNS 스푸핑   = 가짜 DNS 응답 → 피싱

세션 하이재킹 = 인증된 세션 탈취
              최우선 방어 = 암호화
```

### 핵심 방어 키워드

| 공격 | 방어 |
|------|------|
| SYN Flood | SYN Cookie, 임계치 설정 |
| SMURF | `no ip directed-broadcast` |
| ARP 스푸핑 | 정적(static) ARP 설정 |
| IP 스푸핑 | IPSec, 패킷 필터링 |
| DNS 스푸핑 | `/etc/hosts` 직접 등록 |
| 스니핑 | 암호화 (SSL, SSH, VPN) |
| 세션 하이재킹 | 암호화 + 지속 인증 |
