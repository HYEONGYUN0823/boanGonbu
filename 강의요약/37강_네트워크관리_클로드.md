## 2. SNMP 개념

### SNMP란?
**Simple Network Management Protocol** — 네트워크 장비를 모니터링/제어하는 프로토콜

- 동작 계층: **응용 계층 (L7)**
- 전송 프로토콜: **UDP**
- 포트: **161** (요청/응답), **162** (TRAP 알림)

### 핵심 구조: 관리자 ↔ 에이전트

```
[관리자 (NMS)]
     ↓ GET (읽기 요청)
     ↓ SET (변경 명령)
[에이전트 (라우터, 서버 등)]
     ↓ RESPONSE (응답)
     ↓ TRAP (문제 발생 시 자동 보고)
[관리자 (NMS)]
```

| 역할 | 설명 |
|------|------|
| **관리자** | NMS 소프트웨어가 돌아가는 호스트. 에이전트를 제어/감시 |
| **에이전트** | 라우터, 서버 등 실제 장비. DB에 성능 정보 저장 |

### 동작 방식 2가지

| 방식 | 설명 | 비유 |
|------|------|------|
| **폴링 (Polling)** | 관리자가 주기적으로 에이전트에 상태 질의 | 선생님이 매 시간 출석 체크 |
| **트랩 (Trap)** | 에이전트에서 이벤트 발생 시 관리자에게 인터럽트 | 학생이 문제 생기면 먼저 연락 |

### SNMP 기반 도구

- **NMS (Network Management System)**: 네트워크 자원 모니터링/제어 도구
- **MRTG**: SNMP 기반 장비 트래픽 모니터링 프로그램

## 3. SNMP 구성요소 (SMI / MIB / SNMP)

인터넷 네트워크 관리는 **SMI + MIB + SNMP** 세 가지의 협동으로 이루어진다.

### 역할 비교

| 구성요소 | 역할 | 프로그래밍 비유 |
|----------|------|----------------|
| **SMI** | 공통 규칙/표현 방식 정의 (타입 정의는 안 함) | 프로그래밍 언어 문법 |
| **MIB** | SMI 규칙에 따라 객체에 이름+타입 연결, DB 형태 | 변수 선언 및 정의 |
| **SNMP** | 실제로 객체(변수)의 값을 읽고(GET) 변경(SET) | 프로그램 코드 실행 |

### 흐름 요약

```
SMI: "데이터는 이 형식으로 표현해라"
        ↓
MIB: "장비 상태 항목들을 이름+타입으로 DB에 저장"
        ↓
SNMP: "MIB에서 값 읽어오고 (GET) / 바꾸고 (SET)"
```

## 4. SNMP PDU (명령 종류)

PDU = **Protocol Data Unit** — 실제 주고받는 메시지 패킷 단위

### 8가지 PDU 유형

| PDU 이름 | 방향 | 설명 |
|----------|------|------|
| **GetRequest** | 관리자 → 에이전트 | 특정 값 읽기 요청 |
| **GetNextRequest** | 관리자 → 에이전트 | 다음 항목 값 읽기 |
| **GetBulkRequest** | 관리자 → 에이전트 | 여러 값 한 번에 읽기 |
| **SetRequest** | 관리자 → 에이전트 | 값 변경 요청 |
| **Response** | 에이전트 → 관리자 | 요청에 대한 응답 |
| **Trap** | 에이전트 → 관리자 | 이벤트 발생 자동 알림 (인터럽트) |
| **InformRequest** | 관리자 → 관리자 | 확인 응답 포함 알림 |
| **Report** | - | 오류 보고 |

### 분류 요약

```
읽기형  : GetRequest, GetNextRequest, GetBulkRequest
쓰기형  : SetRequest
응답형  : Response
통보형  : Trap (SNMPv1), InformRequest (v2/v3)
```

## 5. SNMP 메시지 구조 & 커뮤니티 스트링

### 메시지 구조

```
[ UDP 헤더 ]
    ↓
[ SNMP 메시지 ]
    ├── 버전 (v1 / v2 / v3)
    ├── 커뮤니티 스트링 (≈ 인증 패스워드)
    └── 데이터 (PDU)
```

### 커뮤니티 스트링

| 스트링 | 기본값 | 권한 |
|--------|--------|------|
| **읽기 전용** | `public` | GET만 가능 |
| **읽기/쓰기** | `private` | GET + SET 가능 |

> ⚠️ **보안 문제**: v1, v2에서 커뮤니티 스트링이 **평문(cleartext)** 으로 전송됨 → 도청 위험

### 버전별 보안 비교

| 버전 | 커뮤니티 스트링 | 암호화 | 인증 | 무결성 |
|------|----------------|--------|------|--------|
| **v1** | 평문 | ❌ | ❌ | ❌ |
| **v2** | 평문 | ❌ | ❌ | ❌ |
| **v3** | 암호화 | ✅ | ✅ | ✅ |

> 💡 기업에서 기본 커뮤니티 스트링(`public`, `private`)을 변경하지 않는 경우가 많아 **사회공학 공격**에 노출될 수 있음

## 6. 원격접속 프로토콜 개요

| 프로토콜 | 포트 | 암호화 | 설명 |

| `FTP` (파일 전송) | `sftp` |

## 7. TELNET

### 개요

- **TErminaL NETwork**의 약자
- TCP **23번 포트** 사용
- 원격 컴퓨터에 **터미널처럼 접속** 가능

### 동작 흐름

```
[내 PC]
  키보드 입력
      ↓
  Telnet Client
      ↓
  TCP/IP
      ↓
  네트워크 (인터넷)
      ↓
  Telnet Server
      ↓
  가상 터미널 드라이버 (Pseudo Terminal Driver)
      ↓
  서버에서 명령 실행 → 결과 반환
```

> 💡 **가상 터미널(Pseudo Terminal)**: 원격에서 접속한 사용자를 마치 로컬 터미널처럼 처리해주는 소프트웨어

### 문제점

> ❌ **암호화 없음** → 아이디, 패스워드, 명령어 모두 **평문 전송** → 도청 취약

## 8. SSH (Secure Shell)

### 개요

- TCP **22번 포트** 사용
- Telnet, FTP, rlogin, rsh 등 **평문 프로토콜을 대체**하기 위해 설계
- **암호화 + 인증 + 무결성** 모두 제공

### Telnet과 비교

| 항목 | Telnet | SSH |
|------|--------|-----|
| 포트 | 23 | 22 |
| 암호화 | ❌ | ✅ |
| 인증 | 평문 PW | 공개키/패스워드 |
| 보안 | 위험 | 안전 |

### SSH 버전

| 버전 | 특이사항 |
|------|---------|
| **SSH-1** | 초기 버전, 보안 취약점 존재, RSA 사용 |
| **SSH-2** | 취약점 개선, SSH-1과 **호환 불가** |
| **OpenSSH** | 오픈소스, SSH-1/SSH-2 모두 지원 |

### SSH 구성 요소 (컴포넌트)

```
Application Layer
┌──────────────────────────────────────────────┐
│  SSH-TRANS   →  SSH-AUTH  →  SSH-CONN        │
│  (암호화 채널)   (상호인증)    (다중화)           │
│                                              │
│         SSH Application (실제 서비스)          │
│   원격로그인(rlogin), 파일전송(SFTP), 포워딩 등  │
└──────────────────────────────────────────────┘
Transport Layer: TCP
```

| 컴포넌트 | 역할 |
|----------|------|
| **SSH-TRANS** | 안전한 암호화 채널 생성 |
| **SSH-AUTH** | 서버 ↔ 클라이언트 **상호 인증** |
| **SSH-CONN** | 하나의 연결에서 **여러 채널 다중화** |
| **SSH-Application** | 원격 로그인, SFTP, 포트 포워딩 등 |

### SSH 주요 기능

- ✅ **인증** (공개키 / 패스워드)
- ✅ **기밀성** (암호화)
- ✅ **무결성** (MAC)
- ✅ **압축**
- ✅ **포트 포워딩** (일종의 터널링)
- ✅ **다중화** (하나의 SSH 연결로 여러 서비스 동시 사용)

---

### 다중화(Multiplexing)란?

> 하나의 SSH 연결(TCP 커넥션)로 **여러 논리 채널을 동시에** 사용하는 것

**예시:**
```
SSH 연결 1개
    ├── 채널 1: 터미널 세션
    ├── 채널 2: SFTP 파일 전송
    └── 채널 3: 포트 포워딩
```
→ 연결을 매번 새로 맺지 않아도 됨 (효율적)

---

### 터널링 / 포트 포워딩이란?

> **터널링**: 암호화된 통로(SSH)를 통해 다른 프로토콜의 트래픽을 안전하게 전달

```
[내 PC] ──SSH 터널──> [SSH 서버] ──> [내부 DB 서버]
  암호화된 구간         복호화 후 전달
```

**포트 포워딩 예시:**
- 외부에서 직접 접근 불가한 내부 서버의 DB 포트를 SSH 터널을 통해 안전하게 접속

## 9. SSH 암호화 통신 원리

### 실제 동작 방식 (핵심)

SSH는 **공개키 암호화**로 키를 안전하게 교환한 뒤, 실제 통신은 **대칭키**로 빠르게 암호화한다.

```
1단계: 서버 공개키 수신
   클라이언트 ──요청──> 서버
   클라이언트 <──서버 공개키── 서버

2단계: 세션 키(대칭키) 생성 및 교환
   클라이언트: 임시 세션 키 생성
   클라이언트: 세션 키를 서버 공개키로 암호화하여 전송
   서버: 자신의 개인키로 복호화 → 세션 키 획득

3단계: 대칭키 암호화로 통신
   이후 통신은 공유된 세션 키로 빠르게 암호화
```

> 💡 **왜 이렇게 하나?**  
> - 공개키 암호화: 느리지만 키 교환에 안전  
> - 대칭키 암호화: 빠르지만 키 공유가 문제  
> → 둘의 장점을 조합한 **하이브리드 암호화 방식**

## 10. 핵심 요약 (시험용)

### 🎯 SNMP

| 항목 | 내용 |
|------|------|
| 동작 계층 | 응용 계층 (L7) |
| 전송 | UDP |
| 포트 | 161 (요청/응답), 162 (Trap) |
| 구성 | 관리자 ↔ 에이전트 |
| 주요 PDU | GET, SET, TRAP |
| v1/v2 | 커뮤니티 스트링 평문 전송 |
| v3 | 암호화 + 인증 + 무결성 |

### 🎯 원격접속

| 항목 | Telnet | SSH |
|------|--------|-----|
| 포트 | TCP 23 | TCP 22 |
| 암호화 | ❌ | ✅ |
| 파일전송 대응 | FTP | SFTP |

### 🎯 SMI / MIB / SNMP 역할

```
SMI  → 규칙 정의 (문법)
MIB  → 데이터 저장 (DB/사전)
SNMP → 실제 통신 (프로그램)
```

### 🎯 SSH 기능 키워드

`암호화` | `인증` | `무결성(MAC)` | `포트포워딩` | `다중화` | `압축`
