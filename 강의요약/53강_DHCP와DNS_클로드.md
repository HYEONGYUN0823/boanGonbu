## 1. 호스트 설정의 개념

컴퓨터가 네트워크에서 통신하려면 아래 **4가지 정보**가 반드시 필요하다.

| 항목 | 설명 |
|------|------|
| ① 컴퓨터의 IP 주소 | 네트워크상 내 주소 |
| ② 서브넷 마스크 | 같은 네트워크인지 판별 |
| ③ 라우터(게이트웨이) IP | 다른 네트워크로 나가는 문 |
| ④ 네임서버(DNS) IP | 도메인 이름 → IP 변환 서버 |

---

## 2. 호스트 설정 프로토콜 종류

### RARP (Reverse ARP)

- **역할**: 물리주소(MAC) → IP 주소 매핑
- ARP는 IP→MAC 이고, RARP는 MAC→IP

> ⚠️ **오늘날 거의 사용하지 않는 이유 (2가지)**
> 1. RARP 서버가 **각 네트워크마다 별도로 존재**해야 함 → 관리 불편
> 2. **IP 주소만 제공** → 서브넷 마스크, 게이트웨이, DNS는 못 줌

---

### BOOTP (Bootstrap Protocol)

- 클라이언트/서버 방식
- 위 4가지 정보를 **모두 제공** 가능
- BOOTP 서버는 인터넷 **어디에나** 위치 가능
- **정적(Static)** 설정 프로토콜 → "이 MAC이면 항상 이 IP"처럼 미리 지정

---

### DHCP (Dynamic Host Configuration Protocol)

- BOOTP의 **후계자** (BOOTP와 역방향 호환성 있음)
- **정적 + 동적** 할당 모두 지원
- 현재 가장 많이 쓰는 방식

---

## 3. DHCP 상세

### 기본 정보

| 항목 | 내용 |
|------|------|
| 계층 | 응용 계층 (L7) |
| 전송 프로토콜 | **UDP** |
| 서버 포트 | **67번** |
| 클라이언트 포트 | **68번** |
| 역할 | TCP/IP의 네트워크 계층(IP)을 보조 |

### 장단점

| 구분 | 내용 |
|------|------|
| ✅ 장점 | 네트워크 구조 변경 시 서버 설정만 바꾸면 클라이언트 자동 반영 |
| ❌ 단점 | 부팅 시 **브로드캐스트** 사용 → 네트워크 트래픽 증가 |
| ❌ 단점 | 전원만 켜져 있어도 IP 할당됨 → 보안상 취약 가능 |

## 4. DHCP 동작 흐름

### 메시지 4단계

```
클라이언트                      서버
    |                              |
    |--- DHCPDISCOVER ----------->|  (브로드캐스트: "IP 주세요!")
    |                              |
    |<-- DHCPOFFER ---------------|  (서버 제안: "이 IP 어때요?")
    |                              |
    |--- DHCPREQUEST ------------>|  (클라이언트 선택: "그 IP 쓸게요")
    |                              |
    |<-- DHCPACK / DHCPNACK ------|  (최종 승인 or 거절)
    |                              |
```

### 각 메시지 설명

| 메시지 | 방향 | 설명 |
|--------|------|------|
| **DHCPDISCOVER** | 클 → 서버 | 브로드캐스트로 DHCP 서버 탐색. 트랜잭션 ID는 임의 숫자 |
| **DHCPOFFER** | 서버 → 클 | 서버가 IP 제안 |
| **DHCPREQUEST** | 클 → 서버 | 클라이언트가 원하는 서버의 제안을 선택 |
| **DHCPACK** | 서버 → 클 | 요청 IP 유효 → 최종 승인 |
| **DHCPNACK** | 서버 → 클 | 요청 IP 무효 → 거절 |

> 💡 **비유**: 편의점 자리 배정
> - DISCOVER: "빈자리 있어요?"
> - OFFER: "여기 자리 있어요!" (여러 직원이 응답)
> - REQUEST: "그중 이쪽 자리로 할게요"
> - ACK: "알겠어요, 사용하세요!"

## 5. DHCP Starvation 공격

### 개념

DHCP 서버가 할당 가능한 **IP를 모두 소진**시켜서, 정상 사용자가 IP를 받지 못하게 하는 공격

### 원리

```
공격자
  ↓
가짜 MAC 주소를 수백~수천 개 생성
  ↓
DHCPDISCOVER 대량 전송 (서로 다른 MAC으로)
  ↓
서버가 각각 DHCPOFFER 전송
  ↓
DHCPREQUEST까지 보냄 (실제로는 사용 안 함)
  ↓
IP 풀 고갈 → 정상 사용자 IP 발급 불가
```

> 💡 **비유**: 가짜 손님 수백 명이 와서 좌석을 모두 예약만 하고 안 앉는 것

## 6. DNS 개요 및 발전 흐름

### DNS란?

- **D**omain **N**ame **S**ystem
- 도메인 이름 ↔ IP 주소를 변환해주는 시스템
- 예: `www.google.com` → `142.250.x.x`

### 발전 흐름

| 단계 | 방법 | 특징 |
|------|------|------|
| 1단계 | 메모에 직접 기록 | 소규모에서만 가능 |
| 2단계 | `hosts` 파일 (로컬 저장) | 변경 시 즉시 반영 어려움 |
| 3단계 | **DNS** | 즉시 반영, 인터넷 규모에 적합 |

### DNS 필요성

- 인터넷 규모가 커지면서 hosts 파일만으로는 관리 불가
- 이름 → 주소, 주소 → 이름 변환 자동화 필요

## 7. DNS 구조 (네임 공간 / Zone / RR)

### 계층 구조

```
.  (루트)
├── com
│   ├── google
│   │   └── www
│   └── naver
├── kr
│   └── co
│       └── naver
└── net
```

### 핵심 용어

| 용어 | 설명 |
|------|------|
| **도메인** | 이름 공간의 단위. 서브도메인으로 분할 가능 |
| **네임 공간** | 이름들의 전체 계층 구조 |
| **Zone** | DNS 서버가 책임지고 관리하는 영역 |
| **Zone 파일** | Zone 데이터를 저장한 실제 파일 |
| **Zone Transfer** | 마스터 → 슬레이브로 Zone 데이터 복사 |

### 마스터 / 슬레이브 서버

| 구분 | 설명 |
|------|------|
| **마스터(Master)** | 원본 Zone 데이터 보유 |
| **슬레이브(Slave)** | 마스터에서 복사한 Zone 데이터 보유 |
| **Zone Transfer** | 마스터 → 슬레이브 동기화 작업. **TCP 53번** 포트 사용 ⚠️ (UDP 아님!) |

> ⚠️ Zone Transfer는 **TCP 53**번 포트를 사용한다. (UDP 53은 일반 DNS 질의에 사용)

### 루트 서버

- DNS 계층의 최상단
- 도메인 정보를 직접 갖고 있지 않고, 하위 서버로 **참조(길 안내)**만 함
- 전 세계 **13개** 존재 (2002년 DDoS 공격 발생)
- 전 세계 분산 배치

### DNS RR (Resource Record) 유형

| RR 코드 | 유형 | 설명 |
|---------|------|------|
| **A** | 주소 | 도메인 → IPv4 주소 (32bit) |
| **AAAA** | 주소 | 도메인 → IPv6 주소 (128bit = 32×4) |
| **NS** | 네임서버 | Zone을 담당하는 권한 있는 네임서버 |
| **SOA** | 권한 개시 정보 | Zone의 시작 정보 (기본 정보 포함) |
| **PTR** | 포인터 | IP → 도메인명 (역방향 조회) |
| **MX** | 메일 교환 | 도메인의 메일 서버 위치 |
| **TXT** | 문자열 | 문자 데이터. **DNS 증폭 DRDoS 공격에 악용** ⚠️ |
| **ANY** | 모든 레코드 | 모든 레코드 요청. **DNS 증폭 DRDoS 공격에 악용** ⚠️ |
| **AXFR** | Full Zone Transfer | 버전 무관하게 전체 Zone 전송 요청 |
| **IXFR** | Incremental Zone Transfer | 버전 비교 후 변경분만 전송 요청 |

## 8. DNS 동작 방식

### 이름 해석 순서 (우선순위)

```
1순위: 캐시 (이전에 조회한 결과)
  ↓ 없으면
2순위: /etc/hosts 파일 (로컬 파일)
  ↓ 없으면
3순위: DNS 서버 질의
```

> ⚠️ **보안 주의**: `hosts` 파일이 파밍(Pharming) 공격으로 변조되면 정상 URL 입력해도 악성 사이트로 이동할 수 있음

### hosts.ics 파일 (Windows)

- 인터넷 연결 공유(ICS) 사용 시 클라이언트 IP를 강제 지정하는 설정 파일
- **hosts 파일보다 DNS 질의 검색 우선순위가 높음** ← 중요

### 재귀적 해석 (Recursive)

```
클라이언트 → DNS 서버에 질문
              DNS 서버가 대신 찾아줌
              (모르면 직접 다른 서버에 물어봄)
              클라이언트는 한 번만 물으면 됨
```

> 💡 "내가 끝까지 찾아서 답 줄게"

### 반복적 해석 (Iterative)

```
클라이언트 → DNS 서버에 질문
              DNS 서버: "나는 모름, 저쪽에 물어봐"
              클라이언트 → 다음 서버에 직접 질문
              반복...
```

> 💡 "길만 알려줄게, 직접 찾아가"

### DNS 서버 유형

| 유형 | 설명 |
|------|------|
| **Recursive DNS 서버** | 클라이언트 대신 재귀적으로 끝까지 찾아주는 서버 |
| **Authoritative DNS 서버** | 특정 Zone을 직접 관리하며, 해당 도메인 질의에만 응답 |

## 9. DNS 이름 해석 과정 예시

`www.example.com`을 찾는 경우

```
① 사용자가 www.example.com 입력
      ↓
② 내 PC의 DNS 캐시 확인
      ↓ 없으면
③ /etc/hosts 파일 확인
      ↓ 없으면
④ Recursive DNS 서버에 질의
      ↓
⑤ Recursive DNS → 루트 서버 질의
      ↓
⑥ 루트 서버: ".com 담당 TLD 서버 주소 알려줌"
      ↓
⑦ .com TLD 서버: "example.com 담당 서버 알려줌"
      ↓
⑧ example.com Authoritative DNS 서버: "최종 IP 응답"
      ↓
⑨ 클라이언트가 해당 IP로 접속
```

## 10. DNS 캐싱

### 개념

한 번 조회한 결과를 **TTL(Time To Live)** 동안 저장해두는 것

| 구분 | 설명 |
|------|------|
| **양성 캐싱 (Positive)** | 조회 성공한 결과를 저장 |
| **음성 캐싱 (Negative)** | 조회 실패(없는 도메인)한 결과도 저장 |

### 장단점

| 구분 | 내용 |
|------|------|
| ✅ 장점 | 속도 향상, 서버 부하 감소 |
| ❌ 단점 | TTL이 길면 변경된 정보가 반영 안 됨 → 잘못된 매핑 가능 |

### 캐시 관련 명령어 (Windows)

```cmd
ipconfig /displaydns   # 캐시된 DNS 정보 확인
ipconfig /flushdns     # 캐시된 DNS 정보 삭제
```

## 11. DNS Lookup 도구

| 도구 | 설명 |
|------|------|
| **nslookup** | 가장 널리 쓰이는 DNS 진단 도구. 도메인/IP 조회 |
| **dig** | nslookup보다 풍부한 정보 제공. 실무/공부용으로 많이 사용 |
| **host** | 단순 조회 시 편리 |
| **whois** | 도메인 등록 정보(소유자, 네트워크 할당 등) 조회 |

### Lookup 유형

| 유형 | 방향 | 설명 |
|------|------|------|
| **Forward DNS Lookup** | 도메인 → IP | 일반 순방향 조회 |
| **Reverse DNS Lookup** | IP → 도메인 | 역방향 조회 (PTR 레코드 사용) |

## 12. DNS 보안 위협

### DNS Spoofing (스푸핑)

가짜 DNS 응답을 보내 사용자를 악성 사이트로 유도하는 공격

#### ① 스니핑 기반 DNS Spoofing

```
희생자가 DNS 질의 전송
      ↓
공격자가 이를 스니핑 (엿보기)
      ↓
정상 DNS 응답보다 먼저 가짜 응답 전송 (시간차 공격)
      ↓
희생자가 정상 URL 입력해도 조작된 IP로 접속
```

> 💡 대응: 중요한 사이트 IP는 **hosts 파일에 직접 등록**해 관리

#### ② DNS Cache Poisoning (캐시 포이즈닝)

```
DNS 서버의 캐시 자체를 조작
      ↓
이후 해당 도메인 질의 시 오염된 IP 응답
      ↓
클라이언트가 악성 사이트로 이동
```

> ⚠️ **매우 중요**

**대응책**

| 대응 방법 | 설명 |
|-----------|------|
| Recursive Query 제한 | Authoritative 서버는 재귀 질의 허용하지 않음. Recursive 서버는 허가된 사용자만 허용 |
| **DNSSEC** 사용 | 전자서명으로 메시지 무결성 및 송신자 인증 |
| Transaction ID + 포트 무작위화 | 예측 어렵게 만들어 위조 방지 |

---

### DNSSEC (DNS Security Extensions)

| 항목 | 내용 |
|------|------|
| **제공하는 것** | 메시지 송신자 인증, 전자서명을 통한 메시지 무결성 |
| **제공하지 않는 것** | 기밀성 (DNS 메시지 암호화 X) |
| **제공하지 않는 것** | 서비스 거부(DoS) 공격 방어 |

---

### DNS Amplification (증폭) DRDoS 공격

- **TXT, ANY** 레코드를 악용
- 출발지 IP를 희생자로 위조해서 DNS 서버에 질의
- DNS 서버가 큰 응답을 희생자에게 전송 → **트래픽 증폭**

## 13. DNS 관련 설정 파일 (Linux)

### `/etc/host.conf` — 질의 순서 설정

```
order hosts, bind
```

| 설정 | 의미 |
|------|------|
| `hosts` | `/etc/hosts` 파일 먼저 검색 |
| `bind` | 그다음 DNS 서버(named) 질의 |

---

### `/etc/resolv.conf` — 사용할 DNS 서버 지정

```
search sulinux.net
nameserver 192.168.0.250
nameserver 168.126.63.1
```

| 항목 | 설명 |
|------|------|
| `search` | 도메인 검색 시 자동 붙여쓸 도메인 |
| `nameserver` | 사용할 DNS 서버 IP. 위에서부터 순서대로 시도 |

---

### `/etc/named.conf` — BIND DNS 서버 핵심 설정

```
options {
    directory "/var/named";  // zone 파일 위치
};

zone "algisa.com" {
    type master;                   // 마스터 서버
    file "algisa.com.db";          // zone 파일명
    #allow-transfer { none; };     // Master만 운영 시: 전송 차단
    allow-transfer { 192.168.20.1; };  // Slave 운영 시: 해당 IP만 허용
};

zone "20.168.192.in-addr.arpa" {  // 역방향 조회용 도메인
    type master;
    file "192.168.20.rev";
};
```

| 항목 | 설명 |
|------|------|
| `directory` | zone 파일이 위치한 디렉터리 |
| `type master` | 마스터 네임서버 |
| `file` | 해당 zone의 데이터 파일명 |
| `allow-transfer` | Zone Transfer 허용 IP 지정 |

---

### 역방향 도메인 (Reverse Domain)

IP → 도메인명 조회를 위한 특수 도메인

```
일반 조회: example.com → IP
역방향 조회: 192.168.0.10 → 도메인명
```

형태: IP 주소를 **역순**으로 쓴 후 `.in-addr.arpa`를 붙임

```
192.168.20.0/24 네트워크
→ 20.168.192.in-addr.arpa
```

---

### IP 주소 얻기까지의 내부 흐름

```
hosts.ics 파일 확인 (최우선, Windows ICS 환경)
      ↓
캐시 확인
      ↓
/etc/hosts 파일 확인
      ↓
/etc/resolv.conf에 지정된 DNS 서버 질의
      ↓
Recursive 해석 → Authoritative 서버 → 최종 IP 응답
```

## 14. 시험 핵심 암기 요약

### DHCP 요약

| 항목 | 내용 |
|------|------|
| 포트 | UDP **67** (서버) / **68** (클라이언트) |
| 계층 | 응용 계층 (L7) |
| 동작 순서 | **D**ISCOVER → **O**FFER → **R**EQUEST → **A**CK (DORA) |
| 공격 | Starvation: 가짜 MAC으로 IP 풀 소진 |

### DNS 요약

| 항목 | 내용 |
|------|------|
| 기능 | 도메인 ↔ IP 변환 |
| 포트 | UDP 53 (일반 질의), TCP 53 (Zone Transfer) |
| 해석 방식 | Recursive (서버가 끝까지), Iterative (길만 알려줌) |
| 캐시 확인 | `ipconfig /displaydns` |
| 캐시 삭제 | `ipconfig /flushdns` |
| 루트 서버 | 전 세계 13개 |
| Zone Transfer | TCP 53 사용 (UDP 아님!) |

### DNS RR 핵심만

| RR | 역할 |
|----|------|
| A | 도메인 → IPv4 |
| AAAA | 도메인 → IPv6 |
| PTR | IP → 도메인 (역방향) |
| MX | 메일 서버 |
| NS | 권한 있는 네임서버 |
| SOA | Zone 시작 정보 |
| TXT / ANY | DRDoS 증폭 공격 악용 ⚠️ |
| AXFR | Full Zone Transfer |
| IXFR | Incremental Zone Transfer |

### DNS 보안 요약

| 공격/기술 | 핵심 |
|-----------|------|
| DNS Spoofing (스니핑 기반) | 정상 응답보다 빠르게 가짜 응답 전송 (시간차 공격) |
| DNS Cache Poisoning | 서버 캐시 자체를 오염 |
| DNSSEC | 전자서명으로 무결성 보장. 기밀성 X, DoS 방어 X |
| DHCP Starvation | 가짜 MAC으로 IP 풀 고갈 |

---

> 📎 **참고**: zone 파일은 `/etc/named.conf`의 `directory` 지시자에 설정된 경로에 위치해야 함
