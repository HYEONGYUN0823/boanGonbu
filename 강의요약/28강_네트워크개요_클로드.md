## 1. 프로토콜

**한 마디로 : 서로 대화하기 위한 약속**

사람끼리 대화할 때도 규칙이 있다. 한국어로 말하기로 했으면 둘 다 한국어를 써야 하고, "안녕"이라고 하면 "안녕"이라고 받아야 한다. 컴퓨터도 마찬가지다. 서로 데이터를 주고받으려면 미리 약속된 규칙이 필요한데, 그게 프로토콜이다.

### 프로토콜의 3요소

| 요소 | 설명 | 비유 |
|---|---|---|
| **구문 (Syntax)** | 데이터를 어떤 구조로 포장할지 | 택배 박스 앞면엔 받는 사람, 뒷면엔 보내는 사람 |
| **의미 (Semantics)** | 각 비트가 무슨 뜻인지 | 앞 8비트는 오류 체크용, 다음 16비트는 데이터 길이 |
| **타이밍 (Timing)** | 언제, 얼마나 빠르게 보낼지 | 송신자 100Mbps인데 수신자가 1Mbps면 조절 필요 |

### 흐름 제어가 필요한 이유

```
송신자 [100Mbps] ──────────▶ 수신자 [1Mbps]
      물 호스 (굵음)              컵 (작음)
```

수신자 버퍼(임시 저장 공간)가 넘쳐서 데이터가 날아간다.  
그래서 수신자가 "좀 천천히 보내"라고 제어 신호를 보내는 게 흐름 제어다.

### 물리적 측면 vs 논리적 측면

- **물리적 (하위 계층)** : 0과 1을 전기 신호, 빛, 전파로 어떻게 보낼지. 케이블 종류, 핀 배치, 전압 등
- **논리적 (상위 계층)** : "이 데이터는 이메일이야", "오류났으면 재전송해" 같은 약속

### 관련 프로토콜

- **RS-232C** : 컴퓨터와 외부 장치(모뎀, 프린터 등)를 연결하는 물리 계층 표준. 지금의 USB 규격 같은 것
- **V.21** : 전화선으로 데이터를 보낼 때 쓰던 모뎀 프로토콜 (속도 300bps)

## 2. OSI 모델

> ⚠️ 프로토콜이 아니라 **모델(설계도, 참조 기준)** 이다.  
> 실제 구현은 TCP/IP 같은 프로토콜이 하고, OSI는 "이렇게 나눠서 생각하자"는 개념 틀이다.

**OSI = Open Systems Interconnection (개방 시스템 상호 연결)**

예전엔 삼성 컴퓨터는 삼성끼리, IBM은 IBM끼리만 통신됐다. ISO(국제표준화기구)가 "제조사 상관없이 다 연결될 수 있게 표준 모델 만들자"해서 나온 게 OSI다.

### 7계층 구조

```
7  응용 계층    (Application)   ┐
6  표현 계층    (Presentation)  ├─ 사용자 지원 / 추상적
5  세션 계층    (Session)       ┘
──────────────────────────────── 종단 시스템에서만 지원
4  전송 계층    (Transport)     ── 경계선 (종단간 제어)
────────────────────────────────
3  네트워크 계층 (Network)      ┐
2  데이터링크 계층 (Data Link)  ├─ 물리적인 면
1  물리 계층    (Physical)      ┘
```

### 종단 시스템에서만 지원한다는 의미

```
[내 PC] ──── [라우터] ──── [라우터] ──── [서버]
 1~7계층       1~3계층       1~3계층      1~7계층
```

중간 라우터는 1~3계층만 본다. 경로 찾아서 넘겨주면 끝이다.  
4~7계층은 출발지(내 PC)랑 목적지(서버)만 처리한다.

> 중간 택배 기사는 주소만 보고 배달한다. 편지 내용은 몰라도 되고, 보낸 사람과 받는 사람만 내용을 읽는 것과 같다.

### OSI vs TCP/IP 비교

```
OSI 7계층            TCP/IP 4계층
─────────────        ──────────────────────
7. 응용              ┐
6. 표현              ├── 응용 계층 (하나로 합침)
5. 세션              ┘
─────────────        ──────────────────────
4. 전송              ── 전송 계층 (TCP, UDP)
─────────────        ──────────────────────
3. 네트워크          ── 인터넷 계층 (IP)
─────────────        ──────────────────────
2. 데이터링크        ┐
                     ├── 네트워크 접근 계층
1. 물리              ┘
```

**핵심 차이**

1. TCP/IP가 먼저 나왔다. OSI는 나중에 "이론 정리용"으로 만든 것. 실제 인터넷은 TCP/IP를 쓴다.
2. OSI 5, 6계층은 TCP/IP에 추가되지 않았다. TCP/IP는 응용 계층 하나로 합쳤다.

### 프로토콜 계층화 = 모듈화

복잡한 통신을 역할별로 쪼개면 다음과 같은 장점이 있다.

- 1계층 고장나도 7계층 코드 안 건드려도 됨
- 계층별로 따로 개발, 따로 업그레이드 가능
- 레고 블록처럼 조립식

## 3. OSI 7계층 상세

택배 보내는 과정으로 비유한다.

### 1계층 — 물리 계층 (Physical)

**전기 신호로 0, 1 보내는 것**

- 케이블, 전압, 핀 배치 같은 진짜 물리적인 것
- 단위 : **비트 (0, 1)**

> 택배로 치면 도로, 트럭 자체

### 2계층 — 데이터링크 계층 (Data Link)

**같은 네트워크 안에서 옆집까지 안전하게**

- **LLC** : 흐름제어, 오류제어 (논리적 담당)
- **MAC** : 물리적 주소(랜카드에 박힌 고유번호)로 누구한테 보낼지
- 단위 : **프레임**

```
[ MAC목적지 | MAC출발지 | 데이터 | 오류체크 ]
```

> 택배로 치면 같은 동네 배달 (아파트 동 호수)

### 3계층 — 네트워크 계층 (Network)

**서울에서 부산까지 경로 찾아가기**

- IP 주소로 목적지를 찾음
- 데이터가 크면 **분할(단편화)** 해서 보냄
- **라우터**가 여기서 동작
- 단위 : **패킷**

| 개념 | 설명 |
|---|---|
| **NAT** | 사설 IP를 공인 IP로 바꿔주는 것 (집 공유기 역할) |
| **ARP** | IP 주소로 MAC 주소를 찾는 것. "192.168.0.5 누구야?" 하고 방송하면 해당 기기가 MAC 주소로 대답 |
| **ICMP** | 오류 메시지 전달용. "목적지 없어", "시간 초과" 등 알림. ping이 ICMP를 사용 |
| **RIP, BGP** | 라우터들이 "나는 이쪽 경로가 빨라" 정보를 교환하는 라우팅 프로토콜 |

> 택배로 치면 전국 물류망, GPS 경로 찾기

### 4계층 — 전송 계층 (Transport)

**앱 ↔ 앱 사이 종단간 연결**

단순히 컴퓨터 → 컴퓨터가 아니라, **크롬 ↔ 웹서버 프로세스** 이렇게 앱끼리 연결한다.

| 프로토콜 | 특징 | 용도 |
|---|---|---|
| **TCP** | 느리지만 안전. 순서 보장, 재전송 | 카톡, 이메일, 파일 다운로드 |
| **UDP** | 빠르지만 불안전. 그냥 쏨 | 유튜브 스트리밍, 게임, 음성통화 |

- 단위 : **세그먼트**

> 택배로 치면 "3동 202호 홍길동 씨 손에 직접 전달"

### 5계층 — 세션 계층 (Session)

**대화 연결을 열고 닫는 것**

- 로그인 ~ 로그아웃 사이를 **세션**이라고 함
- 연결 확립 → 유지 → 동기화 → 종료

> 택배로 치면 "배달원이 문 두드리고, 사인받고, 인수 완료 처리"

### 6계층 — 표현 계층 (Presentation)

**데이터 포맷 변환**

- 암호화/복호화 (HTTPS)
- 압축/해제
- 독립적인 프로토콜은 없음 (다른 계층에 붙어서 동작)

> 택배로 치면 "내용물 포장 방식 (진공포장, 압축포장)"

### 7계층 — 응용 계층 (Application)

**사용자가 직접 쓰는 것**

| 프로토콜 | 용도 |
|---|---|
| HTTP | 웹 브라우저 |
| SMTP | 이메일 |
| FTP | 파일 전송 |
| DNS | 도메인 이름 → IP 주소 변환 |
| TELNET | 원격 접속 |

> 택배로 치면 "쇼핑몰 앱에서 주문 버튼 누르는 것"

### 전체 흐름 예시 (구글 접속)

```
응용      → "google.com 줘" (HTTP)
전송      → TCP로 연결 맺음
네트워크  → ARP로 MAC 찾고, IP로 경로 결정
데이터링크 → 프레임에 포장
물리      → 전기신호로 전송
```

아래로 내려가면서 **포장(캡슐화)** 하고, 구글 서버에서 위로 올라가면서 **뜯는(역캡슐화)** 것이다.

## 4. 캡슐화와 역캡슐화

### 캡슐화 = 택배 포장

데이터가 아래 계층으로 내려갈수록 포장지가 하나씩 더 씌워진다.

```
응용       →  [데이터]
전송       →  [TCP헤더 | 데이터]
네트워크   →  [IP헤더  | TCP헤더 | 데이터]
데이터링크 →  [MAC헤더 | IP헤더  | TCP헤더 | 데이터 | 꼬리]
물리       →  전기신호 (010101011...)
```

> 물건 → 뽁뽁이 → 박스 → 송장 붙이기 → 트럭에 싣기

### 역캡슐화 = 택배 뜯기

도착지에서 위 계층으로 올라갈수록 포장지를 하나씩 벗겨낸다.

```
물리       →  전기신호 받음
데이터링크 →  MAC헤더 벗김  → "나한테 온 거 맞네"
네트워크   →  IP헤더 벗김   → "목적지 IP 맞네"
전송       →  TCP헤더 벗김  → "80번 포트, 웹이네"
응용       →  [데이터] 최종 도착!
```

### 장비별 캡슐화 처리

```
내 PC      →  캡슐화만     (보낼 때 포장)
구글 서버  →  역캡슐화만   (받을 때 뜯기)
라우터     →  역캡슐화 + 캡슐화  (뜯고 → 다시 포장)
스위치     →  아무것도 안함 (그대로 통과)
```

### 라우터가 둘 다 하는 이유

```
[내 PC] ──── wifi ──── [라우터] ──── 광케이블 ──── [구글 서버]
```

라우터는 중간 경유지라 네트워크가 바뀔 수 있다.

1. 들어온 프레임 뜯음 (역캡슐화) → MAC 헤더 제거
2. IP 주소 확인 → "다음은 어디로?"
3. 새 MAC 헤더 씌움 (캡슐화) → 다음 구간용으로
4. 내보냄

> 중간 물류센터에서 박스 뜯고 → 다음 목적지 송장으로 다시 포장하는 것

### 스위치는 캡슐화가 없는 이유

스위치는 MAC 주소만 슬쩍 보고 **그대로 넘겨버린다**.

```
프레임 들어옴
   ↓
"MAC 주소가 AA:BB:CC네 → 3번 포트로 넘겨"
   ↓
프레임 그대로 전달 (건드리지 않음)
```

> 아파트 1층 무인택배함 — 박스 안 뜯고 그냥 호실별로 넣어두는 것

## 5. 헤더

**한 마디로 : 데이터 앞에 붙는 송장**

```
[ 헤더 | 진짜 데이터 ]
  ↑
"이거 어디서 왔고, 어디로 가고, 어떻게 처리해" 정보가 담김
```

### 계층별 헤더 내용

| 계층 | 헤더에 담긴 것 |
|---|---|
| 전송 계층 | 출발지/목적지 포트 번호, TCP/UDP 여부 |
| 네트워크 계층 | 출발지/목적지 IP 주소 |
| 데이터링크 계층 | 출발지/목적지 MAC 주소 |

### 포장되는 과정

```
          [데이터]
     [TCP헤더][데이터]
  [IP헤더][TCP헤더][데이터]
[MAC헤더][IP헤더][TCP헤더][데이터]
```

양파처럼 계층마다 헤더가 하나씩 껴입는다.

## 6. 스위치와 라우터

### 스위치 = 각 동 전담 택배기사

**같은 네트워크 안에서** 기기들 사이를 연결한다.

```
A동 택배기사(스위치)
  ├── 101호 📦
  ├── 202호 📦
  └── 303호 📦
```

- MAC 주소만 보고 어느 포트로 보낼지 결정
- 관련 계층 : **1, 2계층**
- B동 일은 신경 안 씀

### 라우터 = 단지 입구 경비아저씨

**서로 다른 네트워크 사이**를 연결한다.

```
외부 택배 도착 → 라우터 → "A동이면 A동 스위치한테"
                        → "B동이면 B동 스위치한테"
                        → "다른 단지면 밖으로"
```

- IP 주소를 보고 경로를 결정
- 관련 계층 : **1, 2, 3계층**
- 집 공유기가 사실 라우터

### 실제 인터넷 흐름

```
내 PC
  ↓
[스위치] ← 공유기에 연결된 기기들 사이 정리
  ↓
[라우터] ← 집 → 인터넷으로 내보냄
  ↓
[라우터] ← 통신사 라우터들이 경로 찾아줌
  ↓
[스위치] ← 서버 건물 내부 정리
  ↓
목적지 서버
```

### 계층 수 비교

```
[스위치]       [라우터]        [내 PC / 서버]
1계층 ✅       1계층 ✅         1계층 ✅
2계층 ✅       2계층 ✅         2계층 ✅
              3계층 ✅         3계층 ✅
                               4계층 ✅
                               5계층 ✅
                               6계층 ✅
                               7계층 ✅
```

할 일이 많을수록 계층이 많다.

## 7. 공인 IP와 사설 IP

### 비유

```
아파트 주소 : 서울시 강남구 테헤란로 123  ← 공인 IP
동호수      : 101호, 102호, 103호...       ← 사설 IP
```

### 공인 IP

- 인터넷에서 실제로 쓰이는 주소
- 전 세계에서 딱 하나만 존재
- KT, SK, LG 같은 통신사가 부여

### 사설 IP

- 집/회사 내부에서만 쓰는 주소
- 공유기 안에서만 통하는 주소
- 밖에서는 보이지 않음

```
192.168.0.1   ← 공유기 자체
192.168.0.2   ← 내 PC
192.168.0.3   ← 폰
192.168.0.4   ← 노트북
```

### NAT (Network Address Translation)

공유기가 내부 사설 IP를 공인 IP 하나로 대표해서 인터넷에 내보내는 것

```
내 PC (192.168.0.2)
      ↓
   공유기 (NAT)  ← 여기서 변환!
      ↓
 인터넷 (공인IP 하나로 나감)
```

### 왜 이렇게 나눴냐

IPv4 기준 IP 주소가 약 **43억 개**밖에 없는데, 전 세계 기기 수가 그보다 훨씬 많기 때문이다.  
공인 IP 1개로 집 안 기기 여러 개를 연결해서 아껴쓴다.

## 8. 홉 (Hop)

**라우터 하나 거칠 때마다 1홉**

```
내 PC → [라우터A] → [라우터B] → [라우터C] → 목적지 서버
          1홉          2홉          3홉
```

### TTL (Time To Live)과의 관계

패킷에 TTL 값을 설정해서 보내면, 라우터 하나 지날 때마다 TTL이 1씩 깎인다.  
TTL이 0이 되면 패킷을 버린다.

```
TTL = 64로 출발
라우터A 통과 → TTL 63
라우터B 통과 → TTL 62
...
TTL = 0 → 패킷 버림
```

TTL이 없으면 길 잃은 패킷이 인터넷에서 영원히 떠돌아 다닌다.

## 9. 논리 주소와 물리 주소

### 변하는 것 vs 안 변하는 것

| | 출발 | 라우터A | 라우터B | 도착 |
|---|---|---|---|---|
| **IP 주소 (논리)** | 내IP→목적지IP | 내IP→목적지IP | 내IP→목적지IP | 내IP→목적지IP |
| **MAC 주소 (물리)** | 내MAC→라우터A | 라우터A→라우터B | 라우터B→목적지MAC | - |

- **IP 주소** = 최종 목적지 주소. 처음부터 끝까지 고정
- **MAC 주소** = 다음 한 칸의 주소. 라우터 지날 때마다 교체

> IP = 최종 받는 사람 주소 (안 바뀜)  
> MAC = 다음 물류센터 주소 (계속 바뀜)

## 10. 포트 주소

### 왜 필요하냐

```
컴퓨터 A가 동시에
  → 컴퓨터 B랑 FTP로 파일 받는 중
  → 컴퓨터 C랑 TELNET으로 원격 접속 중
```

데이터가 A한테 도착했을 때 "이거 FTP꺼야, TELNET꺼야?" 구분이 필요하다.

### 비유

```
IP 주소   = 아파트 주소 (어느 건물)
포트 번호 = 호실 번호   (몇 호실)
```

### 잘 알려진 포트 번호

| 포트 | 프로토콜 | 용도 |
|---|---|---|
| 21 | FTP | 파일 전송 |
| 23 | TELNET | 원격 접속 |
| 25 | SMTP | 이메일 |
| 80 | HTTP | 웹 |
| 443 | HTTPS | 보안 웹 |

### 전체 주소 개념

```
IP 주소   →  어느 컴퓨터?
포트 번호 →  그 컴퓨터의 어느 프로그램?

컴퓨터 A (IP: 192.168.0.2)
  ├── 포트 21  ←  FTP 프로그램
  ├── 포트 23  ←  TELNET 프로그램
  └── 포트 80  ←  웹 브라우저
```

## 11. 다중화와 역다중화

### 다중화 = 고속도로 진입로

```
국도A ↘
국도B → [진입로] → 고속도로 (하나로 합쳐짐)
국도C ↗
```

여러 앱에서 온 데이터를 **포트 번호 붙여서** 하나의 통신선으로 내보내는 것

### 역다중화 = 고속도로 출구

```
고속도로 → [출구] → 국도A
                  → 국도B
                  → 국도C
```

도착한 데이터를 **포트 번호 보고** 맞는 앱에 나눠주는 것

### 전체 흐름

```
[송신측 - 다중화]
FTP    → [포트21 붙임] ↘
TELNET → [포트23 붙임] → 전송
HTTP   → [포트80 붙임] ↗

[수신측 - 역다중화]
              ↘ 포트21이네 → FTP한테
전송 도착 →   → 포트23이네 → TELNET한테
              ↗ 포트80이네 → HTTP한테
```

포트 번호가 다중화/역다중화의 핵심 열쇠다.
