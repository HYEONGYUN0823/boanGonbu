# 🔑 비대칭키 암호 (공개키 암호)

---

## 📌 먼저 읽기! 자주 나오는 기호 설명

| 기호 | 의미 |
|---|---|
| P | 평문 (Plaintext) = 원본 데이터 |
| C | 암호문 (Ciphertext) = 암호화된 데이터 |
| n, e, d | RSA에서 사용하는 키 관련 숫자 |
| p, q | RSA에서 사용하는 소수 (비밀!) |
| φ(N) | 오일러 피 함수 = (p-1)(q-1) |
| mod | 나머지 연산 (예: 7 mod 3 = 1) |

> 💡 **mod(나머지 연산)이 뭔데?**
> 7 ÷ 3 = 2 나머지 1
> 그래서 7 mod 3 = 1
> 암호학에서 엄청 자주 나오니까 꼭 알아둬!

## 1. 키 배송 문제 ⭐⭐⭐⭐⭐

> 💡 **비유 : 자물쇠와 열쇠 문제**
> 앨리스가 밥에게 비밀 편지를 보내려면
> 자물쇠(암호화 키)를 먼저 안전하게 전달해야 해
> 근데 자물쇠를 전달하는 과정에서 도둑(공격자)이 훔쳐갈 수 있잖아!
> 이게 바로 **키 배송 문제**야

대칭키 암호를 쓰려면 반드시 키를 상대방에게 전달해야 하는데,
이 과정에서 키가 탈취될 수 있는 문제!

### 키 배송 문제 해결 방법 4가지

```
1. 키의 사전 공유
2. 키배포 센터(KDC)
3. Diffie-Hellman 키 교환
4. 공개키 암호

❌ 전자서명으로는 키 배송 문제를 해결할 수 없음! (시험 주의!)
```

---

### 1) 키의 사전 공유 ⭐⭐⭐

> 💡 **비유 : 만나기 전에 미리 암호 약속하기**
> 앨리스와 밥이 직접 만나서 "우리 앞으로 이 암호 쓰자!" 하고 미리 정해두는 것

- 비밀 경로를 통해 미리 키를 전달
- 키 : K(A,B) = K(B,A) (서로 같은 키 공유)

**문제점** : 사람이 많아지면 키 개수가 폭발적으로 증가!

```
필요한 키 개수 = n(n-1)/2

예) 사람이 10명이면 → 10 × 9 / 2 = 45개의 키 필요
    사람이 100명이면 → 100 × 99 / 2 = 4,950개!
```

---

### 2) 키배포 센터(KDC) ⭐⭐⭐⭐⭐

> 💡 **비유 : 열쇠 전문 가게**
> 앨리스와 밥이 직접 키를 교환하지 않고
> 신뢰할 수 있는 열쇠 가게(KDC)에서 그때그때 키를 받아오는 것

**동작 방식**
```
1. KDC가 모든 사용자의 키를 DB에 보관
2. 앨리스가 밥과 통신하고 싶으면 KDC에 요청
3. KDC가 세션 키(K) 생성해서 앨리스와 밥에게 전달
4. 앨리스와 밥이 세션 키로 통신
```

**단점**
```
❌ KDC가 고장나면 → 전체 시스템 마비
❌ KDC가 공격자의 주요 공격 대상이 됨
```

---

### 3) Diffie-Hellman 키 교환 ⭐⭐⭐⭐⭐

> 💡 **비유 : 물감 섞기**
> 앨리스와 밥이 각자 비밀 물감(개인 비밀값)을 가지고 있어
> 공개된 물감(공개값)에 자기 비밀 물감을 섞어서 상대방에게 보내
> 상대방도 자기 비밀 물감을 섞으면 → 둘 다 같은 색이 나옴!
> 중간에 도둑이 봐도 물감만 봐서는 원래 비밀 물감을 알 수 없어!

**핵심 특징**
- 공개키 암호방식을 **최초**로 제안 (디피, 헬먼)
- KDC 없이 두 사람이 직접 대칭키를 만들어냄
- **키 합의 프로토콜** (키 교환이 아니라 함께 만드는 것!)

**사용하는 값**
```
공개 (도청자도 알 수 있음) : p (큰 소수), g (원시근), R1, R2
비밀 (절대 공개 안 함)    : x, y, k (최종 공유 키)
```

**수식**
```
앨리스 : R1 = g^x mod p  (x는 앨리스의 비밀값)
밥     : R2 = g^y mod p  (y는 밥의 비밀값)

공유 키 K = R2^x mod p  (앨리스가 계산)
         = R1^y mod p  (밥이 계산)
         → 둘 다 같은 값이 나옴!
```

**왜 안전하냐?**
```
→ 쉬운 방향 : g^A mod p = 결과값  (계산하기 쉬움)
← 어려운 방향 : 결과값으로 A를 역으로 구하기 = 이산대수 문제 (매우 어려움!)
```

#### 타원 곡선 Diffie-Hellman (ECDH)

- 이산대수 문제를 **타원곡선 위**에서 푸는 방식
- 더 **짧은 키**로 같은 보안성 제공
- 스마트폰 등 자원이 적은 환경에 적합

---

### Diffie-Hellman 취약점과 대응 ⭐⭐⭐⭐⭐

#### 중간자 공격 (Man-in-the-Middle Attack)

> 💡 **비유 : 편지 중간에서 바꿔치기**
> 앨리스가 밥에게 편지를 보내는데
> 중간에 이브가 가로채서 자기 공개키로 바꿔치기!
> 앨리스는 밥과 통신한다고 생각하지만 실제론 이브와 통신 중

```
앨리스 ←→ 이브(중간자) ←→ 밥
(앨리스는 밥이랑 통신한다고 착각)
```

**왜 취약하냐?** → 인증 단계가 없어서!

**대응 방법**
```
✅ 전자서명 (무결성 + 인증 + 부인방지)
✅ 공개키 인증서 (신분증 역할)
✅ 국-대-국 프로토콜 (STS) : 전자서명 + 인증서 사용
```

#### DoS 공격 대응
```
✅ 쿠키(Cookie) 사용 → 키 생성 요청자 확인
```

**실제 사용처**
```
IPSec (VPN), SSL/TLS (웹 보안)
```

### 4) 공개키 암호에 의한 해결 ⭐⭐⭐⭐⭐

> 💡 **비유 : 우체통**
> 밥이 자기 우체통(공개키)을 길거리에 공개해둠
> 누구든지 편지(데이터)를 넣을 수 있어 (암호화)
> 하지만 열쇠(개인키)는 밥만 가지고 있어서 밥만 꺼낼 수 있음 (복호화)
> 도둑이 우체통을 봐도 안에 내용을 꺼낼 수 없어!

```
1. 밥이 공개키를 앨리스에게 공개 (도청자가 알아도 괜찮음!)
2. 앨리스가 공개키로 암호화해서 밥에게 전송
3. 밥만 개인키로 복호화 가능
→ 개인키를 전송할 필요가 없어서 키 배송 문제 해결!
```

## 2. 공개키 암호 기본 개념

### 공개키 사용 원칙 ⭐⭐⭐⭐⭐

```
1. 암호화키와 복호화키는 반드시 같은 사람의 키쌍이어야 함
   예) 밥의 공개키로 암호화 → 밥의 개인키로만 복호화 가능

2. 키는 암호화 또는 복호화 중 한 번만 사용

3. 타인의 개인키는 절대 사용 불가
```

### 공개키 알고리즘 분류

```
            공개키 알고리즘
      ┌──────────┴──────────┐
  소인수분해 문제        이산대수 문제
   ┌────┴────┐        ┌─────┼──────┐
  RSA      Rabin   ElGamal  DSA   ECC
                     ↑      ↑      ↑
                   (DH확장)(서명전용)(짧은키)
```

### 공개키 암호 역사
```
1976년 : 디피, 헬먼이 공개키 암호 아이디어 발표
1978년 : RSA 발표 (Rivest, Shamir, Adleman)
현재   : RSA = 공개키 암호의 사실 표준 (De Facto Standard)
```

### 공개키 암호에 대한 잘못된 상식 ⭐⭐⭐

```
❌ 잘못된 생각 1 : 공개키가 대칭키보다 더 안전하다
→ 꼭 그렇지 않음! 용도가 다를 뿐

❌ 잘못된 생각 2 : 공개키가 대중화되면 대칭키는 안 쓴다
→ 공개키는 느려서 대칭키를 대체할 수 없음
→ 둘은 상호보완 관계!

❌ 잘못된 생각 3 : 공개키는 중앙 관리 기관이 필요 없다
→ 공개키 저장소 같은 중앙 에이전트가 필요할 수 있음
```

## 3. 대칭키 vs 공개키 비교 ⭐⭐⭐

| 항목 | 대칭키 | 공개키 |
|---|---|---|
| 키 관계 | 암호화키 = 복호화키 | 암호화키 ≠ 복호화키 |
| 안전한 키 길이 | 128bit 이상 | 2048bit 이상 |
| 암호화키 | 비밀 | **공개** |
| 복호화키 | 비밀 | 비밀 |
| 키 개수 | **n(n-1)/2** | 2n (개인키+공개키) |
| 속도 | **빠름** | 느림 |
| 키 배송 | 필요 (문제 있음) | **불필요** |
| 주요 용도 | 데이터 암호화 | 키 교환, 전자서명 |
| 전자서명 | 복잡 | **간단** |
| 경제성 | 높다 | 낮다 |
| 제공 서비스 | 기밀성 | 기밀성, 인증, 부인방지 |
| 대표 알고리즘 | DES, AES, SEED | RSA, ECC, DSA |

> 💡 **핵심 요약**
> 대칭키 = 빠르지만 키 전달이 문제
> 공개키 = 느리지만 키 전달 문제 해결
> → 그래서 둘을 합친 하이브리드 방식을 주로 사용!

## 4. RSA 암호시스템 ⭐⭐⭐⭐⭐

> 💡 **비유 : 엄청 큰 수 쪼개기 퀴즈**
> 15 = 3 × 5  → 쉽게 풀 수 있어
> 하지만 300자리 숫자 = ? × ?  → 슈퍼컴퓨터도 수백 년 걸림!
> RSA는 이 **소인수분해의 어려움**을 이용한 암호야

**사용처** : 웹 브라우저(SSL), PGP(이메일), 전자서명, 정부 시스템

### RSA 키 생성 ⭐⭐⭐⭐⭐

> 💡 **비유 : 자물쇠 만들기**
> 1. 재료 고르기 (p, q 선택)
> 2. 자물쇠 몸통 만들기 (N = p × q)
> 3. 공개 자물쇠 번호 정하기 (e 선택)
> 4. 비밀 열쇠 만들기 (d 계산)

```
[1단계] 서로 다른 두 소수 p, q 선택 (p ≠ q)
        → 비밀로 보관!

[2단계] N = p × q 계산
        → 공개해도 됨

[3단계] φ(N) = (p-1)(q-1) 계산
        → 비밀로 보관!

[4단계] e 선택 : 1 < e < φ(N), gcd(φ(N), e) = 1
        → 공개해도 됨 (공개키의 일부)

[5단계] d 계산 : d × e ≡ 1 (mod φ(N))
        → 비밀로 보관! (개인키의 일부)
```

**결과**
```
공개키 (모두에게 알려줌) : {e, n}
개인키 (절대 비밀)       : {d, n}
비밀값 (절대 비밀)       : p, q, φ(N)
```

### RSA 암호화 / 복호화

```
암호화 (공개키 e, n 사용)
C = P^e mod n

복호화 (개인키 d, n 사용)
P = C^d mod n
```

> 💡 **왜 안전하냐?**
> 공개키 n, e를 알아도 개인키 d를 구하려면
> n을 소인수분해(p × q로 분리)해야 하는데
> n이 수백 자리 숫자면 현재 컴퓨터로 불가능!

### RSA 키 길이 권장사항

```
1024bit : ❌ 신규 사용 금지
2048bit : ✅ 2030년까지 사용 가능
4096bit : ✅ 2031년 이후 권장
```

### RSA 시간 복잡도 (k = 키 길이)

```
암호화   : k^2
복호화   : k^3
키 생성  : k^4
→ 키가 길어질수록 엄청 느려짐!
```

### RSA 권장사항 ⭐⭐⭐

```
✅ p와 q는 거의 같은 크기의 소수여야 함
✅ p-1 과 q-1은 각각 큰 소인수를 가져야 함
✅ p-1 과 q-1의 최대공약수는 작아야 함
✅ 메시지는 OAEP 패딩을 사용해야 함
```

## 5. RSA 공격 방법 ⭐⭐⭐

### 수학적 공격 (소인수분해 공격)
```
n을 p × q로 분해하면 RSA 해독 가능
→ 소인수분해 알고리즘이 개발되지 않는 한 RSA는 안전
```

### 타이밍 공격 (시간 공격)
```
복호화 시간을 측정해서 개인키를 추측하는 공격
대응 : 랜덤 지체(Random Delay) 추가
```

### Factoring Attack (인수분해 공격)
```
n을 구성하는 p, q 숫자들의 인수분해를 풀어내서 키를 찾는 시도
→ 큰 소수를 사용할수록 방어 가능
```

### 선택 암호문 공격 (CCA) + OAEP

> 💡 **비유 : 은행 직원을 속여서 정보 빼내기**
> 공격자가 가짜 암호문을 계속 만들어서 서버에 보냄
> 서버의 오류 메시지나 응답 시간을 분석해서
> 조금씩 키나 평문 정보를 알아내는 공격

**대응 : OAEP (최적 비대칭키 암호 패딩)**
```
평문 앞에 해시값 + 랜덤값을 붙여서 암호화
→ 같은 평문이라도 매번 다른 암호문이 생성됨
→ 패턴 분석 불가능!
```

## 6. Rabin 암호시스템

```
RSA의 변형
RSA  : 지수 합동에 근거
Rabin : 2차 합동에 근거

특징 : 성능이 낮은 기기에서 활용 가능
보안 : 소인수분해 문제에 기반
```

## 7. ElGamal 방식 ⭐⭐⭐

```
이산대수 문제에 기반
Diffie-Hellman의 확장판

사용 가능한 곳
✅ 디지털 서명
✅ 암호화
✅ 키 교환
```

**단점**
```
❌ 암호문 길이 = 평문의 약 2배
❌ 메모리 많이 필요
❌ 전송 속도 느림
```

## 8. 타원 곡선 암호 (ECC) ⭐⭐⭐⭐⭐

> 💡 **비유 : 다이어트한 RSA**
> RSA랑 비슷한 보안성인데 키 크기가 훨씬 작음
> 스마트폰, 스마트카드처럼 자원이 적은 곳에 딱!

**핵심 특징**
```
✅ 짧은 키로 높은 보안성
✅ 빠른 속도
✅ 적은 메모리 사용

예) ECC 256bit = RSA 4096bit 수준의 보안성!
```

**수식**
```
타원곡선 방정식 : y² = x³ + ax + b
조건 : 4a³ + 27b² ≠ 0 (이 조건을 만족해야 타원곡선!)
```

**두 점의 덧셈**
```
1. 점 P와 Q를 직선으로 연결
2. 그 직선이 곡선과 만나는 점을 찾음
3. 그 점을 x축 대칭으로 뒤집음
4. 그 점이 R = P + Q
```

**ECDLP 문제 (타원곡선 이산대수 문제)**
```
Q = kP 에서 k를 구하는 문제
(Q와 P는 알고 있지만 k를 역으로 구하는 건 매우 어려움!)
→ 이 어려움을 이용해서 암호화
```

## 9. 하이브리드 암호 시스템

> 💡 **비유 : 햄버거 배달**
> 주문서(대칭키)를 안전한 봉투(공개키 암호화)에 넣어서 전달
> 봉투를 열면 주문서(대칭키)가 나오고
> 그 주문서로 실제 햄버거(데이터)를 주문(복호화)!
>
> 즉, 공개키로 대칭키를 안전하게 전달하고
> 실제 데이터는 빠른 대칭키로 암호화!

**왜 필요하냐?**
```
공개키 암호의 문제점
(a) 대칭키보다 훨씬 느림
(b) 중간자 공격에 취약

하이브리드 = (a) 문제 해결!
(b) 문제는 인증서로 해결
```

**동작 방식**
```
1. 앨리스가 세션키(대칭키) 생성
2. 세션키를 밥의 공개키로 암호화해서 전송  ← 공개키 암호 사용
3. 밥이 개인키로 세션키 복호화
4. 이후 실제 데이터는 세션키(대칭키)로 암호화  ← 대칭키 암호 사용
```

```
큰 데이터  → 대칭키(세션키)로 암호화  (빠름!)
세션키 전달 → 공개키로 암호화         (안전!)
```

**실제 사용 예**
```
PGP     : 이메일 암호화
SSL/TLS : 웹 HTTPS 통신
```

**키 설계 원칙**
```
대칭키(세션키) < 공개키
→ 공개키를 더 강하게 설계!
  (공개키는 오래 사용, 세션키는 한 번만 사용하고 버림)
```

## 📌 최종 핵심 암기 요약

```
✅ 키 배송 문제 해결 방법 4가지
   1. 키 사전 공유
   2. KDC (키배포 센터)
   3. Diffie-Hellman 키 교환
   4. 공개키 암호
   ❌ 전자서명으로는 해결 안 됨!

✅ Diffie-Hellman
   = KDC 없이 두 사람이 함께 대칭키를 만드는 방법
   = 이산대수 문제 기반
   = 중간자 공격에 취약 → 전자서명 + 인증서로 해결

✅ RSA
   = 소인수분해 문제 기반
   = 공개키 {e, n} / 개인키 {d, n}
   = C = P^e mod n (암호화)
   = P = C^d mod n (복호화)
   = 2048bit 이상 권장

✅ ECC
   = 타원곡선 이산대수 문제 기반
   = 짧은 키로 높은 보안성
   = 모바일/IoT 환경에 적합
   = ECC 256bit ≈ RSA 4096bit

✅ 하이브리드 암호
   = 대칭키(빠름) + 공개키(안전) 조합
   = 세션키를 공개키로 암호화해서 전달
   = PGP, SSL/TLS에서 사용
```
