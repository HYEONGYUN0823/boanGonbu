# 🔐 인증 (Authentication) & SSO

---

## 📌 먼저 읽기! 핵심 개념 한눈에

| 인증 유형 | 방식 | 예시 |
|---|---|---|
| Type 1 (지식) | Something you **know** | 패스워드, PIN |
| Type 2 (소유) | Something you **have** | 토큰, 스마트카드, OTP |
| Type 3 (존재) | Something you **are** | 지문, 홍채, 얼굴 |
| Type 4 (행위) | Something you **do** | 서명, 움직임 |
| Type 5 (위치) | Somewhere you **are** | 사용자 IP, 콜백 |

> 💡 **Two Factor vs Multi Factor**
> Two Factor = 타입이 **다른** 2가지 조합 (예: PIN + 카드)
> Multi Factor = 3가지 이상 조합 (예: PIN + 카드 + 지문)

## 1. 메시지 인증

```
메시지 인증 = 전달되는 메시지의 이상 유무를 확인하는 기능

확인 대상
✅ 메시지 내용 변경 여부
✅ 메시지 순서 변경 여부
✅ 메시지 삭제 여부
```

**메시지 인증 방식**
```
1. 메시지 암호화 방식 (관용 암호 / 공개키 암호 인증모드)
2. MAC 방식 ← 가장 많이 사용
3. 해시 함수 방식
```

> 💡 **소극적 공격 vs 적극적 공격**
> 암호화 → 소극적 공격(도청) 방어
> 메시지 인증 → 적극적 공격(위조, 변조) 방어

## 2. 개인 식별 vs 사용자 인증

| 구분 | 사용자 인증 | 개인 식별 |
|---|---|---|
| 수준 | 기본 | 발전된 방식 |
| 제3자 가장 | **가능** (B가 D에게 A로 가장 가능) | **불가능** |

> 💡 **개인 식별이 더 강한 이유**
> 사용자 인증 : 밥이 다른 사람 다비드에게 "나 앨리스야"라고 속일 수 있음
> 개인 식별   : 이런 가장 자체가 불가능하도록 설계됨

## 3. 사용자 인증 유형 ⭐⭐⭐⭐⭐

| 유형 | 설명 | 예시 | 공격 방법 |
|---|---|---|---|
| **Type 1** (지식) | Something you **know** | 패스워드, PIN | 추측 |
| **Type 2** (소유) | Something you **have** | 토큰, 스마트카드, OTP | 강탈, 도난 |
| **Type 3** (존재) | Something you **are** | 지문, 홍채, 얼굴 | 도려냄 😱 |
| (행위) | Something you **do** | 서명, 움직임 | - |
| **Two Factor** | 다른 타입 2가지 조합 | 토큰 + PIN | - |
| **Multi Factor** | 3가지 이상 조합 | 토큰 + PIN + 지문 | - |

**4 Factor 예시**
```
PIN (Type 1) + 카드 (Type 2) + 홍채 (Type 3) + 위치 (Type 5)
```

**ATM 예시**
```
PIN (Type 1) + ATM 카드 (Type 2) = Two Factor 인증
```

## 4. 지식 기반 인증 (Type 1) ⭐⭐⭐⭐⭐

> 💡 **보안성은 비밀번호의 크기와 랜덤성에 달려있음!**

### 장단점

| 장점 | 단점 |
|---|---|
| 다양한 분야 사용 가능 | 망각 가능 |
| 검증 확실성 | 추측 공격 가능 |
| 관리 비용 저렴 | 사회공학 공격에 취약 |
| 편리함 | 부주의한 관리 |

### 패스워드 저장 방법 발전 과정 ⭐⭐⭐⭐⭐

```
1단계 : 평문 저장
        → 파일에 ID/PW를 평문으로 저장
        → ❌ 도청, 파일 탈취로 즉시 노출

2단계 : 해시 저장
        → 패스워드의 해시값을 저장
        → ❌ 사전 공격 가능 (자주 쓰는 비번 해시값 미리 계산해둠)

3단계 : 솔팅(Salting) ← 현재 사용 방식 ⭐⭐⭐⭐⭐
        → 패스워드 + 랜덤 솔트 → 해시
        → ✅ 사전 공격 / 레인보우 테이블 공격 차단
        → UNIX : /etc/shadow 파일의 $id$salt$encrypted-password
```

**솔트의 기능**
```
✅ 오프라인 사전 공격을 획기적으로 어렵게 만듦
✅ 여러 시스템에서 같은 비밀번호를 쓰는 사람도 해시값이 달라짐
   → 한 곳이 털려도 다른 곳은 안전
```

### 패스워드 안전성 수식

$$P = L \times \frac{P}{S}$$

```
P : 패스워드 추측(공격) 확률
L : 패스워드 사용 기간 (짧을수록 안전)
S : 패스워드 길이 (길수록 안전)
R : 사용 빈도 (낮을수록 안전)

→ 길고, 자주 바꾸고, 덜 사용할수록 안전!
```

### 패스워드 보안 정책 ⭐⭐⭐

```
✅ 로그인 실패 횟수 임계치 설정 (브루트포스 방어)
✅ 일주일~한 달 이상 미사용 계정은 사용 불가 처리
✅ 더 이상 안 쓰는 계정은 삭제
✅ 주기적 패스워드 변경 요구 (Password Aging)
```

## 5. 일회용 패스워드 (OTP) ⭐⭐⭐⭐⭐

> 💡 **비유 : 매번 바뀌는 햄버거 쿠폰 번호**
> 오늘 쿠폰 번호는 1234, 내일은 5678
> 어제 번호를 훔쳐봐도 오늘은 못 씀!

```
= 동적 패스워드 (Dynamic Password)
✅ 재전송 공격 (Replay Attack) 방어
✅ 사전공격, 재생공격, 스니핑에 가장 안전한 인증방법
⚠️ 전용기기 필요 → Type 2로 분류
```

### OTP 생성 방식 3가지

**1) 질의응답 방식 (Challenge-Response = 비동기식)**

| 항목 | 내용 |
|---|---|
| 특징 | 은행이 질의값(난수)을 주면 사용자가 OTP 생성 |
| 장점 | 인증서버와 동기화 불필요 |
| 단점 | 질의값 직접 입력 → 번거로움, 서버 관리 필요 |

**2) 시간/이벤트 동기화 방식 (동기식)**

| 구분 | 시간 동기화 | 이벤트 동기화 |
|---|---|---|
| 기준값 | 시간(Timestep) | 계수기(Counter) |
| 대표 예 | RSA Security의 **SecurID** | 버튼 누르면 다음 값 생성 |
| 장점 | 질의값 입력 불필요 | 시간 동기화 덜 필요 |
| 단점 | 시간 동기화 필수 | 계수기 동기화 필수 |

**3) S/Key 방식 (Leslie Lamport) ⭐⭐⭐**

> 💡 **비유 : 해시 체인으로 만든 번호표**
> 비밀키를 n번 해시해서 n개의 OTP를 미리 만들어 서버에 저장
> 사용할 때마다 역순으로 하나씩 꺼내 씀

```
벨 통신 연구소 개발 / 유닉스 계열 OS에서 사용

동작 방식
1. 비밀키를 seed로 해시 체인 방식으로 n번 반복
2. 생성된 n개의 OTP를 서버에 저장
3. 사용할 때마다 역순으로 사용

단점
⚠️ 별도 소프트웨어 설치 필요
⚠️ n번 다 쓰면 재설정 필요 (보통 1000번부터 시작)
```

## 6. 시도-응답 개인 식별 프로토콜 ⭐⭐⭐

> 💡 **비유 : 암호 대화**
> 서버 : "사과" (시도, challenge)
> 사용자 : "바나나" (응답, response)
> → 비밀 정보를 직접 보내지 않고 알고 있다는 것만 증명!

```
기반 : 대칭키 암호 또는 공개키 암호(인증모드)
목적 : 비밀 정보를 간접적으로 증명 (직접 전송 X)
```

### 시도-응답 방식 비교

| 방식 | 기반 암호 | 특징 |
|---|---|---|
| 대칭키 기반 | 대칭키 | 공유 비밀키 사용 |
| 공개키 기반 | 공개키(인증모드) | 개인키로 서명 |

**상호 개인 식별 프로토콜**
```
일반 인증 : 서버가 클라이언트 신원 확인
상호 인증 : 서버 ↔ 클라이언트 서로 신원 확인
           (클라이언트도 서버가 진짜인지 확인!)
```

## 7. 영지식 개인 식별 프로토콜

> 💡 **비유 : 색맹 테스트**
> "나는 빨강과 초록을 구분할 수 있어"를 증명하는데
> 실제로 색을 알려주지 않고 계속 맞추면서 증명하는 것!
> 비밀(색 구분 능력)은 절대 노출 안 함

```
= Zero-Knowledge Identification Protocol

핵심 : 비밀 정보를 서버에 제공하지 않고 신분 증명!
증명자(Prover) = 클라이언트
검증자(Verifier) = 서버
```

**영지식 증명의 3가지 조건**
```
1. 완전성 : 진짜면 항상 성공 (정당한 증명자는 항상 통과)
2. 건전성 : 가짜면 항상 실패 (가짜는 통과 불가)
3. 영지식성 : 비밀은 절대 안 샘 (검증자도 비밀 정보 못 앎)
```

**동굴 예** (Quisquater & Guillou 고안)
```
동굴 안에 문이 있고, 문을 열 수 있는 사람이 증명자
증명자는 문을 여는 법(비밀)을 알려주지 않고
"나 문 열 수 있어"를 반복해서 증명
```

## 8. 소유 기반 인증 (Type 2)

> 💡 **주의 : 단독으로 쓰지 않음!**
> 도난/복제가 가능해서 반드시 Type 1 또는 Type 3과 함께 사용

### 메모리 카드 vs 스마트카드

| 구분 | 메모리 카드 | 스마트카드 |
|---|---|---|
| 저장 | ✅ | ✅ |
| 처리(연산) | ❌ | ✅ |
| 예시 | 마그네틱 은행카드 | IC칩 내장 카드 |
| 특이사항 | 값비싼 카드리더기 필요 | 부당변경 방지 기능 필요 |

**스마트카드 인증 흐름**
```
단말기              리더기
CA 개인키로 서명  →  CA 공개키로 검증
```

### 소유기반 장단점

| 장점 | 단점 |
|---|---|
| 입증된 기술 | 소유물 없으면 인증 불가 |
| 생체인식보다 경제적 | 복제 가능 |
| 일반적 사용 | 자산 관리 필요 |
| - | 사회공학 공격 가능 |

## 9. 생체 인증 (Type 3) ⭐⭐⭐⭐⭐

> 💡 **비유 : 나 자신이 열쇠**
> 잊어버리거나 잃어버릴 수도 없음
> 하지만 한번 유출되면 바꿀 수 없다는 게 치명적 단점!

### 생체 특징 분류

```
생체적(생리학적) 특징 : 지문, 얼굴, 손 모양, 홍채, 망막, 정맥
행동적 특징           : 서명, 키보드 입력, 음성
```

### 생체인증 기술 평가항목 ⭐⭐⭐⭐⭐

| 특성 | 설명 |
|---|---|
| 보편성 | 모든 사람이 가지고 있는가? |
| 유일성 | 동일한 특징을 가진 타인이 없는가? |
| 지속성 | 시간에 따른 **변화가 없는가**? |
| 획득성 | **정량적 측정**이 가능한가? |
| 성능 | 환경변화와 무관하게 높은 정확성? |
| 수용성 | 사용자 거부감이 없는가? (홍채/망막 = 거부감 큼) |
| 반기만성 | 고의적 부정사용으로부터 안전한가? |

### 생체인증 유형 비교

| 유형 | 장점 | 단점 | 주 응용 |
|---|---|---|---|
| 지문 | 안전성 우수 | 훼손 시 인식 곤란, **오탐률 높음** | 범죄수사, 일반산업 |
| 얼굴 | 거부감 적음 | 조명/표정 변화에 민감 | 출입 통제 |
| 망막/홍채 | 복제 불가 | **사용 불편, 거부감, 백내장 취약** | 핵시설, 의료, 교도소 |
| 음성 | 원격 사용 가능 | 정확도 낮음, 도용 가능 | 원격 은행, ARS |
| 서명 | 거부감 적음 | 서명 습관에 따라 인식률 격차 큼 | 은행, PDA |

### 오류율 ⭐⭐⭐

```
FRR (False Rejection Rate) = 오거부율 = Type 1 Error
→ 등록된 사용자를 거부하는 오류
→ 보안성 강화 시 FRR 증가

FAR (False Acceptance Rate) = 오인식률 = Type 2 Error
→ 등록 안 된 사용자를 허용하는 오류
→ 사용자 편의성 강화 시 FAR 증가
```

> ⚠️ **FRR vs FAR 관계**
> 보안 강화 → FRR ↑, FAR ↓
> 편의성 강화 → FAR ↑, FRR ↓
> 둘은 트레이드오프 관계!

## 10. 통합 인증 체계 (SSO, EAM, IAM)

### SSO (Single Sign-On) ⭐⭐⭐

> 💡 **비유 : 놀이공원 종일 이용권**
> 입장할 때 한 번 검표하면
> 롤러코스터, 회전목마, 바이킹 등 어디든 추가 검표 없이 이용 가능!

```
한 번 인증 → 여러 시스템에 재인증 없이 접근
```

**SSO 장단점 ⭐⭐⭐⭐⭐**

| 장점 | 단점 |
|---|---|
| 운영비용 감소 | **SSO 서버 = 단일실패지점(SPoF)** |
| 보안성 향상 | **ID/PW 노출 시 전체 시스템 위험** |
| 사용자 편의성 증가 | 자원(객체)별 권한 관리 미비 |
| 중앙집중 관리로 효율성 | (사람 중심이라 세밀한 권한 관리 어려움) |

### EAM, IAM 비교

```
          SSO               EAM                 IAM
──────────────────────────────────────────────────────
기능  하나의 ID/PW    SSO + 정책기반          ID관리
      로 관리         + 접근제어(차등)         + Provisioning
                                             + 자동화
관리  수작업           수작업                  자동화
특징  통합 로그인      차등 권한 부여          역할 기반 계정 관리
```

> 💡 **관계**
> IAM = SSO + EAM을 **확장/보완**한 것
> EAM = SSO + 차등 권한 관리

## 11. 커버로스 (Kerberos) ⭐⭐⭐⭐⭐

> 💡 **비유 : 놀이공원 + 각 놀이기구 탑승권**
> KDC = 종합 안내소 (한 번 인증 후 티켓 발급)
> 티켓 = 각 놀이기구 탑승권
> 탑승권 있으면 줄 서지 않고 바로 탑승!

```
개발 : MIT Athena 프로젝트
방식 : 클라이언트/서버 모델
기반 : 대칭키 암호 (v4 : DES 사용)
특징 : 티켓 기반 인증 프로토콜 + KDC
분류 : 분산 환경을 위한 SSO의 한 예
```

### 커버로스 구성요소

| 구성요소 | 설명 |
|---|---|
| **KDC** | 키 분배 서버 (핵심! 모든 주체가 신뢰) |
| **AS** | 인증 서버 (실질적 인증 수행) |
| **TGS** | 티켓 부여 서비스 (서비스 티켓 발급) |
| **Ticket** | 사용자 신원/인증 확인 토큰 |
| **Principals** | 커버로스를 사용하는 모든 실체 |

### 커버로스 동작 흐름 ⭐⭐⭐

```
[1단계] 사용자 → AS : TGT(티켓 승인 티켓) 요청
[2단계] AS → 사용자 : TGT 발급 (암호화되어 있어 사용자가 내용 못 봄)
[3단계] 사용자 → TGS : 서비스 티켓 요청 (TGT 제출)
[4단계] TGS → 사용자 : 서비스 티켓 발급
[5단계] 사용자 → 서버 : 서비스 요청 (서비스 티켓 제출)
[6단계] 서버 → 사용자 : 서비스 제공

티켓 유효기간 내에는 같은 서버에서 여러 서비스 이용 가능!
타임스탬프(인증자) 사용 → 재전송 공격 방어
```

### 커버로스 장단점

| 장점 | 단점 |
|---|---|
| 데이터 기밀성 + 무결성 보장 | 패스워드 사전 공격에 취약 |
| SSO 제공 | 비밀키/세션키가 단말기에 임시 저장 → 탈취 위험 |
| - | 타임스탬프 → **시간 동기화(NTP) 필수** |
| - | TGS/AS가 물리적 공격에 취약 |

> ⚠️ **커버로스 제공 서비스**
> ✅ 기밀성, 무결성
> ❌ 가용성, 부인방지 (KDC가 대신 알고 있음)

### 커버로스 v4 vs v5

| 항목 | v4 | v5 |
|---|---|---|
| 암호 알고리즘 | DES 고정 | **모든 암호 알고리즘 사용 가능** |
| 네트워크 주소 | IP만 지원 | **모든 유형 지원** |
| 티켓 유효기간 | 별도 유효기간 있음 | **시작~만료 명시 (유연함)** |
| 오버헤드 | 적음 | **v4보다 많음** |

## 12. SESAME

```
커버로스의 단점을 보완한 인증 시스템

커버로스 : 대칭키만 사용, 티켓 기반
SESAME  : 대칭키 + 비대칭키 모두 사용, PAC 기반

PAC (Privileged Attribute Certificate)
= 주체의 신원 + 객체 접근 능력 + 접근 기간 + PAC 유효기간
```

## 📌 최종 핵심 암기 요약

```
✅ 인증 유형
   Type 1 (지식) : 패스워드, PIN → 추측 공격
   Type 2 (소유) : 토큰, OTP   → 강탈/도난
   Type 3 (존재) : 지문, 홍채  → 도려냄
   Two Factor = 타입이 다른 2가지 / Multi Factor = 3가지 이상

✅ 패스워드 저장 3단계
   평문 → 해시 → 솔팅(해시) ← 현재 사용
   솔팅 = 사전공격 + 레인보우 테이블 차단

✅ OTP 방식
   질의응답 = 비동기식 (동기화 불필요, 번거로움)
   시간/이벤트 동기화 = 동기식
   S/Key = 해시 체인 방식 (Leslie Lamport)

✅ 생체인증 오류율
   FRR (오거부율) : 등록자를 거부 → 보안 강화 시 증가
   FAR (오인식률) : 미등록자를 허용 → 편의성 강화 시 증가
   보안↑ = FRR↑, FAR↓ / 편의↑ = FAR↑, FRR↓

✅ SSO 단점
   단일실패지점(SPoF) → ID/PW 노출 시 전체 시스템 위험
   자원별 세밀한 권한 관리 어려움

✅ SSO < EAM < IAM (기능 확장 순서)

✅ 커버로스
   대칭키 기반 / 티켓 기반 / KDC / SSO
   제공 : 기밀성 + 무결성 (가용성, 부인방지 X)
   단점 : 사전공격 취약, 시간 동기화(NTP) 필수
   v5 > v4 : 모든 암호 지원, 모든 주소 지원, 오버헤드 증가

✅ 커버로스 vs SESAME
   커버로스 : 대칭키 + 티켓
   SESAME  : 대칭키+비대칭키 + PAC
```
