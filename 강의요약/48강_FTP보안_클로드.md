## 1. FTP 개요

**FTP (File Transfer Protocol)** : 호스트 간 파일을 전송하기 위한 프로토콜.

다른 프로토콜과 다르게 **연결을 2개** 사용한다.

| 연결 | 포트 | 역할 | 유지 여부 |
|------|------|------|-----------|
| 제어 연결 (Control) | **21번** | 명령·응답 (USER, PASS, LIST 등) | 세션 동안 계속 유지 |
| 데이터 연결 (Data) | **20번** 또는 1024 이후 | 실제 파일 전송 | 파일 전송마다 생성·폐쇄 |

```
[클라이언트]          [FTP 서버]

 ──── 제어 연결 (21) ────▶
 ◀─── 데이터 연결 (20) ───   ← Active 기준
```

### 로그인 순서

```
클라이언트 → USER kim
서버       → OK
클라이언트 → PASS 1234   ← ⚠️ 평문 전송 (암호화 없음)
서버       → 로그인 성공
```

> ⚠️ 비밀번호가 **평문**으로 전송되어 스니핑에 취약하다.

## 2. FTP 연결 구조

```
제어 연결 (21번)
 - USER, PASS, LIST, GET 등 명령 주고받음
 - 세션 시작~종료까지 유지

데이터 연결 (20번 또는 1024+)
 - 실제 파일이 이동하는 통로
 - 파일 전송할 때만 열렸다가 닫힘
```

## 3. FTP Active / Passive 모드

> ⭐ 시험 단골 — 3가지 차이점 반드시 암기

### Active 모드 (능동 모드) — 기본값

**서버 → 클라이언트** 방향으로 데이터 연결을 건다.

```
1단계: 클라이언트 ──────── 제어 연결(21) ──────▶ 서버
2단계: 클라이언트 ──── PORT 5001 명령 전달 ────▶ 서버  ("나 5001 열어놨어")
3단계: 서버(20)   ──────── connect(5001) ──────▶ 클라이언트
```

**문제점:**  
클라이언트에 방화벽이 있으면 외부(서버)에서 들어오는 연결이 차단된다.  
→ 제어 연결(로그인)은 되지만 **파일 전송만 실패**

---

### Passive 모드 (수동 모드)

**클라이언트 → 서버** 방향으로 데이터 연결까지 직접 건다.

```
1단계: 클라이언트 ──────── 제어 연결(21) ──────▶ 서버
2단계: 클라이언트 ──────── PASV 명령 ───────────▶ 서버  ("수동모드로 할게")
3단계: 서버 ──── "나 4900 열어놨어" 응답 ──────▶ 클라이언트
4단계: 클라이언트 ──── connect(4900) ─────────▶ 서버
```

**장점:** 클라이언트 방화벽 문제 없음 → **현대 인터넷 표준**

---

### Active vs Passive 비교

| 구분 | Active | Passive |
|------|--------|---------|
| 사용 명령 | `PORT` | `PASV` |
| 서버 데이터 포트 | **20번** | **1024 이후 랜덤** (예: 4900) |
| 데이터 연결 주체 | **서버 → 클라이언트** | **클라이언트 → 서버** |
| 방화벽 문제 | 클라이언트 방화벽 문제 가능 | 서버 방화벽 관리 필요 |

> ✅ Active/Passive 모드 선택은 **FTP 서버가 아닌 클라이언트**가 결정한다.

## 4. 익명 FTP & TFTP

### 익명 FTP (Anonymous FTP)

계정 없이 접속 가능한 FTP. 파일 배포·기술지원 용도로 주로 사용.

| 구분 | FTP | 익명 FTP | TFTP |
|------|-----|----------|------|
| ID | O | `anonymous` 또는 `ftp` | X |
| PW | O | 이메일 주소 (로그 기록용) | X |

> 비밀번호는 실제 인증이 아니라 **누가 접속했는지 서버 로그에 기록**하는 용도이다.

---

### TFTP (Trivial FTP)

FTP의 간소화 버전. 기능이 굳이 다 필요 없는 환경에서 사용.

| 구분 | FTP | TFTP |
|------|-----|------|
| 전송 프로토콜 | TCP | **UDP** |
| 포트 | 20, 21 | **69** |
| 인증 | O | **X** |
| 기능 | 다양 | 한정적 |
| 주 사용처 | 일반 파일 전송 | 공유기·스위치 펌웨어, 디스크 없는 워크스테이션 |

**특이점:**  
UDP는 신뢰성이 없으므로, TFTP가 자체적으로 흐름 제어·오류 제어 메커니즘을 구현한다.

## 5. NFS & Samba

### NFS (Network File System)

리눅스/유닉스 계열 간 파일 공유 프로토콜.  
원격 서버의 디렉터리를 **마치 로컬 디스크처럼** 마운트해서 사용.

```
[NFS 서버] 공유 폴더 제공
      ↓ 네트워크
[NFS 클라이언트] /mnt/share 를 열면 서버 파일이 보임
```

---

### Samba

**SMB (Server Message Block)** 프로토콜을 사용하여  
**유닉스/리눅스 ↔ 윈도우** 간 파일·프린터 자원을 공유하는 기능.

```
윈도우 PC  ←── SMB ──▶  리눅스 서버
           파일 공유 가능
```

> cf) `sesame` = 유럽형 인증 프로토콜 (혼동 주의)

## 6. FTP 보안 위협 및 대책

### FTPS vs SFTP

| 이름 | 기반 | 특징 |
|------|------|------|
| **FTPS** | FTP + SSL/TLS | FTP에 암호화 레이어 추가 |
| **SFTP** | SSH | FTP와 완전히 다른 별개 프로토콜, SSH의 일부 |

> SSH 구조: Transport(채널 형성) → Auth(상호인증) → Connection(다중화) → Application(응용)  
> **다중화** = 하나의 SSH 연결 안에서 터미널·파일전송·포트포워딩 등을 동시에 처리

---

### Bounce Attack ⭐ 매우 중요

**개요:**  
FTP의 설계 취약점(데이터 채널 목적지를 확인하지 않음)을 이용한 공격.

**공격 원리:**  
Active 모드에서 `PORT` 명령의 IP·포트를 **공격자가 원하는 임의의 주소로 조작**  
→ FTP 서버가 엉뚱한 목적지로 데이터를 전송  
→ FTP 서버를 **중계 대포**로 활용

```
공격자 → FTP 서버에 PORT 192.168.0.5:80 전달
FTP 서버가 → 192.168.0.5:80 으로 데이터 전송 (내부 네트워크 스캔·공격 가능)
```

주로 **익명 FTP 서버**를 이용해 내부 네트워크 포트 스캔에 활용된다.

---

### Anonymous FTP 취약점 및 대책

**위협:** 쓰기 권한이 있으면 공격자가 악성코드·웹쉘 업로드 가능.

**대책:**
- 반드시 필요한 경우가 아니면 **서비스 제거**
- anonymous 루트 디렉터리·하위 디렉터리 퍼미션 최소화
- 쓰기 권한 제거

---

### TFTP 보안

**문제:** 인증 자체가 없어 보안 고려가 전혀 없음.

**대책: secure mode (chroot 활용)**

```
chroot로 /home/tftp 를 최상위 디렉터리로 지정
→ 지정 디렉터리 상위로 접근 불가 (chroot 감옥)
```

> **chroot 감옥:** 루트 디렉터리를 변경하여 특정 경로 밖으로 나가지 못하게 제한

---

### FTP 접근 제어

| 방법 | 파일 | 역할 |
|------|------|------|
| ftpusers | `/etc/ftpusers` | 접속 **거부** 계정 목록 |
| TCP Wrapper | `hosts.allow`, `hosts.deny` | IP 기반 접근 제어 |

## 7. FTP 서버 운영 (ProFTPD / vsftpd)

### ProFTPD

- `xinetd` 방식 / `standalone` 방식 모두 지원

**ftp 접속 시 거부 조건:**

| 조건 | 결과 |
|------|------|
| `/etc/ftpusers`에 ID 존재 | 접속 거부 |
| `/etc/shells`에 없는 쉘 사용자 | 접속 거부 |

**주요 설정 옵션:**

| 옵션 | 의미 |
|------|------|
| `MaxClients` | 최대 동시 접속 허용 수 |
| `TimeoutLogin` | 인증 완료까지 제한 시간 |
| `TimeoutIdle` | idle 상태 허용 시간 |
| `TimeoutSession` | 세션 최대 유지 시간 |

---

### vsftpd (Very Secure FTP Daemon)

보안 강화에 초점을 맞춘 FTP 서버.

**주요 기능:**
- PAM 지원
- `xferlog` 자체 상세 로그 지원
- standalone / xinetd 모두 지원

**관련 파일:**

| 파일 | 역할 |
|------|------|
| `/etc/vsftpd/ftpusers` | FTP 접속 제한 계정 목록 |
| `/var/log/xferlog` | 업로드·다운로드 기록 로그 |

## 8. 부록

### standalone vs xinetd

| 방식 | 동작 | 장점 | 단점 |
|------|------|------|------|
| **standalone** | 서버가 항상 실행 중 대기 | 빠른 응답 | 메모리 상시 점유 |
| **xinetd** | 요청 올 때만 서버 실행 | 자원 절약 | 상대적으로 느림 |

> 요즘은 하드웨어 성능이 충분해서 **standalone**을 주로 사용.

---

### PAM (Pluggable Authentication Modules)

리눅스의 **통합 인증 시스템**.

FTP·SSH·sudo 등 여러 프로그램이 PAM에 인증을 위임 → 인증 정책을 한 곳에서 통합 관리 가능.

```
사용자 로그인
     ↓
vsftpd
     ↓
PAM에 인증 요청
     ↓
"비밀번호 맞음" → 로그인 성공
```

**PAM 장점:**
- 인증 정책 통합 관리
- OTP·계정 잠금·로그인 실패 제한 등 보안 기능 추가 용이
- 각 프로그램이 인증 로직 직접 구현 불필요

---

### 포트포워딩 (Port Forwarding)

공유기(NAT) 환경에서 **외부 요청을 내부 PC로 전달**하는 설정.

```
인터넷 사용자
      ↓
공유기:80
      ↓
192.168.0.10:80  ← 포트포워딩으로 지정한 내부 PC
```

**SSH 포트포워딩:** SSH 연결 내부에 암호화된 터널을 생성하여 외부에서 내부 서비스에 접근.

```
내 PC ─── SSH 터널 ──▶ 회사 서버 ──▶ 내부 DB
     localhost:3306                  DB:3306
```

> FTP Active 모드에서 클라이언트가 NAT 뒤에 있으면 서버가 접속할 수 없어 포트포워딩 설정이 필요하다.  
> 이 문제 때문에 현대 환경에서는 **Passive 모드**가 표준처럼 쓰인다.

## 핵심 암기 정리

| 항목 | 내용 |
|------|------|
| FTP 제어 포트 | **21** |
| FTP 데이터 포트 | **20** (Active) / 1024+ (Passive) |
| TFTP 포트 | **UDP 69** |
| Active 데이터 연결 | 서버 → 클라이언트 (`PORT`) |
| Passive 데이터 연결 | 클라이언트 → 서버 (`PASV`) |
| 모드 결정 주체 | **클라이언트** |
| FTPS | FTP + SSL/TLS |
| SFTP | SSH 기반 (FTP 아님) |
| Samba | 리눅스 ↔ 윈도우, SMB 사용 |
| Bounce Attack | PORT 명령 조작, FTP 서버를 대포로 이용 |
| chroot 감옥 | TFTP secure mode, 지정 디렉터리 외 접근 차단 |
