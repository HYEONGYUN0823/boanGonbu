## 1. /etc/shadow 파일 & 패스워드 에이징

### 패스워드 에이징이란?
비밀번호 유효기간을 강제로 관리하는 것  
예) "90일마다 바꿔라", "최소 7일은 유지해라"

### /etc/shadow 구조 (9개 필드)

```
형윤 : $6$abc : 19900 : 7 : 90 : 7 : 30 : 20000 :
 ①       ②       ③    ④   ⑤   ⑥   ⑦     ⑧      ⑨
```

| 필드 | 이름 | 설명 |
|------|------|------|
| ① | 계정명 | 로그인 ID |
| ② | 패스워드 | 암호화된 비밀번호 (`*` = 계정 잠금) |
| ③ | 마지막 변경일 | 1970년 1월 1일 기준 일수 |
| ④ | **최소 사용기간** | 변경 후 최소 이 기간은 유지해야 함 |
| ⑤ | **최대 사용기간** | 이 기간 안에 반드시 변경 |
| ⑥ | **경고 기간** | 만료 며칠 전부터 경고 메시지 |
| ⑦ | **비활성 기간** | 만료 후 유예기간 (지나면 계정 잠금) |
| ⑧ | 계정 만료일 | 계정 자체가 만료되는 날짜 |
| ⑨ | 예약 필드 | 현재 미사용 |

> 💡 암기법: **계 비 마 / 최소 최대 / 경 유 계 예**

### 패스워드 에이징 타임라인
```
[최소기간 7일] → [사용가능 90일까지] → [경고 7일] → [유예 30일] → [계정잠금]
  변경 불가         자유롭게 사용        만료 알림    마지막 기회
```

### /etc/passwd vs /etc/shadow

| 파일 | 접근 | 패스워드 |
|------|------|----------|
| `/etc/passwd` | 누구나 읽기 가능 | `x` 로만 표시 |
| `/etc/shadow` | root만 읽기 가능 | 실제 암호화값 저장 |

## 2. 암호화 패스워드 형식

```
$id $ salt $ encrypted_password
 ①     ②           ③
```

| 구분 | 역할 |
|------|------|
| **$id** | 어떤 알고리즘으로 암호화했는지 |
| **$salt** | 랜덤 문자열 (레인보우 테이블 공격 방어) |
| **$encrypted_password** | 실제 암호화된 해시값 (복원 불가) |

### 알고리즘 종류 (암기 필수)

| $id | 알고리즘 | 특징 |
|-----|----------|------|
| `$1$` | **MD5** | 128bit / 취약 (현재 사용 지양) |
| `$2$` | Blowfish | 시험 범위 외 |
| `$5$` | **SHA-256** | 256bit / 안전 |
| `$6$` | **SHA-512** | 512bit / 현재 가장 많이 사용 |

> 💡 암기법: **숫자가 클수록 더 안전** (1=낡은것 / 5=중간 / 6=최신)

### salt가 필요한 이유
```
salt 없음: "1234" → 항상 같은 해시값 → 레인보우 테이블로 역추적 가능
salt 있음: "1234" + "랜덤문자열" → 매번 다른 해시값 → 역추적 불가
```

## 3. RUID / EUID / SetUID

| 구분 | 의미 |
|------|------|
| **RUID** | 실제로 누가 실행했냐 (변하지 않음) |
| **RGID** | 실행한 사용자의 그룹 |
| **EUID** | 지금 어떤 권한으로 실행 중이냐 (SetUID면 바뀜) |

### 일반 실행 vs SetUID 실행

```
일반 실행:
형윤(UID=1001) → ls 실행 → RUID=1001 / EUID=1001 (같음)

SetUID 실행:
형윤(UID=1001) → passwd 실행 → RUID=1001 / EUID=0(root)
```

### 왜 SetUID가 필요해?
`passwd` 명령어로 비번 바꾸려면 `/etc/shadow` 수정 필요  
→ `/etc/shadow`는 root만 수정 가능  
→ SetUID 덕분에 passwd 실행 중에만 EUID=root 가 됨

### SetUID 확인 방법
```bash
ls -l /usr/bin/passwd
-rwsr-xr-x   ← 소유자 x 자리가 s 로 바뀜

# SetUID 걸린 파일 전체 찾기
find / -perm -4000
```

| 특수권한 | 숫자 | 표시 |
|----------|------|------|
| SetUID | **4**755 | 소유자 x → **s** |
| SetGID | **2**755 | 그룹 x → **s** |
| Sticky Bit | **1**777 | 기타 x → **t** |

> ⚠️ 보안 위험: SetUID 파일이 해킹당하면 root 권한 탈취로 이어질 수 있음

## 4. Sticky Bit

### 개념
공용 디렉터리에서 **본인이 만든 파일만 삭제 가능**하게 하는 특수권한

### 동작 방식
```
/tmp 디렉터리 (sticky bit 적용)

형윤의 파일 → 형윤만 삭제 가능 ✔
채현의 파일 → 형윤이 삭제 불가 ✖
민수의 파일 → 형윤이 삭제 불가 ✖
root         → 누구 파일이든 삭제 가능 (예외)
```

> 💡 비유: 공용 사물함인데 **내 짐은 내가만 꺼낼 수 있음**

### 확인 및 설정
```bash
ls -l /tmp
drwxrwxrwt   ← 맨 끝이 t 이면 sticky bit 적용

# 설정 방법
chmod +t /tmp
chmod 1777 /tmp   ← 앞에 1이 붙음
```

## 5. inetd 슈퍼 데몬

### 데몬(Daemon) 종류
| 종류 | 설명 |
|------|------|
| 활동적 프로세스 | 지금 내가 쓰는 프로그램 |
| 백그라운드 프로세스 | 뒤에서 조용히 돌아가는 것 |
| **데몬** | 서비스 요청 대기 중인 서버 프로세스 |

### stand-alone vs inetd 방식

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **stand-alone** | 각 서비스가 항상 메모리 상주 | 속도 빠름 | 자원 많이 점유 |
| **inetd** | 슈퍼데몬이 요청 오면 해당 서비스 실행 | 자원 절약 | 속도 느림 |

> 💡 비유  
> stand-alone = 패스트푸드점 (직원이 항상 카운터에 서 있음)  
> inetd = 호출벨 식당 (벨 누르면 그때 직원이 옴)

### inetd.conf 파일 구조 (7개 필드)

```
ftp  stream  tcp  nowait  root  /usr/sbin/ftpd  ftpd
 ①     ②      ③     ④      ⑤         ⑥           ⑦
```

| 필드 | 설명 |
|------|------|
| ① 서비스명 | ftp, telnet, ssh 등 |
| ② 소켓 타입 | stream(TCP) / dgram(UDP) |
| ③ 프로토콜 | tcp / udp |
| ④ 대기 여부 | wait / nowait |
| ⑤ 실행 계정 | 어떤 계정으로 실행할지 (보통 root) |
| ⑥ 실행 경로 | 실제 프로그램 파일 위치 |
| ⑦ 인수 | 실행 시 넘길 옵션 |

> 💡 암기법: **서 소 프 대 계 경 인**  
> `#` 으로 시작하면 주석 → 해당 서비스 비활성화

### wait vs nowait
- `wait` → 한 요청 끝날 때까지 다음 요청 못 받음 (UDP)
- `nowait` → 동시에 여러 요청 처리 가능 (TCP)

## 6. TCP Wrapper

### 개념
접속 허용/차단을 담당하는 **문지기** 역할

### 전체 흐름
```
외부 접속 → inetd(전화받기) → tcpd(신분검사) → 실제 서비스(ssh, ftp...)
```

| 용어 | 역할 |
|------|------|
| **inetd** | 서비스 대신 전화 받아주는 콜센터 |
| **tcpd** | IP 보고 허용/차단 결정하는 경비 |
| **TCP Wrapper** | tcpd 기능 전체를 부르는 이름 |
| **in.telnetd** | 실제 텔넷 서비스 프로그램 |

### 설정 파일

```bash
# /etc/hosts.allow (허용 목록)
sshd: 192.168.0.10        # 이 IP만 ssh 허용

# /etc/hosts.deny (차단 목록)
sshd: ALL                 # 나머지 전부 차단
```

### 검사 순서 (중요!)
```
접속 시도
  → hosts.allow 확인 → 있으면 즉시 허용 (끝)
  → hosts.deny 확인  → 있으면 즉시 차단 (끝)
  → 둘 다 없으면     → 기본 허용 (default)
```

> ⚠️ 함정: default = **허용** (차단 아님!)

### ALL EXCEPT 문법
```bash
sshd: ALL EXCEPT 192.168.0.10
# → 192.168.0.10 빼고 전부 적용

# hosts.allow에 있으면: 10번 빼고 다 허용
# hosts.deny에 있으면:  10번 빼고 다 차단
```

## 7. PAM (장착형 인증 모듈)

### 개념
인증 방식을 **플러그처럼 꽂았다 뺐다** 할 수 있는 구조  
→ 응용프로그램 수정 없이 인증 방식만 교체 가능

```
사용자 → ssh → PAM → 인증모듈 (비번 / OTP / 지문...)
```

> 💡 비유: PAM = 멀티탭 (콘센트만 바꾸면 다른 기기 꽂을 수 있음)

### PAM 설정파일 구조
```
type  control  module  옵션
auth  required pam_unix.so
```

### type (뭘 검사하냐)

| type | 역할 | 예시 |
|------|------|------|
| **auth** | 인증 (너 누구야?) | 비번, OTP, 지문 |
| **account** | 계정 유효성 (로그인해도 돼?) | 만료여부, 접근시간 제한 |
| **password** | 비밀번호 정책 | 길이, 복잡도 |
| **session** | 로그인 전후 작업 | 홈디렉터리 생성, 로그 기록 |

> 💡 암기: **auth → account → password → session**

### control (실패하면 어떻게 할 거냐)

| control | 동작 | 비유 |
|---------|------|------|
| **requisite** | 실패하면 **즉시** 인증 거부 | 한 문제 틀리면 시험 종료 |
| **required** | 실패해도 끝까지 진행 후 거부 | 끝까지 풀고 채점 |
| **sufficient** | 성공하면 즉시 인증 승인 | 이 문제 맞으면 바로 합격 |
| **optional** | 결과 무시 | 채점 안 함 |

### pam_securetty.so
- root가 로그인 가능한 **터미널 위치 제한**
- `/etc/securetty` 파일 참조
- 목록에 없는 터미널(원격 SSH 등)에서 root 로그인 차단

```bash
# /etc/pam.d/login
auth required pam_securetty.so

# /etc/securetty
console
tty1
tty2
```

## 8. 파일시스템 ext2 / ext3 / ext4

| 파일시스템 | 특징 | 단점 |
|------------|------|------|
| **ext2** | 빠름, 저널링 없음 | 전원 꺼지면 파일 깨짐 → fsck 필요 |
| **ext3** | 저널링 추가 | ext2보다 약간 느림 |
| **ext4** | 저널링 + 지연된 할당 | 현재 가장 많이 사용 |

### 핵심 용어

**fsck (File System Check)**
- 파일 시스템 검사 + 복구 도구
- 전원이 갑자기 꺼졌을 때 손상된 파일 찾아서 복구
- ext2에서 주로 사용
- 비유: **사고 난 후 보험 처리** 🚑

**저널링 (Journaling)**
- 작업하기 전에 미리 기록을 남기는 방식
- 전원이 꺼져도 기록 보고 복구 가능 → fsck 거의 불필요
- 비유: **블랙박스 달아놓음** 📹

**지연된 할당 (Delayed Allocation)**
- 데이터를 바로 저장하지 않고 모았다가 한 번에 저장
- 성능 향상 + 디스크 효율 증가
- 비유: **택배 모아서 한 번에 보내는 것** 📦

## 9. OS별 root 로그인 제한

| OS | 설정 파일 | 설정값 |
|----|----------|--------|
| **Linux** | `/etc/ssh/sshd_config` | `PermitRootLogin no` |
| **Linux** | `/etc/pam.d/login` | `pam_securetty.so` |
| **SunOS** | `/etc/default/login` | `CONSOLE=/dev/console` |
| **AIX** | `/etc/security/login.cfg` | rlogin 제한 |
| **HP-UX** | `/etc/securetty` | console 설정 |

### 왜 root 직접 로그인을 막냐?
```
root 바로 접속 가능 → 해커가 바로 최고 권한 획득 💀
→ 일반계정으로 로그인 후 su / sudo 사용하도록 강제
```

### Linux SSH 설정
```bash
# /etc/ssh/sshd_config
PermitRootLogin no    # root 직접 로그인 금지

# 적용
systemctl restart sshd
```

### PermitRootLogin vs pam_securetty.so 차이

| 방식 | 의미 |
|------|------|
| `PermitRootLogin no` | root 로그인 **아예** 금지 |
| `pam_securetty.so` | 특정 터미널에서만 허용 |

## 10. OS별 보안 점검 파일 위치

### Linux

| 점검 항목 | 파일 위치 | 확인 내용 |
|----------|----------|----------|
| root SSH 로그인 | `/etc/ssh/sshd_config` | `PermitRootLogin no` |
| root 터미널 제한 | `/etc/securetty` | console 등록 여부 |
| PAM 설정 | `/etc/pam.d/` | `pam_securetty.so` |
| 패스워드 정책 | `/etc/login.defs` | 실패횟수, 유효기간 |
| 패스워드 에이징 | `/etc/shadow` | 각 필드값 확인 |

### SunOS (솔라리스)

| 점검 항목 | 파일 위치 | 확인 내용 |
|----------|----------|----------|
| root 로그인 제한 | `/etc/default/login` | `CONSOLE=/dev/console` |
| 로그인 실패 횟수 | `/etc/default/login` | `RETRIES=5` |
| 패스워드 에이징 | `/etc/shadow` | 각 필드값 확인 |

### AIX

| 점검 항목 | 파일 위치 | 확인 내용 |
|----------|----------|----------|
| 계정 잠금 정책 | `/etc/security/user` | 실패횟수, 잠금 정책 |
| root 원격 로그인 | `/etc/security/login.cfg` | rlogin 제한 |
| 패스워드 정책 | `/etc/security/passwd` | 패스워드 정책 |

### HP-UX

| 점검 항목 | 파일 위치 | 확인 내용 |
|----------|----------|----------|
| root 로그인 제한 | `/etc/securetty` | console 설정 |
| 계정 잠금 임계값 | `/tcb/files/auth/system/default` | 실패 허용 횟수 |

## 11. PATH 환경변수 취약점

### PATH란?
명령어 입력 시 OS가 **프로그램을 찾는 디렉터리 목록**
```bash
PATH=/usr/bin:/usr/sbin:/bin
# ls 치면 이 순서대로 ls 프로그램을 찾아서 실행
```

### 취약점: PATH에 . (점) 포함

`.` = 현재 디렉터리를 의미

```bash
# 위험한 설정
PATH=.:/usr/bin:/bin    ← 맨 앞에 .

# 공격 시나리오
1. 해커가 /tmp에 가짜 ls 파일 생성 (악성코드)
2. root가 /tmp 디렉터리에서 ls 명령 실행
3. PATH 앞에 . 이 있어서 현재 디렉터리(/tmp)를 먼저 탐색
4. 가짜 ls가 root 권한으로 실행됨 💀
```

### 조치 방법
```bash
# 위험
PATH=.:/usr/bin:/bin

# 안전 (. 제거)
PATH=/usr/bin:/bin

# 차선책 (. 을 맨 뒤로)
PATH=/usr/bin:/bin:.
```

> ⚠️ 보안 권고: `.`을 PATH에서 **아예 제거**하는 것이 최선

## 핵심 암기 포인트 요약

| 항목 | 암기 내용 |
|------|----------|
| shadow 필드 순서 | **계 비 마 / 최소 최대 / 경 유 계 예** (9개) |
| 암호화 알고리즘 | `$1`=MD5 / `$5`=SHA-256 / `$6`=SHA-512 |
| salt 역할 | 레인보우 테이블 공격 방어 |
| EUID | 실행 권한 / SetUID 걸리면 파일 소유자 권한으로 변경 |
| SetUID 찾기 | `find / -perm -4000` |
| 특수권한 숫자 | SetUID=**4** / SetGID=**2** / Sticky=**1** |
| Sticky bit 표시 | `ls -l` 에서 맨 끝이 **t** |
| inetd.conf 필드 | **서 소 프 대 계 경 인** (7개) |
| TCP Wrapper 검사 순서 | allow → deny → default(허용) |
| ALL EXCEPT | "얘 빼고 전부" |
| PAM type | auth → account → password → session |
| PAM control | requisite(즉시) / required(끝까지) / sufficient(성공시합격) |
| Linux root 제한 | `sshd_config` → `PermitRootLogin no` |
| SunOS root 제한 | `/etc/default/login` → `CONSOLE=` |
| AIX root 제한 | `/etc/security/login.cfg` → rlogin |
| HP-UX root 제한 | `/etc/securetty` → console |
| ext2 vs ext3 | ext2=저널링없음(fsck필요) / ext3=저널링추가 |
| PATH 취약점 | `.` 이 포함되면 현재 디렉터리 먼저 탐색 → 제거 필요 |
