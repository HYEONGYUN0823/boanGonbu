## 1. 악성 소프트웨어 개요

### 핵심 정의

- **악성 프로그램**: 의도적으로 컴퓨터의 보안 속성을 침해할 목적으로 작성된 프로그램
- 개발자 ≠ 사용자 → 개발자가 사용자가 원하지 않는 기능을 몰래 포함

### 3대 악성코드 비교

| 구분 | 바이러스 | 웜 | 트로이목마 |
|---|---|---|---|
| 침해 속성 | **무결성** | **가용성** | **기밀성** |
| 자기 복제 | 있음 | **매우 강함** | 없음 |
| 숙주 필요 | 필요 (파일/부트섹터) | 불필요 (독립 실행) | 불필요 |
| 전파 방법 | 사용자가 감염 파일 옮김 | 네트워크로 스스로 전파 | 사용자가 직접 실행 |
| 주요 피해 | 파일·시스템 손상 | 네트워크 성능 저하 | PC 성능 저하, 좀비PC |

### 비유로 기억하기

- **바이러스** = 감기 바이러스처럼 숙주(파일)에 붙어야 퍼짐
- **웜** = 혼자서 네트워크를 타고 기어다니는 벌레. 아무도 안 옮겨도 스스로 퍼짐
- **트로이목마** = 유용한 프로그램인 척 위장. 복제 없이 몰래 나쁜 짓

### 분류 기준

#### 독립형 vs 기생형

| 유형 | 설명 | 예시 |
|---|---|---|
| 기생형 | 다른 프로그램 없이 독립 존재 불가 | 바이러스, 논리폭탄, 백도어 |
| 독립형 | 자체적으로 구동 가능 | 웜, 좀비 프로그램 |

#### Payload(목적)별 분류

| 목적 | 악성코드 종류 |
|---|---|
| 시스템 파괴 | 논리폭탄, Stuxnet, 랜섬웨어 |
| 공격 에이전트 | 좀비, 봇(bot) |
| 정보 유출 | 키로거, 피싱, 스파이웨어 |
| 잠입(스텔스) | 백도어, 루트킷 |

## 2. 바이러스 (Virus)

### 세대별 분류 — 암기법: **원→암→은→갑→매**

| 세대 | 이름 | 핵심 특징 |
|---|---|---|
| 1세대 | 원시형 (Primitive) | 구조 단순, 고정 크기, 분석 쉬움 |
| 2세대 | 암호화 (Encryption) | 코드 일부 또는 대부분 암호화 |
| 3세대 | 은폐형 (Stealth) | 감염 후 파일 크기 안 변한 척, 백신 속임 |
| 4세대 | 갑옷형 (Armour) | 백신 개발자 공격, 분석·개발 지연시킴 |
| 5세대 | 매크로 (Macro) | MS Office 등 매크로 기능 악용, **현재 가장 많음** |

### 일시적 vs 상주형

- **일시적 바이러스**: 호스트 프로그램 실행될 때만 동작, 종료 시 끝
- **상주형 바이러스**: 메모리에 상주, 독립적으로 활성/비활성

### 다형성 바이러스 (Polymorphic Virus)

- 자신의 모습을 스스로 바꿔 바이러스 스캐너를 속임
- 스캐너가 새로운 바이러스로 인식 → 탐지 어려움

### 매크로 바이러스 특징

- **플랫폼 무관** (Windows/Mac 모두 감염 가능)
- 주로 **문서를 감염**시킴 (실행 파일 아님)
- 전파 경로: 주로 **전자메일**
- Adobe PDF도 피해 대상 (JavaScript 포함 가능)

## 3. 기타 악성코드 유형 전체 정리

### 숨는 놈 (잠입/은폐)

#### 백도어 (Backdoor / 트랩 도어)

- 개발자가 몰래 뚫어놓은 **뒷문**
- 정상 인증 과정 완전히 우회
- 원격에서 아무 때나 접속 가능
- 탐지 매우 어려움 (정상 트래픽처럼 보임)
- 대표 사례: **Back Orifice** (1998, Cult of the Dead Cow 제작)

#### 루트킷 (Rootkit)

- OS 깊숙이 숨어서 **자기 존재를 숨기는** 악성코드
- 백신 프로그램에도 안 보임
- 다른 악성코드를 숨겨주는 역할도 함
- 탐지하려면 OS 바깥(부팅 전)에서 검사해야 함

> **백도어 vs 루트킷**
> - 백도어 = 들어오는 **문** (침입 경로)
> - 루트킷 = **숨는 기술** (은폐 도구)
> - 보통 백도어로 침입 → 루트킷으로 숨는 방식으로 같이 사용됨

### 조종당하는 놈

#### 좀비 (Zombie)

- 해커 명령을 기다리는 **원격 조종 PC**
- 평소엔 정상 PC처럼 작동 → 주인은 감염 사실 모름
- 명령 받으면 DDoS 공격, 스팸 발송 등에 동원
- 여러 좀비PC를 묶은 것 = **봇넷(Botnet)**

### 퍼붓는 놈

#### 스패머 프로그램 (Spammer)

- **대량 스팸 메일 자동 발송** 도구
- 피싱 메일, 광고 메일 대량 살포
- 좀비PC를 이용해서 발송하는 경우 많음

#### 플러더 (Flooders)

- **대량 패킷으로 서버/네트워크를 마비**시키는 도구
- DoS/DDoS 공격에 사용
- 가용성(Availability) 침해
- 비유: 편의점 계산대 앞에 사람 1000명이 몰려서 아무도 계산 못하게 만드는 것

### 납치하는 놈

#### 브라우저 하이재커 (Browser Hijacker)

- 브라우저를 **납치**하는 악성코드
- 홈페이지 → 광고 사이트로 강제 변경
- 검색엔진 → 가짜 검색엔진으로 변경
- 광고 클릭 수익이 목적

### 웹에서 실행되는 놈

#### 모바일 코드 (Mobile Code)

웹에서 자동으로 다운받아 실행되는 코드

| 종류 | 방어 시점 | 방어 방식 | 비유 |
|---|---|---|---|
| **ActiveX** | **실행 전** | 서명 검증으로 차단 | 입장 전 신분증 확인 |
| **Java Applet** | **실행 중** | 샌드박스 안에 격리 | 입장은 시키되 격리실에 가둠 |

### 침투/실행 도구

#### Exploit
- 취약점을 공격해 시스템 제어권 획득

#### Downloaders
- 감염 후 **추가 악성코드 자동 다운로드** (선발대 역할)

#### Auto-rooter
- 취약점 자동 탐색 + 루트킷 자동 설치까지 자동화

#### Kit (공격킷)
- 악성코드 자동 생성 툴
- 비전문가도 악성코드 제작 가능하게 함

### 정보 수집/유출

#### 키로거 (Keylogger)
- 타이핑 내용 몰래 기록
- 비밀번호, 카드번호 등 탈취

#### 스파이웨어 (Spyware)
- 사용자 행동 감시 후 외부로 전송
- 번들(무료 프로그램에 끼워서) 설치되는 경우 많음

#### 애드웨어 (Adware)
- 강제 광고 노출 목적
- 스파이웨어 기능 내장된 경우 많음

| | 스파이웨어 | 애드웨어 |
|---|---|---|
| 목적 | 정보 훔침 | 광고 강제 노출 |
| 수익구조 | 훔친 정보 판매 | 광고 클릭 수익 |

#### 크라임웨어 (Crimeware)
- 금융정보 탈취 목적
- 금전적 이득이 최종 목표

### 기타

#### 다이얼러 (Dialer)
- 몰래 유료 전화 연결 → 전화 요금 폭탄 유발

#### 조크 (Joke)
- 실제로 실행되는 장난 프로그램

#### Hoax
- 프로그램 없이 그냥 가짜 바이러스 경고 메시지

> **조크 vs Hoax**
> - 조크 = 실제로 실행되는 장난 프로그램
> - Hoax = 허위 경고 문자/메일 (프로그램 없음)

## 4. 안티바이러스 방법

### 탐지 3단계

1. **탐지(Detection)**: 바이러스 있는지 판단하고 위치 파악
2. **식별(Identification)**: 어떤 바이러스인지 특정
3. **제거(Removal)**: 모든 감염 시스템에서 제거

### 필터링 방법 비교

| 방법 | 탐지 방식 | 장점 | 단점 |
|---|---|---|---|
| **Signature Scanning** | 알려진 바이러스 시그니처 검색 | 정확도 높음 | 신종/변종 대응 불가 |
| **Behavioral Scanning** | 이상한 행동 패턴 감지 | 신종·Zero-day 대응 가능 | 오탐 가능성 있음 |

### 웜 방어 — 네트워크 모니터링

- **Inbound 모니터링**: 외부 → 내 시스템 트래픽 감시 (침입 방지)
- **Outbound 모니터링**: 내 시스템 → 외부 트래픽 감시 (확산 방지)

> 웜은 양방향 모니터링 필요

## 5. HTTP 트랜잭션

### 트랜잭션이란?

**요청(Request) + 응답(Response)** 한 쌍 = 트랜잭션 1번

### HTTP는 Stateless (무상태)

- HTTP는 트랜잭션 끝나면 이전 요청을 **기억하지 못함**
- 매 요청이 서버 입장에서 완전히 처음 보는 사람
- 보완 수단: **쿠키, 세션**

### HTTP 버전별 연결 방식

| 버전 | 연결 방식 | 특징 |
|---|---|---|
| **HTTP 1.0** | 비영속적 연결 | 요청마다 새로 연결 → 느림, 서버 오버헤드 큼 |
| **HTTP 1.1** | 영속적 연결 | 한 번 연결 후 계속 사용 → 빠름 |

> **오버헤드**: 원래 목적 외에 추가로 드는 비용 (연결 맺고 끊는 작업)
> **버퍼**: 데이터를 임시 저장하는 메모리 공간 → 연결마다 할당 필요

**Connection 헤더**
```
Connection: keep-alive  →  영속적 연결 (연결 유지)
Connection: close       →  비영속적 연결 (응답 후 끊음)
```

> HTTP/1.1은 기본값이 keep-alive
> HTTP/1.0은 기본값이 close → 명시적으로 써야 유지됨

### 특수 문자

| 기호 | 이름 | 의미 |
|---|---|---|
| SP | Space | 띄어쓰기 (0x20) |
| CR | Carriage Return | 커서를 줄 맨 앞으로 (0x0D) |
| LF | Line Feed | 커서를 한 줄 아래로 (0x0A) |
| **CR+LF** | 줄바꿈 | HTTP 공식 줄바꿈 표준 |

## 6. HTTP 요청 메시지

### 구조

```
[요청 라인]  GET /index.html HTTP/1.1 CR+LF
[헤더]       Host: www.naver.com CR+LF
[헤더]       Accept: text/html CR+LF
[헤더]       Accept-Language: ko CR+LF
[헤더]       Accept-Encoding: gzip CR+LF
[헤더]       Connection: keep-alive CR+LF
[헤더]       Referer: https://google.com CR+LF
[헤더]       User-Agent: Chrome/120 (Windows NT) CR+LF
[빈 줄]      CR+LF
[본문]       (GET은 보통 없음)
```

### 주요 헤더 설명

| 헤더 | 의미 | 비유 |
|---|---|---|
| `Host` | 서버 도메인 + 포트번호 | 편지 받는 사람 주소 |
| `Accept` | 받을 수 있는 콘텐츠 형식 | "채식주의자라 고기 빼주세요" |
| `Accept-Language` | 원하는 언어 | 한국어로 줘 |
| `Accept-Encoding` | 압축 방식 | gzip으로 압축해서 줘 |
| `Referer` | 어디서 왔는지 (이전 URL) | "구글에서 왔어요" |
| `User-Agent` | 브라우저 + OS 정보 | "나 크롬이고 윈도우야" |
| `Connection` | 연결 유지 여부 | keep-alive / close |

### GET vs POST 본문(Body)

```
GET  → URL에 데이터 담음 → 본문 없음
       https://naver.com/search?q=보안  ← 데이터가 URL에 노출

POST → 본문에 데이터 담음 → 본문 있음
       id=형윤&password=1234  ← URL 아닌 Body에 숨겨짐
```

### HTTP 메서드

| 메서드 | 용도 | 비고 |
|---|---|---|
| **GET** | 데이터 조회 | URL에 데이터 노출 → 보안 취약 |
| **POST** | 데이터 생성/전송 | Body에 데이터 → 최소한의 보안 |
| **PUT** | 데이터 전체 수정/생성 | 201 반환 가능 |
| **PATCH** | 데이터 일부 수정 | - |
| **DELETE** | 데이터 삭제 | - |
| **OPTIONS** | 지원 메서드 확인 | 공격자 정보수집에 악용 가능 |
| **HEAD** | 헤더만 조회 | 본문 없음 |
| **TRACE** | 요청 그대로 반사 | XST 공격에 악용 → **반드시 비활성화** |

> **TRACE 위험성**: HttpOnly로 보호된 쿠키도 TRACE 응답에 실려나와 탈취 가능 (XST 공격)
> **OPTIONS 위험성**: 공격자가 사용 가능한 메서드 목록 파악 → 불필요한 메서드 차단 필요

## 7. HTTP 응답 메시지

### 구조

```
[상태 라인]  HTTP/1.1 200 OK CR+LF
[헤더]       Content-Type: text/html; charset=utf-8 CR+LF
[헤더]       Content-Length: 1024 CR+LF
[헤더]       Date: Thu, 26 Mar 2026 09:00:00 GMT CR+LF
[헤더]       Server: nginx CR+LF
[헤더]       Connection: keep-alive CR+LF
[헤더]       Set-Cookie: session=abc123; HttpOnly CR+LF
[빈 줄]      CR+LF
[본문]       <html><body>안녕하세요</body></html>
```

### 상태 코드 — 암기법: **앞글자**

| 코드 | 앞글자 | 키워드 | 설명 |
|---|---|---|---|
| **100** | - | 정보 | 계속 진행해도 됨 |
| **200** | - | 성공 | 요청 정상 처리 |
| **201** | - | PUT | 새 자료 생성됨 |
| **301** | - | 항시(영구) | URL 영구 변경 (이사) |
| **302** | - | 임시 | URL 임시 변경 (출장) |
| **400** | **B** | Bad Request / 문법 오류 | 요청 문법 잘못됨 |
| **401** | **U** | Unauthorized / 인증 부족 | 로그인 필요 |
| **403** | **F** | Forbidden / 접근 차단 | 권한 없어서 거부 |
| **404** | **N** | Not Found / 자료 없음 | 해당 URL 없음 |
| **405** | **M** | Method Not Allowed / 미지원 | 해당 URL에 메서드 안 됨 |
| **500** | - | 손상·오류 발생 | 서버 내부 오류 |

> **4xx 앞글자 암기**: B-U-F-N-M (버스 unauthorized 타면 forbidden? no man!)
> **401 vs 403**: 401 = 누군지 몰라(인증), 403 = 알지만 거부(권한)
> **301 vs 302**: 301 = 영구 이사, 302 = 임시 출장

---

## 8. HTTP 메서드

### GET이 보안에 취약한 이유

```
https://naver.com/login?id=형윤&password=1234
```

1. URL에 데이터 노출 → 옆 사람이 화면 보면 보임
2. 브라우저 히스토리에 저장
3. 서버 로그에 URL 그대로 기록

### POST가 최소한의 보안성이 있는 이유

- 데이터가 Body에 담겨 URL에 안 보임
- 브라우저 히스토리에 안 남음
- 단, HTTPS 없으면 패킷 스니핑으로 볼 수 있음

```
GET  = 엽서 (내용이 밖에서 보임)
POST = 봉투 편지 (봉투 안에 숨겼지만 뜯으면 보임)
HTTPS = 암호화된 봉투 (뜯어도 해독 불가)
```

## 9. 쿠키 (Cookie)

### 정의

- HTTP Stateless를 보완하는 수단
- 사용자가 웹사이트 방문 시 생기는 **4KB 이하의 파일**
- **브라우저(클라이언트)** 에 저장
- 요청마다 자동으로 서버에 전송

### 동작 흐름

```
1. 처음 방문
브라우저 ──── 요청 ────────────────→ 서버
브라우저 ←─── Set-Cookie: id=형윤 ── 서버  (응답 헤더)

2. 다음 방문
브라우저 ──── Cookie: id=형윤 ────→ 서버  (요청 헤더, 자동 첨부)
```

> Set-Cookie = 서버가 브라우저에 쿠키 전달하는 **응답 헤더 이름**
> Cookie = 브라우저가 서버에 쿠키 보내는 **요청 헤더 이름**

### 쿠키 구성 요소

```
Set-Cookie: id=형윤; expires=2026-12-31; domain=naver.com; path=/; HttpOnly; Secure
```

| 속성 | 의미 |
|---|---|
| `이름=값` | 실제 쿠키 데이터 |
| `expires` / `max-age` | 만료일 |
| `domain` | 어느 사이트에서 쓸지 (naver.com) |
| `path` | 어느 경로에서 쓸지 (/mypage) |
| `HttpOnly` | JavaScript 접근 차단 |
| `Secure` | HTTPS에서만 전송 |

### domain vs path

```
domain = 건물 주소  (naver.com)
path   = 건물 내 층 (/mypage)
둘 다 맞아야 쿠키 전송!
```

### 쿠키 보안 속성

| 속성 | 방어 대상 | 설명 |
|---|---|---|
| **HttpOnly** | XSS 공격 | JavaScript로 쿠키 접근 완전 차단 |
| **Secure** | 스니핑(도청) | HTTPS에서만 쿠키 전송, HTTP면 전송 안 함 |
| **SameSite** | CSRF 공격 | 다른 사이트에서 쿠키 못 쓰게 차단 |

### 영속 쿠키 vs 세션 쿠키

| | 영속 쿠키 (Persistent) | 세션 쿠키 (Session Cookie) |
|---|---|---|
| 만료 | 설정한 날짜까지 유지 | 브라우저 닫으면 삭제 |
| 저장 | 하드디스크에 파일로 저장 | 메모리에만 저장 |
| 헤더 | expires= 또는 max-age= 있음 | expires 없음 |
| 용도 | 자동 로그인, 설정 저장 | 임시 로그인 유지 |

> **주의**: "세션 쿠키"와 "세션"은 다른 개념!
> - 세션 쿠키 = 브라우저 닫으면 사라지는 쿠키 (저장 위치는 브라우저)
> - 세션 = 실제 데이터가 서버에 저장되는 방식

### 쿠키 vs 세션 비교

| | 쿠키 | 세션 |
|---|---|---|
| 저장 위치 | **브라우저** | **서버** |
| 용량 | 4KB 이하 | 제한 없음 |
| 보안 | 낮음 (위변조 가능) | 높음 |
| 속도 | 빠름 | 상대적으로 느림 |
| 만료 | 설정한 날짜까지 | 브라우저 닫으면 보통 만료 |

```
쿠키  = 도서관 책을 집에 가져감 (데이터가 내 손에)
세션  = 책은 도서관에 있고 대출증(ID)만 가져감 (데이터는 서버에)
```

## 10. 세션 (Session)

### 정의

- 서버가 사용자를 기억하기 위해 **서버에 저장해두는 정보**
- 브라우저엔 세션ID만 전달, 실제 데이터는 서버 보관

### 동작 흐름

```
1. 로그인
브라우저 ──── id=형윤, pw=1234 ────→ 서버
                                      세션 생성
                                      {abc123: {user: 형윤}}
브라우저 ←─── Set-Cookie: sessionId=abc123 ───

2. 다음 요청
브라우저 ──── Cookie: sessionId=abc123 ────→ 서버
                                              abc123 조회 → "형윤이구나!"
```

### 세션 만료

```
브라우저 닫으면      →  세션 종료
일정 시간 안 쓰면   →  서버가 자동 삭제 (timeout)
로그아웃 하면        →  서버가 세션 강제 삭제
```

## 11. 세션 하이재킹

### 정의

> 공격자가 **남의 세션ID를 훔쳐서** 그 사람인 척 하는 공격

### 공격 원리

```
정상: 형윤 ──── sessionId=abc123 ────→ 서버 → "형윤이구나!"
공격: 공격자가 abc123 훔침
      공격자 ── sessionId=abc123 ────→ 서버 → "형윤이구나!" (속음)
```

### 세션ID 훔치는 방법

| 방법 | 설명 |
|---|---|
| 스니핑 | HTTP 공용 와이파이에서 패킷 캡처 |
| XSS | 악성 스크립트로 document.cookie 탈취 |
| 쿠키 탈취 | 브라우저 쿠키 파일 직접 훔치기 |

### 대응책

| 공격 방법 | 대응책 |
|---|---|
| 스니핑 | HTTPS 사용 (Secure 속성) |
| XSS | HttpOnly 속성 |
| 세션ID 예측 | 세션ID를 랜덤하게 길게 생성 |
| 장기간 탈취 | 세션 timeout 짧게 설정 |
| IP 변경 감지 | 세션에 IP 정보 묶기 |

> 비유: 놀이공원 입장 팔찌(세션ID)를 복사해서 차고 들어가는 것
> 직원(서버)은 팔찌만 보고 입장시켜줌

## 12. 브라우저 보안 영역

Windows IE 기반 **보안 영역(Security Zone)** 개념
브라우저가 사이트마다 다른 보안 수준을 적용하는 것

### 4가지 영역

| 영역 | 설명 | 보안 수준 | 예시 |
|---|---|---|---|
| **인터넷** | 일반 외부 웹사이트 | 중간 | google.com, naver.com |
| **로컬 인트라넷** | 회사/학교 내부 네트워크 | 낮음 | 192.168.x.x |
| **신뢰할 수 있는 사이트** | 관리자가 직접 등록한 사이트 | 가장 낮음 | 협력사 사이트 |
| **제한된 사이트** | 위험하다고 판단해 차단한 사이트 | 가장 높음 | 악성코드 배포 사이트 |

### 보안 수준 순서

```
제한된 사이트  (가장 엄격)
      ↓
인터넷  (중간)
      ↓
로컬 인트라넷
      ↓
신뢰할 수 있는 사이트  (가장 느슨)
```
