# ✍️ 전자서명 & PKI

---

## 📌 먼저 읽기! 핵심 개념 한눈에

| 기술 | 제공 서비스 | 키 종류 |
|---|---|---|
| 해시함수 | 무결성 | 없음 |
| MAC | 무결성 + 인증 | 대칭키 |
| **전자서명** | **무결성 + 인증 + 부인방지** | **비대칭키** |

> 💡 **전자서명 = 도장 + 신분증**
> 내 도장(개인키)으로 찍은 서명을
> 누구나 내 공개 신분증(공개키)으로 확인할 수 있는 것!

## 1. 전자서명 개요

### 전자서명의 두 가지 방식

| 방식 | 설명 | 단점 |
|---|---|---|
| 공개키 서명방식 | 서명자의 검증 정보를 공개 → 누구나 검증 가능 | - |
| 중재서명 방식 | 관용 암호방식 사용, 제3자가 서명/검증 중재 | 서명할 때마다 제3자 참여 필요 (불편!) |

### 전자서명 과정 ⭐⭐⭐⭐⭐

```
[서명 과정]
송신자(앨리스)
  ↓
메시지 + 서명 알고리즘 + 앨리스의 개인키
  ↓
서명값 생성
  ↓
메시지 + 서명값 전송

[검증 과정]
수신자(밥)
  ↓
메시지 + 서명값 + 검증 알고리즘 + 앨리스의 공개키
  ↓
결과가 참(True) → 메시지 수락 ✅
결과가 거짓(False) → 메시지 거절 ❌
```

> 💡 **핵심 포인트**
> 암호화 : 수신자의 공개키로 암호화 → 수신자의 개인키로 복호화
> 전자서명 : **송신자의 개인키**로 서명 → **송신자의 공개키**로 검증
> → 키의 사용 방향이 반대!

## 2. 전자서명 서비스 ⭐⭐⭐

| 서비스 | 내용 |
|---|---|
| 메시지 인증 | 밥이 받은 메시지가 앨리스로부터 왔음을 확신 |
| 메시지 무결성 | 해시함수를 사용해서 메시지가 변조되지 않았음을 보장 |
| 부인방지 | 앨리스가 나중에 "나 안 보냈어!"라고 해도 센터가 증거 제시 가능 |
| 기밀성 ⭐⭐⭐⭐⭐ | **❌ 전자서명만으로는 기밀성 보장 안 됨!** 기밀성이 필요하면 별도로 암호화 필요 |

---

## 3. 전자서명의 주요 기능 ⭐⭐⭐⭐⭐

> 💡 **비유 : 계약서에 도장 찍기**
> 1. 내 도장이 아니면 못 찍음 (위조불가)
> 2. 누가 찍었는지 알 수 있음 (서명자 인증)
> 3. 내가 찍은 건 부인 못함 (부인방지)
> 4. 찍은 후에 내용 못 바꿈 (변경불가)
> 5. 이 계약서 도장을 다른 계약서에 못 씀 (재사용불가)

| 기능 | 영문 | 설명 |
|---|---|---|
| 위조불가 | Unforgeable | 합법적인 **서명자만** 서명 생성 가능 |
| 서명자 인증 | User Authentication | 서명자를 **누구든지** 검증 가능 |
| 부인방지 | Non-repudiation | 서명 후 서명 사실을 **부인 불가** |
| 변경불가 | Unalterable | 서명한 문서의 **내용 변경 불가** |
| 재사용불가 | Not Reusable | 이 서명을 **다른 문서에 사용 불가** |

## 4. 전자서명 구조별 종류

### RSA 전자서명 ⭐⭐⭐

```
암호화 : 수신자 공개키로 암호화 → 수신자 개인키로 복호화
서명   : 송신자 개인키로 서명  → 송신자 공개키로 검증  ← 역할 반대!
```

### ElGamal 전자서명

```
이산대수 문제를 이용한 최초의 서명 방식
실제로는 거의 안 씀 → 대신 DSA(변형)를 많이 사용
```

### Schnorr 전자서명

```
ElGamal 기반
서명 크기를 줄인 구조
```

### DSS (Digital Signature Standard) ⭐⭐⭐

```
미국 전자서명 표준
ElGamal을 개량 → 계산량 획기적으로 줄임
이산대수 문제 기반
오직 전자서명 기능만 제공 (암호화/키교환 ❌)
NIST 제안
서명 길이 : 320bit (ElGamal보다 짧음)
```

### ECDSA (타원곡선 전자서명) ⭐⭐⭐

```
ECC 기반의 전자서명
짧은 비트 길이 → 빠른 처리 + 짧은 서명
미국 표준화
```

### KCDSA (국내 전자서명)

```
국내 표준
ElGamal 전자서명 개선
이산대수 문제 기반
DSS 변형
```

### 알고리즘별 기능 비교 ⭐⭐⭐

| 알고리즘 | 암호/복호 | 전자서명 | 키 교환 |
|---|---|---|---|
| RSA | ✅ | ✅ | ✅ |
| Diffie-Hellman | ❌ | ❌ | ✅ |
| DSS | ❌ | **✅ (전용)** | ❌ |
| 타원 곡선 | ✅ | ✅ | ✅ |

> ✅ **미국 표준 전자서명 3종**
> RSA 서명 / DSS / ECDSA

## 5. 전자서명 방식

### 메시지 복원형 전자서명

```
기존 공개키 암호방식 그대로 사용
별도 프로토콜 불필요
단점 : 시간이 많이 걸려서 실제로는 안 씀
```

### 부가형 전자서명 ⭐⭐⭐⭐⭐

> 💡 **비유 : 택배 운송장**
> 햄버거(메시지)를 배달할 때
> 내용물은 그대로 보내고
> 운송장(전자서명)을 따로 붙여서 보내는 것!

```
메시지 → [해시함수] → 해시값 → [개인키로 서명] → 서명값
                                                      ↓
메시지 + 서명값 함께 전송

장점 : 메시지가 아무리 길어도 서명은 한 번만!
단점 : 메시지 + 서명값 따로 전송 → 전송량 약간 증가
→ 효율적이라서 실제로 많이 사용!
```

## 6. 특수 전자서명

> 💡 **왜 특수 서명이 필요하냐?**
> 공개키 서명은 누구나 검증 가능 → 사생활 침해 우려
> 특수한 상황에서 검증을 제한하고 싶을 때 사용!

| 종류 | 설명 | 핵심 |
|---|---|---|
| 부인방지 전자서명 | 검증 시 **서명자의 도움** 필수 | D.Chaum 제안 |
| 의뢰 부인방지 서명 | **재판관만** 부인 과정 수행 가능 | 거짓말 탐지기 기능 |
| 수신자 지정 서명 | **지정된 수신자만** 확인 가능 | 서명 남용을 수신자가 통제 |
| 은닉 서명 (블라인드 서명) ⭐⭐⭐⭐⭐ | 서명자가 **내용을 모르는 채로** 서명 | 익명성 유지, 예) 특허등록 |
| 위임 서명 | 대리인이 **대신 서명** | 위임 관계 |
| 다중 서명 | 동일 문서에 **여러 사람이 서명** | 서명 검증 시간 단축 |

> 💡 **은닉 서명 비유 : 봉투에 넣고 서명받기**
> 중요한 문서를 봉투에 넣고
> 서명자는 봉투 바깥에만 서명 (내용을 못 봄)
> 봉투를 열면 서명이 문서에 전사되어 있는 것!
> → 익명 투표, 특허 등록에 활용

## 7. 전자서명의 응용

### 전자투표

```
선거의 4원칙 : 보통, 비밀, 직접, 평등
```

**전자투표 요구사항** ⭐⭐⭐

```
1. 완전성   : 모든 투표가 정확하게 집계
2. 익명성   : 투표 결과로 투표자 구별 불가
3. 건전성   : 부정 투표자에 의한 방해 없음
4. 이중투표방지 : 한 번만 투표 가능
5. 정당성   : 외부 영향 없음
6. 적임성   : 투표 권한 가진 자만 투표 가능
7. 검증 가능 : 누구나 결과 확인 가능
```

**전자투표 방식** ⭐⭐⭐

| 방식 | 투표 장치 | 관리 | 기술 난이도 | 특징 |
|---|---|---|---|---|
| PSEV | 전자투표기 | 상 | 하 | 기존 투표소에서 투표 → 기록장치를 개표소로 **직접 옮겨서** 집계 |
| 키오스크 | 전자투표기 | 중 | 중 | 비지정 투표소 → 개표소와 **온라인 연결**해서 자동 집계 |
| REV | 모바일/PC/TV | 하 | 상 | **가정/직장에서 인터넷**으로 투표 → 자동 집계 |

### 전자입찰 시스템

```
가장 싼 가격을 제시한 응찰자와 계약하는 입찰을 인터넷으로 구현
입찰 금액을 해시함수로 해시한 후 제출!
```

**요구사항** ⭐⭐⭐ (참고용)

```
독립성 / 비밀성 / 무결성 / 공평성 / 안전성
```

**문제점**
```
❌ 메시지 유출
❌ 입찰자-서버 공모
❌ 입찰자간 공모
❌ 입찰자-입찰 공고자 공모
```

## 8. 전자서명으로 해결할 수 없는 문제 → PKI 등장

> 💡 **문제 상황**
> 앨리스의 공개키라고 받았는데... 진짜 앨리스 거 맞아?
> 이브가 "이게 앨리스 공개키야!" 하고 가짜를 줬을 수도 있잖아!

```
해결책 : 인증서 (신분증)
→ 신뢰할 수 있는 제3자(CA)가 "이 공개키는 진짜 앨리스 거!" 라고 보증
→ 이 체계 전체를 PKI (공개키 기반구조) 라고 함
```

---

## 9. PKI (Public Key Infrastructure) ⭐⭐⭐⭐⭐

> 💡 **비유 : 인터넷 주민등록 체계**
> CA = 동사무소 (신분증 발급)
> 공인인증서 = 주민등록증
> PKI = 이 전체 체계

**PKI가 제공하는 보안 서비스**
```
기밀성, 무결성, 접근제어, 인증, 부인방지
```

**PKI 구성요소**
```
인증기관(CA), 등록기관(RA), 검증기관(VA), 사용자, 저장소
```

---

## 10. PKI 구성요소 ⭐⭐⭐⭐⭐

### 인증기관 계층 구조

```
PAA (정책 승인기관)
 ↓ PKI 전반 정책 수립, 루트 CA 역할
PCA (정책 인증기관) ← KISA
 ↓ 도메인 내 정책 수립, CA의 공개키 인증
CA (인증기관)
 ↓ 사용자 인증서 발급/폐지
사용자
```

### 각 구성요소 역할 ⭐⭐⭐⭐⭐

| 구성요소 | 영문 | 역할 | 비유 |
|---|---|---|---|
| 정책 승인기관 | PAA | PKI 전체 정책 수립, 루트 CA | 행정안전부 |
| 정책 인증기관 | PCA | 도메인 정책 수립, CA 공개키 인증 | 시청 |
| 인증기관 | CA | **인증서 발급/폐지** | 구청 |
| 등록기관 | RA | CA 대신 **신분/소속 확인** (선택 요소!) | 은행 |
| 검증기관 | VA | **인증서 유효성 확인** (감사기관) | 감사원 |
| 저장소 | Repository | 인증서/CRL **저장** (LDAP 사용) | 주민등록 DB |
| 사용자 | End Entity | 인증서 사용, 서명 생성/검증 | 일반 시민 |

> ⚠️ **RA는 선택 요소 (Optional)**
> RA가 없으면 CA가 RA 역할까지 수행!

## 11. PKI 형태 ⭐⭐⭐

### 계층 구조

```
         [루트 CA]
        /         \
    [CA1]         [CA2]
    /   \         /   \
[사용자] [사용자] [사용자] [사용자]

트리 형태 (상하 관계 명확)
상위 CA가 하위 CA에 인증서 발행
```

### 네트워크 구조

```
[CA1] ←→ [CA2]
  ↕           ↕
[CA3] ←→ [CA4]

각 CA가 독립적으로 운영
상위 CA 없음 (수평 관계)
```

### 계층 vs 네트워크 장단점 ⭐⭐⭐⭐⭐

| 구분 | 계층 구조 | 네트워크 구조 |
|---|---|---|
| 장점 | 인증경로 탐색 **용이** | 인증경로 **단순** / CA 비밀키 노출 시 **국소적** 피해 |
| 단점 | 최상위 CA 비밀키 노출 시 **피해 막대** | 인증경로 탐색 **복잡** |

## 12. 인증서 (X.509) ⭐⭐⭐⭐⭐

> 💡 **인증서 = 공개키 신분증**
> 이름/소속/공개키 + CA의 서명이 담긴 전자 문서
> CA의 개인키로 서명되어 있어서 위조 불가!

**사용처**
```
IPSec (VPN), SSL/TLS (웹 보안), SET (전자상거래), S/MIME (이메일 보안)
```

### X.509v3 인증서 필드 ⭐⭐⭐⭐⭐

| 필드 | 필수/선택 | 내용 |
|---|---|---|
| 버전 | 필수 | X.509 버전 (v1/v2/v3) |
| 일련번호 | 필수 | CA가 발급한 고유 번호 (반드시 유일!) |
| 서명 알고리즘 식별자 | 필수 | 서명에 사용된 알고리즘 |
| 발행자 이름 | 필수 | 인증서 발급한 CA 이름 |
| 유효기간 | 필수 | **notBefore** (시작) ~ **notAfter** (종료) |
| 주체 이름 | 필수 | 인증서 소유자 이름 (하위 CA에 발급 시 소유자 = CA) |
| **주체의 공개키 정보** | 필수 | **인증서의 핵심!** 소유자의 공개키 |
| 발행자 유일 식별자 | 선택 | 이름 중복 시 구별 (사용 비권장) |
| 주체 유일 식별자 | 선택 | 이름 재사용 시 구별 |
| 확장 | 선택 | v3에서 추가, 사적 정보 포함 가능 |
| **서명** | 필수 | **CA의 개인키**로 전체 필드를 암호화한 서명값 |

> 💡 **서명 필드 생성 과정**
> 나머지 전체 필드 → [해시함수] → 해시코드 → [CA의 개인키로 암호화] → 서명값

### 인증서 확장 영역 ⭐⭐⭐

```
1. 키와 정책 정보
   - 기관키 식별자 : CA가 여러 키 쓸 때 구별
   - 사용자키 식별자 : 사용자가 여러 인증서 가질 때 구별
   - 키 사용 : 암호화/서명/인증서 서명 용도 정의
   - 인증서 정책 : CA의 발급 정책

2. 사용자/발행자 속성 (대체 이름)
   - 이메일, DNS이름, IP주소, URI 등

3. 인증 경로 제약
   - 기본 제한 : 사용자가 CA인지 여부 구분
   - 이름 제한 : 특정 이름 형태에 제약
   - 정책 제한 : 정책 관련 제한
```

## 13. 인증서 폐지 목록 (CRL) ⭐⭐⭐

> 💡 **비유 : 분실 신고된 주민등록증 목록**
> 유효기간이 남아있어도 이미 도용됐다면 사용 못하게 막아야 함!

**인증서 폐지 사유**
```
1. 사용자의 개인키가 노출/훼손
2. CA가 사용자를 더 이상 인증 안 함
3. CA의 개인키가 침해됨
```

**CRL 특징**
```
- 폐지된 인증서 목록
- 인증기관 저장소/디렉터리에 등재
- 각 폐기 인증서는 일련번호로 확인
- 갱신 주기 : 보통 12~48시간
```

## 14. 인증서 운영 프로토콜 ⭐⭐⭐⭐⭐

### OCSP (온라인 인증서 상태 검증 프로토콜)

```
실시간으로 인증서 상태 확인 (CRL보다 빠름!)
갱신 주기 : 약 1분 단위
백그라운드에서 자동 실행

구성요소 : OCSP 클라이언트 + OCSP 서버 + 인증 서버
```

### CMP (인증서 관리 프로토콜)

```
PKI 환경에서 인증서 관리 서비스를 위한
PKI 실체들 간의 통신 프로토콜
```

## 📌 최종 핵심 암기 요약

```
✅ 전자서명
   서명 : 송신자 개인키 사용
   검증 : 송신자 공개키 사용
   (암호화와 키 방향 반대!)
   기밀성은 ❌ → 별도 암호화 필요

✅ 전자서명 5대 기능
   위조불가 / 서명자인증 / 부인방지 / 변경불가 / 재사용불가

✅ 알고리즘 기능
   RSA      = 암복호화 + 서명 + 키교환
   DH       = 키교환만
   DSS      = 서명만 (전용!)
   타원곡선  = 암복호화 + 서명 + 키교환

✅ 특수 서명
   은닉 서명 = 내용 모르고 서명, 익명성 유지
   위임 서명 = 대리 서명
   다중 서명 = 여러 명이 같은 문서에 서명

✅ PKI 구성요소
   PAA → PCA(KISA) → CA → 사용자
   RA  = 선택 요소 (신분 확인 대리)
   VA  = 인증서 유효성 검증 (감사 역할)
   저장소 = LDAP 사용

✅ PKI 구조 비교
   계층형 : 탐색 쉬움, 최상위 CA 노출 시 피해 막대
   네트워크 : 탐색 복잡, CA 노출 시 국소적 피해

✅ X.509 인증서 핵심 필드
   주체의 공개키 정보 = 인증서의 핵심!
   서명 = CA의 개인키로 암호화
   유효기간 = notBefore ~ notAfter

✅ 인증서 폐지
   CRL : 12~48시간 갱신
   OCSP : 실시간(1분 단위), 더 빠름
```
