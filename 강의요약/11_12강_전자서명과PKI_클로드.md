# ✍️ 전자서명 & PKI

---

## 📌 먼저 읽기! 핵심 개념 한눈에

| 기술 | 제공 서비스 | 키 종류 |
|---|---|---|
| 해시함수 | 무결성 | 없음 |
| MAC | 무결성 + 인증 | 대칭키 |
| **전자서명** | **무결성 + 인증 + 부인방지** | **비대칭키** |

> 💡 **전자서명 = 디지털 도장**
> 현실에서 도장을 찍으면 "내가 동의했다"는 증거가 되듯이
> 전자서명도 "내 개인키로 서명했다"는 증거가 됨!
> 위조 불가능하고, 나중에 "나 안 했어"라고 부인할 수 없음

## 1. 전자서명 개요

### 전자서명 방식 분류

```
전자서명
  ├── 공개키 서명방식  → 누구나 검증 가능, 간편
  └── 중재서명 방식    → 서명할 때마다 제3자(중재자) 참여 필요 (불편!)
```

### 전자서명 과정 ⭐⭐⭐⭐⭐

> 💡 **비유 : 공증 도장**
> 앨리스(송신자)가 자기 개인 도장(개인키)으로 문서에 찍음
> 밥(수신자)이 앨리스의 공개 도장 정보(공개키)로 진짜인지 확인

```
[서명 과정 - 송신자]
메시지 → [서명 알고리즘 + 송신자 개인키] → 서명값 생성
메시지 + 서명값 → 수신자에게 전송

[검증 과정 - 수신자]
메시지 + 서명값 → [검증 알고리즘 + 송신자 공개키] → 참/거짓
참  → 메시지 수락 ✅
거짓 → 메시지 거절 ❌
```

> ⚠️ **암호화 vs 전자서명 키 방향 비교! (시험 단골)**
> 암호화  : 수신자 **공개키**로 암호화 → 수신자 **개인키**로 복호화
> 전자서명 : 송신자 **개인키**로 서명  → 송신자 **공개키**로 검증
> → 전자서명은 키 사용 방향이 반대!

## 2. 전자서명 서비스 ⭐⭐⭐

| 서비스 | 내용 |
|---|---|
| 메시지 인증 | 메시지가 진짜 앨리스로부터 왔다는 것을 확인 (데이터 근원 인증) |
| 무결성 | 해시함수 사용 → 메시지가 변조되지 않았음을 보장 |
| 부인방지 | 나중에 "나 안 보냈어"라고 부인 불가 (센터가 저장한 메시지로 증명) |
| **기밀성** | **❌ 전자서명만으로는 기밀성 보장 안 됨!** |

> ⚠️ **시험 주의! 기밀성이 필요하면?**
> 전자서명에 추가로 **암호화(대칭키 or 공개키)** 를 별도로 적용해야 함!

## 3. 전자서명의 주요 기능 5가지 ⭐⭐⭐⭐⭐

> 💡 **비유 : 좋은 도장의 조건**
> 1. 나만 만들 수 있어야 하고 (위조불가)
> 2. 누가 찍었는지 알 수 있어야 하고 (서명자 인증)
> 3. 나중에 "내가 안 찍었어" 못하고 (부인방지)
> 4. 도장 찍은 내용을 못 바꾸고 (변경불가)
> 5. 다른 문서에 재사용 못함 (재사용불가)

```
1. 위조불가 (Unforgeable)
   → 합법적인 서명자만이 전자서명 생성 가능

2. 서명자 인증 (User Authentication)
   → 서명자를 누구든지 검증 가능

3. 부인방지 (Non-repudiation)
   → 서명 후 서명한 사실을 부인 불가

4. 변경불가 (Unalterable)
   → 서명한 문서 내용을 변경 불가

5. 재사용불가 (Not reusable)
   → 한 문서의 서명을 다른 문서에 사용 불가
```

## 4. 전자서명 알고리즘 종류

### 알고리즘별 기능 비교 ⭐⭐⭐

| 알고리즘 | 암호/복호 | 전자서명 | 키 교환 |
|---|---|---|---|
| RSA | ✅ | ✅ | ✅ |
| Diffie-Hellman | ❌ | ❌ | ✅ |
| DSS (DSA) | ❌ | ✅ | ❌ |
| 타원 곡선 (ECC) | ✅ | ✅ | ✅ |

> 💡 **DSS = 서명 전용!**
> DSS/DSA는 전자서명 기능만 제공하도록 설계됨
> 암호화나 키 교환에는 사용 불가!

### RSA 전자서명 ⭐⭐⭐

```
일반 RSA 암호화와 키 방향이 반대!

서명 : 송신자 개인키로 서명  (S = M^d mod n)
검증 : 송신자 공개키로 검증  (M = S^e mod n)
```

### ElGamal 전자서명

```
이산대수 문제를 이용한 최초의 서명 방식
같은 키를 사용하지만 알고리즘은 암호화와 다름
실제로는 거의 사용 안 함 → DSA가 대신 사용됨
```

### Schnorr 전자서명

```
ElGamal 기반이지만 서명 크기를 줄인 방식
```

### DSS / DSA ⭐⭐⭐

```
미국 전자서명 표준 (NIST 제안)
ElGamal을 개량한 방식 → 계산량 대폭 감소
이산대수 문제 기반
서명 길이 : 320bit (ElGamal보다 짧음)
전자서명 기능만 제공 (암호화 불가!)
```

### ECDSA (타원곡선 전자서명) ⭐⭐⭐

```
ECDSA = Elliptic Curve DSA
짧은 비트 길이 → 빠른 처리 + 짧은 서명
미국 표준화
```

### KCDSA (국내 전자서명)

```
국내 표준
ElGamal 전자서명을 개선한 방식
이산대수 문제 기반
DSS를 변형한 방식
```

### 미국 전자서명 표준 3종

```
1. RSA 서명
2. DSS (DSA)
3. ECDSA
```

## 5. 전자서명 방식

### 메시지 복원형 전자서명

```
공개키 암호방식 그대로 사용
별도 프로토콜 불필요 → 장점
시간이 많이 걸림 → 실제로는 거의 사용 안 함
```

### 부가형 전자서명 ⭐⭐⭐⭐⭐

> 💡 **비유 : 햄버거 영수증**
> 햄버거(메시지)를 주문하고
> 영수증(해시값)에 도장(서명)을 찍어서 햄버거에 붙여서 전달
> 받는 사람은 햄버거도 받고 영수증 도장도 확인!

```
[동작 방식]
메시지 → [해시] → 해시값 → [개인키로 서명] → 서명값
메시지 + 서명값 → 함께 전송

[특징]
메시지가 아무리 길어도 해시값만 서명하면 됨 → 효율적!
실제로 가장 많이 사용되는 방식
단점 : 메시지와 서명을 따로 전송 → 전송량 약간 증가
```

## 6. 특수 전자서명

> 💡 **왜 특수 전자서명이 필요하냐?**
> 일반 전자서명은 공개키가 공개되어 있어서 누구나 검증 가능
> → 때로는 특정 사람만 검증하게 하거나, 익명성이 필요한 상황이 있음!

### 부인방지 전자서명

```
서명을 검증할 때 반드시 서명자의 도움이 있어야만 검증 가능
(D.Chaum 제안)
```

### 의뢰 부인방지 서명

```
재판관만이 부인 과정을 수행할 수 있음
일종의 거짓말 탐지기 기능 제공
```

### 수신자 지정 서명

```
지정된 수신자만 서명 확인 가능
필요시 제3자에게 서명이 진짜임을 증명 가능
서명 남용을 수신자(검증자)가 통제
```

### 은닉 서명 (블라인드 서명) ⭐⭐⭐⭐⭐

> 💡 **비유 : 봉투 속 도장**
> 문서를 봉투에 넣고 봉투 위에 도장을 찍음
> 서명자는 봉투 안 내용을 모르는 채로 서명!
> 봉투를 열면 문서에 도장이 찍혀 있음

```
서명자가 서명 내용을 모르는 상태에서 서명
익명성 유지 가능 (서명자와 서명문을 연결 불가)
예시 : 특허등록, 전자투표
```

### 위임 서명

```
서명자가 위임 서명자에게 대리 서명 권한 부여
```

### 다중 서명

```
동일한 전자문서에 여러 사람이 서명
단순 반복 서명보다 효율적
(단순 반복 시 → 서명 길이 증가 + 검증 시간 오래 걸림)
```

## 7. 전자서명의 응용

### 전자투표

**선거의 4원칙** : 보통, 비밀, 직접, 평등

**전자투표 시스템 요구사항 ⭐⭐⭐**

```
1. 완전성     : 모든 투표가 정확하게 집계
2. 익명성     : 투표결과로 투표자 식별 불가
3. 건전성     : 부정 투표자가 선거 방해 불가
4. 이중투표방지 : 두 번 이상 투표 불가
5. 정당성     : 투표에 외부 영향 없어야 함
6. 적임성     : 투표 권한자만 투표 가능
7. 검증가능   : 누구나 투표 결과 확인 가능
```

**전자투표 방식 ⭐⭐⭐**

| 구분 | 장치 | 선거관리 | 기술쟁점 | 특징 |
|---|---|---|---|---|
| PSEV | 전자투표기 | 상 | 하 | 기존 투표소 → 기록장치를 개표소로 **옮겨와** 집계 |
| 키오스크 | 전자투표기 | 중 | 중 | 임의 투표소 → 개표소와 **온라인 연결** → 자동 집계 |
| REV | 모바일/PC/TV | 하 | 상 | 가정/직장에서 **인터넷**으로 투표 → 자동 집계 |

### 전자입찰 시스템

```
가장 싼 가격 제시한 응찰자와 계약하는 입찰방식을 인터넷으로 구현
입찰금액을 해시함수로 해시해서 제출
```

**요구사항 ⭐⭐⭐ (참고용)**
```
독립성 / 비밀성 / 무결성 / 공평성 / 안전성
```

**문제점**
```
네트워크 메시지 유출 / 입찰자↔서버 공모 / 입찰자간 공모 / 입찰자↔공고자 공모
```

## 8. 전자서명으로 해결할 수 없는 문제

> 💡 **문제 : 공개키가 진짜인지 어떻게 아냐?**
> 앨리스의 공개키라고 받았는데 실제로는 이브의 공개키일 수도 있음!
> → 이걸 해결하기 위해 **인증서(신분증)** 가 필요!

```
인증서 = 공개키에 신뢰할 수 있는 제3자(CA)가 전자서명을 해준 것
       = 공개키의 신분증
       = PKI의 근간
```

## 9. PKI (공개키 기반 구조) ⭐⭐⭐⭐⭐

> 💡 **비유 : 인터넷 주민센터**
> CA(인증기관) = 동사무소 (신분증 발급)
> 인증서(공인인증서) = 주민등록증 (공개키 신분증)
> PKI = 전체 행정 체계 (주민센터 → 구청 → 시청)

**PKI 제공 서비스**
```
기밀성 / 무결성 / 접근제어 / 인증 / 부인방지
```

**PKI 구성 요소**
```
인증기관(CA) / 등록기관(RA) / 사용자 / 신뢰 당사자 / 저장소
```

## 10. PKI 구성요소 상세 ⭐⭐⭐⭐⭐

### CA 계층 구조

```
PAA (정책승인기관)        ← 루트 CA, 최상위
 └── PCA (정책인증기관)   ← KISA 담당
      └── CA (인증기관)   ← 실제 인증서 발급
           └── 사용자
```

| 기관 | 역할 |
|---|---|
| **PAA** (Policy Approving Authority) | PKI 전반 정책 수립, **루트 CA** 역할 |
| **PCA** (Policy Certification Authority) | PAA 아래, CA가 따를 정책 수립, CA 공개키 인증 |
| **CA** (Certification Authority) | 사용자 인증서 **발급/폐지(CRL)** 관리, 상호인증 제공 |
| **VA** (Validation Authority) | 인증서 유효성 **검증** (감사기관 역할) |
| **RA** (Registration Authority) | 사용자 **신분/소속 확인** (선택적 요소!) |
| **저장소** | 인증서 저장 DB (LDAP 이용, X.500 표준) |

> ⚠️ **RA는 선택적(Optional) 요소!**
> RA가 없으면 CA가 RA 기능을 직접 수행
> 예시 : 은행 지점 = RA 역할

**사용자 역할**
```
✅ 개인키/공개키 쌍 생성 (우리나라는 CA가 생성)
✅ 전자서명 생성 및 검증
✅ 인증서 획득 및 상태 확인
✅ 인증서 취소목록(CRL)으로 상태 검증
```

## 11. PKI 형태 ⭐⭐⭐⭐⭐

### 계층 구조

```
         [루트 CA]
        /         \
    [CA1]         [CA2]
   /    \        /    \
[사용자] [사용자] [사용자] [사용자]

트리(Tree) 형태
상위 CA가 하위 CA에 인증서 발행
하위 CA는 상위 CA의 정책에 영향 받음
```

### 네트워크 구조

```
[CA1] ←→ [CA2]
  ↕           ↕
[CA3] ←→ [CA4]

각 CA가 독립적으로 존재
상위 CA 영향 없이 직접 상호인증
```

### 혼합형 구조

```
계층 구조 + 네트워크 구조 혼합
```

### 계층 vs 네트워크 장단점 ⭐⭐⭐⭐⭐

| 구분 | 계층적 구조 | 네트워크형 구조 |
|---|---|---|
| 장점 | 인증경로 탐색 **용이** | 인증경로 **단순**, CA 비밀키 노출 시 **국소적** 피해 |
| 단점 | 최상위 CA 비밀키 노출 시 **피해 막대** | 인증경로 탐색 **복잡** |

## 12. 인증서 (X.509) ⭐⭐⭐⭐⭐

> 💡 **인증서 = 공개키 신분증**
> 인증서 안에는 이름, 소속, 이메일, **공개키**가 들어있고
> CA가 자기 개인키로 전자서명 해줌
> → "이 공개키는 진짜 이 사람 거야!"라고 CA가 보증

**인증서 사용처**
```
IP 보안 / SSL/TLS (웹 HTTPS) / SET (전자상거래) / S/MIME (이메일 보안)
```

### X.509v3 인증서 프로파일 ⭐⭐⭐⭐⭐

| 필드 | 필수/선택 | 내용 |
|---|---|---|
| 버전 | 필수 | X.509 버전 |
| 일련번호 | 필수 | CA가 부여하는 **유일한** 번호 |
| 서명 알고리즘 식별자 | 필수 | CA가 서명에 사용한 알고리즘 |
| 발행자 이름 | 필수 | 인증서 발행한 CA 이름 |
| 유효기간 | 필수 | **notBefore** (시작) ~ **notAfter** (종료) |
| 주체 이름 | 필수 | 인증서 소유자 이름 (하위 CA 발급 시 소유자 = CA) |
| **주체의 공개키 정보** | **필수** | **인증서의 핵심! 소유자의 공개키** |
| 발행자 유일 식별자 | 선택 | 이름 중복 시 구별 (사용 비권장) |
| 주체 유일 식별자 | 선택 | 이름 재사용 시 구별 |
| 확장 | 선택 | 추가 정보 (v3에서 추가) |
| **서명** | **필수** | **전체 필드를 해시 후 CA 개인키로 암호화** |

> 💡 **서명 필드 동작 방식**
> 나머지 모든 필드 → [해시] → 해시값 → [CA 개인키로 암호화] → 서명값

### 인증서 확장 영역 ⭐⭐⭐

```
[키와 정책 정보]
  기관키 식별자   : 서명에 사용된 개인키에 대응하는 공개키 구분 (여러 개일 때)
  사용자키 식별자 : 여러 인증서 중 특정 공개키 포함 인증서 빠르게 구분 (여러 개일 때)
  키 사용         : 키 용도 정의 (암호화 / 서명 / 인증서 서명)
  인증서 정책     : CA가 발행 정책 명기

[사용자/발행자 속성]
  사용자 대체 이름 : 이메일, DNS, IP주소, URI 등 추가 신원
  발급자 대체 이름 : 발급자의 다양한 대체 이름

[인증 경로 제약 조건]
  기본 제한 : 사용자가 인증기관인지 여부 구분
  이름 제한 : 사용자 이름에 제약 조건
  정책 제한 : 인증기관에게 발급된 인증서의 정책 제한
```

## 13. 인증서 폐지 목록 (CRL) ⭐⭐⭐

> 💡 **비유 : 블랙리스트**
> 유효기간이 남아있어도 문제가 생기면 인증서를 강제로 취소해야 함
> 취소된 인증서 목록 = CRL (Certificate Revocation List)

**폐지 사유**
```
1. 사용자 개인키가 노출/훼손
2. CA가 사용자를 더 이상 인증 불가
3. CA의 개인키가 침해된 경우
```

**CRL 특징**
```
갱신 주기 : 보통 12~48시간
인증서 확인 : 일련번호로 식별
```

## 14. 인증서 운영 프로토콜 ⭐⭐⭐⭐⭐

### OCSP (온라인 인증서 상태 검증 프로토콜)

```
실시간으로 인증서 상태 확인 (CRL보다 훨씬 빠름!)
갱신 주기 : 약 1분 단위
구성 : OCSP 클라이언트 + OCSP 서버 + 인증 서버
백그라운드에서 자동 수행
```

### CMP (인증서 관리 프로토콜)

```
PKI 환경에서 인증서 관리 서비스를 위한
PKI 실체들 간의 통신 프로토콜
```

### CRL vs OCSP 비교

| 구분 | CRL | OCSP |
|---|---|---|
| 방식 | 목록 다운로드 후 확인 | 실시간 조회 |
| 갱신 주기 | 12~48시간 | ~1분 |
| 속도 | 느림 | 빠름 |

## 📌 최종 핵심 암기 요약

```
✅ 전자서명 기능 5가지
   위조불가 / 서명자 인증 / 부인방지 / 변경불가 / 재사용불가

✅ 전자서명 vs 암호화 키 방향
   암호화  : 수신자 공개키 → 수신자 개인키
   전자서명 : 송신자 개인키 → 송신자 공개키 (반대!)

✅ 전자서명 제공 서비스
   무결성 + 인증 + 부인방지
   ❌ 기밀성은 별도 암호화 필요!

✅ 알고리즘 기능
   DSS/DSA  = 서명 전용 (암호화 불가!)
   DH       = 키 교환 전용 (서명/암호화 불가!)
   RSA, ECC = 암호화 + 서명 + 키 교환 모두 가능

✅ 특수 전자서명
   은닉서명 = 서명자가 내용 모르고 서명 → 익명성 보장
   위임서명 = 대리 서명
   다중서명 = 여러 사람이 같은 문서에 서명

✅ PKI 구성요소
   PAA → PCA(KISA) → CA → 사용자
   RA = 선택적 요소! (없으면 CA가 대신 수행)

✅ 인증서 핵심 필드
   주체의 공개키 정보 = 인증서의 핵심!
   서명 = 전체 필드 해시 후 CA 개인키로 암호화

✅ 인증서 폐지
   CRL  : 12~48시간 갱신
   OCSP : 실시간(~1분) 갱신 → CRL보다 빠름

✅ PKI 형태
   계층적  = 탐색 쉬움, 루트CA 노출 시 피해 막대
   네트워크 = 탐색 복잡, CA 노출 시 국소적 피해
```
