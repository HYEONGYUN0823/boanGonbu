# 각종 애플리케이션 보안 위협 및 대응책

---

## 1. 살라미 공격 (Salami Attack)

> **"티끌 모아 태산"** — 눈치채지 못할 만큼 아주 작은 금액을 여러 번에 걸쳐 빼내는 금융 사기 수법

### 개념

- 이탈리아 살라미 소시지를 **조금씩 얇게 썰어 먹는 모습**에서 이름을 따옴
- 한 번에는 극히 소액이라 피해자가 인식하기 어렵지만, 누적되면 큰 금액이 됨

### Round Down (반올림 취약점 악용)

- 소수점 이하를 **반올림 또는 절삭**할 때 발생하는 미세한 차이를 악용
- 예시: `0.75$ → 0.5$` 로 처리 → **0.25$를 공격자가 취득**
- 수많은 거래에서 반복되면 상당한 금액이 누적됨

### 핵심 한 줄

```
아주 작은 금액을 반복적으로 빼내어 큰 피해를 유발하는 공격
```

## 2. 데이터 디들링 (Data Diddling)

> **원시 데이터 자체를 몰래 조작**하는 공격 — 입력 전후 또는 처리 중 데이터를 위조/변조

### 개념

- 처리되기 전의 원본 데이터를 **슬쩍 바꿔치기**하는 수법
- 디스크에 대체 자료를 미리 만들어 두었다가 교체하기도 함

### 특징

| 항목 | 내용 |
|------|------|
| 주요 행위자 | 데이터에 접근 가능한 **내부자** (전산 처리 담당자 등) |
| 대상 | 급여 데이터, 계좌 잔액, 성적, 재고 수량 등 |
| 발견 난이도 | 높음 (원본과 위조본의 구분이 어려움) |

### 핵심 한 줄

```
데이터를 처리하기 전에 원본을 몰래 바꾸는 내부자 공격
```

## 3. 이블 트윈 공격 (Evil Twin Attack)

> **가짜 AP(액세스 포인트)** 를 만들어 사용자를 속이는 무선 네트워크 공격

### 개념

- 공격자가 **정상 Wi-Fi와 동일하거나 유사한 이름의 Rogue AP**를 생성
- 사용자가 가짜 AP에 접속하면 **중간에서 정보를 가로챔 (MITM)**

### 공격 시나리오

```
정상 AP: Cafe_WiFi
가짜 AP: Cafe_WiFi  또는  Cafe_WiFi_Free   ← 사용자가 구별하기 어려움
          ↓
사용자 접속 → 공격자가 트래픽 도청 · 조작
```

### 관련 예시

- 페이스북은 이메일 주소를 식별자로 사용하기 때문에, 이메일만 다르면 **동일한 이름·생년월일로 가입 가능** → 이블 트윈처럼 **동일 신원을 사칭**할 수 있는 허점과 유사

### 핵심 한 줄

```
진짜와 똑같이 생긴 가짜 Wi-Fi로 사용자를 속여 정보를 탈취하는 공격
```

## 4. 자바(Java) 보안

### 4-1. 보안 3요소 (샌드박스의 구성 컴포넌트)

| 컴포넌트 | 역할 | 비유 |
|----------|------|------|
| **클래스 로더** (Class Loader) | JVM에 클래스를 로드·업로드 / 올바른 클래스 로드 여부 검사 | 부품을 창고에서 꺼내오는 사람 |
| **바이트코드 검사기** (Bytecode Verifier) | 로드된 클래스가 정상 바이트코드인지 검증, 타입 안정성 보장, 메모리 경계 체크 | 부품이 불량인지 검사하는 검사원 |
| **보안 관리자** (Security Manager) | 신뢰할 수 없는 클래스가 시스템 자원(파일·네트워크 등)에 접근하지 못하도록 제한 | "여기까지만 허용" 하는 경비원 |

> ⚠️ **세 컴포넌트 중 하나라도 우회되면 보안 메커니즘 전체가 무너질 수 있다**

### 4-2. 샌드박스 (Sandbox)

> 외부에서 받은 프로그램을 **제한된 울타리(JVM) 안에서만 실행**시켜 시스템을 보호하는 방식

```
외부 코드 다운로드
       ↓
  [클래스 로더]  → 클래스를 JVM 안으로 불러옴
       ↓
[바이트코드 검사기] → 코드가 안전한지 검증
       ↓
  [보안 관리자]  → 시스템 자원 접근 통제
       ↓
  안전한 실행 환경 (Sandbox)
```

**비유**: 아이가 아무 데나 뛰어다니지 못하게 **놀이터 울타리 안에서만 놀게 하는 것**

## 5. 모바일 코드 (Mobile Code)

> 네트워크를 통해 **다른 플랫폼으로 이동하여 동일하게 실행되는 프로그램**

> 여기서 "모바일"은 스마트폰이 아닌, **스스로 이동하며 퍼지는** 의미

### 종류

#### ① Java Applet
- 브라우저에서 동작하는 작은 자바 프로그램
- 서버에서 다운로드 → 클라이언트 PC의 자원으로 실행
- **대응책: 샌드박스(Sandbox)** → JVM 내 보호 영역에서만 실행

#### ② Active-X
- 마이크로소프트가 Windows 사용자를 위해 개발한 브라우저 확장 기술
- **대응책: 인증코드(Authenticode)**
  - 전자서명 + 신뢰된 인증기관(CA)으로 출처 확인
  - 단, 인증 자체가 **완벽한 보안을 보장하지는 않음**
  - 사용자에게 "실행하시겠습니까?" 대화상자를 띄우는 방식

```
Active-X 실행 흐름:
서버 코드 → [CA 서명 확인] → 사용자 수락(OK 클릭) → 실행
                 ↑
         서명만 확인할 뿐,
         코드 내용이 안전하다는 보장은 없음
```

### 악성 모바일 코드

| 유형 | 특징 |
|------|------|
| **바이러스** | 다른 파일에 붙어서 전파 |
| **웜** | 자가 복제로 네트워크를 통해 확산 |
| **매크로 바이러스** | 문서 매크로 기능을 이용해 전파 |

## 6. 익스플로잇 (Exploit) 공격

> 소프트웨어·하드웨어의 **보안 취약점을 실제로 이용**하는 공격 코드 또는 방법

### 주요 예시

- **버퍼 오버플로우 (BOF)**: 입력값이 버퍼 크기를 초과해 메모리를 덮어씀
- 입력값 검증 실패, 인증 우회, 권한 상승 등

### 핵심 한 줄

```
취약점이 존재한다는 것을 알고, 그것을 실제 공격에 이용하는 행위
```

## 7. 핵심 요약 (시험 대비)

| 공격/개념 | 핵심 키워드 |
|-----------|-------------|
| 살라미 공격 | 소액 반복 절취, 눈치채기 어려움 |
| Round Down | 반올림/절삭 오차 악용 |
| 데이터 디들링 | 원본 데이터 위조·변조, 내부자 |
| 이블 트윈 | 가짜 AP, Rogue AP, MITM |
| 클래스 로더 | 클래스를 JVM에 로드 |
| 바이트코드 검사기 | 코드 정상 여부 검증, 타입 안정성 |
| 보안 관리자 | 시스템 자원 접근 통제 |
| 샌드박스 | 제한된 환경에서 코드 실행 |
| Java Applet 대응 | 샌드박스 |
| Active-X 대응 | 인증코드(전자서명 + CA) |
| 악성 모바일 코드 | 바이러스, 웜, 매크로 바이러스 |
| 익스플로잇 | 취약점 직접 공격, BOF |
