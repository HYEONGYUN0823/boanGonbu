# 🏰 접근통제 모델 (MAC / DAC / RBAC & 보안 모델)

---

## 📌 먼저 읽기! 3가지 접근통제 한눈에 비교 ⭐⭐⭐⭐⭐

| 항목 | MAC | DAC | RBAC |
|---|---|---|---|
| 정의 | 보안 등급 비교로 접근 제어 | 주체의 신분(ID)으로 접근 제어 | 역할(Role)을 부여해서 접근 제어 |
| 권한 부여 주체 | **시스템** | **데이터 소유자** | **중앙 관리자** |
| 접근 결정 기준 | 보안 레이블(등급) | 신분(ID) | 역할(Role) |
| 정책 유연성 | **경직** | **유연** | **유연** |
| 보안 관리 방식 | **중앙집중형** | **분산형** | **중앙집중형** |
| 장점 | 안정적, 중앙집중 | 유연함, 구현 용이 | 관리 용이 |
| 단점 | 구현 어려움, 고비용 | 멀웨어 취약, ID 도용 취약 | - |
| 적용 사례 | 방화벽, **군 시스템** | **ACL** (리눅스/윈도우) | HIPAA |

## 1. 강제적 접근통제 (MAC) ⭐⭐⭐⭐⭐

> 💡 **비유 : 군대 보안등급**
> "1급 비밀" 문서는 1급 허가를 받은 사람만 볼 수 있어
> 내가 허가를 받았다고 해서 친구한테 "너도 봐도 돼"라고 마음대로 할 수 없음!
> 오직 보안 관리자(시스템)만이 허가를 정할 수 있음

```
핵심 : 보안 레이블 vs 보안 허가증을 비교해서 접근 결정
       객체 소유자도 접근 권한을 변경할 수 없음 (강제!)
```

**주요 개념**
```
보안 레이블 (Security Label/Class)
= levels × categories
= 자원이 얼마나 민감하고 중요한지를 나타냄

보안 허가증 (Security Clearance)
= 어떤 개체가 어떤 자원에 접근할 수 있는지를 나타냄
```

**특징**
```
✅ 중앙집중형 보안관리
✅ MAC 제약사항이 복사된 객체에도 전파됨 (DAC은 전파 안 됨!)
✅ 주로 군 시스템에서 사용
❌ 상업적 환경 적용 어려움
❌ 성능 저하 우려
❌ 비용이 많이 듦
```

**MAC 기반 운영체제 예시**
```
SELinux (NSA, Secure Computing 개발)
Trusted Solaris
```

## 2. 임의적 접근통제 (DAC) ⭐⭐⭐⭐⭐

> 💡 **비유 : 내 방 열쇠 복사**
> 내 방(객체) 열쇠는 내가(소유자/주체) 마음대로 복사해서
> 친구한테 줄 수도 있고 뺏을 수도 있음
> 하지만 친구가 내 방을 빌려서 내용을 다른 곳에 복사하면
> 그 복사된 내용에는 원래 접근 제한이 따라가지 않음!

```
핵심 : 객체 소유자(주체)가 다른 주체의 접근 권한을 자유롭게 설정
       신분(ID/그룹)에 근거해서 접근 제어
```

**특징**
```
✅ 분산형 보안관리
✅ 매우 유연한 접근제어
✅ 대부분의 OS에서 사용 (Windows, Linux, Unix, Mac)
❌ 멀웨어에 취약 (바이러스, 웜, 루트킷)
❌ ID 도용 시 통제 방법 없음
❌ 트로이목마 공격에 취약
❌ 복사 시 접근통제 정보가 전파되지 않음 (MAC과 다름!)
```

**DAC vs MAC 복사 시 차이**
```
DAC : A → 복사 → B  (B에는 원래 접근제한 없음, 정보 유출 가능)
MAC : A → 복사 → B  (B에도 원래 MAC 제약사항 전파됨, 안전)
```

### DAC 구현 기법

#### 1) 접근제어 행렬 (Access Control Matrix)

> 💡 **비유 : 엑셀 표**
> 행(Row) = 주체(사람), 열(Column) = 객체(파일)
> 칸에 권한(읽기/쓰기/실행) 표시

```
         파일A   파일B   파일C
앨리스    r,w     r       -
밥        r       r,w     r
찰리       -      r       r,w

단점 : 주체/객체 많아지면 행렬이 너무 커짐
       빈 칸(희소행렬)이 많아서 비효율적
→ 해결 : ACL 또는 CL(자격목록) 사용
```

#### 2) 자격 목록 (Capability List, CL)

```
= 접근제어 행렬을 행(주체) 단위로 관리
= 주체 관점에서 "내가 어떤 객체에 어떤 권한이 있나"

앨리스의 자격목록 : [파일A:r,w] [파일B:r]
밥의 자격목록     : [파일A:r]   [파일B:r,w] [파일C:r]

예시 : 커버로스의 티켓이 CL의 예!
```

#### 3) 접근제어 목록 (ACL, Access Control List)

```
= 접근제어 행렬을 열(객체) 단위로 관리
= 객체 관점에서 "이 파일에 누가 어떤 권한이 있나"

파일A의 ACL : [앨리스:r,w] [밥:r]
파일B의 ACL : [앨리스:r]   [밥:r,w] [찰리:r]

예시 : Linux/Unix 파일 시스템이 ACL의 예!
```

> 💡 **ACL vs CL 관계**
> ACL과 CL의 편리한 점과 불편한 점은 서로 반대!
> ACL = 객체 중심 (특정 파일에 누가 접근 가능한지 관리 용이)
> CL  = 주체 중심 (특정 사용자가 무엇에 접근 가능한지 관리 용이)

## 3. 역할기반 접근통제 (RBAC) ⭐⭐⭐⭐⭐

> 💡 **비유 : 햄버거 가게 직책**
> 알바생 = 조리 + 서빙 권한
> 매니저 = 조리 + 서빙 + 재고관리 + 매출확인 권한
> 사장   = 모든 권한
>
> 사람이 바뀌어도 직책(역할)에 따른 권한은 그대로!
> 알바생이 사라지고 새 알바생이 오면 그냥 "알바생 역할" 부여하면 끝!

```
핵심 : 사용자에게 직접 권한 부여 X
       역할(Role)에 권한 부여 → 사용자에게 역할 할당

역할 = 사용자 집합 + 권한 집합의 매개체
```

**RBAC의 특징**
```
✅ 중앙 관리자가 역할/권한 설정 (사용자가 임의로 변경 불가)
✅ 사용자:역할 = 다대다 관계 (1:1 필요 없음)
✅ 역할:자원 = 다대다 관계
✅ MAC + DAC의 단점을 보완
✅ Non-DAC이라고도 불림 (사용자에게 피할 수 없는 역할 할당)
```

**RBAC vs 그룹의 차이**
```
그룹 = 사용자들의 집합 (권한 집합은 아님)
역할 = 사용자들의 집합 + 권한 집합
```

**RBAC 3가지 기본 보안 정책**
```
1. 특권의 최소화 (Least Privilege)
2. 직무의 분리 (Separation of Duties)
3. 데이터 추상화 (Data Abstraction)
```

## 4. 보안 모델

### 벨라파듈라 모델 (BLP) ⭐⭐⭐⭐⭐

> 💡 **비유 : 기밀 문서 규칙**
> "위에 있는 기밀 문서를 아래로 흘리면 안 돼!"
> 높은 등급의 정보가 낮은 등급으로 내려가는 걸 막는 것이 핵심

```
목적  : 기밀성 (Confidentiality)
유형  : 상태머신 모델 + 정보흐름 모델
특징  : 다중등급 보안(MLS)의 최초 수학적 모델
적용  : 미국 군 시스템 (time sharing 메인 시스템)
```

**BLP 보안 규칙**

| 규칙 | 이름 | 내용 | 기억법 |
|---|---|---|---|
| **No Read Up (NRU)** | 단순 보안속성 (ss-property) | 주체는 같거나 **낮은** 등급 객체만 **읽기** 가능 | 위에 건 못 읽어! |
| **No Write Down (NWD)** | *(별) 속성 (*-property) | 주체는 같거나 **높은** 등급 객체에만 **쓰기** 가능 | 아래엔 못 써! |
| **Strong *-property** | 강한 별 속성 | 읽기+쓰기 모두 하려면 **등급이 동일**해야 함 | 읽쓰 동시는 같은 등급만! |

```
주체의 level = clearance  (허가증)
객체의 level = classification (분류)

핵심 요약 :
높은 등급 정보 → 낮은 등급으로 이동 불가 (기밀 유지!)
```

**BLP 장단점**
```
✅ 기밀성 유지에 매우 강함
✅ 트로이목마 공격 방어 가능

❌ 무결성 고려 안 함 (Blind Writing 가능)
❌ 은닉채널(Covert Channel)에 취약
❌ 가용성 고려 안 함
```

> 💡 **은닉채널 (Covert Channel)**
> 기본 채널에 기생하는 비인가 통신 채널
> BLP의 가장 큰 약점!
> (= 스테가노그래피와 유사한 개념)

### 비바 무결성 모델 (Biba) ⭐⭐⭐⭐⭐

> 💡 **비유 : 가짜 뉴스 차단**
> "신뢰도 낮은 정보가 신뢰도 높은 데이터를 오염시키면 안 돼!"
> 증권 투자 시 거짓 정보(낮은 무결성)를 듣고 신뢰할 수 있는 투자 계획(높은 무결성)을 망치는 것을 막는 것!

```
목적  : 무결성 (Integrity)
유형  : 상태머신 모델
특징  : 최초의 수학적 무결성 모델
       BLP와 유사한 구조 (Sensitivity level ↔ Integrity level)
적용  : 상업용 (회계법인 등)
```

**Biba 보안 규칙**

| 규칙 | 내용 | 기억법 |
|---|---|---|
| **단순 무결성** (No Read Down) | 주체는 **같거나 높은** 무결성 객체만 읽기 가능 | 낮은 거 못 읽어! |
| **무결성 제한** (No Write Up, *-integrity) | 주체는 **같거나 낮은** 무결성 객체에만 쓰기 가능 | 위에는 못 써! |
| **호출 속성** | 주체는 **낮은** 무결성 주체에게만 서비스 요청 가능 | 위에는 못 부탁해! |

> ⚠️ **BLP vs Biba 비교 (시험 단골!)**
>
> | | BLP (기밀성) | Biba (무결성) |
> |---|---|---|
> | 목적 | 기밀성 | 무결성 |
> | No Read | Up (위 못 읽음) | Down (아래 못 읽음) |
> | No Write | Down (아래 못 씀) | Up (위에 못 씀) |
> | 방향 | 정보 하향 흐름 차단 | 정보 상향 오염 차단 |

### 클락-윌슨 무결성 모델 (Clark-Wilson) ⭐⭐⭐

> 💡 **비유 : 은행 회계 처리**
> 잔액을 직접 수정할 수 없고 반드시 입금/출금/이체 프로세스(TP)를 통해서만 변경 가능!
> 이렇게 하면 실수나 부정 조작을 막을 수 있음

```
목적  : 무결성 (Integrity)
대상  : 상업용 (Biba보다 더 현실적)
특징  : Biba의 확장판, 무결성 3가지 목표 모두 다룸
       (Biba는 첫 번째 목표만 다룸)
```

**핵심 개념**
```
CDI (Constrained Data Item)   : TP를 통해서만 수정 가능한 데이터
UDI (Unconstrained Data Item) : 일반 사용자가 TP 없이 조작 가능한 데이터
TP  (Transformation Procedure): CDI를 수정하는 소프트웨어 프로시저

Well-formed Transaction = TP를 이용해서 CDI를 수정하는 것
```

**무결성 모델의 3가지 목적**
```
1. 허가되지 않은 사용자의 수정 예방
2. 허가된 사용자의 부적절한 수정 예방 (직무분리)
3. 내부 및 외부 일치성 유지 (정확한 트랜잭션)
```

### 만리장성 모델 (Chinese Wall = Brewer-Nash Model) ⭐⭐⭐

> 💡 **비유 : 컨설턴트의 이해충돌 방지**
> 삼성 컨설팅을 맡은 컨설턴트가 LG 컨설팅도 동시에 맡으면 안 됨!
> 한 쪽 정보가 다른 쪽에 흘러가면 이해충돌 발생
> → 만리장성처럼 둘 사이에 벽을 세움!

```
목적  : 기밀성 + 무결성 (이해충돌 방지)
특징  : MAC + DAC 개념을 모두 사용
적용  : 상업용 (이해충돌이 발생할 수 있는 환경)
       컨설팅 회사, 금융 기관 등
```

---

### 보안 모델 비교 ⭐⭐⭐⭐⭐

| 모델 | 목적 | 유형 | 특징 |
|---|---|---|---|
| **BLP** | 기밀성 | MAC | 최초의 수학적 모델, No Read Up / No Write Down |
| **Biba** | 무결성 | MAC | BLP 이후 개발, No Read Down / No Write Up |
| **Clark-Wilson** | 무결성 | MAC | Biba 확장판, 상업용, TP/CDI 개념 |
| **만리장성** | 기밀성+무결성 | MAC+DAC | 이해충돌 방지, 상업용 |

---

### 기타 보안 모델

#### 정보흐름 모델

```
BLP  : 높은 등급 → 낮은 등급으로 정보 흐름 차단
Biba : 낮은 무결성 → 높은 무결성으로 정보 흐름 차단

은닉채널(Covert Channel)이 존재하지 않음을 보장하는 규칙 제공
```

#### 상태기계 모델

```
시스템은 상태전이를 통해 항상 안전한 상태 유지
모든 보안 모델에 기본적으로 적용됨
```

## 📌 최종 핵심 암기 요약

```
✅ MAC : 시스템이 강제로 결정 (보안등급 비교)
         중앙집중형, 군 시스템, MAC 제약사항 복사 시 전파됨
         SELinux, Trusted Solaris

✅ DAC : 객체 소유자가 권한 결정 (신분/ID 기반)
         분산형, 대부분 OS (Windows/Linux/Unix)
         멀웨어/트로이목마 취약, 복사 시 전파 안 됨
         ACL이 대표적 구현

✅ RBAC : 역할 기반 접근 제어
          관리자가 중앙에서 역할 설정
          최소권한 + 직무분리 충족 용이
          사용자:역할, 역할:자원 = 다대다 관계

✅ BLP (기밀성)
   No Read Up   : 위에 건 못 읽어 (ss-property)
   No Write Down: 아래엔 못 써   (*-property)
   은닉채널 취약, 무결성/가용성 고려 안 함

✅ Biba (무결성)
   No Read Down  : 낮은 거 못 읽어
   No Write Up   : 위에는 못 써 (*-integrity)
   BLP와 방향 반대!

✅ Clark-Wilson (무결성, 상업용)
   Biba 확장판
   TP를 통해서만 CDI 수정 가능 (Well-formed Transaction)

✅ 만리장성 (기밀성+무결성)
   MAC + DAC 혼합
   이해충돌 방지 목적

✅ 보안 모델 최초 순서
   최초 기밀성 수학 모델 = BLP
   최초 무결성 수학 모델 = Biba
```
